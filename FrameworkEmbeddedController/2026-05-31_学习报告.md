# 每日嵌入式学习：Framework EmbeddedController

## GitHub 链接

https://github.com/FrameworkComputer/EmbeddedController

## 项目一句话介绍

FrameworkComputer/EmbeddedController 是 Framework Laptop 使用的嵌入式控制器固件仓库，主要用 C 语言实现笔记本底层电源时序、键盘、热管理、电池充电、调试接口以及与主机系统协作的 EC 功能。

## 为什么今天选它

今天避开了历史项目列表中的 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK、InfiniTime、RMK、TinyUSB、Ariel OS 和 OpenSK，选择 Framework EmbeddedController，是因为它很适合作为“真实产品级嵌入式固件”的学习样本。很多嵌入式项目要么偏演示，要么偏单一外设驱动，而这个仓库直接服务于一台量产笔记本电脑，既有裸机或轻量系统的底层代码，也有电源、键盘、风扇、充电、电池、安全启动、调试口等典型硬件协同问题。它不像完整 Linux 内核那样庞大到难以下手，又比普通 blink demo 更接近工程现场。

这个项目的另一个价值在于，它不是凭空开始的固件，而是基于 Google Chromium EC 体系发展而来。README 明确提到，Framework 会把适合 Chromium OS 设计理念的通用功能上游，但也保留一些不适合上游、却对 Framework Laptop 有用的改动，例如部分未被 Chrome OS 使用的内存映射区域和 UCSI 相关驱动。这种“上游基础加产品定制”的模式，在真实嵌入式开发中非常常见，值得学习。

## 项目解决的问题

EC，也就是 Embedded Controller，中文常译作嵌入式控制器。它在笔记本里通常不是主 CPU，而是一颗更小的微控制器，负责许多“电脑还没完全开机也必须能工作”的底层任务。Framework README 中明确提醒，EC 会处理低层功能，包括系统电源时序。修改 EC 代码可能导致设备无法上电、无法启动，甚至损坏主板、电池或外接设备。这段警告很重要，因为它说明 EC 固件和普通上层应用完全不同，它直接面对硬件状态机和电气约束。

在 Framework Laptop 场景里，EC 要解决的问题包括：按下电源键后如何按照正确顺序拉起各路电源；电池和充电器状态如何监控；键盘矩阵如何扫描；风扇和温度如何控制；主系统和 EC 之间如何通信；调试时如何通过 SWD 或 UART 观察内部状态；固件镜像如何通过校验和头部验证保证可以启动。这些问题共同构成了一台笔记本的“底层生命支持系统”。

## 技术架构分析

从 README 和目录结构看，这个项目属于典型的分层式 EC 固件。最上层是面向具体产品的 board 目录，Framework Laptop 的不同代际被抽象成不同 board，例如 Intel 11th Gen Core 处理器对应 hx20，Intel 12th 和 13th Gen Core 处理器对应 hx30。board 目录通常放置板级引脚、外设连接、电源时序、传感器组合、键盘布局和产品差异化配置。

下层是 chip 目录，README 指出 Framework Laptop EC 使用 MEC1521H-B0-I-SZ WFBGA144，并有 256KB RAM。chip/mchp 这类目录负责微控制器本身相关的寄存器、启动、时钟、中断、低功耗和片上外设适配。board 关心“这块板子怎么接线”，chip 关心“这颗芯片怎么工作”，两者分开是嵌入式工程保持可维护性的关键。

common、core、power、driver、include、util、test、zephyr 等目录共同组成了工程主体。common 通常沉淀通用服务和协议逻辑，core 处理运行时和体系结构相关部分，power 聚焦电源状态机，driver 放各类外设驱动，include 提供头文件接口，util 提供构建、烧录或调试工具，test 和 fuzz 则体现项目对测试的支持。仓库根目录的 Makefile、Makefile.rules、Makefile.toolchain 和 firmware_builder.py 表明它仍然保留了传统嵌入式 C 项目的构建风格，同时也接入了自动化构建。

构建方面，README 给出的 Ubuntu 环境依赖包括 gcc-arm-none-eabi、libftdi1-dev、build-essential、pkg-config 和 gawk。hx20 可用 BOARD=hx20 与 CROSS_COMPILE=arm-none-eabi- 构建，hx30 则使用 BOARD=hx30。产物示例为 build/hx20/ec.bin，并可刷写到 EC SPI flash ROM。README 还特别说明了 Intel 11th Gen 机型刷写时不要擦除或覆盖特定 flash 扇区，这正体现了固件分区和设备安全边界的重要性。

## 源码结构导读

阅读这个仓库可以先从 README.md 开始，理解 EC 的风险、构建方式、支持板型、flash 分区、调试接口和 Chromium EC 背景。随后进入 board/hx20 与 board/hx30，观察真实产品如何表达板级差异。再看 chip/mchp，理解 Framework 使用的 Microchip EC 芯片如何被抽象进通用框架。driver 目录适合按兴趣挑选，例如加速度计、电池、充电器、温度传感器、I2C、SPI、USB-C 或键盘相关驱动。power 目录值得重点阅读，因为笔记本电源时序是 EC 固件中最有工程含量的部分之一。

如果想理解项目如何从源码变成可刷写镜像，可以阅读 Makefile、Makefile.rules、Makefile.toolchain 和 README.fmap。README 里提到 EC 镜像有 checksum 和 header verification，随意用逆向工具改二进制会导致启动失败，而从源码编译会正确生成所需校验信息。这给学习者一个非常实际的提醒：固件构建并不只是把 C 文件编成 bin，还包含镜像布局、头部、校验、分区和启动约束。

## 学习价值

这个项目的学习价值不在于照着刷自己的笔记本，而在于理解真实硬件产品的嵌入式控制逻辑。第一，它能帮助学习者把 C 语言、Makefile、交叉编译、外设驱动和板级配置联系起来。第二，它展示了 EC 在现代笔记本中的职责边界：主 CPU 负责高性能计算，EC 负责低功耗、常开、实时性更强的控制任务。第三，它能训练工程风险意识。README 的警告不是形式主义，而是告诉你固件错误会直接影响硬件安全。第四，它体现了上游开源项目与厂商产品分支之间的关系，适合学习如何在大代码库中维护产品定制。

## 3～5 步动手路线

第一步，先只读不刷机。阅读 README.md，记录 hx20、hx30、MEC1521H、EC SPI flash、SWD、UART、RO/RW 分区这些关键词，建立项目地图。

第二步，搭建只编译环境。在 Ubuntu 或 WSL 中安装 README 提到的交叉编译工具和依赖，尝试选择一个 board 进行构建，目标是生成 ec.bin，而不是立刻写入硬件。

第三步，对照源码结构画一张分层图。把 board、chip、driver、power、common、core、include、util 的职责写清楚，找出一个具体功能从板级配置到驱动实现再到公共接口的路径。

第四步，挑一个低风险模块阅读。例如先看键盘扫描、风扇设置、温度传感器或电池状态读取相关代码，不急着碰启动、flash 分区和电源关键路径。

第五步，如果将来真的要在硬件上实验，必须先准备可恢复方案、调试接口、原厂镜像备份，并确认哪些 flash 扇区不能擦除。本报告不建议把学习阶段的第一目标设为刷写真机。

## 工程点评

Framework EmbeddedController 的工程特征很鲜明：它保留了 Chromium EC 的成熟基础，又面向 Framework Laptop 做了板级和产品级改造。优点是代码结构完整，目录层次能体现真实固件的复杂度，文档至少给出了构建、板型、flash 和调试入口，适合有一定基础的人深入。缺点是门槛并不低，学习者需要理解 C、交叉编译、微控制器外设、笔记本电源管理和固件镜像布局；同时它和具体硬件强绑定，不适合当作普通开发板教程来照抄。

从工程安全角度看，这个项目尤其适合用来培养“先模拟、先编译、先读图、再上板”的习惯。很多初学者把嵌入式开发理解为下载代码、编译、烧录、看效果，但 EC 固件要求你先知道自己在改什么、会影响哪条电源轨、会不会破坏启动校验、有没有恢复路径。这样的约束反而让它成为很好的进阶学习材料。

## 今日 10 个嵌入式概念/名词

### 1. EC，嵌入式控制器

通俗说，EC 是藏在笔记本里的小管家。主 CPU 还没启动时，它已经在处理电源键、电池、充电、键盘和风扇等任务。在本项目中，整个仓库就是 Framework Laptop 的 EC 固件。嵌入式工程师需要懂 EC，因为很多产品的稳定性不是由主系统决定，而是由这些常开控制器决定。

### 2. Power Sequencing，电源时序

电源时序是指不同电源轨按正确顺序打开和关闭。Framework README 明确提到 EC 会处理系统电源时序。它的作用类似“开机仪式”，顺序错了可能导致芯片不启动或硬件受损。嵌入式工程师必须理解它，因为真实硬件往往不是一上电就万事大吉。

### 3. Board Support，板级支持

板级支持就是把通用固件适配到某一块具体硬件板。项目中的 hx20 和 hx30 就是不同 Framework Laptop 代际的 board。工程师需要懂它，因为产品差异通常体现在引脚、电源、传感器和外设连接上，而不是每次重写整个系统。

### 4. Chip Port，芯片移植层

芯片移植层负责适配具体微控制器。README 提到的 MEC1521H 相关代码主要会落在 chip/mchp 等位置。它屏蔽寄存器、中断、时钟和片上外设差异，让上层模块可以复用。工程师需要懂它，因为换芯片时，最先要处理的就是这一层。

### 5. Cross Compile，交叉编译

交叉编译是在电脑上编译给另一种处理器运行的程序。README 使用 gcc-arm-none-eabi 和 CROSS_COMPILE=arm-none-eabi-，说明目标不是电脑本机，而是 EC 微控制器。嵌入式工程师离不开交叉编译，因为开发机和目标硬件几乎总不是同一种运行环境。

### 6. SPI Flash ROM，SPI 闪存

SPI Flash ROM 是保存固件镜像的非易失存储器。README 提到 ec.bin 可以刷写到 EC SPI flash ROM，并提醒某些扇区不能擦除。它在项目中承载 EC 启动代码和固件分区。工程师需要懂它，因为刷写、升级和恢复都离不开存储布局。

### 7. RO/RW 固件区域

RO 通常表示只读区域，RW 表示可写或可更新区域。README 中给出了 bootsector、lfwro 和 rw 的地址范围，并提到当前 EC 从 ro region 运行。这个概念关系到安全启动、升级回滚和设备恢复。工程师需要懂它，因为固件不是一个随便覆盖的单一文件。

### 8. SWD 调试

SWD 是 ARM 常见的两线调试接口。README 提到 EC debug connector 配置为 2 wire SWD，同时还有 UART_TX 和 UART_RX。它在项目中用于底层调试和恢复。工程师需要懂它，因为当固件无法正常启动时，串口日志和 SWD 往往是最后的排查入口。

### 9. Checksum 和 Header Verification

校验和与头部验证用于确认固件镜像格式和内容是否可信。README 说明随意修改 EC 二进制会导致 EC 启动失败，而从源码编译会生成正确校验信息。工程师需要懂它，因为固件安全不仅是代码逻辑，还包括镜像打包、签名或校验流程。

### 10. Upstream 与产品分支

Upstream 指上游开源项目，产品分支则是厂商为了具体硬件维护的改动。Framework README 提到项目基于 Google Chromium EC，并会把适合的通用功能上游，同时保留不适合 Chrome OS 的改动。工程师需要懂这个概念，因为长期维护产品固件时，如何跟随上游又不丢失产品特性，是非常现实的工程问题。
