# 每日嵌入式学习：TinyUSB

## GitHub 链接

https://github.com/hathach/tinyusb

## 项目一句话介绍

TinyUSB 是一个面向嵌入式系统的开源跨平台 USB Host 和 Device 协议栈，核心目标是在资源受限的微控制器上提供可移植、小体积、实时性较好、同时兼顾内存安全和线程安全的 USB 能力。

## 为什么今天选它

今天选择 TinyUSB，首先是因为它不在历史项目列表中，和之前讲过的 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK、InfiniTime、RMK 都不是同一项目，也不是这些项目的 fork。其次，USB 是嵌入式工程里非常常见、但又经常被低估的一块内容。很多开发者能很快点亮串口、屏幕、传感器，却在做 USB 虚拟串口、U 盘、键盘鼠标、音频、视频、固件升级、主机枚举时遇到大量细节问题。TinyUSB 正好把这些细节用一个相对完整的工程呈现出来，适合有 C 语言、单片机、RTOS 或裸机基础的人深入学习。

它的难度也比较合适。TinyUSB 不是一个只会闪灯的 demo，而是一个真正可复用的中间件项目；但它又不像完整 Linux USB 子系统那样庞大到难以下手。学习它，可以同时接触协议栈分层、端点传输、描述符、类驱动、板级支持包、RTOS 适配、静态内存设计、持续集成和硬件在环测试等工程主题。

## 项目解决的问题

TinyUSB 解决的是“如何在不同 MCU、不同板卡、不同操作系统模型上复用 USB 协议栈”的问题。USB 看起来只是插一根线，但真正写固件时会涉及设备枚举、配置描述符、接口描述符、端点描述符、控制传输、批量传输、中断传输、等时传输、类协议、主机和设备角色、低功耗恢复、远程唤醒、缓存一致性、DMA、端点零长度包、endpoint stall、热插拔、RTOS 调度以及中断上下文安全等许多细节。

如果每个芯片厂商、每块板子、每个应用都重新写一套 USB 代码，维护成本会非常高，行为也很难一致。TinyUSB 的做法是把上层 USB 类功能、协议栈核心、操作系统抽象层和底层控制器驱动分开，让应用开发者尽量通过统一 API 使用 CDC、HID、MSC、MIDI、Audio、MTP、WebUSB 等功能，同时让移植者可以针对具体 MCU 填写底层 DCD 或 HCD 驱动。

README 中明确强调它的设计重点包括：中断延后到非中断任务函数中处理，以获得线程安全；不使用动态内存分配，所有缓冲区静态化，以提高内存安全性和可预测性；支持 50 多个 MCU families；可在裸机、FreeRTOS、RT-Thread、Mynewt 等环境中工作。这些目标都非常符合嵌入式工程对可预测、可裁剪、可移植的要求。

## 技术架构分析

从架构上看，TinyUSB 可以理解为四层。第一层是应用和示例层，位于 `examples` 目录，里面有 device、host、dual、typec 等示例，帮助开发者理解如何在工程里打开 CDC、MSC、HID、MIDI、UVC 等功能。README 建议新手可以从 `cdc_msc` 示例开始，这个示例把虚拟串口和大容量存储结合起来，非常适合观察 USB 复合设备如何组织接口与端点。

第二层是协议栈核心，主要在 `src` 目录。这里有 `tusb.c`、`tusb.h`、`tusb_option.h`，以及 `device`、`host`、`class`、`common`、`osal`、`portable`、`typec` 等子目录。`device` 处理 USB Device 栈的通用逻辑，`host` 处理 USB Host 栈，`class` 存放 CDC、HID、MSC、Audio、MIDI 等类驱动，`common` 放公共工具和类型，`osal` 是操作系统抽象层，`portable` 则是和具体 USB 控制器、MCU family 相关的适配代码。

第三层是硬件支持层，主要在 `hw` 目录。`hw/bsp` 包含支持板卡的启动、时钟、引脚、链接脚本或板级初始化信息，`hw/mcu` 则放低层 MCU core 和外设驱动相关内容。对于学习者来说，这一层特别值得看，因为它回答了一个关键问题：同一个 USB 协议栈怎样落到 STM32、RP2040、nRF、ESP、CH32、LPC 等不同硬件上。

第四层是测试、文档和工具链。`docs` 存放 Sphinx 文档，包括 getting started、integration、porting、troubleshooting、reference 等内容；`test` 下有 unit-test、fuzz、hil，分别对应单元测试、模糊测试和硬件在环测试；`tools` 存放内部工具。仓库根目录还能看到 `.clang-format`、PVS-Studio、SonarQube 配置等工程质量相关文件。这说明 TinyUSB 不只是写协议代码，也在重视长期维护和自动化验证。

## 源码结构导读

如果第一次读 TinyUSB，不建议直接从最复杂的控制器驱动开始。更好的路线是先看根目录 README，理解项目的总体定位和支持能力；然后进入 `examples/device/cdc_msc` 或相近示例，观察应用层怎样初始化 TinyUSB、怎样定义描述符、怎样在主循环或 RTOS task 中调用协议栈任务函数。

接着可以查看 `src/tusb.h` 和 `src/tusb_option.h`。前者通常是应用接触协议栈的入口之一，后者与编译期配置关系密切。TinyUSB 大量能力通过配置宏打开或关闭，这种方式非常典型：在嵌入式项目里，不能像桌面软件那样把所有模块都编进去，必须根据目标设备的 flash、RAM、端点数量和应用需求进行裁剪。

再往下可以读 `src/class`。如果你关心虚拟串口，就看 CDC；关心键盘鼠标或自定义输入设备，就看 HID；关心 U 盘，就看 MSC；关心音频设备，就看 Audio。类驱动是学习 USB 的好入口，因为它既保留协议规范的影子，又比直接读控制器寄存器更接近应用。

最后再看 `src/portable` 和 `hw/bsp`。这里会看到 DCD 与 HCD 的实现思路。DCD 可以理解为 Device Controller Driver，负责把设备栈的请求转成具体 USB 外设控制器操作；HCD 是 Host Controller Driver，负责 USB 主机侧控制器操作。读这部分时最好手边有目标芯片参考手册，否则容易被寄存器和 FIFO 细节淹没。

## 学习价值

TinyUSB 的学习价值在于，它把“协议、驱动、系统、工程实践”放在同一个项目里。对嵌入式工程师而言，USB 往往不是单纯 API 调用问题，而是系统协作问题。比如一次 CDC 数据收发，表面上是串口读写，背后却包含端点缓冲、USB 帧调度、中断回调、任务上下文处理、环形缓冲、主机驱动行为和 PC 端应用读取速度。TinyUSB 强调中断延后处理和静态内存，就是在处理这些工程约束。

它也适合训练代码阅读能力。项目既有清晰目录，又有足够复杂度；既有应用示例，又有底层适配；既能从单板 demo 跑起来，也能深入到多 MCU 复用。学习者可以通过它理解一个成熟嵌入式中间件如何划分公共层和平台层，如何用配置宏管理功能裁剪，如何用示例证明功能，如何用测试防止回归。

## 3～5 步动手路线

1. 先阅读 README 和在线文档的 Getting Started，确认自己的开发板是否在 TinyUSB 支持列表中。如果手头没有特别合适的板子，可以优先选择 RP2040、STM32、nRF52840 等社区资料较多的板卡。

2. 从 `cdc_msc` 示例入手，先不改协议栈，只完成编译、烧录、枚举。观察电脑端设备管理器、串口工具和磁盘挂载行为，记录设备描述符、配置描述符、接口数量和端点数量。

3. 修改描述符和配置宏，例如调整产品字符串、打开或关闭某个 class、改变 CDC 缓冲区大小。每改一次都重新枚举，观察主机端行为变化。这个过程能帮助你理解 USB 描述符不是装饰信息，而是主机理解设备能力的依据。

4. 进入 `src/class/cdc` 或 `src/class/hid`，跟踪一次数据收发路径。重点看应用 API、类驱动缓冲、endpoint 传输提交、传输完成事件之间的关系。遇到中断回调时，特别注意哪些逻辑在 ISR 中，哪些逻辑被延后到 task 上下文。

5. 如果有精力，再选择一个 `src/portable` 下与你芯片相关的控制器驱动，结合芯片手册看 endpoint、FIFO、DMA、stall、reset、suspend、resume 等处理。不要一开始就试图读完整个 portable 目录，而是围绕一个具体板卡和一个具体示例纵向打通。

## 工程点评

TinyUSB 的优点是定位清晰、可移植性强、支持设备类型多、代码组织有层次，且非常重视嵌入式约束。它不使用动态分配，适合需要确定性内存行为的固件；它把中断工作延后到任务上下文，有助于减少 ISR 里的复杂逻辑；它支持裸机和多种 RTOS，说明 OSAL 抽象不是摆设；它有 examples、docs、test、CI 和静态分析配置，显示出长期维护的意识。

它的挑战也很真实。USB 本身规范复杂，TinyUSB 虽然帮开发者屏蔽了许多细节，但一旦遇到枚举失败、端点 stall、主机兼容性、吞吐量不足、缓存一致性、低功耗恢复等问题，仍然需要工程师理解协议和硬件。另一个挑战是支持平台多，平台适配代码不可避免地包含许多条件编译、寄存器差异和历史兼容。学习时要接受这种复杂性，不要期待它像教学 demo 一样线性简单。

总体看，TinyUSB 很适合作为“嵌入式 USB 协议栈与中间件工程”的学习项目。它能让学习者从一个可运行示例出发，逐步深入到类驱动、协议栈核心、OS 抽象和硬件控制器驱动，最终理解一个跨平台嵌入式库如何在现实约束中保持可维护。

## 今日 10 个嵌入式概念/名词

1. USB Device：通俗说就是插到电脑或主机上的外设角色，比如虚拟串口、键盘、U 盘。在 TinyUSB 中，device 栈负责设备枚举、描述符响应和端点传输。嵌入式工程师需要懂它，因为很多 MCU 产品都要通过 USB 和 PC 或上位机通信。

2. USB Host：通俗说就是主动管理 USB 总线的一方，比如电脑、开发板主机口或带 OTG 主机能力的设备。在 TinyUSB 中，host 栈负责发现设备、枚举设备并加载对应类驱动。工程师需要懂它，因为越来越多嵌入式产品要接键盘、U 盘、4G 模块或其他 USB 外设。

3. 描述符 Descriptor：它是 USB 设备向主机递交的“自我介绍”，告诉主机自己是什么设备、有几个接口、每个接口有哪些端点。在 TinyUSB 示例中，描述符通常是最先需要修改的部分。工程师需要懂它，因为很多枚举问题本质上是描述符不匹配或描述符表达不清。

4. 端点 Endpoint：端点可以理解为 USB 设备内部的数据通道，不同端点承载控制、批量、中断或等时数据。TinyUSB 的 CDC、MSC、HID 等类驱动都要使用端点。工程师需要懂它，因为端点数量、方向和包大小会直接影响功能实现和吞吐量。

5. CDC：Communication Device Class，常见用途是 USB 虚拟串口。TinyUSB 的 CDC 示例非常适合作为入门，因为它能让 MCU 像串口一样和电脑收发数据。工程师需要懂它，因为调试、日志、命令行和简单数据传输经常依赖虚拟串口。

6. MSC：Mass Storage Class，常见表现是 U 盘或可挂载磁盘。TinyUSB 支持 MSC，也有 CDC MSC 复合示例。工程师需要懂它，因为固件升级、数据导出、配置文件读写等场景常用 USB 存储接口。

7. HID：Human Interface Device，常见设备包括键盘、鼠标、游戏手柄，也可以做自定义输入输出。TinyUSB 的 HID 类驱动能帮助开发者实现标准输入设备。工程师需要懂它，因为 HID 通常免驱，适合做控制器、按键板、测试夹具和低速数据通道。

8. OSAL：Operating System Abstraction Layer，也就是操作系统抽象层。TinyUSB 通过 OSAL 适配裸机、FreeRTOS、RT-Thread、Mynewt 等环境。工程师需要懂它，因为可移植中间件不能把自己绑死在某一个 RTOS API 上。

9. DCD 与 HCD：DCD 是 Device Controller Driver，HCD 是 Host Controller Driver，它们分别连接设备栈或主机栈与具体 USB 控制器硬件。TinyUSB 的 `src/portable` 目录会涉及这些内容。工程师需要懂它，因为移植新芯片或排查底层问题时，最终会落到控制器驱动。

10. 静态内存分配：TinyUSB 强调不使用动态分配，缓冲区尽量静态化。通俗说，就是在编译期或初始化阶段把需要的内存安排好，运行时不临时 malloc。工程师需要懂它，因为嵌入式系统 RAM 小、实时性强，动态分配可能带来碎片、失败和不可预测延迟。
