今天的嵌入式学习项目，我们来看 TinyUSB。它是一个面向嵌入式系统的开源跨平台 USB 协议栈，既支持 USB 设备模式，也支持 USB 主机模式。简单说，如果你希望一块微控制器开发板在电脑上表现成虚拟串口、键盘鼠标、U 盘、音频设备、视频设备，或者让一块板子自己去管理外接的 USB 设备，TinyUSB 就是一个很值得研究的工程样本。

今天选它有两个原因。第一，它不是历史里已经讲过的那些项目，也不是那些项目的同名变体或 fork。第二，USB 在嵌入式开发中非常常见，却经常被低估。很多人点亮串口、屏幕、传感器都比较顺手，但一到 USB 虚拟串口、复合设备、主机枚举、端点传输、低功耗恢复、固件升级，就会发现事情远比想象复杂。TinyUSB 的好处在于，它既不是过于简单的演示程序，也不像桌面操作系统里的完整 USB 子系统那样庞大。它处在一个适合学习的位置，既有真实工程复杂度，又能从示例逐步读进去。

从当前公开信息看，TinyUSB 的仓库是 hathach 维护的 tinyusb，主要语言是 C，许可证是 MIT。通过 GitHub 接口查询时，它大约有六千八百多个 star、一千四百多个 fork、两百多个开放 issue，仓库在二零二六年五月底仍有更新和推送。它启用了 issue、project 和 discussion，也就是说，它不是一个长期无人维护的静态代码库，而是一个还在处理社区反馈、硬件适配和协议栈问题的项目。

它最近的活跃度也比较能说明问题。近期提交里可以看到合并请求，例如修复 CDC 和 MSC 的 FreeRTOS 示例里一个栈大小常量的拼写问题。开放的合并请求里，还有 STM32F4 板级支持开启 flash cache 的讨论，里面提到 CDC 和 MSC 吞吐示例从大约每秒五百千字节提升到九百千字节。还有一些近期已经合并的修复，比如 CH32 USBHS 控制器在 endpoint stall 处理上的问题。这些都不是表面上的文档更新，而是和真实硬件、真实端点、真实吞吐量、真实兼容性直接相关。

TinyUSB 解决的核心问题，是如何在不同微控制器、不同板卡、不同操作系统模型上复用一套 USB 协议栈。USB 看起来只是一根线，但固件里要处理的东西非常多。设备要向主机介绍自己，要提供设备描述符、配置描述符、接口描述符和端点描述符。数据要通过控制传输、批量传输、中断传输或者等时传输完成。设备类又分成虚拟串口、键盘鼠标、大容量存储、音频、视频、媒体传输、厂商自定义接口等等。再往下，还有中断、任务上下文、端点缓冲、DMA、缓存一致性、零长度包、stall、热插拔、挂起、恢复和远程唤醒。每一个细节都可能导致设备在电脑上枚举失败，或者看似能用但性能和稳定性很差。

如果每个芯片厂商、每块板子、每个应用都重新写一套 USB 代码，维护成本会非常高。TinyUSB 的思路是把应用层、USB 类驱动、协议栈核心、操作系统抽象层和底层控制器驱动拆开。应用开发者尽量通过统一接口使用 CDC、HID、MSC、MIDI、Audio、MTP、WebUSB 等功能。移植者则针对具体芯片实现底层控制器驱动，也就是设备控制器驱动和主机控制器驱动。

它的设计目标很嵌入式。项目说明里强调，USB 中断不会在中断服务函数里做太多复杂工作，而是延后到非中断任务上下文处理，这样有利于线程安全。它也强调不使用动态内存分配，所有缓冲区尽量静态安排，这样运行时行为更可预测，不容易因为内存碎片或临时分配失败导致问题。它支持五十多个 MCU families，也可以和裸机、FreeRTOS、RT-Thread、Mynewt 等环境配合。对嵌入式工程师来说，这些设计取舍比单纯会调用一个接口更有学习价值。

看源码结构时，可以把 TinyUSB 分成几层。最上面是示例层，主要在 examples 目录。这里有 device、host、dual、typec 等示例。新手可以优先看 CDC 加 MSC 的复合设备示例，因为虚拟串口和 U 盘都很直观，插到电脑上就能看到枚举结果、串口行为和磁盘挂载行为。通过这个示例，可以理解一个复合 USB 设备怎样组织接口和端点。

往下一层是协议栈核心，主要在 src 目录。这里有 tusb.c、tusb.h、tusb_option.h，也有 device、host、class、common、osal、portable、typec 等子目录。device 负责设备栈的通用逻辑，host 负责主机栈，class 放各种 USB 类驱动，common 放公共工具和类型，osal 是操作系统抽象层，portable 则和具体 USB 控制器、具体芯片家族有关。这个目录结构很适合学习一个可移植中间件如何分层。

再下一层是硬件支持，主要在 hw 目录。hw 下面的 bsp 包含板级支持，比如启动、时钟、引脚、链接脚本或板级初始化。mcu 目录则涉及底层微控制器核心和外设相关代码。这里回答了一个很实际的问题：同一套 USB 协议栈，怎样落到 STM32、RP2040、nRF、ESP、CH32、LPC 等不同硬件上。对于想从应用层走向驱动层的学习者，这一层非常关键。

还有一层是文档、测试和工具链。docs 目录包含 getting started、integration、porting、troubleshooting、reference 等文档。test 目录里能看到 unit test、fuzz 和硬件在环测试，也就是不仅在电脑上测试逻辑，还关注真实硬件上的行为。根目录还可以看到代码格式化、静态分析和质量检查相关配置。这个项目让人看到，一个成熟嵌入式库不仅要能跑，还要能长期维护。

如果你准备动手学习，不建议一开始就钻进最底层控制器寄存器。更合适的路线，是先阅读项目说明和入门文档，确认自己的开发板是否被支持。如果手头有 RP2040、STM32、nRF52840 这类资料多的板子，会更容易开始。接着选择 CDC MSC 示例，完成编译、烧录和枚举，观察电脑端出现的串口和磁盘。这个过程先不要急着改协议栈，目标是让一个已知示例完整跑起来。

示例跑通后，可以开始修改描述符和配置宏。比如改产品字符串，调整是否启用某个设备类，改变 CDC 缓冲区大小，或者观察接口数量和端点数量的变化。USB 描述符不是装饰信息，它就是主机理解设备能力的依据。很多枚举失败，最后都会回到描述符、端点和主机期望之间的关系。通过小步修改再重新枚举，你会很快建立直觉。

第三步可以进入 class 目录，挑一个自己关心的类驱动读。虚拟串口就看 CDC，键盘鼠标或自定义输入就看 HID，U 盘就看 MSC。跟踪一次数据从应用接口进入类驱动，再到端点传输提交，最后通过传输完成事件返回的过程。这个过程能帮助你理解，所谓 USB 读写并不是简单的函数调用，而是协议状态、缓冲区、端点和事件驱动共同配合。

第四步再去看 portable 目录和具体板卡支持。这里会出现 DCD 和 HCD。DCD 是 Device Controller Driver，也就是设备控制器驱动，负责把设备栈请求转成具体 USB 外设操作。HCD 是 Host Controller Driver，也就是主机控制器驱动，负责主机侧的控制器操作。读这部分最好结合芯片手册，围绕一个具体板卡、一个具体示例纵向打通，不要试图一口气读完所有平台。

TinyUSB 的工程优点很明显。它定位清晰，代码分层明确，支持设备类型多，也非常重视嵌入式约束。不使用动态内存分配，让运行时行为更稳定。把中断工作延后到任务上下文，减少中断服务函数里的复杂逻辑。通过操作系统抽象层支持裸机和多种实时操作系统，说明它不是只为某一个平台写死。再加上示例、文档、测试和静态分析配置，它体现了长期维护型开源嵌入式中间件的基本样子。

它的挑战也同样真实。USB 规范本身就复杂，TinyUSB 虽然屏蔽了许多细节，但一旦遇到枚举失败、endpoint stall、主机兼容性、吞吐不足、缓存一致性或者低功耗恢复问题，工程师仍然需要理解协议和硬件。支持平台多，也意味着代码里不可避免会有条件编译、寄存器差异和历史兼容。学习时不要期待它像教学 demo 一样线性简单，而要把它当作现实工程来读。

最后，用十个概念把今天的学习主线串起来。USB Device 是插到电脑或主机上的外设角色，比如虚拟串口、键盘和 U 盘。USB Host 是主动管理总线的一方，比如电脑或带主机口的开发板。描述符是设备交给主机的自我介绍，说明设备是什么、有几个接口、每个接口有哪些端点。端点是 USB 设备内部的数据通道，方向、类型和包大小都会影响功能和吞吐。CDC 是通信设备类，最常见用途是 USB 虚拟串口。MSC 是大容量存储类，常见表现就是 U 盘。HID 是人机接口设备类，适合键盘、鼠标、游戏手柄和一些免驱自定义设备。OSAL 是操作系统抽象层，让同一套协议栈能适配裸机和不同实时操作系统。DCD 和 HCD 分别连接设备栈、主机栈与具体 USB 控制器硬件。静态内存分配则体现了嵌入式系统对可预测性的追求，运行时尽量不临时申请内存，减少碎片、失败和不可控延迟。

总体来说，TinyUSB 很适合作为嵌入式 USB 协议栈和中间件工程的学习项目。它能让你从一个可以运行的示例开始，逐步深入到类驱动、协议栈核心、操作系统抽象和硬件控制器驱动。学完它，你得到的不只是“会用一个 USB 库”，更是对嵌入式跨平台中间件如何设计、移植、调试和维护的一套完整认识。
