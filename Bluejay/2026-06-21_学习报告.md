# 每日嵌入式学习：Bluejay

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**: https://github.com/mathiasvr/bluejay
- **一句话介绍**: Bluejay 是一套面向多旋翼无人机的数字电调（ESC）固件，运行在 SiLabs EFM8BB Busy Bee 微控制器上，用汇编语言实现无刷电机的高精度控制。

## 2. 为什么今天选它

- **活跃度**: 拥有 552+ star，2026 年仍在持续更新，社区活跃（Discord 社群 + ESC Configurator 等配套工具）。
- **工程完整度**: 从汇编源码、Makefile 构建系统（通过 Wine 调用 Keil C51 工具链）、批量编译脚本、到烧录工具（efm8load.py）和配套的图形化配置器（Bluejay Configurator），是一条完整的嵌入式产品线。
- **学习价值**: 全汇编实现，意味着每一行代码都直接对应硬件寄存器操作和时序控制。对"纯软件工程师转型嵌入式"而言，Bluejay 是难得的"把 RTOS/协议/外设驱动/实时控制四个维度压缩到一颗 8 位 MCU 上"的实战案例。理解它之后，再看 Zephyr/Embassy 等现代嵌入式框架会更有底层感知。

## 3. 项目解决的问题

Bluejay 解决的真实工程问题：**如何在资源极度受限的 8 位 MCU（EFM8BB1，256 字节 RAM、8KB Flash）上，以微秒级精度控制无刷电机换相、响应飞控的 DShot 数字协议、并实时回传 RPM 遥测。**

与普通软件项目的关键差异：

| 维度 | 后端/前端项目 | Bluejay (嵌入式) |
|------|-------------|-----------------|
| 存储 | GB 级磁盘、MB 级内存 | 256 字节 RAM、8KB Flash |
| 时序 | 毫秒级 API 响应可接受 | 微秒级 PWM 周期，错过节拍电机就失步 |
| 并行 | OS 线程调度 | 6 个中断优先级，手工管理上下文 |
| 调试 | GDB/Sentry/日志 | LED 闪烁、beep 旋律、USB 转串口 |
| 更新 | CI/CD 自动部署 | 专用烧录器 / 串口 bootloader |
| 安全 | Web 安全、数据加密 | 硬件看门狗、掉电保护、堵转检测 |

## 4. 技术架构分析

基于公开源码分析，Bluejay 的架构由以下几个核心层次构成：

### 4.1 硬件层
- **MCU**: SiLabs EFM8BB10 (24MHz) / EFM8BB21 (48MHz)，8051 架构
- **外设**: 4 个定时器（Timer0-3）、可编程计数器阵列 PCA（用于 PWM 生成）、ADC（温度采样）、比较器（BEMF 过零检测）、UART（烧录/调试）
- **FET 驱动**: 6 个 MOSFET 组成三相全桥，驱动无刷电机

### 4.2 固件层

```
┌──────────────────────────────────────────────┐
│                   Main Loop                   │
│  (wait for DShot frame → decode → update PWM) │
├──────────────────────────────────────────────┤
│           Interrupt Service Routines          │
│  Timer0: DShot 遥测信号生成（高优先级）        │
│  Timer1: DShot 帧解码 + 油门更新               │
│  Timer2: 换相计时 + RPM 计算                   │
│  Timer3: 过零检测超时 + 提前角计时               │
│  PCA:    硬件 PWM 输出                         │
│  Int0/1: DShot 信号边沿捕获                     │
├──────────────────────────────────────────────┤
│              BLDC 换相引擎                      │
│  启动阶段 → 初始运行 → BEMF 同步运行            │
│  6-step 换相 + 自适应提前角 + 退磁检测            │
├──────────────────────────────────────────────┤
│              配置系统                           │
│  Flash 模拟 EEPROM + 默认参数表                 │
│  DShot 命令协议（编程/读取/保存）                │
├──────────────────────────────────────────────┤
│              通信层                             │
│  DShot 150/300/600（数字协议解码）              │
│  DShot 遥测（GCR 编码回传 RPM/温度/电压）       │
└──────────────────────────────────────────────┘
```

### 4.3 关键设计决策

1. **全汇编实现**: EFM8BB 的 256 字节 RAM 跑不了 C 运行时（堆栈+全局变量+寄存器组就吃掉大半）。汇编允许精确控制每条指令的周期数和每个字节的内存布局。
2. **中断驱动的实时控制**: DShot 信号捕获依赖 Int0/Int1 边沿中断，DShot 解码在 Timer1 中断中完成，PWM 由 PCA 硬件自动输出——CPU 只在换相决策点做运算，其余时间等待。
3. **Flash 模拟 EEPROM**: EFM8 没有独立 EEPROM，Bluejay 使用 Flash 的一片区域（1A00h-1A7Fh）保存配置参数，通过改写 Flash 实现参数持久化。
4. **宏驱动的可配置架构**: 所有硬件差异（26 种 PCB 布局、2 种 MCU、12 种死区时间、3 种 PWM 频率）都在编译时通过宏定义选择，Makefile 会生成全部组合的 HEX 文件。

### 4.4 构建系统

Makefile 调用 Wine 运行 Keil C51 工具链（AX51 汇编器 + LX51 链接器 + Ohx51 HEX 生成器）。对于每种布局 × MCU × 死区时间 × PWM 频率组合，生成独立的 `.hex` 文件。批量编译会生成数百个目标文件，用户只需选择对应自己 ESC 硬件的那个。

## 6. 核心模块/函数导读

### 6.1 中断向量表与优先级（Common.inc: `Interrupt_Table_Definition`）

```
地址  向量   中断源             优先级
0000h reset    上电复位          最高
0003h Int0     DShot 信号输入    高
000Bh Timer0   遥测信号输出      高
0013h Int1     DShot 互补信号    高
001Bh Timer1   DShot 帧解码      中
002Bh Timer2   换相/RPM 计时     中
005Bh PCA      PWM 周期同步      低
0073h Timer3   过零检测超时      低
```

设计精妙之处：DShot 信号捕获（Int0/Int1）和遥测输出（Timer0）设为最高优先级，保证数字协议不错位；换相计时（Timer2/Timer3）设为中低优先级，即使偶尔延迟几个微秒，电机也不会立即失步。

### 6.2 DShot 解码引擎（Bluejay.asm ~686-800 行）

DShot 是一种数字串行协议，每帧 16 位：12 位油门值 + 4 位校验和，使用曼彻斯特风格编码。

关键函数 `t1_int`（Timer1 中断）：
1. 停止 Timer2，读取帧开始时间戳（`DShot_Frame_Start_L/H`）
2. 检查帧长度是否在有效窗口内（防止噪声误触发）
3. 用 `Decode_DShot_2Bit` 宏逐位解码：测量高电平宽度判断 0/1
4. XOR 校验和验证
5. 反码转换 + 偏移修正（减 96），得到 0-2047 的油门值
6. 检查是否为 DShot 命令（如编程/读取），更新 PWM 输出

### 6.3 BLDC 换相引擎（Bluejay.asm 各处分散）

无刷电机需要 6 步换相（6-step commutation）。Bluejay 采用**无传感器**方案，通过检测反电动势（BEMF）过零点判断换相时机。

核心流程：
```
油门输入 → 启动阶段（强制换相，逐步增加功率）
         → 初始运行（检测 BEMF 信号强度）
         → 同步运行（BEMF 过零检测驱动换相）
              ↑ 每电周期 6 步，循环
```

关键参数：
- `Comm_Period4x_L/H`: 最近 4 次换相的时间间隔，用于 RPM 估算
- `Wt_Advance`: 提前角（15° 标称），补偿 BEMF 检测延迟
- `Flag_Demag_Detected`: 退磁检测标志，退磁严重时自动降功率
- `Startup_Cnt`: 启动换相计数器，用于判断是否进入同步阶段

### 6.4 PWM 生成与 dithering（Common.inc: `Initialize_PCA` + 各布局文件）

PCA（可编程计数器阵列）被配置为 8-11 位 PWM 模式。`PWM_BITS_H` 决定周期长度和分辨率：

| PWM 频率 | 分辨率   | 时钟分频 |
|---------|----------|---------|
| 24kHz   | 11 bit   | 无      |
| 48kHz   | 10 bit   | 无      |
| 96kHz   | 9 bit    | 无      |

**PWM dithering**: Bluejay 实现了 11 位有效油门分辨率（即 2048 级），但在 96kHz 模式下硬件只能提供 9 位。dithering 通过逐帧交替使用两个相邻的 PWM 值，时间平均后达到 11 位精度——类似音频 DAC 的 sigma-delta 调制。

### 6.5 配置系统与 DShot 命令（Bluejay.asm ~2400-2600 行）

Bluejay 通过 DShot 命令实现"免编程器参数配置"：

| DShot 命令 | 功能 |
|-----------|------|
| 0         | 停转 |
| 1-47      | 保留 |
| 48-55     | 编程参数 |
| 56-63     | 参数保存/读取 |

飞控可以通过 DShot 协议直接修改 ESC 的 `Pgm_Comm_Timing`、`Pgm_Demag_Comp`、`Pgm_Startup_Power_Min` 等参数，持久化到 Flash。这是"软件定义硬件"在嵌入式领域的典型实践。

## 7. 软件工程师类比解释

### 7.1 DShot 协议 vs HTTP/2 帧
DShot 是 16 位数字帧（12 位数据 + 4 位 CRC），通过单线串行传输，没有 ACK/NACK。之所以能用单线完成双向通信（同时发送油门 + 接收遥测），是因为收发分时复用同一信号线。

**类比的局限**: HTTP/2 有完整的流量控制、头部压缩、多路复用；DShot 的"双向"只是利用信号间隙回传数据，不是真正的全双工。

### 7.2 中断优先级 vs 操作系统线程优先级
Bluejay 的 6 级中断优先级相当于一个硬实时 RTOS 的线程优先级——高优先级中断（Int0/DShot 捕获）可以抢占低优先级中断（Timer3/过零检测）。但区别在于中断是硬件电路直接触发的，延迟在纳秒级而非微秒级。

**类比的局限**: 操作系统线程可以 sleep、等待锁、被调度器换出；硬件中断处理器必须"快速进、快速出"，不能做阻塞操作。

### 7.3 BEMF 过零检测 vs 单元测试中的 mock
软件测试中 mock 用来模拟依赖。BEMF 检测是用比较器观察"悬空相"的电压，当它穿过电机中点电压时触发换相——相当于用硬件电路"mock"了霍尔传感器的信号。

**类比的局限**: Mock 是确定性行为——给定输入总是返回预期输出；BEMF 信号受转速、负载、温度影响巨大，工程师需要做大量的软件滤波和自适应算法才能可靠检测。

### 7.4 Flash 模拟 EEPROM vs 配置文件持久化
后端应用将配置写入 JSON/YAML 文件。Bluejay 因为没有独立 EEPROM，只能把 Flash 的一页擦除后写入配置数据。Flash 擦写寿命约 1 万次（vs EEPROM 的 10 万次），且擦除前必须整页（512 字节）操作。

**类比的局限**: 配置文件写入不关心次数或原子性；Flash 写入需要关心写入前擦除、写入中掉电导致的数据丢失、磨损均衡——这些都是嵌入式特有难题。

### 7.5 PWM dithering vs 软件限流算法
后端 API 限流用令牌桶算法平滑请求速率。Bluejay 的 PWM dithering 在时序上做类似工作——96kHz 模式下硬件只有 9 位分辨率（512 级），通过在相邻帧间交替使用 `N` 和 `N+1` 的 PWM 占空比，时间平均后等效 11 位（2048 级）。

**类比的局限**: 令牌桶使用内存计数器控制速率；PWM dithering 受硬件定时器精度和电机电感特性的物理约束，实现时还必须保证相邻帧的切换不产生可闻噪声。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1: BEMF (反电动势)
- **一句话**: 电机旋转时，未通电相线圈在磁场中切割磁感线产生的电压信号
- **类比**: 像发电机的输出电压——电机转动时，停在那里的绕组也在发电
- **场景**: 无传感器 BLDC 电机控制，通过检测 BEMF 过零点判断换相时机
- **坑**: BEMF 信号在低速时极弱（接近噪声），必须用启动时序先把电机转到一定速度才能可靠检测

### 卡片 2: 6-Step Commutation (六步换相)
- **一句话**: BLDC 电机每电周期需要 6 次通电状态切换
- **类比**: 像手动换挡变速箱——每到一个转速区间就需要切换齿轮组合
- **场景**: 所有无刷电机驱动，从风扇到无人机到电动车
- **坑**: 换相时机偏差超过几度电角度，电机会剧烈振动、发热甚至退磁

### 卡片 3: Dead Time (死区时间)
- **一句话**: 同一桥臂上下 MOSFET 切换时，强制让两者都关断的短暂时间，防止直通短路
- **类比**: 数据库迁移中的"排他锁过渡期"——旧写停止、新写还没开始，确保数据不冲突
- **场景**: 任何 H 桥/三相逆变器的 PWM 输出
- **坑**: 死区太长增加失真和发热，太短导致 MOSFET 直通烧毁

### 卡片 4: PCA (可编程计数器阵列)
- **一句话**: 独立于 CPU 运行的硬件定时器模块，可自动输出 PWM 信号
- **类比**: 像 GPU 卸载渲染——CPU 设置好参数后，PCA 自行产生波形
- **场景**: 电机 PWM、LED 调光、红外遥控发射
- **坑**: PCA 的 PWM 分辨率和频率互相制约（高频率→低分辨率），需要在产品设计时提前规划

### 卡片 5: DShot 协议
- **一句话**: 飞控和电调之间的数字通信协议，通过单线串行传输油门值
- **类比**: 用 curl 发 HTTP 请求代替模拟电压信号——数字通信比 PWM/PPM 更精确且不受噪声干扰
- **场景**: 现代 FPV 无人机飞控→ESC 通信
- **坑**: DShot 没有硬件级错误重传；信号线受到干扰时电调会直接收到错误油门

### 卡片 6: GCR 编码 (Group Code Recording)
- **一句话**: 将二进制数据编码为特定宽度的脉冲序列，用于 DShot 遥测回传
- **类比**: 类似 Base64 编码——把二进制数据转成适合在受限信道传输的字符集
- **场景**: DShot 遥测（RPM/温度/电压通过信号线回传到飞控）
- **坑**: GCR 解码对时钟精度敏感，EFM8 内部振荡器有 ±2% 偏差，长时运行后可能累积误差

### 卡片 7: Flash 模拟 EEPROM
- **一句话**: 在无硬件 EEPROM 的 MCU 上，用 Flash 的一页区域通过特殊读写流程来保存配置参数
- **类比**: 把 JSON 配置文件放在只读的容器镜像层里，每次 config 变更都要重建镜像
- **场景**: 需要保存用户配置但成本敏感的 MCU 方案
- **坑**: Flash 擦除寿命（10k 次）远低于 EEPROM（100k 次），频繁写入会快速耗尽

### 卡片 8: 看门狗定时器
- **一句话**: 硬件定时器，如果软件在规定时间内没有"喂狗"，系统自动复位
- **类比**: 健康检查探针——如果应用不响应 /health，K8s 自动重启 Pod
- **场景**: 任何需要 7×24 可靠运行的嵌入式设备
- **坑**: 看门狗复位后 GPIO 状态不可预测，外设可能处于半配置状态，需要精心设计复位处理

### 卡片 9: 比较器 (Analog Comparator)
- **一句话**: 硬件模块，比较两个模拟电压并输出数字高低电平
- **类比**: SQL 中的 WHERE 子句——A > B 输出 true/false
- **场景**: BEMF 过零检测、电池电量监测、过流保护
- **坑**: 比较器输出有传播延迟（几十 ns），在高速电机控制中必须校准补偿

### 卡片 10: 寄存器组切换 (Register Banking)
- **一句话**: 8051 架构通过 PSW 寄存器快速切换当前工作寄存器组（R0-R7）
- **类比**: 线程本地存储（TLS）——每个中断有自己的"局部变量"，不用压栈保存
- **场景**: 中断服务例程的快速上下文切换
- **坑**: 寄存器组只有 4 组（0-3），嵌套中断超过 4 层会冲突；调用子程序时仍需要手动栈操作

## 9. 30～60 分钟动手任务

### 目标
深入理解 DShot 协议的编解码流程。

### 不需要硬件的方案
用 Python 模拟 DShot 帧的编码和解码。

### 步骤
1. 阅读 `Bluejay.asm` 中 `Decode_DShot_2Bit` 宏（约 458-482 行），理解 2 位 GCR 解码逻辑
2. 阅读 Timer1 中断中 DShot 帧的 XOR 校验算法（约 766-779 行）
3. 打开终端，用 Python 实现一个 DShot 帧模拟器：

```python
THROTTLE = 1050  # 0-2047
CRC = THROTTLE ^ (THROTTLE >> 4) ^ (THROTTLE >> 8)  # 4-bit XOR
frame = (THROTTLE << 4) | (CRC & 0x0F)
frame_inv = (~frame) & 0xFFFF  # DShot uses inverted signaling
print(f"Raw throttle: {THROTTLE}, Frame: {frame:016b}, Inverted: {frame_inv:016b}")

# Decode
received = (~frame_inv) & 0xFFFF
data = received >> 4
checksum = received & 0xF
verify = data ^ (data >> 4) ^ (data >> 8)
print(f"Decoded data: {data}, Checksum valid: {(verify & 0xF) == checksum}")
```

## 12. 工程点评

### 可维护性
- **优点**: 宏定义清晰、注释详尽（每段代码都有英文注释说明时序意图），常量集中管理
- **不足**: 全汇编导致代码复用困难；Layou文件间的差异难以 diff

### 可移植性
- **差**: 深度绑定 EFM8BB1/2 的 8051 外设（SFR 地址），移植到 STM32/GD32 需要完全重写

### 可靠性
- **强项**: 多层故障检测（堵转、退磁、温度保护、看门狗），启动时序有完整的超时补偿机制
- **潜在风险**: 汇编缺乏类型检查，错误的内存寻址可能直到运行时才暴露

### 可测试性
- **弱**: 无单元测试（汇编+硬件依赖），依赖手动飞行测试和社区反馈
- **外部补偿**: 有 ESC Configurator 可模拟配置流程，但无法模拟电机负载

### 量产/运维
- **优秀**: 26 种布局 + 12 种死区时间 + 3 种 PWM 频率 = 上千种预编译 HEX，用户无需自己编译
- **固件更新**: 通过 dshot_bootloader 机制（串口烧录），无需拆卸 ESC
