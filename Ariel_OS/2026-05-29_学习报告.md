# 每日嵌入式学习：Ariel OS

## GitHub 链接

https://github.com/ariel-os/ariel-os

## 项目一句话介绍

Ariel OS 是一个用 Rust 从底层构建的嵌入式物联网操作系统，目标是在 Cortex-M、RISC-V、Xtensa 等 32 位微控制器上提供安全、低功耗、可移植，并且更接近“开箱即用”的开发体验。

## 为什么今天选它

今天选择 Ariel OS，主要是因为它正好站在嵌入式系统学习的几个交叉点上：嵌入式 Rust、低功耗物联网、异步运行时、抢占式调度、多架构板级支持、网络协议和安全通信。它不是一个只会点灯的入门 demo，也不是庞大到很难下手的全功能 Linux 发行版，而是一个工程结构清晰、模块拆分较细、文档和示例比较完整的微控制器操作系统。对已经有 C/C++、单片机、RTOS 或硬件基础的人来说，Ariel OS 很适合用来观察“现代嵌入式操作系统”如何把语言安全、任务模型、板级描述、构建系统和外设抽象组合在一起。

另一个重要原因是它没有出现在本任务的历史项目列表中。历史上下文中已经讲过 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK、InfiniTime、RMK 和 TinyUSB，Ariel OS 与这些项目不是同一个仓库，也不是这些项目的 fork，因此满足今日项目去重要求。

## 项目解决的问题

传统 MCU 开发常见的问题是碎片化。你可能在一个项目里使用某个 HAL，在另一个项目里换成完全不同的初始化方式；网络、随机数、存储、日志、传感器、定时器等功能往往都要从各类示例中复制拼接。C 语言生态里的 Zephyr、RIOT、FreeRTOS 已经积累了很多系统化经验，但在 Rust 嵌入式生态中，很多组件虽然质量很高，却更像“积木块”，需要开发者自己把 Embassy、esp-hal、defmt、probe-rs、sequential-storage、embedded-test 等项目组织起来。

Ariel OS 想解决的正是这个集成层问题。它不是要替代所有底层生态，而是把已有的嵌入式 Rust 组件整合成一个更完整的操作系统框架。README 中提到，它在现有项目之上补齐抢占式多核调度器、可移植外设 API、额外的网络安全能力，以及名为 laze 的元构建系统。换句话说，它希望让开发者用更少的样板代码写出可移植的嵌入式 Rust 应用，同时保留低功耗和实时性的基本诉求。

## 技术架构分析

Ariel OS 的核心思路可以理解为“在 Embassy 之上做系统级集成”。Embassy 提供异步执行器、硬件抽象、网络、定时器等重要能力，而 Ariel OS 在此基础上加入更强的系统初始化、线程与异步协作、统一外设接口和多板构建支持。FAQ 中明确说，Ariel OS 以 Embassy 为核心，但增加了多核抢占式调度器，使同步代码和异步代码可以在同一个应用中混合使用；它还抽象了外设 API，让不同 MCU 家族之间的应用迁移更容易。

从架构层次看，boards 目录保存结构化板级描述文件，用来生成或维护板级支持；src 目录下分成多个 crate，例如 ariel-os、ariel-os-threads、ariel-os-hal、ariel-os-embassy、ariel-os-rt、ariel-os-power、ariel-os-storage、ariel-os-sensors、ariel-os-random、ariel-os-coap、ariel-os-boards、ariel-os-stm32、ariel-os-nrf、ariel-os-esp、ariel-os-rp 等。这样的拆分方式很值得学习，因为它把“统一入口”“运行时”“调度器”“硬件适配”“协议或功能模块”分开，避免所有代码挤在一个大目录里。

构建系统也是 Ariel OS 的重点。README 提到它使用 laze 作为元构建系统，将 Cargo 难以直接表达的目标三元组、runner、probe-rs 配置、链接脚本和板卡差异绑定起来。对嵌入式工程师来说，这一点很现实：真正的跨板开发不只是代码能编译，还要处理烧录器、芯片型号、内存布局、调试工具、feature 组合和外设初始化。

## 源码结构导读

仓库根目录中的 README.md 是第一入口，适合快速了解项目目标、设计动机、文档位置和许可证。ROADMAP.md 展示项目为什么强调安全软件、开放标准、低功耗通信、异步 Rust、实时能力和未来的 OTA、网络链路层、设备入网、硬件在环测试等方向。FAQ.md 则适合横向比较 Ariel OS 与 Embassy、Tock、RIOT、Zephyr、FreeRTOS 的差异。

boards 目录保存不同开发板的结构化描述，例如 nRF52、nRF5340、Raspberry Pi Pico、ESP32-C3、ESP32-S2、STM32 Nucleo、Arduino Uno Q 等相关板卡。学习时可以把它当成“板级支持包”的入口，观察一个操作系统如何表达芯片、外设、runner 和构建参数。

examples 目录非常适合动手。里面有 hello-world、blinky、gpio、i2c-scanner、http-client、http-server、coap-client、coap-server、tcp-echo、udp-echo、ble-advertiser、ble-scanner、storage、power、random、threading、threading-channel、threading-event、threading-multicore、thread-async-interop、usb-keyboard、usb-serial、sensors-debug 等示例。它们覆盖了从基础 GPIO 到网络、蓝牙、存储、线程、异步协作和 USB 的多个主题。

src 目录是核心源码区。ariel-os crate 是用户侧主要入口，Cargo.toml 里可以看到 alloc、storage、threading、time、random、csprng、hwrng、sensors、ipv4、ipv6、dhcpv4 等 feature。ariel-os-threads 提供通用嵌入式调度器和 IPC，支持 single-core、multi-core、core-affinity、idle-threads 等能力。ariel-os-embassy 负责与 Embassy 生态衔接，ariel-os-hal 提供统一硬件抽象，ariel-os-rt 处理运行时相关内容，ariel-os-stm32、ariel-os-nrf、ariel-os-esp、ariel-os-rp 则体现不同芯片家族的适配层。

## 学习价值

学习 Ariel OS 的最大价值，是理解现代嵌入式项目如何从“能跑”走向“可维护、可移植、可测试”。它把语言安全、异步模型、抢占式线程、板级描述、feature 配置、网络协议和测试体系放在同一个工程里。对 C/C++ 背景的工程师来说，它可以帮助你换一个角度理解 RTOS：任务调度不只是创建线程，外设抽象不只是封装寄存器，构建系统也不是简单执行 make 或 cargo build。

它还适合用来学习嵌入式 Rust 的工程化落地。很多人学 Rust 嵌入式时会停留在 no_std、ownership、borrow checker、HAL 点灯这些概念上，而 Ariel OS 展示了一个更接近真实产品的组织方式：如何把硬件随机数接入系统随机源，如何把网络栈初始化封装起来，如何在异步任务和抢占式线程之间传递消息，如何给多种板卡维护统一示例，如何让测试和文档持续跟随代码演进。

## 3～5 步动手路线

第一步，先阅读 README、FAQ 和 ROADMAP，不急着改代码。重点搞清楚 Ariel OS 与 Embassy 的关系，以及它为什么强调低功耗、安全、异步和抢占式调度并存。

第二步，选择一块项目明确支持的开发板，优先从 Raspberry Pi Pico、nRF52/nRF52840、ESP32-C3 或 STM32 Nucleo 这类常见板卡入手。先跑 hello-world 或 blinky，确认工具链、probe-rs、runner、目标配置和烧录流程都能工作。

第三步，进入 examples 目录，按“GPIO、I2C、日志、随机数、线程、异步协作、网络”的顺序逐个阅读示例。每读一个示例，都回到 src 目录查找对应模块，理解应用 API 如何一路映射到底层 HAL 或 Embassy 组件。

第四步，重点研究 ariel-os-threads 与 thread-async-interop。尝试写一个小应用：一个线程周期性读取传感器或模拟数据，另一个异步任务负责通过网络或串口输出，二者通过 channel 或 event 通信。这个练习能帮助你真正理解混合同步和异步模型的意义。

第五步，如果时间充裕，可以研究 boards 目录和 laze-project.yml，尝试理解新增一块板卡大概需要描述哪些信息。即使不真的移植，也能提升你对嵌入式构建系统、内存布局和板级支持的敏感度。

## 工程点评

Ariel OS 的优点是方向清晰：它不满足于提供若干 Rust crate，而是试图把碎片化的嵌入式 Rust 生态整合成一个有操作系统体验的框架。它的 examples 和 tests 都比较丰富，目录结构也能看出模块化边界。对学习者来说，这比只看单一驱动库更有价值。

它的挑战也很明显。Rust 嵌入式生态仍在快速变化，Ariel OS 又叠加了 Embassy、laze、probe-rs、多芯片 HAL、多 feature 组合和操作系统抽象，初学者第一次搭环境可能会比传统 Arduino 或单一 MCU SDK 更费力。它适合有一定基础的人循序渐进学习，不适合作为完全零基础的第一块单片机教程。

从工程角度看，我最推荐关注三件事：第一，feature 如何控制系统能力，例如 threading、storage、random、network；第二，板级描述如何影响构建、烧录和外设初始化；第三，同步线程与 async Rust 如何共存。掌握这三点后，再去看网络、BLE、CoAP、HTTP、传感器和低功耗模块，会更容易建立整体图景。

## 今日 10 个嵌入式概念/名词

1. 嵌入式 Rust：通俗说，就是把 Rust 语言用于没有完整操作系统、资源有限的 MCU 或 SoC。Ariel OS 从底层使用 Rust 构建，强调内存安全和类型约束。在项目中，它贯穿 src 下各个 crate。嵌入式工程师需要懂它，因为越来越多安全敏感和联网设备开始尝试用 Rust 降低内存错误风险。

2. no_std：它表示程序不依赖完整标准库，适合没有普通操作系统的微控制器。Ariel OS 面向 MCU，自然需要考虑 no_std 约束。工程师需要懂 no_std，因为裸机和 RTOS 环境下没有文件系统、进程、动态内存等桌面假设。

3. Embassy：这是 Rust 嵌入式生态中重要的异步框架。Ariel OS 明确建立在 Embassy 之上，用它处理硬件抽象、异步执行、网络和定时器等能力。工程师需要懂 Embassy，因为它代表了一种用 async/await 管理嵌入式并发的现代方式。

4. 抢占式调度器：通俗说，就是系统可以在任务自己让出 CPU 之前打断它，把处理器切给更高优先级或其他任务。Ariel OS 在 Embassy 之上加入抢占式多核调度能力。工程师需要懂它，因为实时系统的响应延迟、优先级反转和线程安全都与调度方式有关。

5. 异步任务：异步任务不是传统线程，它在等待 I/O 或定时器时可以让出执行权。Ariel OS 支持 async Rust，并通过示例展示 hello-world、网络和线程异步互操作。工程师需要懂它，因为联网、低功耗和多外设场景里，异步模型能减少阻塞和资源浪费。

6. HAL 硬件抽象层：HAL 把具体寄存器和芯片差异封装成统一接口。Ariel OS 通过 ariel-os-hal 和不同芯片家族适配层提高可移植性。工程师需要懂 HAL，因为可维护的项目不能把业务逻辑到处写死在某个寄存器地址上。

7. 板级支持包：板级支持包描述某块开发板的芯片、外设、引脚、内存和调试方式。Ariel OS 的 boards 目录保存结构化板级描述，并用于生成 board support。工程师需要懂它，因为同一颗芯片换一块板，引脚和外设连接可能完全不同。

8. feature 配置：Rust Cargo 的 feature 可以按需开启线程、存储、随机数、网络、传感器等功能。Ariel OS 的 ariel-os crate 中有大量 feature。工程师需要懂它，因为嵌入式资源有限，不能把所有功能无脑编进固件。

9. CoAP：CoAP 是面向受限设备的应用层协议，常用于物联网场景。Ariel OS 提供 coap-client 和 coap-server 示例，并有 ariel-os-coap 模块。工程师需要懂它，因为低功耗、低带宽设备不一定适合直接使用重量级 HTTP。

10. 硬件在环测试：它指在真实硬件或硬件测试台上自动运行测试，而不仅是在 PC 上模拟。Ariel OS roadmap 提到扩展 CI 基础设施与 hardware-in-the-loop 自动化。工程师需要懂它，因为嵌入式 bug 往往只在真实外设、电源、时钟和中断环境中暴露。

## 今日结论

Ariel OS 值得作为今天的嵌入式学习项目，因为它同时具备近期活跃度、工程完整度和学习深度。它把 Rust 嵌入式生态中常见的优秀组件进一步组织成操作系统式体验，让我们能从一个真实仓库里学习低功耗物联网系统如何处理板级支持、异步运行、抢占式调度、网络协议、构建系统和测试。建议今天先把 README、FAQ、ROADMAP 和 examples 目录读通，再选择一个示例跑起来；真正的收获不只是“会用 Ariel OS”，而是理解一个现代 MCU 操作系统项目如何被设计和维护。
