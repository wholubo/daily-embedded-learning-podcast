# 每日嵌入式学习：ZMK

## GitHub 链接

https://github.com/zmkfirmware/zmk

## 项目一句话介绍

ZMK，全称 Zephyr Mechanical Keyboard Firmware，是一个基于 Zephyr 实时操作系统构建的开源机械键盘固件项目，目标是在无线、低功耗、可定制和许可证友好之间取得平衡，让自制键盘、分体键盘和蓝牙键盘拥有更现代的固件基础。

## 为什么今天选它

今天选择 ZMK，是因为它处在嵌入式学习中一个很有代表性的交叉点上：它不是简单的点灯演示，也不是庞大到难以下手的工业控制平台，而是一个真实用户很多、硬件形态清晰、软件结构完整、同时涉及 RTOS、蓝牙、USB HID、输入扫描、电源管理、设备树、配置系统和持续集成的中等难度项目。对于已经掌握 C 语言、单片机基础、GPIO、中断和基本构建流程的学习者来说，ZMK 很适合作为“从裸机思维走向工程化固件”的案例。

它的应用对象也足够具体：键盘。键盘看起来简单，只是按键输入，但真正做成可日常使用的固件时，需要处理矩阵扫描、消抖、组合键、层切换、宏、蓝牙配对、USB 输出、睡眠唤醒、电池续航、分体键盘左右通信，以及不同开发板和键盘布局的适配。ZMK 把这些问题放在 Zephyr 的工程体系里解决，因此学习它不只是学习键盘固件，更是在学习一个现代嵌入式项目如何组织驱动、配置、事件和应用逻辑。

## 项目解决的问题

ZMK 解决的核心问题，是让机械键盘固件在现代微控制器上更容易实现无线化、低功耗和复杂输入逻辑。传统键盘固件经常围绕 USB 有线输入展开，而 ZMK 从设计上就强调蓝牙和 Zephyr RTOS，尤其适合 nRF52 这类支持低功耗蓝牙的芯片。它把键盘输入抽象成按键位置、行为、事件和 HID 报告，再通过 USB 或蓝牙发送给主机。

另一个问题是可配置性。键盘爱好者往往会修改键位、层、组合键、长按短按行为、旋钮、传感器和显示内容。如果每一次修改都要深入改 C 源码，门槛会很高。ZMK 借助 Zephyr 的设备树、Kconfig 和 keymap 配置，把大量硬件差异和键位逻辑放到配置文件中表达，让同一套核心固件可以适配很多不同键盘。

第三个问题是工程可维护性。一个键盘固件如果只面向一块板子，代码可以写得很直接；但当它要支持大量开发板、分体键盘、不同矩阵接线、不同通信方式和不同输入设备时，就必须有清晰的模块边界。ZMK 的源码结构体现了这种边界：启动、电源、活动状态、矩阵变换、物理布局、传感器、事件管理、HID、行为系统、蓝牙、USB 和 split 逻辑分布在不同模块中。

## 技术架构分析

ZMK 的底层基础是 Zephyr RTOS。Zephyr 提供线程、设备模型、驱动框架、Kconfig 配置系统、设备树硬件描述、蓝牙协议栈、USB 栈和构建系统。ZMK 不是从零实现一个操作系统，而是把键盘固件作为 Zephyr 应用构建，这一点非常适合学习“站在 RTOS 之上写产品固件”的方式。

从构建入口看，`app/CMakeLists.txt` 首先通过 `find_package(Zephyr REQUIRED HINTS ../zephyr)` 接入 Zephyr 构建系统，然后设置 ZMK 相关模块，把各类源码加入 app 目标。这里可以看到 `src/activity.c`、`src/behavior.c`、`src/matrix_transform.c`、`src/physical_layouts.c`、`src/sensors.c`、`src/event_manager.c`、`src/hid.c` 等关键文件，也能看到大量 `target_sources_ifdef` 条件编译。这说明 ZMK 的功能不是一次性全部编进固件，而是通过 Kconfig 选项按需启用，例如电源管理、外部电源、WPM、指针设备、分体键盘、不同按键行为等。

输入路径大致可以理解为：硬件层负责扫描按键矩阵或读取输入设备，矩阵变换和物理布局把硬件位置映射成逻辑位置，keymap 和 behavior 系统决定这个位置在当前层应该触发什么行为，事件管理器把状态变化传播给其他模块，最后 HID 模块把按键、消费控制键、鼠标或其他输入转换成主机能理解的报告，并通过 USB 或 BLE 发出去。

ZMK 的事件系统值得重点看。键盘输入并不是简单的“按下就发一个字符”。比如长按短按、连击、粘滞键、Caps Word、层切换、宏和组合键，都需要根据时间、状态和上下文判断。ZMK 把很多变化表达成事件，例如 activity state changed、position state changed、sensor event、USB connection state changed 等，这种事件驱动风格让模块之间不必互相硬编码调用，比较适合复杂固件长期演进。

电源管理也是架构中的重要部分。无线键盘对续航很敏感，固件需要在没有输入时降低活动状态，必要时进入睡眠，并通过按键或 GPIO 唤醒。ZMK 中的 activity、pm、ext_power、gpio_key_wakeup_trigger 等模块都与这条主线有关。学习这些模块，可以把“低功耗”从抽象概念落到实际状态机和驱动调用上。

## 源码结构导读

ZMK 仓库根目录中，README 介绍项目目标、许可证、文档站点和社区入口。真正的固件应用主体在 `app` 目录。`app/CMakeLists.txt` 是理解项目构建的第一站，它展示了 ZMK 如何接入 Zephyr、如何添加源码、如何按配置选项启用功能模块。`app/Kconfig` 是理解功能开关的入口，它定义键盘名称、USB 设备信息、蓝牙设备名、HID 报告类型以及大量 ZMK 相关配置。

`app/src` 是主要业务逻辑所在。`activity.c` 处理活动状态，帮助判断设备处于活跃、空闲还是可睡眠状态。`behavior.c` 和 `src/behaviors` 目录是按键行为系统的核心，例如普通按键、重置、软关机、保持点击、粘滞键、切层等能力都可以在这里找到线索。`matrix_transform.c` 和 `physical_layouts.c` 负责把硬件矩阵位置转换成用户理解的键位布局。`sensors.c` 与旋钮、触摸板或其他传感输入相关。`event_manager.c` 连接各类状态变化，是理解模块协作的关键文件。

`hid.c` 负责把内部输入状态转换成 HID 报告。HID 是键盘和主机之间的标准语言，理解它可以帮助学习者明白为什么固件里要区分键盘按键、消费控制键、鼠标输入和报告大小。蓝牙、USB、split、display、pointing 等目录则对应不同外设能力和产品形态。对于初学阅读，不建议一开始就追所有路径，而是先选一条“按键按下到主机收到字符”的链路，再逐步扩展到蓝牙、电源和分体键盘。

## 学习价值

ZMK 的学习价值首先在于工程完整性。它不是孤立算法，而是一个长期维护的固件系统，包含构建、配置、驱动适配、事件分发、协议输出、低功耗和文档社区。学习它可以帮助嵌入式工程师理解，真正的产品固件往往不是把所有逻辑写在一个 main 循环里，而是依赖可配置的模块体系和清晰的数据流。

其次，它是学习 Zephyr 的好入口。直接阅读 Zephyr 主仓库可能会被大量内核和驱动代码淹没，而 ZMK 提供了一个明确应用场景：键盘。通过 ZMK，可以反向理解 Zephyr 的 Kconfig、设备树、CMake、驱动模型、蓝牙和 USB 栈在应用项目中如何被使用。

第三，它能训练输入设备和人机接口方面的思维。HID 报告、键盘矩阵、消抖、NKRO、BLE 配对、USB 描述符、分体通信和休眠唤醒，都是嵌入式工程中非常常见但又容易被教程简化的主题。ZMK 把它们组织到一个真实项目中，值得反复拆解。

## 3～5 步动手路线

1. 先阅读 README、ZMK 官方文档首页和 `app/CMakeLists.txt`，不要急着改代码。目标是弄清楚它为什么依赖 Zephyr、固件入口在哪里、哪些源码是无条件编译、哪些源码由 Kconfig 控制。

2. 选择一个官方支持或社区常见的 nRF52 开发板，按照文档完成一次最小键盘配置构建。如果暂时没有硬件，也可以先把构建跑通，把生成的配置文件和构建日志看明白。

3. 沿着按键输入链路读代码：从矩阵扫描或输入事件开始，经过矩阵变换、keymap、behavior、event manager，最后到 HID 报告。阅读时可以画一张数据流图，标出“硬件位置”“逻辑键位”“行为”“事件”“HID 报告”这几个概念。

4. 做一个小改动，例如修改键盘名称、切换 HID 报告类型、添加一个简单按键行为，或观察某个 Kconfig 选项打开前后的编译差异。这个阶段的重点不是功能多炫，而是理解配置如何影响编译和运行。

5. 进阶时再研究低功耗和蓝牙。可以重点看 activity、pm、BLE 连接状态和唤醒相关代码，理解无线键盘为什么必须把输入响应和省电策略同时考虑。

## 工程点评

ZMK 的优点是站在 Zephyr 之上，复用了成熟的 RTOS、蓝牙栈、USB 栈和构建体系，同时把键盘领域的行为系统做得比较细。它的模块划分清晰，配置化程度高，很适合支持多板卡、多键盘和多功能组合。对于学习者来说，它能展示一个现代固件项目如何在“灵活”和“复杂”之间取舍。

它的学习门槛也真实存在。第一次接触时，Zephyr 的 west、设备树、Kconfig 和 CMake 可能会同时出现，让人感觉入口很多。ZMK 自己的行为系统、事件系统和键位配置也需要时间消化。因此建议不要把目标设成“一天读完全部源码”，而是把它当成一个持续拆解的项目。先理解构建，再理解按键路径，然后看 HID，最后看蓝牙和低功耗，这样会更稳。

从嵌入式工程角度看，ZMK 最值得学习的不是某一个具体函数，而是它的组织方式：用 RTOS 和配置系统管理硬件差异，用事件系统管理复杂输入状态，用标准协议对接主机，用条件编译控制功能体积，并始终围绕实际产品体验处理功耗、连接和可维护性。

## 今日 10 个嵌入式概念/名词

### 1. Zephyr RTOS

通俗解释：Zephyr 是一个面向资源受限设备的实时操作系统，提供任务调度、驱动框架、网络和蓝牙等能力。在 ZMK 中，它是整个固件的地基，ZMK 作为 Zephyr 应用运行。嵌入式工程师需要懂它，是因为越来越多复杂设备不再适合纯裸机写法，RTOS 能帮助管理并发、驱动和可移植性。

### 2. Kconfig

通俗解释：Kconfig 是一套配置选项系统，用来决定哪些功能打开、哪些代码参与编译。在 ZMK 中，`app/Kconfig` 定义了键盘名称、HID 类型、蓝牙和电源相关选项。工程师需要懂它，因为现代固件常常用配置生成不同产品变体，而不是为每块板复制一份代码。

### 3. 设备树

通俗解释：设备树用结构化文本描述硬件资源，比如 GPIO、I2C、SPI、矩阵行列和外设连接。在 ZMK 中，不同键盘和开发板可以通过设备树表达硬件差异。工程师需要懂它，因为它把硬件信息从 C 代码中分离出来，是 Zephyr、Linux embedded 等体系的常见方法。

### 4. HID

通俗解释：HID 是 Human Interface Device，人机接口设备标准，键盘、鼠标和游戏手柄常用它和主机通信。在 ZMK 中，按键最终会被转换成 HID 报告，通过 USB 或蓝牙发给电脑、平板或手机。工程师需要懂它，因为很多输入设备都不是随便发字符，而是要遵守主机能识别的报告格式。

### 5. BLE

通俗解释：BLE 是低功耗蓝牙，适合电池供电设备做短距离连接。在 ZMK 中，无线键盘通常依赖 BLE 与主机配对和发送输入。工程师需要懂它，因为连接稳定性、功耗、配对体验和延迟都会直接影响产品质量。

### 6. 键盘矩阵扫描

通俗解释：键盘为了节省引脚，会把按键排成行列矩阵，通过逐行逐列扫描判断哪个键被按下。在 ZMK 中，矩阵扫描是输入链路的起点之一。工程师需要懂它，因为矩阵接线、鬼键、二极管方向和扫描频率都会影响键盘可靠性。

### 7. 消抖

通俗解释：机械按键按下或松开时，电信号会在很短时间内抖动，固件要过滤掉这些毛刺。在 ZMK 这类键盘固件中，消抖决定了按键是否稳定、是否误触发。工程师需要懂它，因为任何机械开关、按钮和继电器输入都可能遇到类似问题。

### 8. NKRO

通俗解释：NKRO 是 N-Key Roll Over，意思是可以同时识别多个按键。在 ZMK 的 Kconfig 中可以看到 HID 报告类型与 NKRO 相关的配置。工程师需要懂它，因为输入设备的协议能力、报告大小和兼容性常常需要权衡，不是越大越好。

### 9. 事件驱动

通俗解释：事件驱动不是让模块到处互相调用，而是把状态变化包装成事件，再由关心它的模块处理。在 ZMK 中，按键位置变化、传感器输入、USB 连接状态等都可以进入事件体系。工程师需要懂它，因为复杂固件靠事件解耦，后期增加功能会更安全。

### 10. 低功耗电源管理

通俗解释：低功耗电源管理就是在设备不用时尽量少耗电，需要时又能快速醒来。在 ZMK 中，activity、pm、外部电源和唤醒触发都围绕这个目标展开。工程师需要懂它，因为无线嵌入式产品的体验往往不只看功能，还看电池能用多久、唤醒是否及时、连接是否可靠。
