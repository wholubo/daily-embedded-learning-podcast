# 每日嵌入式学习：RIOT OS

## 1. GitHub 项目链接与一句话介绍

- GitHub 项目链接：https://github.com/RIOT-OS/RIOT
- 一句话介绍：RIOT OS 是一个面向物联网与微控制器设备的开源实时操作系统，目标是在 8 位、16 位、32 位 MCU 上提供统一、低功耗、模块化、带网络能力的嵌入式软件平台。

## 2. 为什么今天选它

- 活跃度：本次获取到的仓库 HEAD 为 `206a295c`，最近一次提交时间显示为 2026-06-10，说明主干仍在维护。由于 GitHub API 已触发限流，本报告对活跃度的判断主要基于 `git ls-remote`、浅克隆后的 `git log -1` 和公开 README。
- 工程完整度：RIOT OS 覆盖内核、线程调度、硬件抽象、板级支持、网络协议栈、驱动、构建系统、示例、测试与文档。它不是一个单点库，而是接近“嵌入式 Linux 发行版 + RTOS 内核 + IoT 协议栈”的组合。
- 学习价值：它把软件工程师熟悉的模块化、接口抽象、包管理、构建配置、网络协议，放进了 RAM、Flash、功耗、实时性和中断约束都很强的 MCU 场景。
- 适合软件工程师的原因：RIOT 提供 `native` 端口，可以在 Linux/BSD 上直接运行 RIOT 应用。今天实际编译并运行了 `examples/guides/threads` 的 native64 示例，因此没有硬件也可以先理解线程、调度、构建和日志输出。

## 3. 项目解决的问题

RIOT OS 解决的是“如何在小型、异构、低功耗的物联网设备上开发可维护系统软件”的问题。典型场景包括传感器节点、无线网络节点、工业或楼宇自动化设备、低功耗边缘设备，以及需要 IPv6、6LoWPAN、CoAP、MQTT-SN、LoRaWAN、BLE、CAN 等通信能力的 MCU 产品。

和普通软件项目相比，它多了几类硬约束。第一是硬件差异，同一套应用可能要跑在 AVR、MSP430、ESP32、RISC-V、ARM Cortex-M 等平台上。第二是资源限制，线程栈、驱动缓冲区、协议栈缓存都直接消耗 RAM。第三是时序限制，定时器、中断、调度延迟会影响协议重传、传感器采样和低功耗唤醒。第四是可靠性限制，设备可能无人值守运行数月，崩溃后未必有人能接键盘重启。第五是功耗限制，定时器精度、无线收发、外设初始化顺序都可能改变电池寿命。

## 4. 技术架构分析

基于公开 README 与仓库文件推断，RIOT OS 可以分成以下层次。

第一层是内核核心，主要位于 `core/`。这里包含线程、调度、IRQ 保护、panic、消息机制等基础能力。`core/sched.c` 中可以看到运行队列、优先级位图、当前线程和上下文切换请求等关键状态；`core/thread.c` 中可以看到线程状态切换、唤醒、睡眠、zombie 线程清理等逻辑。

第二层是 CPU 与板级支持，主要位于 `cpu/` 和 `boards/`。CPU 层处理体系结构相关的启动、上下文切换、中断、外设寄存器适配；board 层描述具体开发板的引脚、时钟、LED、按钮、外设实例。软件工程师可以把它理解成“运行时适配层”，但这里的适配不是操作系统 API 差异，而是寄存器、时钟树、引脚复用和启动文件差异。

第三层是驱动与外设抽象，主要位于 `drivers/`、`drivers/periph_common/` 和相关 include。README 明确提到 SPI、I2C、UART、USB、CAN、RTC、PWM、Timer、ADC、DAC、传感器、执行器、显示屏、GNSS 等能力。它的价值在于让应用尽量依赖统一接口，而不是直接写每个芯片的数据手册细节。

第四层是系统服务与协议栈，主要位于 `sys/` 和 `pkg/`。例如 `sys/include/ztimer.h` 描述了 ztimer 高层定时器抽象：应用可以使用微秒、毫秒、秒级时钟，也可以通过链式 clock、转换器和底层硬件 timer 构成不同精度的计时能力。README 还列出 IPv6、IPv4 via lwIP、6LoWPAN、UDP、TCP、RPL、CoAP、MQTT、LwM2M、NimBLE、LoRaWAN、UWB、CAN 等网络能力。

第五层是构建与配置系统。`Makefile.include` 是整个工程的构建入口之一，它设置 RIOTBASE、RIOTCPU、RIOTBOARD、RIOTMAKE、RIOTPKG、BUILD_DIR、APPDIR、BINDIR 等路径变量，并处理 board alias、外部 board/module/pkg 目录、Makefile.local、CI 信息等。对软件工程师来说，这类似大型后端 monorepo 的构建编排；但嵌入式构建还要把目标板、CPU、链接脚本、外设模块和二进制大小一起纳入约束。

## 5. 源码陪读：今天只读哪几个文件

1. `README.md`：先读项目定位、设计原则、支持平台和功能列表，建立全局地图。
2. `examples/guides/threads/Makefile`：再读一个最小示例如何声明应用名、选择 `BOARD ?= native`、设置 `RIOTBASE`、打开 `DEVELHELP`，最后 include 顶层构建系统。
3. `examples/guides/threads/main.c`：接着读线程示例，重点看线程栈如何显式分配，`thread_create` 如何设置优先级、入口函数和名称。
4. `core/sched.c`：然后读调度器，重点看 `sched_runqueues`、`runqueue_bitcache`、`sched_run()` 与优先级选择。
5. `sys/include/ztimer.h`：最后读定时器抽象，理解为什么不同时间精度会影响功耗、延迟和数据结构复杂度。

阅读顺序建议从 README 到示例，再从示例跳入线程和调度，最后补 ztimer。这样不会一开始陷入全部内核细节。

## 6. 核心模块/函数导读

- `thread_create`：应用创建线程的入口。在示例中，它接收显式分配的栈、栈大小、优先级、标志、入口函数、参数和线程名。嵌入式里线程栈不是无限增长的抽象资源，而是 RAM 里的固定数组。
- `sched_run()`：调度器核心路径之一。它通过运行队列位图找到下一个优先级队列，选择下一个线程，并维护当前线程状态。这里能看到 RIOT 为了减少查找成本使用 bitcache，这和高性能服务器里用位图加速资源选择很像。
- `thread_sleep()` 与 `thread_wakeup()`：线程状态迁移的基础函数。它们在修改状态时会关闭或恢复中断，说明并发不仅来自线程，也来自中断上下文。
- `ztimer_set()`、`ztimer_sleep()`、`ztimer_remove()`：ztimer 的基础 API。公开文档强调回调会在中断上下文中执行，因此不能把它当普通业务回调随意阻塞或做重活。
- `Makefile.include`：RIOT 应用构建的枢纽。它把应用目录、目标板、CPU、构建目录、包目录、工具目录和外部模块组织起来，最终决定哪些模块被编译、链接到目标固件。

## 7. 软件工程师类比解释

- RTOS 线程像后端服务里的轻量 worker。局限是：后端 worker 的栈和内存通常由 OS 宽松管理，RIOT 线程栈是你在源码里显式给出的固定数组，溢出就是严重故障。
- Board 支持包像云平台里的环境适配层。局限是：云环境差异常常是配置、权限和网络；开发板差异还包括时钟、引脚复用、电源域、中断号和外设实例。
- ztimer 像任务调度器或延迟队列。局限是：服务器延迟队列通常不关心微安级功耗，而 MCU 上使用高精度计时可能阻止低功耗睡眠。
- 驱动抽象像数据库驱动或 HTTP 客户端 SDK。局限是：外设驱动直接面对寄存器、DMA、中断、电气时序和总线仲裁，错误可能表现为硬件无响应而不是异常栈。
- `BOARD ?= native` 像本地 mock 环境。局限是：native 能验证逻辑和构建路径，但不能完全模拟真实 MCU 的时钟漂移、中断延迟、Flash 写入寿命和无线链路质量。

## 8. 今日 10 个嵌入式概念卡片

1. 概念名：RTOS。解释：面向实时任务的操作系统，强调确定性调度和资源可控。类比：后端任务运行时。场景：多传感器采样、通信协议、控制循环并行运行。坑点：不要以为有 OS 就等于资源无限。
2. 概念名：线程栈。解释：每个线程独立使用的一段固定 RAM。类比：每个 worker 的私有运行空间。场景：创建采样线程、通信线程、shell 线程。坑点：栈太小会溢出，太大会浪费 RAM。
3. 概念名：优先级调度。解释：调度器优先运行更高优先级的可运行线程。类比：消息队列里的优先级队列。场景：无线收包处理高于日志打印。坑点：低优先级任务可能饥饿。
4. 概念名：中断上下文。解释：硬件事件打断当前代码后执行的特殊上下文。类比：不可阻塞的高优先级回调。场景：定时器到期、串口收字节、GPIO 边沿。坑点：不能在中断里做耗时操作。
5. 概念名：硬件抽象层。解释：把芯片寄存器差异包装成统一接口。类比：数据库访问层。场景：同一应用切换 STM32、ESP32、nRF52。坑点：抽象不代表硬件能力完全一致。
6. 概念名：板级支持包。解释：描述某块板子的引脚、时钟和外设连接。类比：部署环境配置。场景：LED 引脚、UART 端口、传感器总线定义。坑点：同一 CPU 的不同板子也可能完全不同。
7. 概念名：tickless scheduler。解释：没有固定系统节拍时也能按事件或定时器唤醒。类比：事件驱动调度。场景：低功耗设备睡眠到下一次采样。坑点：调试时间问题更复杂。
8. 概念名：MTD。解释：Memory Technology Device，抽象 Flash 等存储设备。类比：块存储接口。场景：配置保存、日志、OTA 镜像。坑点：Flash 有擦写粒度和寿命限制。
9. 概念名：6LoWPAN。解释：让 IPv6 能在低功耗无线个域网上运行的一组适配机制。类比：为受限网络做的协议压缩和分片层。场景：传感器 mesh 网络。坑点：不能按以太网带宽和 MTU 来设计。
10. 概念名：native port。解释：把嵌入式系统作为普通进程跑在主机上。类比：本地仿真环境。场景：无硬件学习、CI、协议逻辑调试。坑点：native 通过不代表真板一定稳定。

## 9. 30～60 分钟动手任务

目标：在没有硬件的情况下，编译并运行 RIOT 的线程示例，观察线程创建、优先级和 native 端口。

步骤：
1. 克隆仓库：`git clone --depth 1 https://github.com/RIOT-OS/RIOT.git`。
2. 进入 `examples/guides/threads`。
3. 执行 `make -j2`，默认会使用 `BOARD ?= native`，在 64 位系统上可能解析为 native64。
4. 运行生成的 ELF，例如 `timeout 3s bin/native64/threads_example.elf`。
5. 修改 `main.c` 中 `THREAD_PRIORITY_MAIN - 1` 为不同优先级，思考输出顺序是否可能变化。

验收标准：能够看到类似 `main(): This is RIOT!`、`Hello, from the thread!`、`Hello, from the main thread!` 的输出；能够说清楚为什么线程栈要在源码里定义，以及为什么 native 运行不能替代真板验证。

本次实际验证结果：`make` 成功生成 `threads_example.elf`，大小统计为 text 28515、data 1080、bss 79640、总计 109235 字节；运行输出包含 RIOT native 初始化、版本号 `206a2`、线程和主线程的打印。

## 10. 3 道深度追问题

1. 如果一个传感器采样线程必须每 10 毫秒处理一次数据，而网络线程偶尔会执行较长的协议栈逻辑，你会如何分配优先级、栈大小和缓冲区？怎样证明不会错过采样窗口？
2. 软件工程里你可能习惯把延迟任务丢给定时器或队列；在 RIOT 的 ztimer 中，为什么文档提醒不要随便使用微秒级时钟？这背后涉及哪些功耗和硬件定时器约束？

## 12. 工程点评

RIOT OS 的可维护性体现在清晰的目录分层和大量示例，应用、内核、sys、drivers、boards、cpu、pkg 分工明显。可移植性体现在多 CPU、多 board 和 native 端口；但这种可移植性依赖严格的抽象边界和持续维护。可靠性方面，调度器、线程状态和 ztimer 文档都体现了对中断上下文、栈溢出和定时误差的关注。可测试性方面，native 端口非常有价值，可以把一部分嵌入式逻辑纳入普通 CI。量产和运维方面，README 提到 OTA via SUIT、PSA Crypto、文件系统和 MTD 抽象，说明项目关注设备生命周期；但实际产品仍需补齐安全启动、密钥管理、日志回收、电源异常恢复和板级验证。
