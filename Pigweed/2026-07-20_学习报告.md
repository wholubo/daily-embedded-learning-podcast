# 每日嵌入式学习：Pigweed

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**: https://github.com/pigweed-project/pigweed
- **一句话介绍**: Pigweed 是 Google 开源的一套面向 32 位微控制器的嵌入式开发模块集合——它解决的核心问题是"在资源极度受限的 MCU 上用现代软件工程方法开发可靠的嵌入式系统"。

## 2. 为什么今天选它

Pigweed 是嵌入式开源社区中一个非常独特的存在。它由 Google 维护，自 2020 年开源以来持续活跃（截至 2026 年 7 月仍在高频迭代），stars 523，拥有超过 120 个独立模块。对于有软件工程背景、正在转向嵌入式的学习者来说，Pigweed 几乎是为你们量身定做的：

- **它把现代工程实践带进了嵌入式世界**：在大多数嵌入式项目还在用 makefile + printf 调试时，Pigweed 带来了完整的单元测试框架（`pw_unit_test`）、代码审查流水线、Bluetooth 大小分析工具、自动化格式检查——这些是后端开发者的日常工具链，但在嵌入式领域极为少见。
- **纯软件类比的学习路径**：如果你熟悉 protobuf + gRPC，那么 `pw_rpc` 会立刻让你觉得亲切——它正是把 RPC 通信模式带到了串口/HDLC 等嵌入式传输层上。如果你了解 file system 但从未碰过 NOR flash，`pw_kvs` 会让你看到 key-value store 在 flash 上是什么样子。
- **强调"正确性"从测试开始**：Pigweed 的 `pw_unit_test` 可以在主机上编译运行，无需硬件——这意味着你可以在没有开发板的情况下运行和调试嵌入式 C++ 代码。

## 3. 项目解决的问题

Pigweed 回答一个现实问题：**当你的 MCU 只有几十 KB RAM、几百 KB Flash，却需要跑 RPC 服务、可靠日志、持久化存储、多线程同步、安全更新时，代码该怎么组织？**

和普通的纯软件项目相比，Pigweed 必须处理的约束包括：

- **MMU-less 环境**：没有虚拟内存、没有进程隔离。所有代码在单一地址空间运行，一个 null pointer 解引用就会搞垮整个系统。
- **Flash 磨损**：NOR flash 有擦写寿命限制（通常 1 万到 10 万次）。`pw_kvs` 必须做 wear leveling（磨损均衡）来延长存储器寿命。
- **中断上下文与普通线程的同步**：在 Cortex-M 上，ISR 中不能调用阻塞 mutex。Pigweed 的 `pw_sync` 提供了明确的 IRQ-safe 和 thread-safe 原语。
- **极端的二进制大小要求**：一个完整功能系统经常被限制在 128KB 以内。Pigweed 的 `pw_tokenizer` 能把日志字符串从占用 ROM 变成二进制 token，大幅缩小固件体积。
- **无标准库依赖**：许多 MCU 上没有 C++ exception、RTTI 或者完整的 STL。Pigweed 提供了 `pw_span`、`pw_function`、`pw_containers` 等自给的数据结构。

## 4. 技术架构分析

Pigweed 采用 **monorepo 结构 + GN/Ninja 构建系统**。整个项目不是单一库，而是约 120 多个独立 `pw_*` 模块的集合，每个模块有独立文档、测试和依赖。以下基于公开文件分析关键模块的结构：

### 4.1 构建系统层
- `pw_build`：定义了 GN 模板（如 `pw_source_set`、`pw_test_group`），统一管理编译选项。
- `pw_toolchain`：工具链定义，支持 GCC、Clang、arm-none-eabi-gcc。
- `pw_bloat`：二进制大小分析工具，可在 CI 中追踪固件体积变化。

### 4.2 硬件抽象层（HAL）
- `pw_digital_io`、`pw_i2c`、`pw_spi`、`pw_uart`：外设抽象接口，每个有多个 backend（Linux 模拟、MCUXpresso、RP2040、Zephyr 等）。
- `pw_clock_tree`、`pw_dma_mcuxpresso`：更高层次的硬件配置抽象。

### 4.3 系统启动与运行时
- `pw_boot` / `pw_boot_cortex_m`：Cortex-M 启动序列，包括向量表设置、静态内存初始化、C++ 构造函数调用，最终进入 `main()`。
- `pw_system`：整合启动、RPC、日志、OS 抽象的完整系统框架（WIP）。
- `pw_cpu_exception`：CPU 异常捕获统一接口，收集寄存器状态后交给应用层处理。

### 4.4 同步与线程
- `pw_sync`：提供 Mutex、TimedMutex、InterruptSpinLock、ThreadNotification 等同步原语。
- `pw_thread`：线程创建与管理抽象，后端支持 FreeRTOS、ThreadX、Zephyr、embOS 和 STL。
- `pw_work_queue`：工作队列，用于延迟执行或中断下半部处理。

### 4.5 存储与持久化
- `pw_kvs`：Flash-backed 键值存储，带磨损均衡和冗余拷贝。日志结构（log-structured）设计，写后追加、压缩回收空间。
- `pw_blob_store`：单 blob 持久化容器，比文件系统更轻量，支持数据完整性校验。
- `pw_persistent_ram`：掉电保留的 RAM 区域数据管理。

### 4.6 通信与 RPC
- `pw_rpc`：嵌入式 RPC 框架，基于 protobuf 定义服务和消息，支持串口、HDLC、Zephyr 等传输层。
- `pw_hdlc`：HDLC 帧封装/解封模块，常用于 UART RPMSG 通道。
- `pw_transfer`：基于 `pw_rpc` 的文件传输协议。

### 4.7 调试与诊断
- `pw_console`：Python REPL 终端 + 日志查看器，可直接通过串口与设备交互。
- `pw_log`：日志框架，支持 tokenized、RPC、basic 等多种后端。
- `pw_tokenizer`：编译时把字符串替换为 token，减少固件体积和 I/O 流量。
- `pw_snapshot`：系统快照/崩溃报告数据结构。
- `pw_trace`：轻量级跟踪框架。

### 4.8 工具与基础设施
- `pw_unit_test`：在主机或设备上运行单元测试。
- `pw_presubmit`：CI presubmit 检查框架。
- `pw_ide`：VS Code / IDE 配置生成。
- `pw_env_setup`：开发环境引导脚本。
- `pw_watch`：文件变更监控，自动重编译。

## 6. 核心模块/函数导读

### 6.1 `pw_boot_Entry()` — 启动入口

- 位置：`pw_boot_cortex_m/boot.cc`
- 工程角色：这是 CPU 复位后执行的第一段 C 代码。它设置中断向量表偏移（VTOR），调用静态内存初始化（BSS 清零、data 段复制），然后调用 C++ 构造函数，最后进入用户 `main()`。
- 与启动流程的关系：所有嵌入式 C/C++ 项目都依赖这样一个启动序列。Pigweed 把它做成可复用模块，而不是让每个项目手写 startup.s。

### 6.2 `KeyValueStore::Put()` / `KeyValueStore::Get()` — Flash KV 存储

- 位置：`pw_kvs/public/pw_kvs/key_value_store.h`
- 工程角色：向 NOR flash 写入或读取持久化配置。Put 操作为每个 key-value 条目创建一个新的 log entry（追加写入），Get 操作从有效日志中查找最新版本。
- 与持久化的关系：flash 擦除粒度是 sector，写入是 page。KVS 内部管理 sector 回收和磨损均衡。如果你的项目需要存 Wi-Fi 配置、校准参数，但又不想引入一个完整的文件系统，这就是标准方案。

### 6.3 `pw_rpc::RawServer::ProcessPacket()` — RPC 服务器处理

- 位置：`pw_rpc/server.cc`
- 工程角色：从传输层接收数据包，解析 protobuf 编码的 RPC 请求，调用注册的服务处理器，并发回响应。
- 与通信协议的关系：这是嵌入式设备"接收命令 - 执行 - 返回结果"的最现代方式。相比传统的 AT 指令或者自定义二进制协议，protobuf + RPC 提供了类型安全、版本兼容、自动序列化/反序列化的优势。

### 6.4 `pw_tokenizer::Encode()` — 编译时日志 Tokenizer

- 位置：`pw_tokenizer/tokenizer.h`
- 工程角色：在编译时把 printf 格式字符串替换为一个 4 字节哈希 token，实际字符串参数单独编码。运行时只发送 token + 参数，解码端用 token 数据库还原原始日志。
- 与构建系统的关系：tokenizer 的实现依赖编译时哈希计算，所以在 GN 构建过程中有额外的 codegen 步骤。这是嵌入式特有的"用编译时间换运行时体积"的典型模式。

### 6.5 `InterruptSpinLock` - 中断安全的自旋锁

- 位置：`pw_sync/interrupt_spin_lock.h`
- 工程角色：一种在 ISR 和任务之间共享数据时使用的锁。它的特点是：禁止中断 + 自旋等待，因为 ISR 里不能调用 blocking mutex。
- 与 RTOS 的关系：在 FreeRTOS 上，`InterruptSpinLock` 的 backend 会调用 `taskENTER_CRITICAL()`/`taskEXIT_CRITICAL()`。这是一个典型的"嵌入式特有同步原语"，纯软件后端开发者不会遇到它。

## 7. 软件工程师类比解释

### 7.1 pw_kvs（Flash KV Store）↔ Redis / LevelDB

Pigweed 的 KeyValueStore 在概念上很像你熟悉的嵌入式中 Redis 或 LevelDB：存 key-value 数据，支持 put/get/delete，还有基本的空间回收。但关键区别是：Redis 在 DRAM 里运作，擦写不限次；pw_kvs 的后端是 NOR flash，**每次写入前必须先擦除整个 sector**，而每个 sector 只能擦除约 1 万次。所以 pw_kvs 必须做磨损均衡——这是后端工程师要理解的新维度：资源寿命管理。

**这个类比的局限是什么？** 像 LevelDB 这样的 LSM-Tree 引擎为读写吞吐做了大量优化，而 pw_kvs 的主要优化目标是"减少 flash 擦写次数"和"在掉电时保证数据一致性"，性能排第二位。

### 7.2 pw_rpc ↔ gRPC / tRPC

你已经熟悉用 protobuf 定义服务，gRPC 生成客户端和服务端代码。pw_rpc 做的是同一件事：protobuf 定义 → 代码生成 → 客户端调用远程方法。区别在于传输层：gRPC 走 HTTP/2，pw_rpc 走 HDLC 帧或自定义串口协议。

**这个类比的局限是什么？** pw_rpc 不支持 streaming RPC 的大部分模式，也没有负载均衡、健康检查、服务发现这些微服务层的功能。它只做最基本的"请求-响应"和"单向流"。

### 7.3 pw_tokenizer ↔ 编译时 i18n / ICU 消息 ID

如果你做过国际化和本地化，你应该见过这种模式：在源码中用消息 ID 代替字符串，运行时根据 ID 查找翻译。pw_tokenizer 做的事情几乎完全一样，只不过它的目标是缩小固件体积而不是翻译。

**这个类比的局限是什么？** i18n 系统的消息 ID 是有结构的（通常带 namespace），目的是维护多语言翻译文件；pw_tokenizer 的 token 是 4 字节哈希，目的是最小化运行时开销，且在 token 碰撞时需要人工干预。

### 7.4 pw_boot_cortex_m 启动序列 ↔ 容器初始化 / main() 前置钩子

你可以把 Cortex-M 启动序列想象成一个容器的 `ENTRYPOINT` + `CMD`：CPU 复位（docker run）→ 设置向量表（配置镜像运行模式）→ BSS/Data 初始化（加载环境变量）→ C++ 构造函数（初始化依赖服务）→ main()（应用入口）。但是嵌入式的启动是**在裸金属上**的，没有操作系统帮你做这些事情。

**这个类比的局限是什么？** 容器启动有操作系统内核提供进程隔离和资源管理，Cortex-M 启动时唯一运行的代码就是自己写的，一个错误的向量表配置就会导致 CPU 在复位后直接跑飞。

### 7.5 pw_sync::InterruptSpinLock ↔ 数据库行锁 + 中断禁用

InterruptSpinLock 的行为是用"关中断 + 自旋"来保护临界区，类似于你在数据库事务中先执行 `SELECT ... FOR UPDATE` 加行锁再操作数据。区别很微妙但很关键：数据库行锁如果冲突，事务会等到锁释放——持锁的任务可能被调度器换出。而 InterruptSpinLock 持锁期间**禁用中断**，这意味着必须在极短时间内完成操作，否则会错过外部事件。

**这个类比的局限是什么？** 数据库行锁是基于操作系统的调度器来实现等待的，持锁线程可以被挂起。InterruptSpinLock 持锁时必须保持 CPU 运行（自旋），不能睡，也不能被抢占。所以它的临界区通常只有几条指令，绝不能做 I/O 或 malloc。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：Wear Leveling（磨损均衡）
- **一句话解释**：把 flash 写入分散到不同 sector，避免同一个 sector 先于其他 sector 达到擦写寿命上限。
- **软件工程类比**：相当于日志轮转（log rotation）——如果总写同一个日志文件，它很快就满了；滚动到不同文件就能延长系统寿命。
- **常见工程场景**：pw_kvs 的 KeyValueStore、LittleFS、MCUboot 的 scratch partition。
- **容易误解/坑点**：磨损均衡不等于"每次都换一个物理地址"——它需要跟踪每个 sector 的擦写次数。设计不当会在高写入频率下迅速耗尽 flash 寿命。

### 卡片 2：Log-Structured Storage（日志结构存储）
- **一句话解释**：所有写入操作都追加到存储介质末尾，不原地修改旧数据。旧数据在新数据覆盖后变成 garbage，等待回收。
- **软件工程类比**：Kafka 的 append-only log 或者 PostgreSQL 的 WAL 日志。
- **常见工程场景**：pw_kvs、LevelDB 的 SSTable、LittleFS。
- **容易误解/坑点**：日志结构意味着读取时需要扫描最新版本——pw_kvs 通过地址映射和 checksum 来加速读取，但不像关系数据库有 B-Tree 索引。

### 卡片 3：Interrupt Spin Lock（中断自旋锁）
- **一句话解释**：一种用于中断（ISR）和普通线程之间同步数据的锁。它在持锁期间禁用本地中断并自旋等待。
- **软件工程类比**：类似于把 `local_irq_disable()` 和 `spinlock` 组合起来用，但后端开发者一般用 mutex 或信号量就够了。
- **常见工程场景**：ISR 向任务队列投递事件、ISR 中修改共享计数器。
- **容易误解/坑点**：自旋锁持锁时间必须极短（< 1us），否则会错过其他中断。也不可递归使用（单核上自旋 = 死锁）。

### 卡片 4：Tokenized Logging（令牌化日志）
- **一句话解释**：编译时把字符串替换为哈希 token，运行时只发送 token 加参数，解码端根据 token 还原日志。
- **软件工程类比**：类似前端工程的代码分割——你的调试字符串"代价"不是在开发机上算的，而是在受限的 MCU 上算的。
- **常见工程场景**：需要大量日志但 flash 空间有限的量产固件。
- **容易误解/坑点**：token 碰撞（两个字符串映射到同一个哈希）会导致日志显示错误。Pigweed 在编译时会检查 token 唯一性。

### 卡片 5：VTOR（Vector Table Offset Register）
- **一句话解释**：Cortex-M 上的一个寄存器，告诉 CPU 中断向量表在 flash/RAM 的哪个地址。
- **软件工程类比**：类似于 x86 的 IDT（中断描述符表）寄存器，但比它简单很多——VTOR 只是一个 32 位地址。
- **常见工程场景**：引导加载程序跳转到应用固件前，需要设置 VTOR 指向应用的向量表。
- **容易误解/坑点**：中断向量表的对齐要求（通常 256 字节或 512 字节对齐）。对齐错误会导致 CPU 在中断发生时执行意外代码。

### 卡片 6：BSS 段（Block Started by Symbol）
- **一句话解释**：可执行文件中存放未初始化全局变量和静态变量的区域，在启动时由启动代码清零。
- **软件工程类比**：相当于 Go 中声明的 `var x int`——编译器保证它在使用前是零值。在嵌入式上这事不能依赖 OS，必须自己写清零循环。
- **常见工程场景**：所有嵌入式 C/C++ 启动代码都要在 main 之前完成 BSS 清零和 Data 段复制。
- **容易误解/坑点**：BSS 在 flash 中不占空间（只记录大小），但 RAM 中占空间。如果链接脚本没正确配置 BSS 段，全局变量不会自动初始化为零。

### 卡片 7：RPMSG（Remote Processor Messaging）
- **一句话解释**：一种在多核异构系统中（如 Cortex-M + Cortex-A）进行核间通信的协议，基于共享内存和 vring 实现。
- **软件工程类比**：类似于 IPC（进程间通信）的共享内存机制，但发生在不同的 CPU 核心之间。
- **常见工程场景**：Zephyr 的 RPMsg 子系统、OpenAMP、Pigweed 的 pw_hdlc 作为传输层。
- **容易误解/坑点**：核间通信需要处理 cache coherence——M 核通常没有数据 cache，但 A 核有，所以需要通过 dmb/sev 指令维护一致性。

### 卡片 8：Protobuf 在嵌入式中的代价
- **一句话解释**：protobuf 编码后的消息体积小、跨语言兼容，但在 MCU 上解码需要额外的 RAM 和 CPU 周期。
- **软件工程类比**：要像给 microservice 做资源预算一样——`protobuf-lite` 可比完整 `protobuf` 小 10 倍，但功能也少。
- **常见工程场景**：Pigweed 的 pw_rpc、nanopb 用于资源更受限的环境。
- **容易误解/坑点**：MCU 上 protobuf 的 reflection（反射）功能通常禁用，因为实现代码太大。这也意味着不能动态生成 message，所有 message 类型必须在编译时注册。

### 卡片 9：GN / Ninja
- **一句话解释**：Google 开发的构建系统（meta-build system），GN 产生 Ninja 构建文件。Pigweed 用它而不是 CMake。
- **软件工程类比**：类似于 Bazel（也是 Google 出的），但更轻量。如果用过 Bazel/Pants，GN 的模式很相似：BUILD.gn 文件声明目标和依赖。
- **常见工程场景**：Chromium（最早的 GN 用户）、Fuchsia、Pigweed。
- **容易误解/坑点**：GN 的 scope 和变量作用域规则比 CMake 严格——在 if/else 里定义的变量不会自动泄漏到外部 scope，需要 `set()` + 参数传递。

### 卡片 10：Code Size Bloat Tracking（固件大小膨胀追踪）
- **一句话解释**：在 CI 中跟踪每次提交对固件二进制体积的影响，发现意外膨胀。
- **软件工程类比**：相当于 npm 包的 bundlephobia——但是针对二进制而非 JS。
- **常见工程场景**：Pigweed 的 pw_bloat、ESP-IDF 的 `idf_size.py`。
- **容易误解/坑点**：一个看似无害的 `#include <iostream>` 可以把固件增大 200KB（因为 iostream 静态构造和异常支持代码）。嵌入式 C++ 开发必须不断检查 include 链。

## 9. 30～60 分钟动手任务

### 目标
在本地克隆 Pigweed 仓库，配置构建环境，编译并运行一个模拟目标上的单元测试——无需任何硬件开发板。

### 步骤

1. **克隆仓库**
   ```bash
   git clone https://github.com/pigweed-project/pigweed.git
   cd pigweed
   ```

2. **引导开发环境**（Pigweed 自带 Python 环境配置）
   ```bash
   source bootstrap.sh
   ```
   这一步会下载 GN、Ninja、Python 依赖等工具，需要几分钟。

3. **激活环境**
   ```bash
   source activate.sh
   ```

4. **编译主机模拟目标**
   ```bash
   gn gen out/desktop --args='dir_pw_third_party_nanopb="//third_party/nanopb/repo"'
   ninja -C out/desktop pw_unit_test:run_tests
   ```
   **注意**：如果需要更快的入门，也可以跳过完整引导，直接编译一个 `pw_kvs` 或 `pw_tokenizer` 测试子集：
   ```bash
   gn gen out/host --export-compile-commands
   ninja -C out/host pw_sync/test/*
   ```

5. **阅读测试输出**
   - 观察 PASS/FAIL 结果。
   - 注意每个测试的大小：观察 `pw_bloat` 生成的 .bloat 文件。

- [ ] `bootstrap.sh` 成功完成，不报错。
- [ ] `gn gen` 成功生成构建配置。

- Pigweed 的 `targets/host_device_simulator` 可以在本地运行完整的系统仿真（包括 pw_rpc 通信）。

## 10. 3 道深度追问题

### 问题 1（时序/并发/资源限制）
Pigweed 的 `pw_kvs` 采用日志结构存储，写入时总是追加到当前 sector。考虑这样一个场景：你的系统每分钟写一次传感器数据到 KVS（每次约 32 字节），flash sector 大小为 4096 字节，每个 sector 可擦写 10000 次。假设你只有 4 个 sector，**在磨损均衡算法正常工作的情况下，这套设备的预期寿命是多少？如果磨损均衡算法失效，始终写同一个 sector，寿命又是多少？**

### 问题 2（软件→嵌入式视角转换）
在 REST API 后端开发中，你通常会这样设计一个"更新配置"的接口：客户端 PUT /config，服务端校验参数、写入数据库、返回 200 OK。在 Pigweed 上用 `pw_rpc` 实现相同的功能，需要考虑哪些 REST API 开发中不存在的约束？请从**原子性、掉电安全、传输可靠性、payload 大小限制**四个方面分析。

### 问题 3（工程判断）
Pigweed 在架构上选择了 GN/Ninja 而不是 CMake。CMake 更普及、有更丰富的生态和 IDE 支持（CLion、VS Code CMake Tools 等），而 GN 在嵌入式领域非常小众。你认为 Google 做这个选择的**真实原因可能是什么**？如果让你在一个 10 人团队中为新产品选择构建系统，你会选 CMake 还是 GN？你的理由是什么？

## 11. 今日复盘问题

**"在今天之前，你可能觉得嵌入式开发就是'写 C、烧录、看串口输出'。Pigweed 展示的是一种完全不同的模式——先写测试、再写实现、用 CI 追踪代码体积、用 RPC 做交互式调试。这种'把软件工程正规军流程搬进嵌入式'的方式，你觉得在实时性要求严格（例如电机控制 1kHz 控制环）的场景下，哪些能用的上，哪些用不上？为什么？"**

## 12. 工程点评

### 可维护性
Pigweed 在可维护性上是满分级别的。每个模块有独立文档、设计决策记录（`.rst`）、代码走查入口（每个 BUILD.gn 有 OWNERS 文件）。`pw_presubmit` 强制执行代码格式（clang-format）、类型检查（clang-tidy）和测试覆盖率。

### 可移植性
采用 facade（外观）模式设计：模块定义统一接口（如 `pw_digital_io`），不同硬件平台提供对应 backend。目前支持的平台包括 STM32、NXP i.MX RT、RP2040、Linux 模拟、Zephyr、Arduino 等。跨平台是 Pigweed 的第一性设计原则。

### 可靠性
- 异常捕获：`pw_cpu_exception` 确保 CPU 异常时能记录完整状态而非静默死机。
- 掉电安全：`pw_kvs` 的日志结构保证电源丢失时最多丢失最后一次写入。
- 冗余拷贝：KVS 支持多副本存储，单 sector 损坏可恢复。

### 可测试性
这是 Pigweed 最突出的优势。`pw_unit_test` 支持在 x86 主机上运行嵌入式逻辑代码，无需硬件。`pw_bloat` 可以在 CI 中间件跟踪固件大小。Host simulator 甚至可以在无硬件环境下测试 RPC 通信。

### 量产/运维
Pigweed 的 `pw_software_update` 提供了安全固件更新框架。`pw_snapshot` 可以在设备崩溃后生成结构化的 dump，便于离线分析。`pw_tokenizer` 减少了日志 I/O，这对高通信道（如 BLE）非常关键。

## 13. 明天可以延展的方向

1. **深入 pw_tokenizer 的实现**：了解编译时哈希（compile-time hash）的计算方式、token 碰撞处理机制、以及如何在自定义 bootloader 中集成 tokenized log。这是"用编译时计算换运行时性能"的经典案例。

2. **pw_rpc 协议深度 + 与 gRPC 的真正差异**：比较 pw_rpc 的包协议设计与 gRPC 的 HTTP/2 帧结构的异同。学习如何在一个没有 TCP/IP 的串口链路上实现可靠的 RPC 通信（超时/重传/保序）。

3. **Pigweed 的 facade 模式与 Zephyr 的 device tree 驱动的对比**：这两种是嵌入式跨平台抽象的主流方案——一个是纯 C++ 接口 + 编译时绑定 backend，一个是编译时从 DTS 生成代码。理解它们的优缺点有助于你在实际项目中选择适合的抽象层。
