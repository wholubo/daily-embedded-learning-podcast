# 每日嵌入式学习：Klipper

## 1. GitHub 项目链接与一句话介绍
- 仓库链接：https://github.com/Klipper3d/klipper
- **一句话介绍**：Klipper 是一个把 3D 打印机运动规划放到通用计算机上、把精确定时执行留给一个或多个微控制器的 3D 打印固件项目。

## 2. 为什么今天选它
- 活跃度：本次公开仓库检查到 HEAD 为 `c707dd1`，提交日期为 2026-06-14，说明项目近期仍在维护。
- 工程完整度：项目同时包含主机端 Python 程序、微控制器端 C 固件、多架构移植层、配置样例、协议文档、运动学文档、测试目录和大量真实打印机配置。
- 学习价值：它不是单片机上“一坨固件”式设计，而是把复杂算法、配置解析、G-code 处理和 API 交给 Linux 主机，把硬实时脉冲输出交给 MCU，天然适合软件工程师理解“软硬协同”的边界。
- 避重说明：历史列表中已有 Marlin，但 Klipper 不是 Marlin 的同仓库、fork 或同源变体；它采用主机加 MCU 的分布式架构，学习重点也不同。

## 3. 项目解决的问题
3D 打印机的核心问题不是简单地“让电机转”，而是要把切片软件生成的 G-code 转成多个步进电机、加热器、风扇、传感器在精确时间上的协同行为。喷头移动速度、加速度、拐角速度、挤出压力、床面高度补偿、振动抑制都会影响成品质量。

和普通软件项目相比，Klipper 多了几个嵌入式特有约束。第一是时序约束，步进脉冲需要在几十微秒级精度内输出，文档中提到步进事件可达到 25 微秒或更好的调度精度。第二是资源约束，很多控制板仍然是 8 位或低端 ARM MCU，不能把复杂运动学全部塞进去。第三是可靠性约束，加热器失控、通信延迟、任务阻塞都可能导致打印失败甚至硬件风险。第四是硬件差异，真实用户的主板、驱动芯片、传感器和机械结构差异巨大，配置系统必须比普通服务端配置更贴近物理世界。

## 4. 技术架构分析
Klipper 的架构可以概括为“主机负责计算，MCU 负责准时执行”。`klippy/` 是主机端，主要用 Python 编写，运行在树莓派或其他 Linux 设备上。它负责读取配置、解析 G-code、维护打印机对象、做运动规划、处理宏、管理多个 MCU 连接，并通过协议把压缩后的低层命令发给控制板。

`src/` 是微控制器端 C 代码。公开文档说明该目录下有 `avr`、`stm32`、`atsam`、`atsamd`、`lpc176x`、`linux`、`pru` 等架构目录，`generic` 提供可复用辅助代码。微控制器端的执行从架构相关入口进入，最终调用 `src/sched.c` 中的 `sched_main()`。初始化函数通过 `DECL_INIT()` 注册，周期任务通过 `DECL_TASK()` 注册，命令函数通过 `DECL_COMMAND()` 暴露给主机调用。

通信层是 Klipper 的关键设计。协议文档把它描述成主机和 MCU 之间的低层通信协议，目标是低延迟、低带宽、低复杂度且尽量无错误。MCU 通过 `DECL_COMMAND()` 声明可被主机调用的命令，通过 `sendf()` 发送响应。构建过程会收集这些声明，生成可编码、可压缩的消息格式。对软件工程师来说，这像是一套针对小 MCU 定制的 RPC 和 IDL 机制，但它服务的是物理动作和时间表，不是普通业务请求。

运行流程上，一个典型移动命令从 `klippy/gcode.py` 解析 G1 开始，进入 `klippy/extras/gcode_move.py`，最终调用 `ToolHead.move()`。`klippy/toolhead.py` 负责 look-ahead，也就是提前观察后续移动，决定每段移动的起止速度和梯形速度曲线。运动学类会检查移动是否合法，再把笛卡尔空间中的移动转换为各步进轴的运动。随后 C helper 计算和压缩 step pulse 的时刻，MCU 端根据收到的排程准时输出脉冲。

硬件抽象上，Klipper 用配置文件表达引脚、MCU、步进电机、加热器、传感器和机械结构。改配置通常不需要重新刷 MCU 固件，这是它区别于许多传统 3D 打印固件的地方。MCU 端只保留足够稳定、可移植、重视时序的底层能力；主机端则通过 Python 模块扩展 `extras` 里的功能，如温度、TMC 驱动、床网格、输入整形、虚拟 SD 卡等。

基于公开文件推断，Klipper 的可维护性重点在三个边界：Python 主机对象模型和配置加载边界，主机到 MCU 的协议边界，以及 MCU 内部任务和定时器边界。任何新增功能如果放错边界，就可能把可维护的高级逻辑变成难调试的实时问题，或者把必须硬实时的动作交给不可靠的软件层。

## 5. 源码陪读：今天只读哪几个文件
1. `docs/Code_Overview.md`：先读项目作者给出的总览，理解目录、主机端和 MCU 端的职责划分。
2. `docs/Features.md`：再读功能文档，重点看高精度步进、多 MCU、配置文件、Pressure Advance、Input Shaping、硬件无关性这些设计目标。
3. `klippy/klippy.py`：阅读主机端启动入口，观察配置读取、对象加载、事件注册、状态切换和错误处理。
4. `klippy/toolhead.py`：阅读运动规划核心，理解 look-ahead、移动对象、速度曲线和运动学检查。
5. `src/sched.c` 与 `src/command.c`：最后读 MCU 端调度和命令分发，理解任务、定时器、关机路径和主机命令如何落到 MCU 函数。

阅读顺序建议是先文档后源码，先主机后 MCU，最后再回到协议文档串联全链路。

## 6. 核心模块/函数导读
- `Printer` 类：位于 `klippy/klippy.py`，负责保存打印机对象、读取配置、加载模块、管理状态和事件。它像主机端的应用容器，所有硬件抽象对象都通过配置进入这个容器。
- `ToolHead.move()`：位于 `klippy/toolhead.py`，承担移动请求的核心入口。它把 G-code 层的“去某个坐标”变成可检查、可预读、可排程的运动对象。
- `LookAheadQueue`：负责提前观察一串移动，决定拐角速度和加减速。没有它，打印路径会在每段线段处频繁停顿，既慢又容易留下瑕疵。
- `DECL_COMMAND()`：MCU 端命令声明宏，是主机和固件之间的协议契约。主机能调用什么底层动作，不靠手写 JSON，而靠构建期收集 C 代码里的声明。
- `sched_add_timer()` 与 `sched_timer_dispatch()`：位于 `src/sched.c`，管理 MCU 端定时回调。它们把“某个时刻输出一个脉冲”变成可执行的中断驱动事件，是硬实时部分的核心。

## 7. 软件工程师类比解释
- 主机端 Klippy 类似后端服务，负责解析请求、加载配置、组织对象和调用下游能力。局限是后端服务通常可以重试或排队，而打印运动错过时刻会直接影响物理结果。
- MCU 端类似一个高可靠执行代理，只执行主机下发的低层命令。局限是普通代理可以有较复杂逻辑，而 MCU 端必须非常节省内存和 CPU，并满足微秒级时序。
- `DECL_COMMAND()` 类似 RPC 接口定义。局限是这里的接口参数最终影响 GPIO、电机和加热器，不是数据库里的业务状态，错误代价更高。
- look-ahead 类似数据库查询优化器或流处理的窗口计算，提前看后续数据再决定当前策略。局限是它优化的是物理运动连续性，需要考虑速度、加速度和机械振动。
- 配置文件类似基础设施即代码。局限是这里的配置直接绑定引脚、电流、温度、机械尺寸和传感器，配置错误可能造成撞机或过热。

## 8. 今日 10 个嵌入式概念卡片
1. 概念名：步进脉冲。解释：步进电机驱动器通过脉冲数量和频率决定转动距离与速度。软件工程类比：消息队列中的事件流。常见场景：3D 打印、CNC、机械臂。坑点：脉冲时刻抖动会变成机械振动或尺寸误差。
2. 概念名：硬实时。解释：动作必须在规定时间点发生，晚了就算失败。软件工程类比：强 SLA 的延迟系统。常见场景：电机控制、采样、通信时隙。坑点：平均延迟好不等于最坏延迟可接受。
3. 概念名：运动学。解释：把喷头坐标转换为各个电机轴的运动。软件工程类比：业务模型到执行计划的转换层。常见场景：Delta、CoreXY、机械臂。坑点：不同机械结构不能套同一套轴映射。
4. 概念名：梯形速度曲线。解释：移动通常分为加速、匀速、减速三段。软件工程类比：限流器的平滑爬坡。常见场景：步进电机加减速。坑点：只看目标速度不看加速度会丢步或震动。
5. 概念名：look-ahead。解释：提前分析后续路径，避免每个线段都停顿。软件工程类比：批处理优化和窗口计算。常见场景：打印路径规划、CNC 插补。坑点：预读太少会慢，预读太激进会超出机械能力。
6. 概念名：Pressure Advance。解释：补偿挤出机压力滞后，减少角落堆料和拉丝。软件工程类比：对延迟系统做前馈补偿。常见场景：高速打印挤出控制。坑点：参数不准会过补偿。
7. 概念名：Input Shaping。解释：通过调整运动输入减少机械共振。软件工程类比：对系统响应做滤波。常见场景：高速打印抑制振纹。坑点：需要结合机器真实共振频率。
8. 概念名：GPIO 抽象。解释：用统一接口隐藏不同芯片引脚寄存器差异。软件工程类比：数据库驱动适配层。常见场景：多 MCU 移植。坑点：抽象不能牺牲关键路径性能。
9. 概念名：看门狗与 shutdown。解释：系统检测到异常后进入安全停机路径。软件工程类比：熔断和故障隔离。常见场景：通信失败、温度异常、任务卡死。坑点：停机动作本身也要安全、确定。
10. 概念名：多 MCU 时钟同步。解释：多个控制板协同时需要校准各自时钟漂移。软件工程类比：分布式系统时钟同步。常见场景：工具头板、主板、扩展板协同。坑点：普通 NTP 思维不足以处理微秒级动作排程。

## 9. 30～60 分钟动手任务
目标：在没有打印机硬件的情况下，读懂一条 G1 移动命令从主机到 MCU 的职责切分。

步骤：第一步，打开 `docs/Code_Overview.md`，画出 `gcode.py -> gcode_move.py -> ToolHead.move() -> LookAheadQueue -> kinematics -> MCU command` 的流程图。第二步，阅读 `klippy/klippy.py` 中 `_read_config()` 和 `_connect()`，标出配置如何变成 printer objects。第三步，阅读 `src/sched.c` 中 `sched_add_timer()` 和 `sched_timer_dispatch()`，写下为什么定时器回调必须很短。第四步，任选 `config/` 下一个示例配置，找出 MCU、stepper、heater、probe 或 fan 相关段落，解释这些配置最终会影响哪些软件对象或硬件行为。

验收标准：你能用自己的话说明“为什么 Klipper 不把所有运动规划都放到 MCU 上”，并能指出一条移动命令里哪些部分是软实时，哪些部分是硬实时。

## 12. 工程点评
可维护性方面，Klipper 的主机端 Python 模块化程度高，复杂功能可以放在 `extras` 中演进，降低了 MCU 固件膨胀风险。可移植性方面，`src/` 下的架构目录和 `generic` 层让它可以支持多种芯片，同时把 GPIO、定时器、串口等差异隔离。可靠性方面，调度文档明确要求任务函数避免长时间阻塞，定时器函数必须在几微秒内完成，并提供 shutdown 路径。可测试性方面，项目有 `test/` 目录和 simulator 支持，但真实运动质量仍依赖机械结构、驱动、电流、共振和传感器标定。量产和运维方面，配置文件降低了刷机频率，Web UI 和 API 生态方便用户维护；但配置自由度越高，现场误配置风险也越高，因此文档、默认配置和安全检查非常关键。
