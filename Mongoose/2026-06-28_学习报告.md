# 每日嵌入式学习：Mongoose（嵌入式网络库）

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/cesanta/mongoose
- **一句话**：Mongoose 是一个从 2004 年延续至今的嵌入式网络库，仅需两个文件（mongoose.c / mongoose.h）就能让任何 MCU 或桌面平台拥有完整的 HTTP、MQTT、WebSocket、TLS 1.3 协议栈和 OTA 固件更新能力，甚至运行在国际空间站上。

## 2. 为什么今天选它

| 维度 | 说明 |
|------|------|
| **活跃度** | 12.9k star，GitHub 上连续活跃近 20 年，最近提交在几小时前 |
| **工程完整度** | 内置 TCP/IP 栈 + 多平台 BSD Socket 兼容层 + 单文件分发模式，单元测试 + oss-fuzz + Cisco Talos/微软/MSRC 安全审计 |
| **对软件工程师的学习价值** | 网络协议（HTTP/TCP/MQTT）你已经在后端熟悉了，但"在 32KB RAM 的 MCU 上跑 HTTP 服务器"是完全不同的约束场景——这正是软件转嵌入式最有价值的视角转换 |

Mongoose 被 Siemens、Schneider Electric、Broadcom、Bosch、Google、NASA 等机构用在产品中；代码设计干净、文档完善、架构分层清晰，是学习嵌入式网络编程的绝佳范本。

## 3. 项目解决的问题

**真实场景**：工业设备需要有一个 Web 管理界面、或者 IoT 传感器需要通过 MQTT 上报数据、或者设备需要远程 OTA 更新固件——但这些设备只有 MCU（微控制器），跑不起完整 Linux 和 nginx。

**嵌入式特有约束**：

- **内存约束**：STM32H7 有 1MB RAM 算大的，但很多 MCU 只有 16KB～256KB RAM。Mongoose 可以在 40KB RAM 的 MCU 上运行 HTTP 服务器。
- **无 MMU**：MCU 没有虚拟内存，所有代码和堆栈共享同一物理地址空间，内存泄漏直接崩系统。
- **无操作系统或轻量 RTOS**：没有 Linux 的进程隔离，Mongoose 自己实现事件循环 + 内置 TCP/IP 栈（也可以跑在 lwIP/Zephyr 上）。
- **持久化约束**：没有磁盘，Flash 擦写次数有限（通常 1 万～10 万次），OTA 更新要精确控制 Flash 写入。
- **时序约束**：网卡中断、DHCP 租约、TCP 重传、MQTT keepalive 全都在裸机中断上下文或 RTOS 线程中处理。
- **低功耗要求**：IoT 设备可能用电池，网卡大部分时间要处于低功耗模式。

## 4. 技术架构分析

以下分析基于源代码（https://github.com/cesanta/mongoose）公开的代码和文档。

### 整体架构分层

```
应用层 (HTTP/MQTT/WebSocket/Modbus/SNTP)
     ↓
事件层 (mg_mgr/event loop / connection mgr)
     ↓
传输层 (BSD socket / builtin TCP/IP stack)
     ↓
硬件驱动层 (SPI ETH / W5500 / STM32 ETH / WiFi)
```

### 核心模块

| 模块 | 文件 | 职责 |
|------|------|------|
| **事件管理器** | `event.c` / `net.c` | 中心事件循环 `mg_mgr_poll()`，管理所有连接的 IO 多路复用 |
| **内置 TCP/IP 栈** | `net_builtin.c` / `net_builtin.h` | 完整 L2/L3/L4 协议栈：ARP、IP、ICMP、UDP、TCP，不依赖外部网络库 |
| **HTTP 引擎** | `http.c` / `http.h` | HTTP/1.1 解析和响应构造，路由匹配，文件服务 |
| **MQTT 客户端/服务器** | `mqtt.c` / `mqtt.h` | MQTT 3.1.1 pub/sub、QoS 0/1、遗嘱消息 |
| **TLS 1.3** | `tls_builtin.c` | 内置 ECC 加密套件（AES-128、ChaCha20、Ed25519），无需 OpenSSL |
| **OTA 模块** | `ota.c` + `ota_stm32h7.c` 等 | 面向具体 MCU 的固件下载和 Flash 写入 |
| **驱动层** | `drivers/` 子目录 | STM32 以太网 MAC、W5500 SPI 转以太网、Cypress WiFi 等 |
| **文件系统抽象** | `fs.c` / `fs_fat.c` / `fs_packed.c` | 嵌入式文件系统访问接口，支持 FATFS 或打包文件系统 |

### 运行流程

1. 用户调用 `mg_mgr_init()` 初始化事件管理器，创建一个连接链表头和定时器链表。
2. 调用 `mg_http_listen()` 或 `mg_mqtt_connect()` 创建监听/连接，注册事件处理函数。
3. 主循环反复调用 `mg_mgr_poll(&mgr, timeout)`，该函数内部：
   - 遍历所有连接，检查 IO 就绪状态（select/epoll 或轮询）。
   - 对每个有数据的连接，读取数据到 `recv` 缓冲区，然后触发 `MG_EV_READ` 事件。
   - 协议层（HTTP/MQTT）在事件处理中解析数据，生成 `MG_EV_HTTP_MSG` / `MG_EV_MQTT_MSG` 等高级事件。
   - 用户的事件处理函数收到高级事件，处理业务逻辑，调用 `mg_http_reply()` / `mg_mqtt_pub()` 等发送响应。
4. `mg_mgr_poll()` 还负责：TCP 重传计时、DHCP 续约、DNS 超时、MQTT keepalive、定时器触发。

### 硬件抽象设计

Mongoose 的架构很优雅：通过 `struct mg_tcpip_driver` 抽象底层硬件，任何新的以太网芯片只需实现 3 个函数指针（init/tx/rx/poll），就可以接入完整的 TCP/IP 栈和上层协议。

### 配置系统

编译时通过 `-DMG_ENABLE_TCPIP=1 -DMG_ENABLE_MQTT=0` 等宏裁剪功能，优化固件体积。所有配置项见 `src/config.h`。

- **为什么最后读**：OTA 是嵌入式产品工程的核心环节，也是纯软件工程师最不熟悉的领域。

## 6. 核心模块/函数导读

### 6.1 `mg_mgr_poll()` — 事件循环引擎
- **声明**：`void mg_mgr_poll(struct mg_mgr *mgr, int ms);`
- **角色**：这是整个库的心脏——每次调用都遍历所有连接，轮询 IO 事件、触发定时器、处理重传。
- **工程要点**：这个函数必须在主循环中以固定间隔反复调用。如果被阻塞超过 TCP 超时时间，连接就会断。在裸机上这就是 while(1) 循环；在 RTOS 中是一个专用线程。
- **类比后端**：相当于 Node.js 的 `event loop`，但每次是手动调用的，并且 timeout 参数直接影响低功耗休眠时长。

### 6.2 `mg_http_listen()` — HTTP 服务启动入口
- **声明**：`struct mg_connection *mg_http_listen(struct mg_mgr *mgr, const char *url, mg_event_handler_t fn, void *fn_data);`
- **角色**：创建一个监听 socket，接收 HTTP 请求。
- **工程要点**：URL 参数可以是 `http://0.0.0.0:80`（桌面/Linux）或 `http://0.0.0.0:80` 在 MCU 上同样有效。Mongoose 在 MCU 上用自己内置的 TCP/IP 栈接收连接。
- **类比后端**：相当于 `app.listen(80)`（Express.js），但底层可能完全基于自制 TCP 栈。

### 6.3 `struct mg_tcpip_driver` — 硬件驱动抽象
- **结构**：4 个函数指针 `{init, tx, rx, poll}`。
- **角色**：这是 Mongoose 能在任何带以太网 PHY 的 MCU 上移植的关键接口。提供一个新硬件驱动只需实现这 4 个函数。
- **工程要点**：`rx` 可以为 NULL（中断驱动模式），此时驱动通过 `mg_tcpip_qwrite()` 从 ISR 推送帧到主循环。这在嵌入式中非常常见：中断收包→主循环处理。
- **局限类比**：类似后端中的"网络接口适配器"模式，但这里的 `tx/rx` 操作的是原始以太网帧（L2 层数据），不是 HTTP 请求。

### 6.4 `mg_ota_begin()` / `mg_ota_write()` / `mg_ota_end()` — OTA 三部曲
- **角色**：从网络接收固件数据 → 分块写入 Flash → 校验后触发跳转。
- **工程要点**：
  - Flash 写入前必须先擦除扇区（通常 4KB～128KB 一个扇区）。
  - 擦写寿命约 1 万次（NOR Flash）～10 万次（NAND）。
  - OTA 期间掉电需要容错：常见方案是双备份区（A/B 分区）或恢复出厂区。
- **类比后端**：相当于 Kubernetes 的 Rolling Update，但 MCU 没有回滚的"容器层"，一旦写入错误分区就可能变砖。

### 6.5 `mg_mqtt_sub()` / `mg_mqtt_pub()` — MQTT 发布/订阅
- **角色**：在嵌入式设备上实现 MQTT 客户端。
- **工程要点**：
  - MQTT 的 QoS 1 需要设备本地存储已发送但未确认的报文（掉电重传），这对无持久化存储的 MCU 是个挑战。
  - MQTT 的 Keepalive 机制直接影响电池寿命：太频繁则耗电，太长则可能被服务器断开。
- **类比后端**：相当于 Redis Pub/Sub 或 AWS IoT Core 的 device shadow，但封包大小通常限制在 256 字节以内，且没有持久化队列。

## 7. 软件工程师类比解释

### 7.1 概念：事件驱动联网（Event-driven networking）

**类比** → Node.js 的事件循环 + Express 中间件。

`mg_mgr_poll()` 相当于 Node.js 的 `event loop`；用户注册的事件处理函数 `ev_handler` 相当于 Express 的路由中间件；事件类型 `MG_EV_HTTP_MSG` 相当于 Express 接收到的 HTTP 请求。

**这个类比的局限**：Node.js 的事件循环由 libuv 封装了 OS 级别的 epoll/kqueue/IOCP，而 Mongoose 在 MCU 上没有 OS 支持，它自己实现了一套 select/poll（甚至在某些平台上直接轮询裸寄存器）。此外 Node.js 的处理函数在 V8 堆上跑，有 GB 级内存；MCU 上事件处理必须在微秒级完成，否则丢包。

### 7.2 概念：内置 TCP/IP 栈

**类比** → 从零实现一个简版 Linux 网络协议栈（socket/bind/listen/accept）。

Mongoose 的 `net_builtin.c` 实现了 ARP 缓存、IP 分片重组、TCP 状态机、流量控制、重传计时器——相当于在 MCU 上"重新发明"了 Linux 内核的网络子系统。

**这个类比的局限**：Linux 的 TCP/IP 栈经过数十年优化，支持复杂的拥塞控制（CUBIC/BBR）、多队列、硬件 offload。Mongoose 的内置栈只实现了最必要的功能子集（每个方向默认 1 个 TCP 窗口，不支持 SACK 等），但这对 IoT 设备已足够。

### 7.3 概念：OTA 固件更新（Over-the-Air Firmware Update）

**类比** → 后端服务的滚动部署（rolling update），但服务是单例的、没有 Blue/Green 的"流量切换"。

OTA 流程：下载新固件 → 校验签名 → 擦除 Flash → 写入 → 验证 → 重启。MCU 上通常有 A 区和 B 区两个备份 Boot 分区，当前运行在 A 区时下载新固件到 B 区，然后切换启动向量指向 B 区。

**这个类比的局限**：后端滚动部署中的旧版本可以随时切回；MCU 的 OTA 一旦写入失败（掉电/校验失败），设备可能变砖（除非有不可擦除的 BootROM 恢复区）。Flash 的擦写次数限制也意味着不能频繁更新。

### 7.4 概念：编译时裁剪（Compile-time configuration）

**类比** → 微服务的"按需打包"——后端项目用 Maven/Gradle dependency exclusion 去掉不需要的库；Mongoose 用编译宏 `MG_ENABLE_*` 裁剪协议支持。

`src/config.h` 中定义了 50+ 个编译开关，比如 `#define MG_ENABLE_MQTT 0` 可以在固件中完全移除 MQTT 代码。

**这个类比的局限**：后端微服务裁剪依赖是为了减少启动时间或漏洞面；嵌入式裁剪是为了把固件体积压缩到几十 KB 的 Flash 里。少了 1KB 代码，可能意味着 MCU 可以降一个型号（成本降低 20%～50%）。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1: 事件驱动（Event-driven architecture）
- **一句话**：MCU 在主循环中反复轮询事件源，不阻塞等待。
- **类比后端**：Node.js event loop 但手动驱动。
- **常见场景**：裸机网络栈、GUI 事件处理。
- **坑点**：事件处理不能阻塞超过一个时间片，否则丢包；后端可以用 async/await 把阻塞任务挂起，裸机上所有操作都是同步的。

### 卡片 2: TCP/IP 卸载（TCP offload）
- **一句话**：MCU 用内置的 MAC+PHY 硬件处理部分网络层，上层协议仍在软件实现。
- **类比后端**：数据库的索引下推（index pushdown）。
- **常见场景**：STM32 的 ETH 外设 + lwIP 或 Mongoose 软件栈。
- **坑点**：不是所有 MCU 都带以太网 MAC；很多低端芯片需要外加 SPI 转以太网芯片（如 W5500）。

### 卡片 3: Flash 擦写寿命（Flash endurance）
- **一句话**：Flash 存储单元有擦写次数上限（NOR 约 1 万次，NAND 约 10 万次）。
- **类比后端**：SSD 的写入寿命（TBW）。
- **常见场景**：日志记录、配置参数保存、OTA 固件分区。
- **坑点**：频繁写入相同地址（如每秒写一次配置）会快速烧坏 Flash。必须用磨损均衡（wear leveling）。

### 卡片 4: DHCP 客户端
- **一句话**：MCU 启动时通过 DHCP 从路由器获取 IP 地址，与后端完全一样——但 MCU 掉电后没有持久化存储 IP，每次重启都需要重新 DHCP。
- **类比后端**：EC2 实例启动时自动获取私有 IP。
- **常见场景**：所有需要联网的 IoT 设备。
- **坑点**：DHCP 超时通常需要 1～5 秒，MCU 在此阶段不能中断。如果 DHCP 服务器没响应，设备需要降级到 Link-Local 地址或静态 IP。

### 卡片 5: ARP 缓存
- **一句话**：IP 到 MAC 地址的本地映射表，MCU 上缓存在内存中，条目通常只有 5～10 个。
- **类比后端**：DNS 缓存 + 路由表合体。
- **常见场景**：每个 TCP 连接建立前都需要 ARP 查询。
- **坑点**：ARP 表满了会覆盖旧条目；ARP 欺骗在 MCU 上更难检测（没有防火墙）。

### 卡片 6: 零拷贝（Zero-copy）——嵌入式版
- **一句话**：网络数据直接在 DMA 缓冲区中处理，避免从硬件寄存器搬移到应用内存。
- **类比后端**：Linux 的 `sendfile()` / Netty 的 `FileRegion`。
- **常见场景**：高速网络处理、音视频流。
- **坑点**：MCU 的 DMA 缓冲区通常是固定大小且对齐的，引用时要小心字节对齐和 Cache 一致性。

### 卡片 7: ISR → 主循环队列（Interrupt to main loop queue）
- **一句话**：硬件中断（ISR）收到数据后不直接处理，而是推到一个队列，主循环轮询时取队列处理。
- **类比后端**：消息队列 + Worker 模式（RabbitMQ + 消费者）。
- **常见场景**：网卡中断、串口接收。
- **坑点**：ISR 中不能调用阻塞函数、不能分配堆内存、不能持有互相阻塞的锁。推队列操作必须是原子的。

### 卡片 8: 磨损均衡（Wear leveling）
- **一句话**：将 Flash 写入均匀分布到所有扇区，避免某一片提前报废。
- **类比后端**：SSD 的 FTL（Flash Translation Layer）。
- **常见场景**：文件系统（littlefs/SPIFFS）、Flash 存储配置参数。
- **坑点**：MCU 上实现磨损均衡通常需要额外的 RAM 维护映射表（64KB RAM 可能只能存几千条映射）。

### 卡片 9: 看门狗定时器（Watchdog Timer）
- **一句话**：硬件定时器，如果超过设定时间没有被"喂狗"（清零），就触发系统复位。
- **类比后端**：健康检查 + 自动重启（Kubernetes liveness probe）。
- **常见场景**：工业设备、汽车电子、远程部署的 IoT 节点。
- **坑点**：`mg_mgr_poll()` 耗时超过看门狗超时时间会导致设备反复复位。必须在每一个主循环中喂狗。

### 卡片 10: 双分区 OTA（A/B partition OTA）
- **一句话**：Flash 分成 A 区和 B 区，运行在 A 区时下载新固件到 B 区，然后切换启动。
- **类比后端**：Blue/Green 部署。
- **常见场景**：所有支持远程固件更新的产品。
- **坑点**：需要 BootROM 级别的基础加载程序（Bootloader）来判断分区有效性。如果两个分区都损坏，就需要 Recovery 模式（通过 USB/串口强制烧录）。

## 9. 30～60 分钟动手任务

### 目标
用 Mongoose 在本地 Linux 环境（WSL）上启动一个简易的 HTTP 服务器，提供设备模拟仪表盘。

### 步骤
1. 下载 mongoose.h 和 mongoose.c：
   ```bash
   curl -L --noproxy '*' -o mongoose.h "https://raw.githubusercontent.com/cesanta/mongoose/master/mongoose.h"
   curl -L --noproxy '*' -o mongoose.c "https://raw.githubusercontent.com/cesanta/mongoose/master/mongoose.c"
   ```
2. 创建 `main.c`，内容如下：
   ```c
   #include "mongoose.h"
   static void ev_handler(struct mg_connection *c, int ev, void *ev_data) {
     if (ev == MG_EV_HTTP_MSG) {
       struct mg_http_message *hm = (struct mg_http_message *) ev_data;
       struct mg_http_serve_opts opts = { .root_dir = "." };
       mg_http_serve_dir(c, hm, &opts);
     }
   }
   int main(void) {
     struct mg_mgr mgr;
     mg_mgr_init(&mgr);
     mg_http_listen(&mgr, "http://localhost:8000", ev_handler, NULL);
     printf("Server started on http://localhost:8000\n");
     for (;;) mg_mgr_poll(&mgr, 1000);
     return 0;
   }
   ```
3. 编译：`cc -o server main.c mongoose.c`
4. 运行：`./server`
5. 浏览器访问 `http://localhost:8000`

## 10. 3 道深度追问题

**问题 1（时序/并发）**：Mongoose 的事件循环是单线程的。如果一个事件处理函数中调用了 `sleep(2)`（模拟阻塞操作），会如何影响 TCP 重传、MQTT keepalive 和同一管理器上的其他连接？在后端 Node.js 中可以用 `setTimeout` 推迟阻塞操作，Mongoose 上应该用什么机制？

**问题 2（资源限制）**：假设你需要在仅有 64KB RAM 的 MCU 上同时运行：①HTTP 服务器（服务静态文件）、②MQTT 客户端（每 10 秒上报一个温度值）、③OTA 下载管理器。估算每个模块至少需要多少 RAM？如果 RAM 不够，你的裁剪策略是什么？

**问题 3（软件→嵌入式视角转换）**：你在后端项目中对 API 切面进行 logging/metrics/tracing 的增强时，通常用 AOP 切面或 middleware 模式。现在把同样的需求放到 STM32H7 上：
- 固件版本：通过 HTTP API `/api/version` 返回
- 日志：通过 WebSocket 实时推送到浏览器
- 指标：CPU 负载、网络吞吐量、温度

请用 Mongoose 的事件模型设计这三个功能，并指出哪个功能在 MCU 上实现时有与后端完全不同的工程约束。
