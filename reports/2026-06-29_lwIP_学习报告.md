# 每日嵌入式学习：lwIP（轻量级 TCP/IP 协议栈）

## 1. GitHub 项目链接与一句话介绍

- 仓库地址：https://github.com/lwip-tcpip/lwip
- 一句话：lwIP 是一个专为嵌入式系统设计的轻量级 TCP/IP 协议栈，让只有几十 KB 内存的微控制器也能实现完整的 TCP/IP 网络通信。

## 2. 为什么今天选它

lwIP 是嵌入式网络领域的事实标准。从 Zephyr、FreeRTOS+TCP 到各种物联网模块，几乎每个需要网络的嵌入式系统都在直接或间接使用 lwIP 的设计思想。

- **活跃度**：GitHub 镜像仓库 1671 Stars、596 Forks，2026 年 6 月仍有活跃提交，核心维护团队持续修复 TCP 选项解析边界检查、ND6 邻居发现等关键问题。
- **工程完整度**：支持 IPv4/IPv6、TCP 拥塞控制、UDP、DHCP、DNS、SNMP、MQTT、HTTP、TLS 等完整协议栈，代码精炼但功能完备。
- **学习价值**：对于从纯软件转向嵌入式的工程师，lwIP 是理解"资源受限环境下的网络编程"的绝佳教材——它展示了如何在数十 KB 内存中实现 TCP 的拥塞控制、重传、RTT 估计等完整功能。
- **Why Now**：此前已经覆盖了 TinyUSB（USB 协议栈）、Mongoose（嵌入式 HTTP/MQTT 库），但底层的 TCP/IP 协议栈本身还没有讲过。理解 lwIP 是理解上层网络应用的基础。

## 3. 项目解决的问题

**真实场景**：一个温度传感器节点，使用 ESP8266/ESP32 或 STM32+W5500 芯片，需要通过 WiFi 或以太网将数据上报到云端 MQTT 服务器。微控制器的 RAM 只有 32KB~256KB，Flash 只有 256KB~2MB。标准的 Linux TCP/IP 协议栈（如 Linux kernel 的网络子系统）根本无法在这里运行。

**lwIP 解决的约束**：

- **内存**：lwIP 核心代码占用约 40KB ROM，运行时 RAM 消耗可配置在 10KB~50KB 之间。通过 pbuf（packet buffer）零拷贝设计、静态内存池、回调驱动而非线程驱动等机制，大幅降低内存开销。
- **时序**：在无 OS（NO_SYS）模式下，全部网络处理在单线程事件循环中完成，避免锁和上下文切换开销。
- **可靠性**：完整的 TCP 拥塞控制（慢启动、拥塞避免、快速重传、快速恢复）、RTT 估计、选择性确认（SACK），在资源受限环境下保持 TCP 的可靠性承诺。
- **可移植性**：通过抽象层（sys.h/arch.h）隔离平台相关代码，支持 FreeRTOS、裸机、ThreadX 等多种底层环境。

**与普通软件项目的差异**：后端工程师写一个 TCP 服务端，可以毫不在意地分配几百 MB 内存、使用阻塞 socket、依赖 OS 内核完成 TCP 分段重传。而在 lwIP 中，每一字节内存都是手工管理的，TCP 发送窗口必须精确计算，pbuf 链的引用计数必须手动跟踪，否则一个内存泄漏就会导致整个设备死机。

## 4. 技术架构分析

**基于公开文件和源码推断。**

### 整体架构分层

```
应用层（HTTP/SNMP/MQTT/TFTP/...）
    ↑↓
Socket API  /  Netconn API  /  Raw API
    ↑↓
    TCP          UDP         RAW
    ↑↓            ↑↓          ↑↓
     -- IP（IPv4 + IPv6）--
    ↑↓
     netif（网络接口抽象层）
    ↑↓
   底层驱动（Ethernet/PPP/6LoWPAN/...）
```

### 核心目录结构

| 目录 | 用途 |
|------|------|
| `src/core/` | TCP/IP 核心协议实现（tcp.c, udp.c, ip.c, pbuf.c, netif.c, mem.c, memp.c, dns.c 等） |
| `src/api/` | 高层 API（netconn：类似 BSD socket 的线程安全 API；sockets：兼容 POSIX socket 的封装层） |
| `src/apps/` | 应用层协议（httpd, snmp, mqtt, sntp, tftp, mdns 等） |
| `src/netif/` | 网络接口驱动（ethernetif.c、PPP、6LoWPAN SLIP 等） |
| `src/include/lwip/` | 全部头文件（57 个 .h，包括 tcp.h, pbuf.h, opt.h, netif.h, ip.h, udp.h 等） |

### 关键设计模式

1. **pbuf 机制**：lwIP 的核心数据抽象。一个网络数据包被表示为一个 pbuf 链表，每个 pbuf 节点要么引用动态分配的 RAM，要么指向 ROM 中的静态数据。pbuf 支持引用计数，多个协议层可以共享同一个数据缓冲区而不需要拷贝数据——这是"零拷贝"设计的关键。

2. **Raw/Callback API**：lwIP 最底层的 API，完全基于回调函数。用户注册 `tcp_recv_fn`（收到数据时触发）、`tcp_sent_fn`（数据被确认时触发）、`tcp_poll_fn`（周期性轮询）等回调。这种设计避免了线程开销，但要求用户"不要在回调中阻塞"。

3. **Netconn API**：在 Raw API 上封装的一层线程安全 API，内部使用邮箱/信号量实现类似 BSD socket 的阻塞操作。适合在有 RTOS 的环境中使用。

4. **配置系统（opt.h）**：lwIP 的所有功能开关和参数都在 `opt.h` 中定义（约 115KB 的配置头文件）。用户通过 `lwipopts.h` 覆盖默认值，精细控制哪些功能编译进固件。这是嵌入式领域典型的"配置即编译"范式。

### 初始化流程（lwip_init 源码分析）

```c
void lwip_init(void) {
    stats_init();       // 统计计数器
    sys_init();         // 系统抽象层（信号量/邮箱初始化）
    mem_init();         // 堆内存管理器
    memp_init();        // 静态内存池
    pbuf_init();        // pbuf 池
    netif_init();       // 网络接口链表
    ip_init();          // IP 层
    etharp_init();      // ARP 缓存
    raw_init();         // Raw socket
    udp_init();         // UDP 控制块链表
    tcp_init();         // TCP 控制块链表 + TCP 定时器
    igmp_init();        // IGMP 组播
    dns_init();         // DNS 客户端
    sys_timeouts_init();// 系统定时器
}
```

这个初始化顺序是有讲究的：先底层后上层（先内存、再 pbuf、再 netif、再 IP、再传输层），因为上层初始化时可能需要调用下层功能。

### 关键数据结构

- **struct tcp_pcb**：TCP 协议控制块，包含发送/接收窗口、序列号、状态（CLOSED/LISTEN/SYN_SENT/ESTABLISHED 等）、回调和超时信息。每个 TCP 连接对应一个 pcb。
- **struct pbuf**：包缓冲区。核心字段：`next`（链表指针）、`payload`（数据指针）、`len`（本节点数据长度）、`tot_len`（链中从本节点开始的总长度）、`ref`（引用计数）、`type_internal`（类型标志）。
- **struct netif**：网络接口，包含 IP 地址、MAC 地址、MTU、输入/输出/链路状态函数指针。

## 5. 源码陪读：今天只读哪几个文件

建议按以下顺序阅读：

**第一站：`src/include/lwip/opt.h`**
- 阅读目标：理解 lwIP 的配置体系。不要逐行读全文件（115KB），而是搜索 `LWIP_TCP`、`TCP_WND`、`TCP_MSS`、`MEMP_NUM_TCP_PCB`、`PBUF_POOL_SIZE` 等关键宏，理解每个配置对内存和功能的影响。
- 为什么先读它：lwIP 的一切行为都由这些宏控制，不理解配置就看不懂代码为什么这样写。

**第二站：`src/include/lwip/pbuf.h` + `src/core/pbuf.c`**
- 阅读目标：理解 pbuf 的数据结构、分配（pbuf_alloc）、释放（pbuf_free）、链式管理、引用计数机制。重点关注 `PBUF_RAM`、`PBUF_POOL`、`PBUF_ROM` 三种类型的不同行为和适用场景。
- 为什么第二步：pbuf 是 lwIP 所有数据流动的载体，不理解 pbuf 就无法理解任何协议处理。

**第三站：`src/include/lwip/tcp.h`**
- 阅读目标：理解 TCP PCB 的核心结构、所有回调函数类型（`tcp_recv_fn`、`tcp_sent_fn`、`tcp_poll_fn`、`tcp_err_fn`、`tcp_accept_fn`）、窗口缩放选项、SACK 结构体。
- 为什么第三步：理解了 PCB 和回调机制就理解了 lwIP TCP 的工作范式。

**第四站：`src/core/tcp.c`**
- 阅读目标：重点关注 `tcp_init`（TCP 子系统的初始化）和 `tcp_bind`、`tcp_listen`、`tcp_connect` 等连接管理函数。不要试图一次读完 86KB，先看入口函数和状态管理。
- 为什么第四步：连接管理是理解 lwIP 如何使用回调方式处理三次握手的起点。

**第五站：`src/core/init.c`**
- 阅读目标：完整通读整个文件（约 16KB）。这里包含了全部编译时检查（防止配置错误）和 `lwip_init` 的调用顺序。看完这个文件你就知道 lwIP 是如何"启动"的。
- 为什么最后读：作为收尾，把前面的模块如何组织成一个整体串起来。

## 6. 核心模块/函数导读

### 1. `lwip_init()`（`src/core/init.c`）
- **角色**：系统初始化入口。在 NO_SYS 模式下由用户 main 函数在初始化时调用一次；在有 RTOS 时由 `tcpip_init()` 替代。
- **关键点**：初始化顺序就是依赖顺序。如果在 `mem_init()` 之前调用 `tcp_init()`，会导致内存池未初始化而崩溃。这种显式依赖管理在嵌入式系统中非常常见。
- **关系**：与 sys_init（系统抽象）、mem_init（堆）、memp_init（内存池）、pbuf_init（包缓冲区）构成系统的"软启动链"。

### 2. `pbuf_alloc()`（`src/core/pbuf.c`）
- **角色**：分配一个 pbuf，是 lwIP 中最频繁调用的函数之一。每个接收到的网络包、每个要发送的数据段都需要通过 pbuf 封装。
- **关键点**：根据 type 参数（RAM/POOL/ROM）从不同的内存来源分配。PBUF_POOL 类型从预分配的固定大小池中分配——分配速度快、无碎片，但单个 pbuf 大小固定。PBUF_RAM 类型从堆中分配——大小灵活但可能碎片化。
- **关系**：所有协议层的输入输出都通过 pbuf。上层调用 `tcp_write()` 时，数据被拷贝到 pbuf 中排队；下层驱动从 netif 的 input 回调中接收到 pbuf，然后通过协议栈向上传递。

### 3. `tcp_process()`（`src/core/tcp.c`）
- **角色**：TCP 状态机的主处理函数。根据当前状态（SYN_SENT、ESTABLISHED、FIN_WAIT_1 等）和接收到的 TCP 标志（SYN、ACK、FIN、RST）执行状态转换。
- **关键点**：该函数在 `tcp_input()` 中被调用（每次收到 TCP 段时），它实现了 RFC 793 定义的 TCP 状态机。在嵌入式环境下，这个函数必须快速返回，不能阻塞。
- **关系**：与 `tcp_input()`（输入段解析）、`tcp_output()`（输出段发送）、`tcp_tmr()`（周期定时器）共同构成 TCP 的核心处理循环。

### 4. `tcp_output()`（`src/core/tcp_out.c`）
- **角色**：发送 TCP 段。检查发送窗口、Nagle 算法、拥塞窗口，决定是否可以发送数据。如果需要发送，构造 TCP 头并通过 `ip_output_if()` 交给 IP 层。
- **关键点**：这是 lwIP 性能的关键。它必须平衡延迟（尽快发送小包）和效率（合并报文减少头开销）。Nagle 算法和延迟 ACK 的相互作用是常见的性能调优点。
- **关系**：被 `tcp_write()`（应用写入数据）、`tcp_process()`（状态变化触发）、`tcp_sent_fn`（收到 ACK 后窗口释放）触发。

### 5. `netif_add()`（`src/core/netif.c`）
- **角色**：注册一个网络接口到 lwIP 的 netif 链表上。每个物理接口（以太网、WiFi、PPP）在初始化时调用此函数。
- **关键点**：参数包括 IP 地址、子网掩码、网关、输入函数指针（`netif_input`）。netif 结构中还包含 `state` 指针（指向驱动私有数据）和 `hwaddr`（MAC 地址）。
- **关系**：底层驱动通过 netif 的 `output` 或 `linkoutput` 函数发送数据；网络层通过 netif 的 `input` 回调接收数据。netif 是整个协议栈和硬件之间的桥梁。

## 7. 软件工程师类比解释

### 1. pbuf 链表 → HTTP 的分块传输（Transfer-Encoding: chunked）
pbuf 链将一个网络包拆分为多个 pbuf 节点，每个节点有自己的 payload 指针和长度。这类似于 HTTP 分块传输中将响应体拆成多个 chunk。
- **类比的局限**：pbuf 链不只是为了"不知道总长度"，更重要的是"零拷贝"——不同协议层可以添加/移除自己的头部而不需要移动数据，只需要调整 payload 指针。HTTP chunk 没有这种指针操作的语义。

### 2. Raw/Callback API → Node.js 的事件循环 + 回调
Node.js 中你注册一个 `http.createServer((req, res) => {...})` 回调，事件循环在请求到达时调用它。lwIP 的 Raw API 完全一样——你注册 `tcp_recv()`、`tcp_sent()` 等回调，主循环在有网络事件时调用它们。
- **类比的局限**：Node.js 的回调在 V8 的 EventLoop 中运行，有完整的堆栈和异常处理。lwIP 的回调在全局的 tcpip_thread 或主循环中运行，**不允许阻塞**（没有类似 `await` 的机制），也不能抛出异常——每个回调必须显式返回错误码。

### 3. memp.c 的内存池 → 数据库连接池
lwIP 使用 `memp.c` 管理多个固定大小的内存池（TCP PCB 池、pbuf 池、网络缓冲池等）。需要时从对应池中分配，用完后归还。这类似于数据库连接池——预先创建一批固定形状的资源，复用它们避免频繁分配和释放的开销。
- **类比的局限**：数据库连接池通常可以动态扩容（"最大连接数"只是一个上限）。lwIP 的内存池在**编译时**就固定了大小（`MEMP_NUM_TCP_PCB`、`PBUF_POOL_SIZE` 等宏），超过上限直接分配失败。你不能在运行时"再开一个连接池"。这反映了嵌入式系统的"静态资源规划"思维。

### 4. opt.h + lwipopts.h → 编译期 Feature Flags
lwIP 中没有运行时配置中心或 JSON 配置文件。所有功能开关都在 `lwipopts.h` 中定义为预处理器宏，编译时决定包含哪些代码。这类似于后端项目中用 `#ifdef FEATURE_X` 或 Cargo features 来条件编译。
- **类比的局限**：后端项目可以在运行时动态加载配置或通过环境变量切换特性。lwIP 的配置是"一次编译，终生使用"——要改变配置必须重新编译整个固件并重新烧录到设备。这也意味着发布固件前必须非常确定配置的正确性。

### 5. NO_SYS 模式下的主循环轮询 → 单线程 Reactor 模式
在 NO_SYS 模式下，lwIP 在用户的主循环中定期调用 `sys_check_timeouts()` 和 `ethernetif_input()`。这本质上就是 Reactor/事件驱动模式——一个线程循环检查多个事件源并分发处理。
- **类比的局限**：软件工程中的 Reactor 模式（如 Netty、libevent）通常支持大量并发连接（数万级别），依靠操作系统的 epoll/kqueue 实现高扩展性。lwIP 的 NO_SYS 模式通常仅支持少量连接（个位数到数十个），因为其设计目标是低资源占用而非高并发。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：pbuf（Packet Buffer）
- **一句话**：lwIP 中所有网络数据的载体，是一个支持零拷贝链式拼接的包缓冲区。
- **软件类比**：Cursor/Stream 的组合体——每个 pbuf 像是一段内存切片（Cursor），多个 pbuf 可以链起来形成完整消息（Stream）。
- **常见场景**：网卡中断收到一个以太网帧，驱动将其放入 pbuf 中，然后通过协议栈逐层向上传递。每经过一层（IP → TCP → 应用），pbuf 的 payload 指针就向前移动到该层首部之后。
- **易错点**：pbuf 使用引用计数管理生命周期，引用计数归零时自动释放。忘记调用 `pbuf_ref()` 增加引用就跨线程传递 pbuf，会导致悬空指针。反之，忘记 `pbuf_free()` 则造成内存泄漏。

### 卡片 2：静态内存池（MEMP）
- **一句话**：在编译时预分配一批固定大小的内存块，运行时按需分配和释放，速度快且不会产生碎片。
- **软件类比**：Go 语言的 `sync.Pool` 或 Java 的对象池，都是为了避免频繁的 GC 开销。
- **常见场景**：lwIP 中 TCP PCB（`MEMP_NUM_TCP_PCB`）、pbuf 池（`PBUF_POOL_SIZE`）、网络缓冲（`MEMP_NUM_NETBUF`）都使用静态内存池。
- **易错点**：池耗尽时分配返回 NULL。如果代码中没有正确处理内存分配失败的情况，会导致空指针解引用。嵌入式开发中必须总是检查分配返回值。

### 卡片 3：Raw API（原生回调 API）
- **一句话**：lwIP 最底层的编程接口，完全基于函数指针回调，没有阻塞操作。
- **软件类比**：前端开发中的事件监听器。你不是去"读数据"，而是注册"当数据到达时调用我"。
- **常见场景**：高性能网络设备（路由器、网关）中使用 Raw API，因为它没有线程切换开销。
- **易错点**：在回调函数中做阻塞操作（如延时等待、打印大量日志、调用可能阻塞的函数）会阻塞整个协议栈的处理线程，导致丢包或超时重传。

### 卡片 4：netif（Network Interface）
- **一句话**：lwIP 对物理网络接口的抽象，每个网口（以太网/WiFi/PPP）对应一个 netif 实例。
- **软件类比**：Linux 中的 `struct net_device` 或 Kubernetes 中的 CNI 接口。
- **常见场景**：ESP32 的 WiFi 初始化后调用 `netif_add()` 注册 WiFi 接口；以太网芯片（如 W5500）的驱动初始化后调用同样的函数注册以太网接口。
- **易错点**：多个 netif 同时存在时，IP 路由选择（通过哪个接口发送数据）由默认路由表决定。如果默认路由设置错误，数据可能从错误接口发出。

### 卡片 5：TCP PCB（Protocol Control Block）
- **一句话**：TCP 连接的"控制结构体"，包含连接状态、窗口、序列号、回调函数指针等全部 TCP 连接上下文。
- **软件类比**：HTTP 服务器中的 `request` 对象——它代表一个特定连接的所有状态。
- **常见场景**：`tcp_new()` 创建一个新 PCB，`tcp_bind()` 绑定端口，`tcp_listen()` 开始监听，`tcp_connect()` 发起连接。
- **易错点**：PCB 分为两种类型——`tcp_pcb`（连接态）和 `tcp_pcb_listen`（监听态）。监听 PCB 在调用 `tcp_listen()` 时从连接 PCB 转换而来，其内存结构不同。对监听 PCB 调用连接态的函数会导致内存访问错误。

### 卡片 6：ARP 缓存
- **一句话**：将 IP 地址映射到 MAC 地址的本地缓存表，用于在以太网段上找到目标设备。
- **软件类比**：DNS 缓存将域名映射到 IP，ARP 缓存将 IP 映射到 MAC。
- **常见场景**：当 lwIP 要通过以太网发送 IP 包时，先查 ARP 缓存找目标 IP 的 MAC 地址。如果缓存未命中，则广播 ARP 请求。
- **易错点**：ARP 条目会老化超时，超时后需要重新查询。在高频通信场景下，如果 ARP 缓存太小（`ARP_TABLE_SIZE` 默认 10），频繁的 ARP 申请和超时会导致网络闪断。

### 卡片 7：零拷贝（Zero Copy）
- **一句话**：数据在协议栈各层之间传递时，不拷贝数据本身，只调整指针和元数据。
- **软件类比**：Rust 中的引用（&T）传递而不是 clone()，Go 中切片（slice）的重新切片操作。
- **常见场景**：TCP 收到数据后，传递给用户回调时只传递 pbuf 指针；用户在回调中处理完数据后调用 `pbuf_free()`。这整个过程有一次内存拷贝（从 DMA 缓冲区到 pbuf），没有额外的拷贝。
- **易错点**：零拷贝意味着"谁最后使用，谁负责释放"。如果用户在回调之外还保存了 pbuf 指针，必须调用 `pbuf_ref()` 增加引用计数。忘记增加引用而跨函数调用使用 pbuf，是 lwIP 中最常见的内存错误之一。

### 卡片 8：配置即编译
- **一句话**：嵌入式系统的功能选择和参数配置在编译时通过预处理宏完成，没有运行时的配置文件读取。
- **软件类比**：Rust 的 `#[cfg(feature = "...")]` 条件编译，或 C++ 的 `#ifdef` 编译期开关。
- **常见场景**：`LWIP_TCP=1` 编译 TCP 支持，`LWIP_UDP=0` 排除 UDP 代码以节省空间，`TCP_WND=4096` 设置 TCP 窗口大小。
- **易错点**：`opt.h` 默认值基于"通用嵌入式设备"设定，直接使用默认值可能导致性能不佳或内存浪费。例如默认的 TCP 窗口（`TCP_WND`）可能太小导致吞吐量低下，默认的 `MEMP_NUM_TCP_PCB` 可能太少导致无法建立足够连接。

### 卡片 9：NO_SYS 模式
- **一句话**：lwIP 在没有 RTOS 或操作系统的裸机环境下运行的模式，所有网络处理在单线程主循环中完成。
- **软件类比**：单线程事件循环（setInterval/poll），类似 Arduino 的 `loop()` 函数。
- **常见场景**：资源极度受限的 MCU（如 Cortex-M0 只有 8KB RAM）或简单的 IoT 传感器节点。
- **易错点**：NO_SYS 模式下没有锁保护，任何中断服务程序（ISR）中访问 lwIP 数据结构都必须极其小心——通常的做法是在 ISR 中只做标记，在主循环中处理 lwIP 调用。

### 卡片 10：sys_timeout / TCP 定时器
- **一句话**：lwIP 内部的软件定时器机制，用于实现 TCP 超时重传（RTO）、保活探测、延迟 ACK 等时间敏感操作。
- **软件类比**：JavaScript 的 setTimeout——注册一个函数在指定时间后执行。
- **常见场景**：TCP 发送数据段后启动 RTO 定时器；如果在定时器到期前没有收到 ACK，触发超时重传并指数退避。
- **易错点**：lwIP 的定时器精度取决于主循环调用 `sys_check_timeouts()` 的频率。如果主循环长时间不返回（比如在处理一个耗时的传感器读取），TCP 定时器无法触发，导致 RTO 失效、连接中断。

## 10. 3 道深度追问题

## 12. 工程点评

**可维护性**：8/10。lwIP 代码风格清晰，模块划分合理，头文件有完善的 Doxygen 注释。但 opt.h 的 115KB 配置宏可能是双刃剑——配置灵活但新手上手困难。变量命名使用匈牙利前缀（如 `u16_t len`、`err_t err`）对 C 语言新手不太友好。

**可移植性**：9/10。通过 `cc.h`（编译器相关）、`sys.h`（OS 相关）、`arch.h`（架构相关）三个抽象文件隔离平台差异。支持从 ARM Cortex-M0 到 x86 Linux 的广泛范围。这也是它成为嵌入式网络事实标准的原因之一。

**可靠性**：8/10。编译时静态检查（init.c 中大量 `#error` 宏）在编译阶段就捕获配置错误。TCP 实现经过 20 多年实战检验，协议处理基本正确。但在内存不足时的错误处理路径上（如 pbuf_alloc 返回 NULL），部分回调路径的 NULL 检查可能遗漏。

**可测试性**：6/10。官方提供了 UNIX 模拟环境可以在 PC 上测试，但缺少完善的单元测试框架和 CI 测试用例。近年来 GitHub Actions 开始构建，但测试覆盖率不透明。对"修改后如何确保没有引入 TCP 协议错误"这个问题，社区更多依赖同行评审和实际使用。

**量产/运维**：7/10。lwIP 的统计计数器和 SNMP 代理提供了基本的运行时监控能力。但缺少类似 TCP 连接的 Prometheus 指标导出、结构化日志等现代可观测性设施。在量产环境中，通常需要二次开发来添加这些能力。
