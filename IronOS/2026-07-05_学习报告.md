# 每日嵌入式学习：IronOS — 开源智能烙铁固件

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/Ralim/IronOS
- **一句话介绍**：IronOS 是一套为多种智能电烙铁硬件（TS100、Pinecil、MHP30 等）提供开源替代固件的嵌入式项目，把一支普通烙铁变成拥有温度控制、功率协商、OLED 显示、自动休眠、BLE 等功能的智能工具。

## 2. 为什么今天选它

**活跃度与工程完整度**：8270+ Stars，持续活跃（2026-07-04 仍有提交），支持 31 种语言翻译，覆盖 10 种硬件变体（包含 STM32、RISC-V/BL702、GD32VF103 多个平台），CI/CD 完善，稳定版 v2.23。项目已经处于"日常可用功能完备"阶段，代码质量高、注释规范。

**适合软件工程师学习嵌入式的原因**：
- 这是"软件思维做硬件产品"的典型范例——从用户角度设计状态机（Soldering / Sleeping / Hibernating / SettingsMenu 等），而非从寄存器出发。
- 覆盖了嵌入式产品开发的全栈知识：ADC 采样 → 热电耦测温 → PID 控制 → PWM 输出 → USB PD 协议协商 → OLED 显示 → FreeRTOS 任务调度 → Flash 存储配置。
- 跨平台硬件抽象层（BSP）设计清晰，是学习嵌入式"一次编写多平台运行"的好教材。

## 3. 项目解决的问题

智能烙铁需要解决的真正工程问题包括：

### 真实产品场景
一支现代智能烙铁（如 Pinecil V2）需要做到：用户插上电，烙铁头在几秒内加热到设定温度（比如 350°C），实时显示温度，在静止时自动降温进入休眠保护烙铁头和用户安全，支持从 USB PD 充电器协商出最高 28V 的电源。

### 与普通软件项目相比多出的约束
- **实时性约束**：温度控制环路必须在每次 ADC 采样完成后的几毫秒内计算出新的 PWM 占空比，否则温度会过冲或不足，直接影响焊接质量甚至烧坏烙铁头。
- **硬件资源约束**：STM32F103 只有 64KB Flash 和 20KB RAM，要在这么小的空间里放下 FreeRTOS、OLED 驱动、USB PD 协议栈、31 种语言翻译文本、PID 控制器和大量配置项。
- **时序敏感性**：PWM 和 ADC 需要精确同步，ADC 采样期间 PWM 必须关闭（dead time），否则采样结果被加热电流噪声淹没。
- **可靠性约束**：烙铁是热工具，温度失控可能导致火灾、烫伤。代码中有 Thermal Runaway 检测（热失控保护）和多重安全校验。
- **电源协商的时序**：USB PD 协议需要在 CC 线上 5ms 内响应，否则充电器会断开。

## 4. 技术架构分析

### 总体架构

IronOS 采用典型的嵌入式分层架构：

```
+----------------------------+
|   Operating Mode State     |  ← GUI 状态机（Soldering/Sleeping/Settings...）
|         Machine            |
+----------------------------+
|   Core / Threads           |  ← FreeRTOS 任务（UI、PID 控制、温度读取）
+----------------------------+
|   Drivers                  |  ← TipThermoModel, USBPD, 加速度计
+----------------------------+
|   BSP (Board Support Pkg)  |  ← 每个硬件一个子目录，包含 HAL 层
|   Miniware / Pinecil /     |
|   Pinecilv2 / MHP30 / etc  |
+----------------------------+
|   Vendor (MCU SDK)         |  ← STM32 HAL / GD32VF103 SDK / BL702 SDK
+----------------------------+
```

### 运行流程

1. **上电** → `main()` → HAL 初始化 → FreeRTOS 启动
2. FreeRTOS 创建多个任务：
   - **PID 控制任务**（高优先级）：周期读取 ADC → 计算当前温度 → PID 算法 → 输出 PWM
   - **UI 任务**（中等优先级）：处理按键输入 → 更新状态机 → 刷新 OLED
   - **电源管理任务**：轮询 PD/QC 协商、检查电压
3. 状态机在 `OperatingModes.h` 中定义，包括 `HomeScreen → Soldering → Sleeping → Hibernating` 等

### 关键模块

- **PID 温度控制（power.cpp）**：David Hilton 设计的控制算法，使用指数移动平均（expMovingAverage）作为 I 项，利用热电耦的热质量常数来预测温度变化趋势。PWM 输出根据电源电压动态缩放（`P=V²/R`），确保在任何输入电压下功率可控。
- **热电耦模型（TipThermoModel）**：将 ADC 原始值转换为微伏，再根据热电耦的 Seebeck 系数换算为摄氏度。
- **USB PD 协商（USBPD.h）**：实现了 USB Power Delivery 协议的物理层和协议层，与充电器协商电压和电流。
- **BSP 硬件抽象**：每个硬件平台（MHP30、Miniware、Pinecil、Pinecilv2、Sequre）都有独立的 BSP 目录，实现了统一的 `BSP.h` 接口。

### 配置系统
所有设置项存储在 `Settings.h` 的枚举中，通过 `getSettingValue()` / `setSettingValue()` 访问，持久化到 Flash（`flash.c`）。

### 通信
- Pinecil V2 通过 BLE（`ble.c` / `ble_peripheral.c`）提供无线控制和固件升级。
- 所有设备通过 USB 固件升级（DFU / 串口）。

- **阅读目标**：理解嵌入式 PID 控制的实际工程实现，包括功率计算（`P=V²/R`）、PWM 动态缩放、快速/慢速 PWM 切换策略。

### ② `source/Core/Inc/Settings.h`
- **阅读目标**：了解嵌入式设备的配置系统设计——如何在只有几十 KB Flash 的空间里存储和读写配置项，枚举驱动的设置索引，以及版本控制（`SETTINGSVERSION`）策略。
- **为什么第二步**：理解了控制回路后，看它读取哪些参数。

### ③ `source/Core/Drivers/TipThermoModel.h`
- **阅读目标**：理解热电耦测温原理——ADC 原始值 → 微伏转换 → 温度查表的完整链路。

### ④ `source/Core/Threads/UI/logic/OperatingModes.h`
- **阅读目标**：理解嵌入式状态机设计——12 种操作模式（从 StartupLogo 到 ThermalRunaway）的定义和转换关系。
- **为什么最后读**：从底层看到上层，最终理解产品行为。

## 6. 核心模块/函数导读

### 6.1 `power.cpp` → `setTipX10Watts()`
这是"执行器"入口。接收以十毫瓦（x10 mW）为单位的功率目标值，调用 `X10WattsToPWM()` 将其转换为 PWM 占空比，然后调用 `setTipPWM()` 实际控制加热。同时它更新功率历史，用于 PID 的 I 项。

### 6.2 `power.cpp` → `availableW10()`
动态计算"在当前输入电压下，烙铁头理论上能吸收多少功率"。公式为 `P = V² / R`，还要乘以 PWM 死区时间的缩放因子。这是嵌入式中的"自适应功率上限"——同样的 50% PWM，接 9V 充电器和接 20V 充电器得到的实际功率完全不同。

### 6.3 `power.cpp` → `X10WattsToPWM()`
反向计算：给定目标功率，算出对应的 PWM 占空比。如果目标功率超过当前可用功率，则限幅为满 PWM。还会在这个过程中实际采样一次电压（`sample` 参数），保持了电压数据的新鲜。

### 6.4 `Settings.h` → `SettingsOptions` 枚举
定义了 40+ 个设置项（如 `SolderingTemp`、`SleepTemp`、`PowerLimit`、`USBPDMode`），每个都有格式化注释。版本号 `SETTINGSVERSION` 用于 Flash 存储的向前兼容——这在嵌入式产品中非常重要，因为固件升级后旧设置格式可能不兼容。

### 6.5 `OperatingModes.h` → `OperatingMode` 枚举 + 状态机函数指针
12 种操作模式，每种模式对应一个函数指针（如 `gui_solderingMode()`、`drawHomeScreen()`）。每个函数接收按键状态和上下文结构，返回下一个操作模式。这是"嵌入式状态机的标准实现"。

### 6.6 启动流程：`freertos.c` → FreeRTOS 任务创建 + `stm32f1xx_hal_msp.c` → HAL MSP 初始化
`freertos.c` 中包含 FreeRTOS 的启动配置，`stm32f1xx_hal_msp.c` 负责 MCU 外设的底层初始化（时钟、GPIO、ADC 定时器等）。目前 `freertos.c` 主要由 STM32 CubeMX 自动生成，但了解它的结构对理解系统全貌有帮助。

## 7. 软件工程师类比解释

### 7.1 PID 温度控制 → 后端 API 的限流器 / 熔断器
- 软件中的熔断器（Circuit Breaker）也是基于当前"负载"调整"输出"的反馈系统。
- **这个类比的局限**：硬件 PID 的采样和输出之间只有几毫秒的延迟，而且输出直接影响物理世界（加热），不像熔断器的状态转换可以接受秒级延迟。PID 还有 I 项积累，这在软件限流器中很少直接使用。

### 7.2 BSP 硬件抽象层 → 接口抽象（Interface / Adapter Pattern）
- BSP 定义了统一的 `BSP_Power.h`、`BSP_PD.h` 等接口，每个硬件平台提供各自的实现。和软件中的 `interface` + 工厂模式完全一样。
- **这个类比的局限**：软件接口替换通常只在编译时或运行时切换一次，而 BSP 的各个实现（MHP30 vs Pinecilv2）背后是完全不同的 MCU 架构（ARM Cortex-M3 vs RISC-V）和 SDK，编译时就需要选好目标平台，无法热切换。而且底层驱动需要处理寄存器位操作和中断优先级等软件开发者不常接触的细节。

### 7.3 FreeRTOS 任务调度 → 事件循环 / Actor 模型
- FreeRTOS 的每个任务类似于一个 Actor，有自己的栈和优先级，通过队列通信。和 Node.js 的 Event Loop 或 Erlang 的 Actor 模型相似。
- **这个类比的局限**：FreeRTOS 任务是抢占式优先级调度——高优先级任务可以随时中断低优先级任务。这与 JavaScript 的协作式事件循环完全不同。而且嵌入式任务的栈空间通常只有几百字节到几 KB，必须精确计算栈深度，溢出会导致不可预测的崩溃。

### 7.4 电量协商（USB PD）→ OAuth 2.0 授权码流程
- 设备（Sink）和充电器（Source）通过 CC 线协商电压电流，类似于 Client 和 Authorization Server 之间的握手协议。
- **这个类比的局限**：USB PD 协商在 5ms 内必须完成，否则充电器断开。OAuth 的握手可以在几百毫秒甚至几秒内完成。而且 USB PD 是物理层协议，CC 线上的电压变化直接编码数据，不像 HTTP 那样有应用层封装。

### 7.5 Flash 存储设置项 → 数据库 Schema 迁移
- `Settings.h` 中的 `SETTINGSVERSION` 类似于数据库 Schema 的版本号。固件升级时，代码根据版本号决定是否需要迁移旧格式。
- **这个类比的局限**：Flash 写入有擦除次数限制（通常 1 万～10 万次），而且每次写入前必须整块擦除。嵌入式设置更新通常采用"写新副本 → 校验 → 激活"策略，如果掉电，需要回滚。数据库迁移通常有事务保证，嵌入式 Flash 操作的"事务"需要手动实现。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：PID 控制器
- **一句话**：根据目标温度和当前温度的差值，通过比例（P）、积分（I）、微分（D）三项计算出加热功率的算法。
- **软件类比**：类似于后端 API 的自动扩缩容——负载高了就加实例，低了就减实例，但还考虑了累积偏差和变化趋势。
- **常见工程场景**：温度控制、电机速度控制、无人机姿态稳定。
- **坑**：P 项太大会震荡，I 项太大会"积分饱和"导致超调严重。很多嵌入式项目只使用 PI，不用 D，因为 D 对噪声太敏感。

### 卡片 2：指数移动平均（expMovingAverage）
- **一句话**：一种加权移动平均滤波，给最近的采样值更高的权重，用于平滑 ADC 噪声。
- **软件类比**：类似于 Prometheus 的 EWMA（指数加权移动平均）用于计算 CPU 负载。
- **常见工程场景**：ADC 去噪、功率历史平滑、RSSI 信号平滑。
- **坑**：权重因子（alpha）的选择需要实验；太小响应慢，太大毛刺多。

### 卡片 3：PWM（脉宽调制）
- **一句话**：通过快速开关电源，用导通时间占总周期的比例（占空比）来控制平均功率。
- **软件类比**：好比一个方波信号中，"干活时间 / 总时间"的比例。类似于 HTTP/2 的流优先级——给高优先级流分配更多的带宽时间片。
- **常见工程场景**：LED 调光、电机调速、加热功率控制。
- **坑**：PWM 频率必须高于传感器/负载的响应带宽，否则负载会看到闪烁或抖动。ADC 采样必须在 PWM 关闭期间进行，否则电源噪声耦合到测量值。

### 卡片 4：ADC（模数转换器）
- **一句话**：将模拟电压值转换为数字值的硬件外设。
- **软件类比**：相当于一个 JSON 解析器，把连续波动的电压"序列化"成有限的数字值。
- **常见工程场景**：温度测量、电池电压检测、触摸感应。
- **坑**：ADC 有分辨率（如 12 位 = 0～4095）、参考电压、采样率限制。噪声和量化误差是固有特性，需要硬件滤波和软件滤波配合。

### 卡片 5：热电耦（Thermocouple）
- **一句话**：两种不同金属接触时产生热电势，通过测量这个微小的电压来计算温度的传感器。
- **软件类比**：类似一个"只读 API"，输入的物理量（温度）通过固定的"传输函数"映射到电压值。
- **常见工程场景**：烙铁温度测量、烤箱温度、工业炉温。
- **坑**：热电耦输出是微伏级的，需要高增益运放放大。冷端补偿（CJC）必须在软件中做，忽略它会导致几十度的误差。

### 卡片 6：USB Power Delivery（USB PD）
- **一句话**：一种通过 USB Type-C 的 CC 线协商供电电压和电流的协议，最高支持 240W。
- **软件类比**：类似一个 CDN 的"协议协商"——客户端和服务器商量使用什么版本（HTTP/1.1 vs HTTP/2）、多大带宽。
- **常见工程场景**：快充、笔记本电脑供电、烙铁/电动工具供电。
- **坑**：PD 协商有时间限制（5ms），需要在中断或高优先级任务中处理。EPR（28V）模式有安全风险，IronOS 默认禁用。

### 卡片 7：FreeRTOS 任务与优先级
- **一句话**：嵌入式实时操作系统中的独立执行单元，每个任务有栈、优先级、状态，由调度器决定谁运行。
- **软件类比**：类似于操作系统线程，但在更小的 RAM 里（每个任务栈可能只有 256 字节）。
- **常见工程场景**：任何使用 RTOS 的嵌入式项目。
- **坑**：优先级反转、死锁、栈溢出是常见 bug。FreeRTOS 的 `configTICK_RATE_HZ` 决定了时间片粒度（通常 1000Hz = 1ms）。

### 卡片 8：BSP（Board Support Package）
- **一句话**：为特定硬件平台编写的一层底层驱动代码，对上提供统一 API，屏蔽不同 MCU 的差异。
- **软件类比**：类似于 ORM 的数据库适配器——统一 API 背后可以切换 MySQL 或 PostgreSQL。
- **常见工程场景**：产品支持多 MCU 平台、SDK 开发板支持包。
- **坑**：BSP 层容易写"薄"——如果 API 设计不能充分表达硬件能力，上层代码会变得臃肿。反之如果 API 太"厚"，移植到新平台成本很高。

### 卡片 9：Flash 存储与磨损均衡
- **一句话**：Flash 存储器在写入前必须先擦除，且每个扇区有有限的擦写次数（典型值 1 万次）。
- **软件类比**：类似一个只支持追加写入的日志文件，不能原地修改，而且磁盘每写 1 万次就报废。
- **常见工程场景**：设置项存储、固件升级（OTA）、日志记录。
- **坑**：频繁写入会快速耗尽 Flash 寿命。解决方案包括：只在值变化时写入、使用磨损均衡算法、备份/回滚策略。

### 卡片 10：热失控保护（Thermal Runaway）
- **一句话**：一种安全机制，监测温度变化速率。如果检测到加热时间过长但温度不上升（比如烙铁头被氧化、传感器脱落），立即关闭加热并报警。
- **软件类比**：类似于后端服务的心跳超时检测——如果服务 30 秒没有心跳，认为它挂了并触发熔断。
- **常见工程场景**：电热设备（烙铁、3D 打印机热床、加热板）。
- **坑**：误报会导致用户体验差——比如在大风环境中烙铁降温快，可能导致不该触发保护时触发了。需要合理的温度变化速率阈值。

## 9. 30～60 分钟动手任务

### 任务：阅读理解 IronOS 的 PID 控制回路，在你的开发机上用 Python 模拟

**目标**：在 Mac/Linux 终端运行一个 Python 脚本，模拟 IronOS 的 PID 温度控制逻辑，观察 P、I、D 参数对温度稳定性的影响。

**步骤**：
1. 阅读 `power.cpp` 中 `setTipX10Watts()` 和 `X10WattsToPWM()` 的代码（已在上面提供）。
2. 创建一个 Python 脚本 `pid_sim.py`，模拟：
   - 一个"虚拟烙铁头"：初始温度 25°C，热容量 1.69 J/°C（来自注释），加热功率随时间增加温度。
   - 一个 PID 控制器：设定点 350°C，用 `expMovingAverage` 作为 I 项。
   - ADC 采样噪声：给温度读数加 ±2°C 的随机噪声。
3. 运行模拟至少 1000 步，打印每步的目标温度、当前温度、PWM 输出。
4. 调整 PID 参数，观察振荡（P 太大）、响应慢（P 太小）、超调（I 太大）等效应。

**验收标准**：
- 模拟器能在 200 步内从 25°C 升温到 350±5°C 稳态。
- 温度稳定后波动不超过 ±3°C（包括 ADC 噪声）。
- 能通过修改参数制造明显的"温度振荡"和"超调"现象。

**无硬件替代方案**：全部在 Python 中完成，不需要任何硬件。

## 10. 3 道深度追问题

### 问题 1（时序/并发）
IronOS 中，ADC 采样和 PWM 输出在时序上是"互斥"的——ADC 采样期间 PWM 必须关闭（dead time）。`power.cpp` 中通过 `totalPWM` 和 `powerPWM` 两个常量来缩放可用功率。如果工程师错误地将 `totalPWM` 设置得比 `powerPWM` 大很多，会导致什么后果？从控制精度的角度分析。

### 问题 2（软件→嵌入式视角转换）
你是一个后端工程师，习惯了使用 Redis 缓存数据库查询结果来降低延迟。现在要用类似的"缓存"思维优化 IronOS 的温度读取——是否可以在温度稳定的情况下，跳过每次 ADC 采样直接使用上次的值来减少功耗？分别从嵌入式功耗、安全性和用户体验三个角度分析这个方案的可行性。

### 问题 3（工程判断）
IronOS 支持 Pinecil V1（GD32VF103 RISC-V）、Pinecil V2（BL702 RISC-V）、TS100（STM32F103 ARM Cortex-M3）三种完全不同 MCU 架构的硬件。从 BSP 设计的角度：如果把所有 10 种设备的 BSP 硬塞进一个固件二进制文件中（运行时检测硬件型号并加载对应 BSP），会面临哪些编译、存储空间、启动时间方面的工程挑战？为什么 IronOS 选择为每个设备单独编译一个固件镜像？

## 12. 工程点评

### 可维护性 ★★★★☆
- 代码结构清晰：核心逻辑在 `Core/`，平台适配在 `BSP/`，第三方库在 `Middlewares/`。
- 注释充分，尤其是解释了 PID 参数选择的工程原理（如"thermal mass divided by 20"）。
- 用枚举而非魔数管理设置项和操作模式，可读性强。
- 不足：部分 STM32 HAL 代码由 CubeMX 自动生成，大量代码重复在 Vendor 目录下。

### 可移植性 ★★★★☆
- 真正做到了跨 MCU 架构（ARM Cortex-M3、RISC-V GD32VF103、RISC-V BL702）。
- BSP 接口设计统一，移植到新设备只需实现少数几个函数（`BSP_Power.h`、`BSP_PD.h`、`BSP.h` 等）。
- 但 FreeRTOS 版本锁定在第三方中间件，升级 RTOS 版本需要较多适配工作。

### 可靠性 ★★★★★
- 热失控保护、欠压检测、多重安全检查。
- EPR（28V）模式默认禁用，需要用户主动开启，符合安全第一原则。
- 设置项有版本号管理，Flash 写入有校验。

### 可测试性 ★★★☆☆
- 嵌入式硬件固有的测试困难——没有硬件模拟器，温度控制回路只能在真实硬件上调试。
- 但核心算法（功率计算、PWM 转换等）逻辑纯，可以单元测试。

### 量产/运维 ★★★★☆
- CI/CD 完备，自动构建所有设备的固件镜像。
- 31 种语言翻译，国际化做得很完善。
- DFU 刷写流程有详细文档。
