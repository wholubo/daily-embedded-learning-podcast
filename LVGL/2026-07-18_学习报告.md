# 每日嵌入式学习：LVGL — 轻量通用嵌入式图形库

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**: https://github.com/lvgl/lvgl
- **一句话介绍**: LVGL（Light and Versatile Graphics Library）是一个纯 C 编写、零外部依赖的嵌入式 UI 图形库，让任何 MCU 或 MPU 都能跑出媲美智能手机的图形界面。

## 2. 为什么今天选它

LVGL 是目前嵌入式领域 **Star 最高（24k+）、社区最活跃**的开源图形库之一，最新 v9.5 已发布，提交频率每日更新。它覆盖了从 32KB RAM 的 Cortex-M0 小 MCU 到带 GPU 的四核 MPU 的全范围硬件，工程完整度极高：

- 有完整的文档站点、100+ 示例代码、论坛、商业支持。
- 架构设计极其模块化：显示驱动、输入设备、样式系统、布局引擎、渲染后端、动画引擎、事件系统、字体引擎、文件系统抽象 —— 每一层都可插拔。
- 适合软件工程师学习嵌入式的原因：UI 是软件工程师最熟悉的领域，但嵌入式 UI 有**显存、刷新率、内存碎片、局部渲染、触摸校准、色彩深度**等纯软件背景没见过的新约束。LVGL 把这些约束封装在清晰的抽象层里，是"从软件工程视角看懂嵌入式约束"的绝佳教材。

## 3. 项目解决的问题

LVGL 解决的核心问题是：**在资源极度受限的 MCU 上，如何提供一套完整的声明式 UI 框架**。

真实场景举例：
- 智能手表表盘（128×128 单色 OLED）
- 家电触摸面板（320×240 TFT，RGB565，16 位色）
- 工业 HMI 终端（带触屏，Flexbox 布局自适应）
- 医疗设备小屏菜单（旋钮编码器输入）

**与普通 Web/App 的差异**：
- 没有操作系统给予的 "无限堆内存" —— LVGL 需要自己在静态数组上管理内存池。
- 没有 GPU 辅助 —— 软件渲染器逐像素绘制矩形、圆弧、字体。
- 没有 DOM 的自动重排 —— 开发者手动控制 `lv_timer_handler()` 调用周期。
- 帧缓冲可能只有屏幕的 1/10 —— 必须分块渲染。
- 输入设备可能不是触摸屏，而是旋转编码器 —— LVGL 抽象了 5 种输入类型。

## 4. 技术架构分析

LVGL v9 的架构分为以下层次（基于公开文档和源码推断）：

### 核心层 (Core)
- **对象系统**：`lv_obj_t` 是所有 UI 元素的基类，采用 C 语言模拟的继承（通过 `lv_obj_class_t` 和 `class_p` 指针实现）。对象有父子树结构，类似 DOM Tree。
- **事件系统**：`lv_event_t` 实现观察者模式，支持事件冒泡和捕获。
- **定时器系统**：`lv_timer_t` 为内部心跳，驱动动画、刷新、输入轮询。用户在 `main loop` 中调用 `lv_timer_handler()`。
- **动画引擎**：支持属性插值、缓动函数、延迟、重复、自动反转。

### 显示与渲染 (Display & Draw)
- **显示抽象**：`lv_display_t` 管理屏幕尺寸、旋转、双缓冲、局部刷新回调。
- **渲染后端**：软件渲染器 (`sw`) 是默认实现，支持 SVG 光栅化 (`LV_USE_SVG`)、NEON/Helium SIMD 优化、以及硬件加速后端（PXP、G2D、DMA2D、OpenGLES、SDL、VG-Lite 等）。
- **绘制命令**：`lv_draw_rect` / `lv_draw_label` / `lv_draw_image` / `lv_draw_arc` / `lv_draw_vector` 等独立绘制 API。

### 输入设备 (Input Devices)
- 5 种类型：触摸/鼠标（POINTER）、键盘（KEYPAD）、硬件按钮（BUTTON）、旋转编码器（ENCODER）。
- 支持手势识别：捏合、滑动、旋转、双指滑动。
- 支持事件驱动模式（`LV_INDEV_MODE_EVENT`）减少功耗，替代周期轮询。

### 样式系统 (Style)
- ~100 个样式属性，支持继承、局部状态（press/checked/focused/disabled）、过渡动画。
- 类似 CSS 的 Flexbox 和 Grid 布局引擎。
- 样式以 `lv_style_t` 存储键值对，支持 `LV_STYLE_CONST_INIT` 编译时常量优化。

### 小部件 (Widgets)
- 30+ 内置小部件：按钮、标签、滑块、进度条、图表、键盘、仪表盘、圆弧、表格、文本区、图像、下拉列表、复选框、开关、弹窗等。
- 小部件通过对象类继承实现：`lv_obj_class_t` → `lv_button_class` → 实例。

### 扩展模块
- 字体引擎：内置字体渲染 + FreeType/MicroPython 后端。
- 文件系统抽象：支持 FatFS、POSIX、SD 卡、SPIFFS/LittleFS。
- 图像解码：PNG、BMP、JPG、GIF、SVG、TGA。
- 国际化：UTF-8 多语言支持，含 BiDi 阿拉伯语和 CJK。
- 3D 模型：通过 OpenGL 渲染 glTF 模型。
- 数据绑定：观察者模式自动同步 UI 与业务数据。

### 构建与配置
- 通过 `lv_conf.h` 静态配置 —— 所有特性在编译时如果未启用则零开销。
- 支持 CMake、Makefile、ESP-IDF component、Arduino library、PlatformIO、Zephyr module、Kconfig。
- 提供 `lv_conf_template.h` 作为配置入口模板。

## 5. 源码陪读：今天只读哪几个文件

建议按以下顺序阅读：

1. **`lv_conf_template.h`** — 入口配置，理解嵌入式 UI 的"零开销原则"：每个特性都有编译开关，不用的不占资源。
2. **`include/lvgl/lvgl.h`** — 主头文件，了解所有模块的命名空间和依赖关系。
3. **`src/display/lv_display.c` + `include/lvgl/display/lv_display.h`** — 显示驱动程序接口，理解帧缓冲、双缓冲、局部刷新、渲染模式。这是嵌入式 UI 和 Web UI 最大的差异点。
4. **`src/indev/lv_indev.c` + `include/lvgl/indev/lv_indev.h`** — 输入设备抽象。看如何用统一接口支持触摸、键盘、编码器。
5. **`src/core/lv_obj.c` + `include/lvgl/core/lv_obj.h`** — 对象系统核心。这是"在 C 语言里模拟 OOP 继承"的经典案例。

## 6. 核心模块/函数导读

### `lv_init()`
LVGL 的全局初始化入口。它做了四件事：（1）清零全局状态 `lv_global_t`；（2）初始化显示和输入设备的链表（`lv_ll_init`）；（3）注册内置布局（flex/grid）和主题；（4）初始化日志、内存、定时器子系统。必须在所有 LVGL 操作之前调用。

### `lv_display_create()` / `lv_display_set_buffers()`
创建显示实例并绑定帧缓冲。关键设计是**缓冲策略**：全帧缓冲（`LV_DISPLAY_RENDER_MODE_FULL`）vs 局部缓冲（`LV_DISPLAY_RENDER_MODE_PARTIAL`）。局部缓冲只分配 1/10 屏幕的内存，LVGL 自动分块渲染 → 调用 `flush_cb` 逐块发送给显示控制器。这对只有几 KB SRAM 的小 MCU 是必须的设计模式。

### `lv_timer_handler()`
这是 LVGL 的"主循环心跳"。在用户的 `while(1)` 超循环中周期调用。它驱动三类工作：（1）处理过期定时器（含动画帧更新）；（2）刷新脏区域（仅重绘上次调用后标记为"脏"的对象，类似 React 的 Virtual DOM diff）；（3）轮询输入设备。返回距离下次调度需要的毫秒数。

### `lv_obj_class_t` 和继承机制
LVGL 在 C 语言中用结构体指针模拟类继承。每个小部件类型（如按钮）定义一个 `lv_button_class` 实例，其中包含构造函数、析构函数、事件处理器、宽度/高度计算的虚函数指针。`lv_obj_t` 内部保存 `class_p` 指针，运行时通过 `LV_OBJ_GET_CLASS(obj)` 查找虚函数表。这是嵌入式 C 中常见的"手动 vtable"模式。

### `lv_style_t` 与属性解析
样式存储为键值对数组。每个属性有一个 ID（枚举值）和一个 `lv_style_value_t`（联合体，可以是数字、指针或颜色）。样式解析时按优先级搜索：本地样式 → 父级继承 → 默认主题。支持 `const` 编译时样式（`LV_STYLE_CONST_INIT`）减少 RAM 占用。属性变更会自动触发脏标记，在下一个 `lv_timer_handler` 周期重绘。

## 7. 软件工程师类比解释

### 类比 1：`lv_obj_t` 对象树 ≈ React 组件树
LVGL 的 UI 结构是一个父子嵌套的对象树，类似 React 的组件树。父对象移动/隐藏，子对象跟随。事件会沿树冒泡和捕获。
- **局限**：React 组件是 JavaScript 闭包，天然支持异步渲染和暂停；LVGL 的对象树用 C 手动维护，没有 GC，开发者必须显式 `lv_obj_delete()`，否则泄漏。

### 类比 2：脏区域重绘 ≈ Virtual DOM diff + 局部更新
LVGL 只重绘发生变化的对象区域（标记为"脏"），而不是整帧重绘。
- **局限**：Virtual DOM 是内存中的结构 diff，复杂度 O(n)；LVGL 的脏区域是矩形裁剪，复杂度 O(k)（k 为脏矩形数量），且受帧缓冲大小限制。如果脏区域碎片化严重，性能会急剧下降，因为没有批量合并优化。

### 类比 3：`lv_conf.h` 编译时配置 ≈ Webpack tree-shaking + feature flags
LVGL 每个模块都用 `#if LV_USE_XXX` 包裹，未启用的模块在编译阶段直接移除，零运行时开销。
- **局限**：Webpack tree-shaking 是构建时静态分析，依赖 ES module 的 import 图；LVGL 是纯宏控制，开发者手动开关，容易遗漏依赖导致链接错误。没有"自动检测未使用代码"的机制。

### 类比 4：`lv_display_set_flush_cb` 回调 ≈ 后端渲染 API 调用
Web 开发写 Canvas 2D 时，浏览器管理帧缓冲和垂直同步；LVGL 让开发者自己实现 flush 回调，把渲染好的像素数组通过 SPI/I2C/8080 并口发送给物理显示屏。
- **局限**：浏览器有双缓冲 + vsync + GPU 合成；LVGL 的 flush 是同步阻塞的，发送数据期间 CPU 不能做其他事。如果显示屏控制器不支持自动刷新，开发者还要自己实现 DMA + 中断来避免 CPU 空转。

### 类比 5：`lv_style_t` 属性继承 ≈ CSS 级联
样式可以设置给对象本身（类似 inline style），也可以设置给主题或屏幕（类似全局 CSS），子对象自动继承可继承属性（类似 `color` / `font-size` 继承）。
- **局限**：CSS 级联有完善的优先级计算（特异性 / `!important`）；LVGL 的优先级是硬编码的（本地 > 主题 > 父级继承），不支持用户自定义优先级规则。

## 8. 今日 10 个嵌入式概念卡片

### 1. 帧缓冲 (Frame Buffer)
- **一句话**：内存中一块区域，存放一张屏幕完整像素数据。
- **类比**：Canvas 元素的 pixels 数组。
- **场景**：`lv_display_set_buffers` 分配的数组。
- **坑**：16 位色深 RGB565 下，320×240 屏幕需要 150KB —— 很多 MCU 总共才 64KB RAM。解法：局部缓冲 + 分块刷新。

### 2. 局部刷新 (Partial Rendering)
- **一句话**：只重绘屏幕上变化的部分，不整帧刷新。
- **类比**：React 只更新变化 DOM 节点，不重新渲染整个页面。
- **场景**：LVGL 默认模式，缓冲大小只需 1/10 屏幕。
- **坑**：如果一个全屏动画（如弹出键盘）导致大面积变化，局部刷新的分块绘制反而比全帧缓冲慢，因为每次 flush 需要发送多个小矩形给屏幕控制器。

### 3. 双缓冲 (Double Buffer)
- **一句话**：两个帧缓冲，一个用于显示，一个用于后台绘制，绘制完成后交换。
- **类比**：数据库的 A/B 表切换。
- **场景**：`lv_display_set_buffers` 传入两个 buffer，避免撕裂。
- **坑**：双缓冲需要两倍 RAM，很多小 MCU 负担不起。LVGL 允许单缓冲 + 局部刷新做折中。

### 4. 色彩深度 (Color Depth)
- **一句话**：每个像素用多少位表示颜色。
- **类比**：图片的 PNG-8 vs PNG-24 vs JPEG。
- **场景**：`LV_COLOR_DEPTH` 配置为 16（RGB565）或 8（L8）。
- **坑**：字体抗锯齿在 8 位色深下几乎不可用；jpg 图片需要 24/32 位色。选择色深直接影响所有内存计算。

### 5. 脏区域标记 (Dirty Area / Invalidation)
- **一句话**：对象属性变化时，标记该对象区域为"需要重绘"。
- **类比**：Redis 的键变更通知。
- **场景**：`lv_obj_invalidate()` 被 `lv_obj_set_size()` 隐式调用。
- **坑**：频繁修改大量小对象会导致脏区域碎片化，增加合并和 flush 开销。

### 6. 硬件抽象层 (HAL，Hardware Abstraction Layer)
- **一句话**：在驱动和应用之间定义统一的接口，方便移植。
- **类比**：JDBC 让 Java 代码不依赖具体数据库。
- **场景**：LVGL 的 `lv_display_set_flush_cb` 和 `lv_indev_set_read_cb` 就是 HAL 回调。
- **坑**：回调函数是同步的，如果底层 SPI 或 I2C 传输慢（几毫秒），会阻塞整个 UI 循环。需要用 DMA 或 RTOS 任务来解耦。

### 7. 编译时零开销原则 (Zero-Cost Abstractions)
- **一句话**：没用到的功能在编译阶段完全移除，不产生运行时开销。
- **类比**：Rust 的 trait 单态化。
- **场景**：`lv_conf.h` 中的 `LV_USE_PNG 0` 让 PNG 解码器代码不被编译。
- **坑**：配置项之间可能有隐式依赖（如 `LV_USE_FREETYPE` 依赖 `LV_USE_STDLIB_MALLOC`），手册不熟时链接错误排查困难。

### 8. 输入设备抽象 (Input Device Abstraction)
- **一句话**：用统一的 `lv_indev_read_cb` 回调封装触摸屏、键盘、编码器。
- **类比**：Linux 的 input 子系统（`evdev`）。
- **场景**：旋钮编码器 → `enc_diff` 字段；触摸 → `point` 字段。
- **坑**：触摸屏需要校准（将 ADC 原始值映射到屏幕坐标），LVGL 不自带校准算法，开发者需自己实现或使用 `lv_lib_input`。

### 9. 手动内存池 (Static Memory Pool)
- **一句话**：LVGL 在静态数组上管理自己的堆，不用系统 malloc。
- **类比**：Node.js 的 Buffer pool。
- **场景**：`LV_MEM_SIZE` 定义内存池大小，LVGL 内置分配器处理所有 lv_obj、lv_style 等分配。
- **坑**：内置分配器是简单的 freelist，**不防碎片**。长时间运行后反复分配/释放可能导致无法分配大块连续内存。

### 10. 渲染后端可插拔 (Pluggable Render Engine)
- **一句话**：LVGL 默认用 CPU 逐像素软件渲染，但可以切换到 GPU 加速后端。
- **类比**：Angular 的 zone.js 替换。
- **场景**：NXP PXP → `LV_USE_DRAW_PXP`；OpenGLES → `LV_USE_DRAW_OPENGLES`。
- **坑**：不同后端对绘制命令的支持程度不同（如 OpenGLES 不支持某些 blend mode）。切换后端后 UI 表现可能不一致。

## 9. 30～60 分钟动手任务

### 任务：在 PC 上运行 LVGL 模拟器，添加一个带动态数据的仪表盘

**目标**：不买任何硬件，在 PC 上体验 LVGL 的开发流程，理解显示驱动、定时器、脏区域刷新的实际运作。

**步骤**：
1. **克隆 LVGL 和 PC 模拟器**（约 5 分钟）：
   ```bash
   git clone https://github.com/lvgl/lv_port_pc_vscode.git
   cd lv_port_pc_vscode
   git submodule update --init --recursive
   ```
2. **构建和运行**（约 10 分钟）：
   ```bash
   mkdir build && cd build
   cmake ..
   make -j4
   ./main
   ```
   应该看到一个带 LVGL demo 的窗口。
3. **阅读 `main/src/main.c`**，找到 `lv_timer_handler()` 的调用位置，理解主循环结构（约 5 分钟）。
4. **添加一个实时仪表盘**（约 20 分钟）：
   - 在 `main.c` 中 `lv_demo_widgets()` 之前，添加代码：
   ```c
   lv_obj_t * meter = lv_meter_create(lv_scr_act());
   lv_obj_set_size(meter, 200, 200);
   lv_obj_center(meter);
   
   /* 添加刻度 */
   lv_meter_scale_t * scale = lv_meter_add_scale(meter);
   lv_meter_set_scale_range(meter, scale, 0, 100, 270, 135);
   lv_meter_set_scale_ticks(meter, scale, 10, 2, 10, lv_palette_main(LV_PALETTE_GREY));
   
   /* 添加指针 */
   lv_meter_indicator_t * indic = lv_meter_add_needle_line(meter, scale, 4, lv_palette_main(LV_PALETTE_RED), -10);
   
   /* 定时更新 */
   lv_anim_t a;
   lv_anim_init(&a);
   lv_anim_set_var(&a, meter);
   lv_anim_set_exec_cb(&a, [](void * var, int32_t v) {
       // 更新指针值
   });
   lv_anim_set_values(&a, 0, 100);
   lv_anim_set_time(&a, 3000);
   lv_anim_set_repeat_count(&a, LV_ANIM_REPEAT_INFINITE);
   lv_anim_start(&a);
   ```
5. **观察**：打开终端 top/htop，观察窗口不动时 CPU 占用 ；窗口有动画时 CPU 占用变化。这就是嵌入式 UI 设计必须关注"空闲功耗"的原因。

**验收标准**：
- 窗口显示一个 200×200 的表盘，指针在 0～100 间循环摆动。
- 理解 `lv_timer_handler` 的调度机制。
- 能指出上述代码中哪些内存是静态分配的，哪些是堆分配的。

**无硬件替代方案**：模拟器方案就是无硬件的 —— 你用 PC 的 SDL 窗口模拟嵌入式显示。

## 10. 3 道深度追问题

1. **时序/并发问题**：假设你在一块 320×240 RGB565 屏幕上用单缓冲+局部刷新模式跑 LVGL，`flush_cb` 通过 SPI 以 20MHz 发送像素数据（约 4MB/s）。动画帧率目标 30fps。问：每次渲染周期内，CPU 花在 flush 上的时间占比是多少？如果此时还有一个 100Hz 的传感器中断，如何保证 UI 不卡顿？

2. **软件→嵌入式视角转换**：你习惯在后端服务里用 `malloc` 随意分配对象，GC 或 RAII 帮你回收。现在 LVGL 要求你在编译时指定 `LV_MEM_SIZE`，并在一个静态数组中手动管理对象生命周期。如果 `lv_mem_alloc` 返回 NULL，你的 UI 会直接崩溃还是优雅降级？请设计一个内存不足时的容错策略。

3. **工程判断**：LVGL v9 引入了新的 `include/lvgl/` 目录结构，废弃了旧的 `src/` 直接包含方式，同时提供了"事件驱动输入模式"（`LV_INDEV_MODE_EVENT`）以减少轮询功耗。对于一款使用 CR2032 纽扣电池供电的智能手表产品，你选择继续使用旧风格的 `src/` 包含方式以保持与现有代码库的兼容性，还是迁移到新结构并启用事件驱动模式？说出你的决策依据。

## 11. 今日复盘问题

如果你的后端服务每秒处理 1 万请求，你不太担心内存碎片 —— 因为每个请求在短时间内分配和释放，操作系统会帮你整理。但在 LVGL 这种长期运行、手动内存池、无 MMU 的嵌入式环境中，`malloc/free` 的碎片效应是真实且不可忽视的。**你今天看到的 LVGL 代码中，哪些设计（或缺少的设计）是在主动对抗内存碎片？如果有，它们有效吗？**

## 12. 工程点评

- **可维护性**：★★★★★ — 代码注释完善（Doxygen 风格）、模块划分清晰、v9 重构了 include 路径使依赖关系更显式。文档从 API 参考到移植指南到 100+ 运行示例覆盖全面。
- **可移植性**：★★★★★ — 纯 C89/C99，无平台依赖，HAL 回调设计让底层驱动完全解耦。已移植到 50+ 平台/RTOS。
- **可靠性**：★★★★☆ — 内存池有 sentinel 防越界断言，大量 `LV_ASSERT` 运行时检查。但内置内存分配器不防碎片，长期运行需要外部监控或定期重启。官方不提供看门狗集成指南。
- **可测试性**：★★★☆☆ — 提供了 `lv_test` 子系统和截图对比工具，但单元测试覆盖率不高，没有 CI 中运行的自动化 UI 测试套件。社区贡献的 widget 测试分散在各移植仓库中。
- **量产/运维**：★★★★☆ — 支持 Kconfig 和编译时裁剪，适合量产前的 feature 剥离。但有商业闭源部分（LVGL Pro），社区版缺少 WYSIWYG 编辑器，复杂的 UI 全靠手写 C 代码。
