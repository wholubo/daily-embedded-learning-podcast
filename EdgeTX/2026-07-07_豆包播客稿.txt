大家好，欢迎收听今天的嵌入式学习分享。今天我们要聊的项目是 EdgeTX，一套为航模遥控器打造的尖端开源固件。

EdgeTX 是从 OpenTX 分叉出来的社区驱动项目，它的目标是给航模和无人机遥控器提供一个功能强大、可定制、开源的固件系统。你可以把它理解成遥控器的操作系统。这个项目在 GitHub 上有超过两千三百颗星、五百五十多个分叉，社区非常活跃，几乎每天都有代码提交。

为什么今天选它来学习呢？因为 EdgeTX 几乎覆盖了嵌入式系统的所有典型模块。它有 ADC 摇杆采样与滤波、混频器的实时计算、支持十几种协议的脉冲输出模块、音频驱动、蓝牙、GPS、遥测处理、SD 卡文件系统、Bootloader、USB 多模式切换，甚至还内嵌了 Lua 脚本引擎。对于一个软件工程师转型嵌入式来说，这是一个绝佳的完整案例。

先说说 EdgeTX 解决的实际问题。一个真实的遥控器，比如 Radiomaster TX16S 或者 FlySky PL18，里面跑的就是 EdgeTX 固件。你拨动摇杆、按动开关，EdgeTX 读取这些物理输入，经过混频器逻辑运算，这些运算包括曲线修正、混控组合、逻辑开关判断和全局变量处理，最终通过高频头输出 PPM 或者 SBUS 或者 CRSF 等协议信号给无人机或者航模的接收机。和普通软件项目相比，EdgeTX 面临很多嵌入式特有的约束。首先是实时性要求，混频器必须在固定时间窗口内完成计算，通常是四到九毫秒，超时就会导致遥控信号抖动甚至失控。其次是资源约束，目标芯片通常是 STM32F2 或者 F4 或者 F7 或者 H7 系列，Flash 从 512KB 到 2MB，RAM 从 128KB 到 1MB，要在这么有限的资源里跑图形界面、Lua 脚本、文件系统、多协议支持和音频播放。第三是安全可靠性，混频器的每个数学运算必须精确可预测，模型文件损坏可能导致飞行事故，所以存储系统都带有 CRC 校验和备份机制。此外还有功耗管理和多样化的人机交互适配。

接下来看看技术架构。EdgeTX 的代码主要放在 radio slash src 目录下，里面按功能分成十几个子目录。硬件抽象层 HAL 定义了 ADC、USB、音频、存储和看门狗等驱动接口。目标板支持目录下有十五个不同的硬件平台，包括 Taranis、Horus、PL18、TX16SMK3 等等。图形用户界面有彩色和单色两套独立的 UI 引擎。任务管理目录封装了 FreeRTOS 的多任务调度。存储子系统负责 SD 卡和 Flash 的读写。脉冲输出目录支持 PPM、SBUS、CRSF、DSM2、GHOST 等多种协议。遥测目录解析接收机发回的传感器数据。Lua 目录内嵌了 Lua 5.2 脚本引擎。驱动目录有 SPI、I2C、触摸屏和音频等外设驱动。翻译目录支持二十一种语言界面。Bootloader 负责 DFU 升级。

运行流程大致是这样的。系统上电后先进入 Bootloader 或者直接启动固件，然后 boardInit 函数初始化时钟、GPIO、ADC、SD 卡和 USB。接着 tasksStart 创建三个 FreeRTOS 任务。Mixer Task 是最高优先级，周期九毫秒，负责硬实时混频计算。Audio Task 优先级次之，负责异步音频播放。Menus Task 优先级最低，周期五十毫秒，处理图形界面和用户交互。主循环里调用 perMain 函数，依次完成读取摇杆值、运行混频器、输出脉冲、更新显示和检查按键。检测到电源键长按时就进入关机流程。

源码陪读环节，我推荐按这个顺序读五个文件。第一个是 radio slash src 下的 main.cpp，看整个固件的启动流程和任务创建。第二个是 edgetx.cpp，看核心全局变量的定义和主循环体。第三个是 tasks.cpp 加上 tasks.h，理解 FreeRTOS 多任务架构，重点是栈大小分配和优先级策略。第四个是 mixer.cpp，看混频器的核心数据结构和计算流程，从摇杆输入到最终通道输出的整条数据管道。第五个是 hal 目录下的 adc_driver.h，看硬件抽象接口的设计模式。

核心模块方面，edgeTxInit 是所有子系统的初始化入口，它在 menusTask 被 RTOS 调度后第一个执行。perMain 是五十毫秒周期的主循环函数，负责读取 ADC、运行混频器、触发脉冲输出、更新 LCD、处理按键和播放音频。混频器的核心管线从摇杆 ADC 值开始，经过校准、混控计算、限幅，最终输出到物理通道。tasksStart 的优先级设计很有意思，Mixer 任务用最高优先级但栈只有四百字节，因为混频器是纯计算没有递归也不分配堆内存。而 Menus 任务因为要处理图形界面，栈需要两到八 KB。至于 SD 卡的文件系统操作，基于 FatFs 库，代码里特意预留了 50MB 的空闲空间缓冲以防止写满崩溃。

为了帮助软件工程师理解这些嵌入式概念，我们来做几个类比。混频器就像 Web 后端的中间件管道，摇杆原始输入经过一系列处理步骤最终输出，但局限是混频器的每一步都有微秒级的时间预算，不能异步等待。FreeRTOS 的任务调度类似于多线程池，但局限是运行在单核 MCU 上且栈是静态分配的。HAL 驱动接口类似于依赖注入模式，不同硬件提供不同实现，但局限是函数指针表在编译时固定，没有运行时多态开销。模型数据存储类似于数据库 ORM，把结构化数据序列化到持久化存储，但局限是没有 SQL 查询或事务支持。多协议高频头模块驱动类似于微服务的 API 网关，把不同接收机协议统一成一个接口，但局限是协议切换通常需要重启模块。

接下来是十个嵌入式概念卡片。第一，ADC 采样与滤波。模数转换器将摇杆的模拟电压转为数字量，EdgeTX 使用 IIR 低通滤波消除机械抖动。滤波太强会引入延迟，太弱则通道值抖动。第二，FreeRTOS 任务栈静态分配。每个任务的栈在编译时就固定了，不能动态增长，栈溢出会导致静默崩溃而且很难调试。第三，硬实时和软实时的区别。Mixer 任务必须在九毫秒内完成计算，错过截止期会导致信号失效，这是硬实时约束。在硬实时任务中调用内存分配或者文件 IO 会破坏时序。第四，PWM 和 PPM 脉冲协议。PWM 通过方波高电平时长编码控制信号，PPM 在一个帧里串行编码多个通道。第五，Bootloader 和 DFU。Bootloader 在开机时检查固件更新请求，通过 USB DFU 模式让用户拖放文件烧录。如果 Bootloader 本身损坏，MCU 就变砖了。第六，CRC 校验与存储可靠性。EdgeTX 在模型文件中嵌入 CRC 校验，加载时验证完整性，损坏时自动回退。第七，Lua 脚本引擎嵌入。EdgeTX 内嵌 Lua 5.2，用户可以运行自定义脚本，但脚本执行有时间配额限制不能阻塞 Mixer 任务。第八，FatFs 文件系统。用于管理 SD 卡上的 FAT 文件系统，但突然断电时容易损坏，长时间写操作也可能阻塞其他任务。第九，遥测的反向数据通道。接收机把传感器数据发回遥控器，EdgeTX 解析后在屏幕上显示并触发告警，帧格式因协议不同而完全不同。第十，CMake Superbuild 跨平台构建。一套配置同时产出宿主机模拟器和 ARM 交叉编译两种目标，交叉编译链稍有错误就会编译失败。

今天推荐一个三十分钟到一小时的动手任务。克隆 EdgeTX 仓库，构建 Native 模拟器目标，也就是在电脑上运行的模拟版本。然后运行模拟器，你可以在里面添加混控条，观察通道数值随摇杆输入的变化。同时打开 mixer.cpp 源码，找到 processMixer 函数，对照模拟器里的数值理解计算逻辑。这个模拟器本身就是无硬件的替代方案，在 Ubuntu 或者 WSL 上直接运行。

最后是三道深度追问。第一题，Mixer 任务栈只有四百字节而 Menus 任务需要两到八 KB，为什么差异这么大？如果混控逻辑意外引入了深度递归，系统会怎样？第二题，在 Web 后端中 API 超时你可以加异步队列或者扩容，但在 EdgeTX 中混频器计算超过了九毫秒周期，你能用什么嵌入式特有的方法排查和解决？第三题，EdgeTX 的彩色和单色 GUI 是两套完全不同的实现，为什么不像 Web 前端那样用响应式设计一套代码适配所有屏幕？从资源约束的角度分析根本原因。

今天的复盘问题是这样：想象你是 EdgeTX 的架构师，现在需要在 Mixer 和 Menus 两个任务之间增加一个低优先级的数据分析任务，每天统计摇杆使用频率并写 SD 日志。你不能改变现有任务的优先级和栈大小。你会怎么设计这个新任务呢？

从工程角度点评，EdgeTX 的可维护性不错，代码风格简洁实用，模块化程度较好。HAL 层的设计可移植性做得很到位，新目标板只需要实现驱动接口即可接入。可靠性方面有关键数据校验和异常重启检测，但没有完善的单元测试框架，这是个风险。好在有 Native 模拟器大大降低了测试门槛。量产和运维链路很成熟，从 CI 自动构建到 Buddy 桌面工具刷机到 SD 卡内容包独立管理，Bootloader 支持 USB DFU 拖放升级，不用拆机就能更新固件。
