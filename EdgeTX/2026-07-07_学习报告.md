# 每日嵌入式学习：EdgeTX

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/EdgeTX/edgetx
- **一句话介绍**：EdgeTX 是为航模/遥控器（RC Radio）打造的尖端开源固件，从 OpenTX 分叉而来，提供了社区驱动、创新快速迭代的遥控器操作系统。

## 2. 为什么今天选它

- **活跃度**：2,345+ Stars、551 Forks、近 800 个活跃 Issue，每日均有代码提交，社区极度活跃（Discord、Facebook、GitHub Discussions）。
- **工程完整度**：完整的企业级嵌入式项目——多平台 CMake Superbuild 构建系统、FreeRTOS 多任务调度、完善的 HAL 驱动抽象、存储子系统（SD+Flash）、USB 协议栈、LUA 脚本引擎、多语言国际化和 UI 框架。
- **学习价值**：适合软件工程师转型嵌入式的重要原因——它几乎覆盖了嵌入式系统的所有典型模块：ADC 摇杆采样与滤波、混频器（Mixer）实时计算、PWM/PPM/SBUS/CRSF 等多协议脉冲输出、音频驱动、蓝牙、GPS、遥测处理、SD 卡文件系统、Bootloader、USB 多模式（Joystick/存储/串口/DFU）。
- **与历史项目不重复**：本项目与之前讲过的 OpenTX、QMK、Tasmota、Zephyr 等同源项目不同——EdgeTX 是独立分叉，专注航模遥控器场景，工程选型和架构有其独特性。

## 3. 项目解决的问题

EdgeTX 解决的核心问题：**为航模/无人机遥控器提供功能强大、可定制、开源的操作系统固件。**

### 真实产品场景

一个实体遥控器（比如 Radiomaster TX16S、FlySky PL18 或 Jumper T-Pro）的内部运行着 EdgeTX 固件。操作者通过摇杆、开关、旋钮输入控制意图，EdgeTX 读取这些物理输入，经过混频器逻辑运算（支持曲线、混控、逻辑开关、全局变量），最终通过高频头（RF Module）输出 PPM/SBUS/CRSF 等协议信号给无人机/航模的接收机。

### 与普通软件项目相比的额外约束

- **实时性**：混频器必须在固定时间窗口内完成计算（通常 4~9ms），否则会导致遥控信号抖动甚至失控。
- **资源约束**：目标 MCU 通常是 STM32F2/F4/F7/H7 系列，Flash 512KB~2MB，RAM 128KB~1MB，需要在有限资源内跑 GUI、Lua、文件系统、多协议、音频。
- **安全可靠性**：遥控器数据不能出错——混频器的每个数学运算必须可预测；模型文件损坏可能导致飞行事故，因此 Flash 存储和 SD 卡操作都有 CRC 校验和备份机制。
- **功耗管理**：电池供电设备，需要精细的休眠/唤醒策略。
- **人机交互多样性**：从黑白单色 LCD 到 480x272 彩色触控屏、从按键到触控 + 滚轮，一个固件要适配多种硬件变体。

## 4. 技术架构分析

*注：以下分析基于公开源代码和文件结构推断。*

### 整体模块划分

```
radio/src/
├── hal/           # 硬件抽象层（ADC、USB、Audio、Storage、Watchdog等）
├── targets/       # 板级支持包（taranis、horus、pl18、tx16smk3 等 15 个目标）
├── gui/           # 用户界面（彩色/单色两套 UI 引擎）
├── tasks/         # FreeRTOS 任务封装（mixer_task、menus）
├── os/            # 操作系统抽象（task、timer、sleep、async）
├── storage/       # 存储子系统（SD卡 + Flash）
├── io/            # IO 模块（串口、蓝牙、GPS、SBUS）
├── pulses/        # 脉冲输出协议（PPM、SBUS、CRSF、DSM2、GHOST 等）
├── telemetry/     # 遥测协议解析
├── lua/           # LUA 脚本引擎（自定义混控、音频、显示屏）
├── drivers/       # 外设驱动（SPI、I2C、Touch、Audio 等）
├── translations/  # 多语言界面（支持 21 种语言）
├── bootloader/    # DFU/ST 引导加载程序
└── boards/        # 板级配置文件
```

### 运行流程

1. **上电** → 进入 bootloader 或直接启动固件
2. **boardInit()** → 初始化时钟、GPIO、ADC、SD 卡、USB
3. **tasksStart()** → 创建三个 FreeRTOS 任务：
   - **Mixer Task**（最高优先级 4）：9ms 周期硬实时混频循环
   - **Menus Task**（优先级 1）：50ms 周期处理 GUI 和用户交互
   - **Audio Task**（优先级 2）：异步音频播放
4. **主循环（menusTask）**：调用 perMain() → 读取摇杆值 → 混频器计算 → 输出脉冲 → 更新显示 → 检查按键
5. **关机检测**：电源按键长按检测 → drawSleepBitmap() → boardOff()

### 硬件抽象层设计

HAL 通过函数指针结构体（如 `etx_hal_adc_driver_t`）定义驱动接口，不同目标板提供各自的实现。这种设计类似 Linux 内核的 platform driver：

```c
struct etx_hal_adc_driver_t {
    const etx_hal_adc_inputs_t* inputs;
    bool (*init)();
    // ...
};
```

### 配置系统

模型和遥控器配置以二进制结构体（`RadioData`、`ModelData`）存储在 Flash 或 SD 卡上，通过 `storage/` 模块完成读写，支持 CRC 校验和版本迁移。

## 6. 核心模块/函数导读

### 6.1 `edgeTxInit()`（edgetx.cpp）
- **角色**：所有子系统初始化入口。
- **调用链**：boardInit() → 各 HAL 驱动 init() → SD 卡挂载 → 模型数据加载 → 混频器初始化 → 定时器启动。
- **与启动流程的关系**：它是 `menusTask()` 被 RTOS 调度后第一个调用的函数，决定了系统的初始状态。

### 6.2 `perMain()`（edgetx.cpp）
- **角色**：50ms 周期的菜单/主循环函数。
- **功能**：读取最新 ADC 值 → 运行混频器 → 触发对应脉冲输出 → 更新 LCD 显示 → 处理按键事件 → 检查遥测数据 → 播放音频队列。
- **与 RTOS 的关系**：由 `menusTask` 每 50ms 调度一次，Mixer 内部又通过 `mixerTask` 以更高速率执行。

### 6.3 混频器核心管线（mixer.cpp）
- **数据流**：
  `摇杆 ADC 值 → calibratedAnalogs[] → anas[]（输入）→ act[]（混控计算中间值）→ ex_chans[]（限幅前）→ chans[]（限幅后）→ channelOutputs[]（最终输出）`
- **关键函数**：`mixerTask`（高优先级任务入口）、`processMixer`（单次混控计算）、`evalInputs`（输入值评估）。
- **与构建系统的关系**：混频器配置通过固件编译时选项（`HELI`、`FLIGHT_MODES`、`CURVES`、`GVARS` 等 CMake 选项）裁剪功能。

### 6.4 `tasksStart()` 与任务优先级分配合约
- **Mixer Task**：优先级 4（最高），栈 400 字节（debug 512），9ms 硬周期。
- **Audio Task**：优先级 2（次高），栈 400 字节，异步播放 WAV/PCM。
- **Menus Task**：优先级 1，栈 2000 字节（彩屏 8KB），50ms 周期。
- **关键设计**：Mixer 栈只有 400 字节——这是嵌入式极简栈分配的经典案例。为什么这么小？因为混频器是纯计算，没有递归，不分配堆内存，所有数据都是全局静态数组。

### 6.5 `sdCheckAndCreateDirectory()`（sdcard.cpp）
- **角色**：SD 卡文件系统操作封装，基于 FatFs。
- **功能**：检查/创建目录、文件存在性判断、扩展名匹配搜索（.bmp / .jpg / .gif 等）。
- **与持久化的关系**：模型文件、位图、声音、日志、Lua 脚本全部依赖 SD 卡存储，SD 卡故障是遥控器最常见的用户侧问题，因此代码中特别保留了 50MB 的空间缓冲以防止写满崩溃。

## 7. 软件工程师类比解释

### 7.1 混频器（Mixer）→ Web 后端的中间件管道
混频器把摇杆原始输入经过一系列处理步骤（曲线、混控、限幅、逻辑开关）输出为最终通道值。这和 Express.js/Koa 中请求经过一系列中间件处理后生成响应非常相似。
*局限：混频器的每一步有严格的时间预算（微秒级），不能异步等待或 IO 阻塞；Web 中间件通常不关心微秒级延迟。*

### 7.2 FreeRTOS 任务调度 → 操作系统线程池 + 消息队列
EdgeTX 的三个任务（Mixer、Audio、Menus）类似于你启动的三个 worker 线程池，各自处理不同优先级的工作。
*局限：FreeRTOS 在单核 MCU 上运行，没有抢占式多核调度；线程池通常假设有足够的 CPU 核来处理并发。另外 RTOS 任务栈是静态分配的，不能在运行时动态扩展。*

### 7.3 HAL 驱动接口（adc_driver_t）→ 依赖注入接口
`etx_hal_adc_driver_t` 结构体包含函数指针，不同硬件板提供不同实现——这相当于后端中的依赖注入（DI）模式。测试时也可以注入模拟实现。
*局限：嵌入式 HAL 通常没有运行时多态的开销（无 vtable），函数指针表在编译时或链接时固定；后端 DI 容器通常在运行时解析依赖。*

### 7.4 模型存储（RadioData/ModelData）→ 数据库 ORM 模型
遥控器和模型的配置数据以二进制结构体序列化到 Flash/SD 卡，带 CRC 校验和版本字段。类似你用 Protobuf 或 FlatBuffers 定义数据模型并持久化。
*局限：嵌入式存储没有 SQL 查询、索引或事务（除了 FatFs 的基本文件操作）；版本迁移必须向前兼容且不能依赖网络升级。*

### 7.5 LM4（多协议高频头模块驱动）→ 微服务 API 网关
EdgeTX 通过 LM4（Multi-Protocol Module）支持数十种不同品牌的接收机协议（FRSKY、FlySky、DSM、Crossfire、GHOST 等），抽象出一个统一的脉冲输出接口。类似 API 网关把不同后端服务统一成一套客户端 API。
*局限：协议切换通常需要重启模块或重新绑定；微服务网关可以热切换而不会影响物理连接。*

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：ADC 采样与滤波
- **一句话**：模数转换器（ADC）将摇杆的模拟电压值转换为数字量，EdgeTX 使用 IIR 低通滤波（JITTER_ALPHA）消除机械抖动。
- **类比**：类似前端表单输入的 debounce（防抖），只不过是在硬件层面连续处理。
- **场景**：几乎所有带摇杆/电位计/滑块的嵌入式产品。
- **坑**：滤波太强会引入可感知的延迟（滞后效应），滤波太弱则通道值抖动、飞行手感差。

### 卡片 2：FreeRTOS 任务栈静态分配
- **一句话**：在嵌入式 RTOS 中，每个任务的栈在编译时静态指定大小（例如 Mixer 仅 400 字节），不能在运行中动态增长。
- **类比**：在 Go 中 goroutine 栈可以动态扩容；在嵌入式中栈溢出会导致静默崩溃。
- **场景**：任何跑 RTOS 的 MCU 项目。
- **坑**：用递归或大局部数组很容易导致栈溢出。调试栈溢出比段错误更难——通常表现为随机死机。

### 卡片 3：硬实时 vs 软实时
- **一句话**：Mixer 任务必须在 9ms 内完成所有混控计算，错过截止期可能导致遥控信号失效——这是硬实时约束。
- **类比**：视频流解码需要每帧 16ms 完成是软实时；飞行控制中的 PWM 信号周期是硬实时。
- **场景**：电机控制、导航信号处理、音频播放。
- **坑**：在硬实时任务中调用 malloc/new、printf、文件 IO 等阻塞操作会破坏时序。

### 卡片 4：PWM/PPM 脉冲协议
- **一句话**：PWM（脉宽调制）通过方波高电平宽度编码控制信号，PPM 则在单个帧内编码多个通道。
- **类比**：类似 HTTP/1.1 的管线化——PPM 在一个"请求"中串行发送多个"通道"数据。
- **场景**：舵机控制、ESC 调速、LED 调光。
- **坑**：PWM 分辨率受定时器时钟和分频系数限制；PPM 帧同步需要精确时序检测。

### 卡片 5：Bootloader + DFU
- **一句话**：EdgeTX 的 bootloader 负责在开机时检查固件更新请求，通过 USB DFU（Device Firmware Upgrade）模式让用户直接拖放 .bin 文件烧录。
- **类比**：类似于 web 应用的蓝绿部署——bootloader 是"不变的基础设施"。
- **场景**：所有需要 OTA 或 USB 升级的嵌入式产品。
- **坑**：如果 bootloader 本身损坏，MCU 将变砖（brick），需要 JTAG/SWD 恢复。

### 卡片 6：CRC 校验与存储可靠性
- **一句话**：EdgeTX 在模型文件和配置数据中嵌入 CRC 校验，加载时验证数据完整性，损坏时自动回退或提示。
- **类比**：等于后端数据库的 checksum 或 Git 的哈希验证——不过是在单机非易失存储上。
- **场景**：配置存储、OTA 固件包、日志完整性。
- **坑**：CRC 只能检测错误，不能纠正（除非使用 ECC）；Flash 写入过程中掉电可能导致半写状态。

### 卡片 7：LUA 脚本引擎嵌入
- **一句话**：EdgeTX 内嵌 LUA 5.2 解释器，用户可以在遥控器上运行自定义脚本实现定制混控、音频提示和 UI 小部件。
- **类比**：相当于在 Chrome 浏览器中嵌入 V8 JavaScript 引擎——只不过运行在 STM32 上，RAM 以 KB 计。
- **场景**：用户可编程的嵌入式设备（计算器、示波器、智能家居面板）。
- **坑**：LUA 脚本执行不能阻塞 Mixer 任务——EdgeTX 对 LUA 有执行时间配额限制；内存分配器需要定制以避免碎片。

### 卡片 8：FatFs 文件系统
- **一句话**：EdgeTX 使用 FatFs 库管理 SD 卡上的 FAT16/FAT32 文件系统，用于加载位图、声音、模型配置和日志。
- **类比**：相当于浏览器中的文件系统 API——读写文件但受限于 FAT 的 8.3 命名规则和 4GB 文件大小限制。
- **场景**：任何需要 SD 卡/NAND Flash 存储的嵌入式设备。
- **坑**：FAT 文件系统在突然断电时容易损坏；长时间的 SD 卡写操作会阻塞其他任务；SPI SD 卡读取时序敏感。

### 卡片 9：遥测（Telemetry）反向数据通道
- **一句话**：接收机把传感器数据（电压、GPS、高度、电流、信号强度等）通过下行链路发回遥控器，EdgeTX 解析后在屏幕上显示并触发音频告警。
- **类比**：等于 HTTP 响应的 Response Body——发射（混频器→接收机）和接收（接收机→遥控器）是双向的。
- **场景**：FPV 飞行、无人机测绘、工业遥测。
- **坑**：下行链路和上行链路共享同一射频通道，数据碰撞和丢包需要容错设计；不同协议（FRSKY、CRSF、SBUS）的遥测帧格式完全不同。

### 卡片 10：CMake Superbuild 跨平台构建
- **一句话**：EdgeTX 的 CMake 配置使用 ExternalProject 模式（Superbuild），在一个顶级 CMake 下同时为 native（宿主机模拟器）和 ARM 目标交叉编译。
- **类比**：等于 Docker 的 multi-stage build——一个配置产出多个目标产物。
- **场景**：需要模拟器 + 实物的嵌入式项目、多 MCU 平台产品线。
- **坑**：交叉编译链配置复杂，Toolchain 文件（arm-none-eabi.cmake）稍有错误就编译失败；调试时 native 目标和 ARM 目标的差异可能踩坑。

## 9. 30～60 分钟动手任务

### 目标
在 EdgeTX 模拟器中运行并观察 Mixer 任务输出。

### 步骤
1. 克隆 EdgeTX 仓库并进入项目目录：
   ```bash
   git clone https://github.com/EdgeTX/edgetx.git
   cd edgetx
   ```
2. 构建 Native（宿主机）模拟器目标：
   ```bash
   mkdir build && cd build
   cmake .. -DNATIVE_BUILD=ON
   make native
   ```
   *如果依赖不足可参考 Docker 构建环境：*
   ```bash
   git clone https://github.com/EdgeTX/build-edgetx
   ```

3. 找到模拟器可执行文件 `build/native/radio/src/native_etx` 并运行。
4. 在模拟器中操作（键盘模拟摇杆 + 鼠标模拟触摸）：
   - 进入模型设置 → Mixer 页面
   - 添加一个混控条（Mix），观察通道数值随摇杆位置变化
5. 源码阅读验证：打开 `radio/src/mixer.cpp`，搜索 `processMixer` 函数，对比模拟器中的数值计算逻辑。

- [ ] 能看到通道数值随模拟摇杆输入实时变化

## 10. 3 道深度追问题

### 问题 1（时序/并发）：
Mixer 任务栈只有 400 字节，而 Menus 任务栈需要 2KB/8KB。为什么 Mixer 的栈可以这么小？如果混控逻辑中意外引入了一个深度递归调用，系统会怎样表现？设计者应该怎么预防？

### 问题 2（软件转嵌入式视角）：
在 Web 后端中，如果你发现某个 API 请求处理超时，你加一个异步队列、或者水平扩容就解决了。在 EdgeTX 中，如果混频器计算耗时超过了 9ms 的任务周期（导致后续脉冲输出延迟），你能用什么手段排查和解决？请至少列出 3 种嵌入式特有的方法。

### 问题 3（资源约束下的设计取舍）：
EdgeTX 的 GUI 系统有彩色和单色两套完全不同的实现。为什么不像 Web 前端那样用"响应式设计"一套代码适配所有屏幕？从嵌入式角度分析：如果一个嵌入式产品同时需要支持 128x64 单色屏和 480x272 彩屏，设计 GUI 架构时有哪些根本性差异需要考虑？

## 11. 今日复盘问题

想象你是 EdgeTX 的固件架构师。现在有一个新需求：在 Mixer 和 Menus 两个任务之间增加一个低优先级的数据分析任务，每天统计摇杆使用频率并写 SD 日志。你需要在**不改变现有任务优先级和栈大小**的前提下实现。你会怎么设计这个新任务——它的栈从哪里来？它和 SD 卡写操作发生冲突时如何协调？

## 12. 工程点评

### 可维护性
EdgeTX 继承了 OpenTX 的代码基础，部分模块（如 mixer.cpp）有超过十年的历史积累。源码风格以简洁实用为主，注释不多但关键路径有说明。CMake 构建系统清晰整洁，模块化程度较好。

### 可移植性
HAL 层的函数指针结构体设计简洁但有效——新目标板只需实现 `etx_hal_adc_driver_t`、`etx_hal_storage_driver_t` 等接口即可接入。不过 GUI 层彩色/单色两套实现完全分离，移植工作量仍然较大。

### 可靠性
关键数据（模型文件、配置）均有 CRC 校验存储；存在异常重启检测机制（`abnormal_reboot.h`）；SD 卡保留最小空闲空间防止写满崩溃。没有单元测试框架（只有 `tests/` 目录下少量集成测试），这一点对规模如此大的嵌入式项目来说是个风险。

### 可测试性
好的一面：Native 模拟器目标让开发者无需硬件即可进行大部分功能验证。不足的一面：混频器的数学计算没有独立的单元测试，遥测协议解析的测试覆盖依赖于集成测试。

### 量产/运维
EdgeTX 具有完善的生产部署链路：CI 自动构建固件 → GitHub Release 发布 → Buddy 桌面工具辅助刷机 → SD 卡内容包独立管理（位图/声音/Lua 脚本）。Bootloader 设计支持 USB DFU 拖放升级，可在无需拆机的情况下完成固件更新，量产运维成熟度较高。
