# 每日嵌入式学习：RMK

## GitHub 链接

https://github.com/HaoboGu/rmk

## 项目一句话介绍

RMK 是一个用 Rust 编写的开源机械键盘固件库，面向 STM32、nRF52、RP2040、ESP32 等微控制器，支持 USB、有线低延迟、BLE 蓝牙、多设备连接、分体键盘、Vial 实时改键、层切换、宏、组合键、鼠标控制和低功耗矩阵扫描。

## 为什么今天选它

今天选择 RMK，是因为它处在嵌入式、Rust、低功耗外设、HID 输入设备和无线连接几个方向的交叉点上，工程规模适中，但内容足够完整。它不是一个只点亮 LED 的入门示例，也不像大型通用 RTOS 那样需要很重的系统背景。对已经有 C 或 C++、单片机、RTOS、Linux 或硬件基础的人来说，RMK 很适合作为“把真实产品固件拆开学习”的对象。

从历史去重看，今天没有选择已经讲过的 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK、InfiniTime，也没有选择这些项目的 fork。RMK 虽然同样属于键盘固件领域，但它是独立仓库，核心语言、架构和目标重点都不同，特别强调 Rust、Embassy 异步运行时、配置生成和 BLE/Vial 结合，适合作为新的学习项目。

## 项目解决的问题

自制机械键盘看起来只是“按键变成 USB 或蓝牙输入”，但真正做成可用产品会遇到很多嵌入式问题。按键矩阵需要扫描和消抖，组合键和长按短按需要状态机，USB HID 报告要符合主机协议，蓝牙连接要处理配对、重连、电池信息和多设备切换，分体键盘要处理左右半区通信，配置还要让普通用户不用重写固件就能改键。

RMK 解决的正是这类问题。它把键盘固件常见能力做成 Rust 库和配置框架，使用者可以通过一个 keyboard.toml 描述芯片、矩阵、布局、通信和行为；如果需要更强定制，也可以直接写 Rust 代码。README 明确提到，RMK 基于 Embassy，支持 stm32、nRF、rp2040、esp32 等系列；支持 Vial 实时修改键位，甚至可以通过 BLE 修改；内置层切换、媒体控制、系统控制、鼠标控制等高级功能；BLE 支持自动重连和多设备；有线模式延迟约 2ms，蓝牙模式约 10ms；开启 async_matrix 后，一个 2000mAh 电池可让键盘续航几个月。

## 技术架构分析

RMK 的核心特点是 no_std Rust 与异步嵌入式框架结合。rmk/src/lib.rs 中可以看到项目默认在非测试场景使用 no_std，这意味着它面向没有标准库、资源受限的微控制器环境。它导出了 embassy_time、embassy_futures、heapless、futures 等嵌入式常用组件，并按功能划分为 ble、boot、channel、config、debounce、driver、event、hid、host、input_device、keyboard、keymap、matrix、processor、split、storage、usb、watchdog 等模块。

项目的基本数据流可以概括为：矩阵或输入设备产生按键事件，事件通过发布订阅通道进入输入处理器和键盘状态机，键盘状态机生成 HID 报告，最后由 USB 或 BLE 的 HID writer 发给主机。这个流向非常适合学习嵌入式事件驱动架构，因为它把硬件采样、事件抽象、业务处理和协议输出分开了。

RMK 也是一个 Cargo 多 crate 工程。仓库里没有根目录 Cargo.toml，主要 crate 分为 rmk、rmk-config、rmk-macro、rmk-types。rmk 是核心固件逻辑；rmk-config 负责解析 keyboard.toml 配置；rmk-macro 提供过程宏，用来简化键盘定义、事件和代码生成；rmk-types 存放 KeyAction、EncoderAction、蓝牙、组合键、鼠标按钮、协议和键码等共享类型。这样的拆分体现了中型嵌入式 Rust 工程的组织方式：核心逻辑、配置、宏和类型边界清晰，便于测试和复用。

## 源码结构导读

README_zh.md 是入门入口，里面列出特性、真实案例、模板初始化方式和 RP2040 构建烧录示例。examples 目录分为 use_config 和 use_rust 两种风格。use_config 适合从 keyboard.toml 开始学习，目录中能看到 esp32_ble_split、nrf52840_ble、pi_pico_w_ble、rp2040、stm32f1、stm32f4、stm32h7 等示例。use_rust 则适合理解纯 Rust API 的写法，里面有 nRF52840、RP2040、RP2350、STM32、ESP32、SF32 等多种目标。

rmk/src 是最值得细读的核心目录。matrix 和 debounce 负责按键矩阵扫描与消抖；event、channel、processor 和 keyboard 体现事件流和状态机；hid、usb、ble、host 负责把键盘行为转成主机能识别的输入报告；keymap、keyboard_macros、layout_macro 关系到层、布局和宏；split 适合研究分体键盘通信；storage 用于保存配置或运行时修改后的键位；watchdog、boot、driver 则更贴近底层工程实践。

rmk-config/src 适合学习“配置如何变成固件常量”。项目会读取 event_default.toml、芯片默认配置和用户 keyboard.toml，再由 build.rs 或宏展开阶段生成 Rust 常量。rmk-types/src 适合作为键盘领域模型词典，里面能看到 action、battery、ble、combo、connection、keycode、modifier、mouse_button、protocol、steno 等类型。rmk-macro/src 则适合有 Rust 过程宏基础后再读，因为它承担了自动生成样板代码的工作。

## 学习价值

对嵌入式工程师来说，RMK 的学习价值不只是“做一把键盘”。第一，它展示了 Rust 在资源受限设备上的实际组织方式，包含 no_std、feature flag、异步任务、静态内存和类型安全。第二，它把 USB HID、BLE HID、矩阵扫描、低功耗、配置生成和状态机整合在一个真实产品场景中。第三，它兼顾工程易用性和底层可控性，既能用 keyboard.toml 快速配置，也能用 Rust API 深度定制。第四，它的示例覆盖多种 MCU，适合作为比较不同芯片外设、构建目标和烧录方式的材料。

## 3～5 步动手路线

第一步，先通读 README_zh.md 和 examples/README.md，理解 RMK 的目标、支持芯片、两种使用方式，以及有线、蓝牙、分体、Vial 等关键词。不要急着改源码，先把项目能解决什么问题想清楚。

第二步，选择一个手边最容易验证的开发板。如果有 RP2040 或 Pi Pico，可以从 rp2040 示例入手；如果有 nRF52840，可以优先看 BLE 示例；如果偏 ESP32，则看 esp32c3_ble、esp32s3_ble 或 esp32c6_ble。先完成 release 构建，再考虑烧录和调试。

第三步，从 use_config 示例开始，重点阅读 keyboard.toml。观察芯片、矩阵行列、USB/BLE、布局、层和行为如何描述。然后对照 rmk-config/src 中的配置结构，理解配置文件如何被解析、合并默认值并转成编译期常量。

第四步，沿着数据流阅读源码：先看 matrix 和 debounce，理解输入怎样产生事件；再看 event、channel、processor、keyboard，理解事件怎样进入键盘状态机；最后看 hid、usb、ble，理解状态机怎样输出主机识别的 HID 报告。

第五步，做一个小改动，例如增加一个自定义层、尝试组合键或宏、打开 Vial 配置、或在某个事件处理路径加日志。目标不是一次性读完整个项目，而是让“配置、构建、烧录、按键输入、主机收到报告”这一条链路跑通。

## 工程点评

RMK 的优点是边界清晰、文档友好、示例覆盖面广，并且很适合展示 Rust 嵌入式工程的现代写法。它没有把全部逻辑堆在一个 firmware.c 里，而是按配置、类型、宏、核心逻辑和示例拆开。它对普通用户保留 keyboard.toml 入口，对高级用户保留 Rust API，这种分层值得学习。

需要注意的是，RMK 也有一定门槛。Rust、Embassy、BLE、USB HID、过程宏和键盘领域模型叠在一起，新手可能会觉得抽象层偏多。学习时不建议从宏或 BLE 协议栈直接开始，而应先从一个有线矩阵键盘示例进入，先理解最小闭环，再逐步加入无线、Vial、分体和低功耗。总体来看，RMK 是一个适中偏上的学习项目，适合用来训练真实嵌入式固件的架构阅读能力。

## 今日 10 个嵌入式概念/名词

1. no_std：通俗说，就是程序不能依赖操作系统提供的标准库，适合裸机或小型 RTOS 环境。RMK 在非测试场景默认 no_std，说明它主要运行在微控制器上。嵌入式工程师需要懂 no_std，因为很多 MCU 没有文件系统、线程和动态内存环境。

2. Embassy：这是 Rust 嵌入式生态中的异步框架，可以把定时、外设和任务写成 async 风格。RMK 基于 Embassy 支持多种 MCU，并用异步方式处理矩阵扫描、通信和时间相关逻辑。工程师理解它，有助于掌握现代嵌入式事件驱动设计。

3. HID：HID 是 Human Interface Device，也就是键盘、鼠标这类人机输入设备协议。RMK 最终要把按键状态变成 HID 报告，通过 USB 或 BLE 发给电脑、平板或手机。做输入设备、遥控器、控制面板时，HID 是非常常见的协议。

4. BLE：BLE 是低功耗蓝牙。RMK 用 BLE 做无线键盘连接，并支持自动重连和多设备能力。嵌入式工程师需要理解 BLE 的连接、配对、服务和功耗特性，因为很多电池供电产品都依赖它。

5. 按键矩阵：键盘不会给每个按键单独接一个引脚，而是用行列矩阵减少 GPIO 数量。RMK 的 matrix 模块与各类示例都围绕矩阵扫描展开。理解矩阵有助于设计按键板、扫描算法和排查鬼键问题。

6. 消抖：机械按键按下时电平会抖动，软件必须判断稳定状态。RMK 中有 debounce 相关模块。工程师需要懂消抖，因为按钮、旋钮、继电器触点等真实硬件都不是理想信号。

7. Keymap：Keymap 是键位映射，决定某个物理按键对应什么输出。RMK 支持实时键位编辑和多层键位。理解 keymap 能帮助工程师把硬件输入和用户体验连接起来。

8. Feature flag：Rust 的 feature flag 可以在编译时开关功能，例如 BLE、storage、split、display 等。RMK 大量使用 feature 控制不同固件能力。嵌入式工程师需要懂它，因为资源受限设备通常必须按需裁剪功能。

9. 构建脚本 build.rs：RMK 使用构建阶段读取配置并生成常量。通俗说，build.rs 像编译前的自动加工步骤，把配置变成代码。嵌入式项目常用这种方式生成引脚、内存、协议或表格数据。

10. 分体键盘：分体键盘把左右半区分成两个控制区域，需要同步按键事件和连接状态。RMK 的 split 模块和相关示例可用于学习这类通信结构。嵌入式工程师理解分体设计，有助于处理多板协同、主从通信和低功耗同步问题。
