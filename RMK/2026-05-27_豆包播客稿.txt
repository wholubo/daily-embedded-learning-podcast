今天的嵌入式学习项目，选的是 RMK。它是一个用 Rust 编写的开源机械键盘固件库，面向 STM32、nRF52、RP2040、ESP32 这类常见微控制器。它能做的事情不只是把按键变成电脑上的字符输入，还包括 USB 有线键盘、BLE 低功耗蓝牙键盘、分体键盘、实时改键、多层键位、宏、组合键、媒体控制、鼠标控制，以及低功耗矩阵扫描。对有一定 C、C++、单片机、RTOS 或硬件基础的人来说，RMK 是一个很适合拆开学习的真实固件项目。

今天之所以选它，是因为它处在几个很有学习价值的交叉点上。它是嵌入式项目，又是 Rust 项目；它涉及低功耗外设，又涉及 USB 和蓝牙这种主机通信协议；它面向真实键盘产品，但工程规模还没有大到完全看不动。和只点亮一个 LED 的例程相比，RMK 更接近实际产品。和大型通用实时操作系统相比，它又更聚焦，主线很清楚，就是从硬件按键出发，经过扫描、消抖、事件处理和键位映射，最后把结果变成主机能识别的输入报告。

从去重角度看，今天没有选择之前已经讲过的 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK 或 InfiniTime，也没有选择这些项目的分支。RMK 虽然也属于键盘固件领域，但它是独立仓库，核心语言、架构重点和学习路径都不同。它更强调 Rust、Embassy 异步运行时、配置生成，以及蓝牙和 Vial 实时改键的结合，所以很适合作为新的学习对象。

联网检索时，GitHub 搜索结果显示 RMK 大约有一千六百多个 star，仓库描述是一个支持层、宏、实时键位编辑、蓝牙无线和分体键盘的 Rust 键盘固件库。搜索结果显示项目近期仍有推送记录。本地克隆仓库后也能看到，最近的提交是一次修复持续集成相关问题的合并提交。README 里有中文说明、用户文档、接口文档、常见问题、更新日志、构建状态和社区入口。也就是说，这不是一个只放在那里展示的代码仓库，而是有文档、有示例、有测试、有实际用户场景的工程。

机械键盘固件看起来很简单，好像只是把一个按键变成一个字符。但真正做成稳定产品，就会遇到一连串嵌入式问题。键盘通常不会给每一个按键都单独接一个引脚，而是用行列矩阵节省输入输出资源。矩阵扫描出来的电平还会抖动，需要做消抖。一个按键可能在不同层里代表不同功能，短按和长按也可能触发不同动作。主机端不能直接理解这些内部状态，所以固件还要把它们转换成标准 HID 报告。到了无线键盘，还要考虑蓝牙配对、自动重连、多设备切换、电池信息和功耗。到了分体键盘，还要考虑左右两边怎样通信和同步状态。RMK 正是把这些问题组织成一个可以复用、可以配置、可以扩展的固件框架。

RMK 的一个重要特点，是它使用 no_std Rust。no_std 可以理解为程序不能依赖操作系统提供的完整标准库，它更适合裸机或资源受限的微控制器。RMK 默认在非测试环境下使用 no_std，这说明它的目标不是普通桌面程序，而是真正要跑在小芯片上的固件。项目中还能看到 Embassy、heapless、futures 等嵌入式 Rust 生态里的常见组件。Embassy 可以把定时、外设和任务写成异步风格，让矩阵扫描、通信和等待事件这些动作更自然地组合起来。

从架构上看，RMK 的数据流非常适合学习。最前面是矩阵或其他输入设备，它们负责感知按键变化。接着，这些变化会变成事件，通过发布订阅通道送进输入处理器和键盘状态机。键盘状态机会结合当前层、组合键、宏、长按短按和用户配置，决定现在应该输出什么。最后，固件把结果生成 HID 报告，再通过 USB 或 BLE 发给电脑、平板或手机。这样一条链路把硬件采样、事件抽象、业务处理和协议输出分开了，是很典型的嵌入式事件驱动设计。

RMK 的仓库结构也很值得看。它不是把所有代码堆在一个文件里，而是一个 Cargo 多 crate 工程。核心部分包括 rmk、rmk-config、rmk-macro 和 rmk-types。rmk 是固件主逻辑，里面有矩阵、消抖、事件、键盘状态机、HID、USB、蓝牙、分体、存储、看门狗等模块。rmk-config 负责解析 keyboard.toml 这类配置文件，把用户写的键盘描述变成程序能理解的数据。rmk-macro 提供过程宏，用来减少样板代码。rmk-types 则存放键盘动作、编码器动作、蓝牙连接、组合键、鼠标按钮、键码和协议等共享类型。这样的拆分很适合中型嵌入式项目学习，因为边界清楚，职责也比较明确。

如果要读源码，可以先从中文 README 开始。README 里列出了 RMK 的主要特性，比如支持 stm32、nRF、RP2040、ESP32，支持 Vial 实时改键，支持层切换、媒体控制、系统控制、鼠标控制，也支持 BLE 自动重连和多设备。它还提到有线模式延迟大约两毫秒，蓝牙模式大约十毫秒；在开启异步矩阵扫描功能后，一个两千毫安时的电池可以让键盘续航几个月。这些数字不需要死记，但它们能帮助我们理解项目追求的目标：低延迟、低功耗、可配置、跨芯片。

examples 目录是第二个入口。它分成 use_config 和 use_rust 两种用法。use_config 更适合初学者，因为它从 keyboard.toml 这种配置文件开始，比较容易看出一个键盘工程需要描述哪些东西，比如芯片、矩阵行列、布局、通信方式和按键行为。use_rust 更适合想深入定制的人，它展示了直接用 Rust 接口组织固件的方式。示例里能看到 RP2040、Pi Pico W、nRF52840、ESP32、STM32 等目标，这对有不同开发板的人很友好。

核心源码可以沿着几个目录慢慢看。matrix 和 debounce 负责按键矩阵扫描与消抖，这部分最贴近硬件。event、channel、processor 和 keyboard 展示了事件如何流动，以及键盘状态机如何决定输出。hid、usb、ble 和 host 负责把内部按键行为转换成主机能识别的输入报告。keymap、layout_macro 和 keyboard_macros 关系到键位层、布局和宏。split 可以用来研究分体键盘通信。storage 则和保存配置、保存实时改键后的状态有关。读这些目录时，不建议一开始就追所有细节，而是先抓住从按键到报告的主链路。

RMK 还有一个很好的学习点，是配置怎样变成固件。很多嵌入式项目都会让用户写配置，但最终固件需要的是编译期常量、静态数组和类型明确的结构。RMK 会读取默认事件配置、芯片默认配置和用户自己的 keyboard.toml，然后在构建阶段或宏展开阶段把这些信息转成 Rust 常量。这种做法很常见，也很重要，因为嵌入式设备资源有限，很多东西最好在编译期决定，而不是运行时再动态解析。

动手学习可以分成几步。第一步，先通读中文 README 和示例说明，只理解 RMK 要解决什么问题，不急着改源码。第二步，选一块手边容易验证的开发板。如果有 RP2040 或 Pi Pico，可以看 RP2040 示例；如果有 nRF52840，可以看蓝牙示例；如果偏 ESP32，可以看 ESP32C3、ESP32S3 或 ESP32C6 的蓝牙示例。第三步，从 use_config 示例开始，重点读 keyboard.toml，看芯片、矩阵、布局和通信是怎么描述的。第四步，再沿着矩阵扫描、消抖、事件通道、键盘状态机、HID 输出这条线读源码。第五步，做一个小改动，比如增加一个自定义层，尝试组合键或宏，打开实时改键能力，或者在某个事件路径加调试输出。目标不是一次读完整个项目，而是先跑通配置、构建、烧录、按键输入和主机收到报告这条闭环。

今天可以顺便记住十个和 RMK 强相关的嵌入式概念。第一个是 no_std，它表示固件不能依赖完整标准库，适合小型微控制器。第二个是 Embassy，它是 Rust 嵌入式异步框架，帮助组织定时、外设和任务。第三个是 HID，也就是人机输入设备协议，键盘和鼠标通常都靠它和主机通信。第四个是 BLE，也就是低功耗蓝牙，RMK 的无线键盘能力就依赖它。第五个是按键矩阵，它用行列结构节省引脚，是键盘硬件设计的基础。第六个是消抖，因为真实机械按键的电平不会干净地从零跳到一。第七个是 keymap，也就是键位映射，它决定物理按键在不同层里输出什么。第八个是 feature flag，Rust 可以用它在编译时裁剪蓝牙、分体、存储、显示等功能。第九个是 build.rs，它像编译前的自动加工步骤，可以把配置变成代码。第十个是分体键盘，它涉及多板协同、主从通信和状态同步。

从工程角度看，RMK 的优点是边界清晰、示例多、文档相对友好，而且能展示 Rust 嵌入式工程比较现代的写法。它既给普通用户提供 keyboard.toml 这种低门槛入口，也给高级用户保留直接写 Rust 的空间。它把键盘固件里常见的矩阵扫描、消抖、状态机、USB、蓝牙、存储和配置生成放在一个真实场景里，比单独看某个外设驱动更容易形成完整认识。

当然，RMK 也不是零门槛项目。Rust、Embassy、USB HID、BLE、过程宏和键盘领域模型叠在一起，刚开始可能会觉得抽象层很多。学习时不要从最复杂的蓝牙协议或宏生成开始，也不要一上来就试图理解所有 feature。更好的方式是先选一个有线键盘示例，先理解矩阵输入怎样变成 HID 报告，再逐步加入蓝牙、实时改键、分体和低功耗。这样学下来，RMK 不只是一个键盘固件项目，也会变成一次很完整的嵌入式系统架构训练。
