# 每日嵌入式学习：OpenSK

## GitHub 链接

https://github.com/google/OpenSK

## 项目一句话介绍

OpenSK 是 Google 开源的安全密钥固件与库，用 Rust 实现 FIDO U2F 和 FIDO2/CTAP 协议，目标是在 Nordic nRF52840 等嵌入式硬件上构建一个从应用到运行环境都尽量开放的硬件安全钥匙实验平台。

## 为什么今天选它

今天避开了历史项目列表中的 LVGL、Trice、Meshtastic firmware、ZSWatch、ZMK、InfiniTime、RMK、TinyUSB 和 Ariel OS，选择 OpenSK 的原因有三个。第一，它不是普通传感器 demo，而是把 USB 通信、闪存持久化、随机数、密码学、用户确认、协议状态机、硬件抽象和构建烧录流程放在同一个工程里，学习密度很高。第二，它使用 Rust 编写，既能观察嵌入式领域正在采用的内存安全语言实践，也能继续对照传统 C/C++ 固件的分层方式。第三，安全密钥是一个非常明确的产品形态，学习时不容易停留在“能跑起来”的层面，而会自然追问认证协议、私钥保护、批量证明、PIN 策略、USB HID 包处理和资源约束之间的取舍。

## 热度/活跃度依据

通过 GitHub API 查询到的当前仓库信息显示，OpenSK 仓库为 `google/OpenSK`，描述为“OpenSK is an open-source implementation for security keys written in Rust that supports both FIDO U2F and FIDO2 standards.”。查询时仓库约有 3350 个 star、330 个 fork、11 个 open issues，主要语言为 Rust，许可证为 Apache-2.0，主题标签包括 embedded、firmware、hardware、rust、security、security-key、fido2、ctap2、u2f、webauthn、tock-os、wasefire 等。仓库最近 pushed_at 为 2026-05-29T15:04:50Z，说明近期仍有更新。README 也明确说 develop 分支面向开发者，新功能在这里开发，编号分支对应较旧的 CTAP 版本。以上信息足以说明它不是陈旧样例，而是一个仍然值得跟踪的嵌入式安全方向项目。

## 项目解决的问题

传统硬件安全密钥往往是黑盒产品，使用者知道它能完成网站登录时的二次认证，却很难看到从 USB 传输、FIDO 协议解析、凭据创建、私钥保存、PIN 验证到用户触摸确认的完整实现。OpenSK 想解决的正是这个透明度问题。它把安全密钥的核心行为放进开源仓库中，让开发者可以研究一个认证器如何与浏览器或操作系统交互，如何处理 CTAP 命令，如何生成和保存凭据，如何通过用户在设备上的物理操作来确认意图，以及如何在资源有限的 MCU 上组织这些安全敏感逻辑。

README 特别强调 OpenSK 是概念验证和研究平台，不建议直接作为日常使用的安全密钥。这个定位很重要：它降低了学习门槛，也提醒我们不要把实验固件等同于经过完整商业认证、供应链和侧信道审计的量产安全产品。对于嵌入式工程师来说，这样的项目适合用来学习工程结构、协议实现和安全边界，而不是直接拿去保护个人关键账号。

## 技术架构分析

从仓库结构看，OpenSK 的核心由 Rust 库、环境抽象、CTAP 协议模块和工具脚本组成。根目录下有 `Cargo.toml`、`Cargo.lock`、`build.rs`、`setup.sh`、`flash.sh`、`docs`、`libraries`、`src`、`tools`、`rules.d` 等内容。README 说明现在 OpenSK 可以作为 Wasefire applet 运行，也可以作为库使用。Wasefire 可以理解为面向安全嵌入式应用的运行框架，OpenSK 把自身的认证器逻辑做成 applet，底层平台则需要提供按钮、时钟、密码学、随机数、LED、存储、定时器、USB CTAP 等板级能力。

`libraries/opensk/src/lib.rs` 里能看到 `Ctap<E: Env>` 这个关键抽象。它把 CTAP 状态、HID 传输和环境对象组合起来，其中 `Env` 是环境接口，负责把具体硬件能力注入到协议逻辑中。这种写法很适合嵌入式学习：上层认证协议不直接依赖某个芯片寄存器，而是通过 trait 访问时钟、存储、随机数、USB 连接和密码学能力。换句话说，OpenSK 的架构不是把所有代码写成一个巨大的 `main`，而是把“协议怎么运行”和“硬件怎么提供能力”分开。

CTAP 相关代码集中在 `libraries/opensk/src/ctap`。这个目录下有 `command.rs`、`response.rs`、`data_formats.rs`、`client_pin.rs`、`credential_management.rs`、`large_blobs.rs`、`pin_protocol.rs`、`storage.rs`、`main_hid.rs`、`hid` 子目录、`status_code.rs`、`ctap1.rs` 等文件。单看文件名就能看出项目主线：先从 USB HID 收到数据包，再解析成 CTAP 命令，进入状态机处理注册、登录、PIN、凭据管理、大对象存储等逻辑，最后组织响应并返回给主机。对于学习者来说，这是一个很好的协议栈剖面。

`libraries/opensk/src/api` 则定义板级和安全能力接口，包括 `clock.rs`、`connection.rs`、`rng.rs`、`persist.rs`、`key_store.rs`、`user_presence.rs`、`firmware_protection.rs`、`fingerprint.rs` 和 `crypto` 子目录。`customization.rs` 里集中放置可定制常量，例如 AAGUID、PIN 协议选择、凭据保护策略、最小 PIN 长度、企业证明、签名计数器等。这个设计让安全策略可以被集中审查，也便于测试脚本检查不变量。

## 源码结构导读

新读者可以先读 `README.md`，了解 OpenSK 的定位、支持硬件和注意事项。然后读 `docs/install.md`，这里说明了 Rust、OpenSSL、Python/uv、udev 规则、存储擦除、功能开关和目标平台。文档中特别提到 OpenSK 会把凭据等数据保存在设备 flash 中，烧录 Wasefire 时可能擦除存储，若要保留数据需要使用更新参数。这个细节体现了嵌入式开发中“程序镜像”和“持久化数据”之间的边界。

接着可以看 `libraries/opensk/src/lib.rs`，重点理解 `Ctap` 结构体如何把环境、状态机和 HID 处理组合起来。然后进入 `libraries/opensk/src/ctap/mod.rs`，顺着模块声明建立地图：命令、响应、数据格式、PIN、凭据管理、HID、存储和错误码分别在哪里。不要一开始就试图读懂所有加密细节，先把数据流跑通更重要。

第三步读 `libraries/opensk/src/api/customization.rs` 和 `libraries/opensk/src/ctap/storage.rs`。前者告诉你安全密钥有哪些可配置策略，后者展示凭据怎样从持久层读写、查找和反序列化。最后再看 `tools` 目录，里面的配置脚本和密钥材料生成脚本能帮助理解生产或实验设备如何注入证书、私钥和配置。

## 学习价值

OpenSK 的学习价值不在于“做一个自己的登录钥匙”这么简单，而在于它把很多嵌入式工程师必须面对的主题放在一个真实项目里。你可以看到 Rust 的 `no_std` 思路如何服务资源受限环境；可以看到协议状态机如何避免和硬件细节耦合；可以看到 USB HID 这种常见外设协议如何承载上层安全协议；可以看到 flash 持久化不仅是写几个配置值，还会牵涉隐私、可恢复性和擦除风险；也可以看到安全产品为什么不能只看功能，还要考虑证明、认证、随机数质量、用户存在确认、调试开关和日志开销。

对于有 C/C++、单片机、RTOS 或 Linux embedded 基础的学习者，OpenSK 难度适中偏上。它不像 blink demo 那样简单，但也没有大到无法入手。只要按模块拆开，先理解项目结构和协议流程，再逐步深入密码学和认证细节，就能收获非常扎实的嵌入式安全工程经验。

## 3～5 步动手路线

1. 先克隆仓库并阅读 `README.md`、`docs/install.md` 和 `docs/debugging.md`，记录支持的硬件、依赖工具、功能开关、烧录脚本和调试方式。暂时不急着烧录，先把工程边界弄清楚。

2. 在本机完成 Rust 工具链和依赖准备，运行项目提供的测试或检查命令。关注 `Cargo.toml` 中的 feature 设计，理解 `ctap1`、`config-command`、`debug`、`ed25519`、`fingerprint` 等开关如何影响二进制功能和资源消耗。

3. 从 `libraries/opensk/src/lib.rs` 跟踪一次 USB HID 数据进入 `Ctap` 的处理路径，再到 `ctap` 模块中的命令解析和响应生成。建议画一张自己的流程图，把主机、USB HID、CTAP 命令、凭据存储、用户确认和响应返回串起来。

4. 如果手头有 Nordic nRF52840-DK 或受支持 dongle，再按文档烧录实验固件。第一次只做最小功能验证：设备能被系统识别，能在 WebAuthn demo 类网站完成注册和登录。之后再打开 debug 日志，观察日志对 RAM、二进制大小和 USB 通信速度的影响。

5. 选择一个小改动做练习，例如调整 LED 行为、阅读并解释最小 PIN 长度策略、分析凭据查找函数，或者为某个错误路径补充测试。安全项目不建议随意大改，越是小而可验证的改动越能训练工程判断。

## 工程点评

OpenSK 最大的优点是边界清晰。CTAP 协议、硬件环境、持久化存储、定制策略和工具脚本各有位置，学习者可以分层进入。Rust 的 trait 抽象让工程更容易测试和迁移，也符合现代嵌入式软件“平台能力接口化”的趋势。项目文档对安全风险也比较克制，没有把研究平台包装成万能量产方案，这是成熟工程文化的体现。

它的挑战也很明显。FIDO2、CTAP、WebAuthn、证明证书、PIN/UV、凭据保护这些概念密度很高，第一次阅读容易被名词淹没。另一个挑战是硬件相关实验需要合适开发板，尤其是安全密钥类项目不仅要能烧录，还要理解 USB、存储擦除和主机侧识别。最后，README 提到硬件加速密码学仍有集成工作，当前会使用 RustCrypto，这也提醒我们：安全固件的算法正确性、实现安全性和硬件加速支持是三个不同层面，不能混为一谈。

## 今日 10 个嵌入式概念/名词

1. **FIDO2**：通俗说，它是一套让网站登录不再单纯依赖密码的认证标准。在 OpenSK 中，它决定安全密钥和浏览器、操作系统之间如何完成注册与登录。嵌入式工程师需要懂它，因为协议标准会直接影响固件状态机、数据结构和错误处理。

2. **CTAP**：CTAP 是客户端到认证器协议，可以理解为主机和安全钥匙之间说话的语言。OpenSK 的 `ctap` 目录就是围绕命令解析、响应生成和状态管理展开。工程师需要懂 CTAP，因为它展示了上层安全协议如何落到 USB HID 这样的底层传输上。

3. **USB HID**：HID 常被用于键盘鼠标，也可以承载安全密钥通信。OpenSK 中的 HID 模块负责把主机发来的包组织成协议数据。嵌入式工程师需要懂 HID，因为它免驱、普遍、但包大小和时序都有限制，设计协议时必须考虑这些约束。

4. **AAGUID**：它是认证器证明全局唯一标识的一部分，用来描述某类认证器。OpenSK 的定制文件中包含相关常量。工程师需要懂它，因为安全设备不只要“能签名”，还要能向外表达自己是什么设备、属于什么模型。

5. **Attestation**：证明机制用于在注册时告诉网站，这个凭据来自某类硬件安全密钥。OpenSK 文档提醒，自制设备的证明密钥可能带来可识别性和隐私风险。工程师需要懂它，因为安全功能经常伴随隐私和供应链信任问题。

6. **PIN/UV**：PIN 是用户输入的口令，UV 是用户验证，可以包含指纹等方式。OpenSK 中有 client pin、pin protocol 和 fingerprint 相关模块。工程师需要懂它，因为用户验证策略会影响安全等级、交互体验和设备资源占用。

7. **持久化存储**：安全密钥必须保存凭据、计数器和配置。OpenSK 使用设备 flash 保存数据，并在文档中提醒烧录可能擦除存储。工程师需要懂它，因为嵌入式设备的存储容量、擦写寿命和升级流程都会影响产品可靠性。

8. **随机数生成器**：安全协议依赖高质量随机数生成密钥和挑战响应。OpenSK 的 API 中有 rng 接口。工程师需要懂它，因为伪随机或低熵随机数会让看似正确的密码学系统失去安全性。

9. **no_std Rust**：在没有完整标准库的环境中使用 Rust，需要依赖核心库、alloc 和明确的硬件接口。OpenSK 的库具备面向嵌入式环境的写法。工程师需要懂它，因为越来越多安全敏感固件开始考虑 Rust，但资源约束仍然存在。

10. **Feature 开关**：嵌入式项目常用编译特性裁剪功能。OpenSK 文档列出 config-command、ctap1、debug、ed25519、fingerprint 等开关。工程师需要懂它，因为每增加一个功能都可能改变二进制大小、RAM 占用、攻击面和测试范围。
