# 每日嵌入式学习：GRBL（CNC 运动控制固件）

## 1. GitHub 项目链接与一句话介绍

- 仓库：`https://github.com/grbl/grbl`
- 一句话：GRBL 是一个运行在 Arduino/AVR 芯片上的高性能 CNC 铣床运动控制固件，它接收标准的 G 代码指令，实时计算出步进电机的脉冲时序，实现精确的 3 轴运动控制和速度规划。

## 2. 为什么今天选它

- **活跃度**：6.2k+ Stars，2026 年 7 月仍有更新提交，社区极其活跃。有成熟的 Wiki、大量的第三方 GUI（如 Candle、Universal G-Code Sender）和硬件衍生版本。Gnea 分支（grbl/grbl）是正统维护版本，持续修复和发布。
- **工程完整度**：从串口协议解析、G 代码解析、运动规划（look-ahead acceleration planner）、步进电机定时器 ISR 到硬件抽象——覆盖了一个嵌入式实时系统从输入到物理执行的全部环节，代码精炼清晰，适合软件工程师逐文件阅读。
- **学习价值**：
  - 纯软件工程师很少接触"硬实时"——GRBL 的步进电机 ISR 必须在 33 微秒内执行完毕。
  - 在 2KB RAM、32KB Flash 的 AVR 芯片上，要用 Bresenham 算法做多轴同步步进，用环形缓冲做运动队列——极致的资源约束训练。
  - 从 G 代码解析 -> 运动规划 -> 定时中断执行，是一条完整的"高层次输入 -> 低层次物理输出"链路。

## 3. 项目解决的问题

- **工程场景**：小型桌面 CNC 雕刻机、激光雕刻机、等离子切割机。用户把设计图交给 CAM 软件生成 G 代码文件，通过串口发给 GRBL，GRBL 控制步进电机驱动器和主轴/激光器来执行加工。
- **嵌入式特有约束**：
  - **硬实时**：步进电机脉冲必须精确到微秒级抖动，不能受到 CPU 其他任务的干扰。GRBL 用 Timer1 的 CTC 中断产生固定频率脉冲，且 ISR 内不允许任何阻塞操作。
  - **资源极限**：Atmega328P 只有 2KB SRAM、32KB Flash、16MHz 主频。运动规划缓冲只能容纳 18 个块，步进段缓冲只有 32 个段，任何超出都需要流水线阻塞等待。
  - **无操作系统**：不使用 RTOS，全部是裸机中断 + 主循环的超循环架构。中断负责时间关键任务，主循环负责串口读入、G 代码解析和运动规划。
  - **物理安全**：限位开关、急停、归零动作、软限位——这些在纯软件产品中不存在，但在 CNC 中断一根限位开关线意味着撞刀损毁机器，必须被当作最高优先级事件处理。

## 4. 技术架构分析

GRBL 的整体架构是一个基于中断的事件驱动裸机系统，不依赖任何 RTOS。主要模块和运行流程如下：

**主循环流程**（protocol.c 中的 `protocol_main_loop`）：
1. **串口接收**（serial.c）：UART 中断把每个字节放入环形接收缓冲，主循环从中读取一行（以 `\n` 结尾）。
2. **实时命令探测**：在读取每个字节时，"截获"特殊字符（`?` 状态查询、`~` 继续、`!` 急停等），实时命令不进入 G 代码解析器。
3. **G 代码解析**（gcode.c）：把一行 ASCII 文本解析成运动参数（目标坐标、进给率、主轴转速、M 指令等）。
4. **运动控制**（motion_control.c）：`mc_line()` 把解析后的目标坐标传入规划器；`mc_arc()` 把圆弧插补成多段直线微段再传入规划器。
5. **运动规划器**（planner.c）：`plan_buffer_line()` 把直线运动放入环形缓冲 `block_buffer[]`，并运行前瞻（look-ahead）算法——检查最近的 18 个块之间的连接速度，调整每个块的入口/出口速度以保证平滑加速。
6. **步进执行**（stepper.c）：Timer1 的中断服务程序 `ISR(TIMER1_COMPA_vect)` 从段缓冲 `segment_buffer[]` 中取出预计算段，执行 Bresenham 多轴步进脉冲。这个 ISR 大约 30kHz 频率，典型执行时间 5 微秒，最大 25 微秒。
7. **辅助控制**：
   - `spindle_control.c`：主轴/激光器 PWM 控制。
   - `coolant_control.c`：冷却液开关控制。
   - `limits.c`：限位开关归零和硬限位检测。
   - `probe.c`：G38.x 探针动作。
   - `settings.c`：$ 参数读写 EEPROM 持久化。

**关键设计模式**：

- **环形缓冲链**：串口接收缓冲 -> 行缓冲 -> 规划器块缓冲（18 个） -> 段缓冲（32 个） -> 步进 ISR。每一层都是生产者/消费者模式，解除各模块间的时序耦合。
- **前瞻加速规划**：不是简单的梯形加减速。规划器在每次新块入队时，从尾到头反向遍历环形缓冲，用物理公式重新计算每个块的最大允许入口速度，考虑相邻块之间的转向角度（"junction"转角限制）和轴最大加速度。
- **Bresenham + AMASS**：使用 Bresenham 线段算法而不是 DDA（数字微分分析器），因为 Bresenham 只用整数加减法、没有累积误差。AMASS（自适应多轴步进平滑）在低步进频率时通过位扩展提高非主导轴的分辨率，减少抖动噪声。
- **实时命令截获**：在串口 ISR 中直接设置 volatile 位标记，主循环在安全点检查执行。中断不直接执行命令，而是通过标志位通知主循环。

> 以上分析基于公开 README、Wiki 和源码推断，没有内部设计文档。

## 6. 核心模块/函数导读

### 6.1 `main()` 函数（main.c）
- **作用**：裸机 C 程序的入口。初始化硬件外设、加载持久化设置、打开中断后进入主循环。
- **关键细节**：`sei()` 开启全局中断后才进入主循环。`HOMING_INIT_LOCK` 强制开机后必须先归零，防止意外移动。
- **嵌入式意义**：裸机启动没有 `main()` 之外的框架，初始化顺序直接影响硬件正确性。如果先初始化 Stepper 再初始化 Serial，串口还没通的时候马达可能已经发出误脉冲。

### 6.2 `plan_buffer_line()`（planner.c）
- **作用**：G 代码运动解析后，把目标坐标、进给率、加速度限制打包成 `plan_block_t`，放入环形缓冲，然后触发前瞻重算。
- **关键细节**：函数里计算了每个轴上的步数（steps[]）、最大步事件数（step_event_count）、名义速度（nominal_speed_sqr）、加速度、夹角限速（max_junction_speed_sqr）等。然后调用 `planner_recalculate()`。
- **嵌入式意义**：前瞻算法是 GRBL 最复杂的地方。在只有 2KB RAM 的环境里做浮点运算并保持 30kHz 脉冲准确率，体现了"算法精简"的设计哲学。

### 6.3 `ISR(TIMER1_COMPA_vect)`（stepper.c）
- **作用**：Timer1 输出比较 A 匹配中断——GRBL 最关键的"心跳"。频率约 8kHz~30kHz，每 tick 产生一个或一组步进脉冲。
- **关键细节**：
  - 使用 `busy` 标志防止重入。
  - 先设方向引脚（DIRECTION_PORT），再设步进引脚（STEP_PORT），然后用 Timer0 做精确的脉冲宽度计时（`settings.pulse_microseconds`）。
  - `sei()` 在 ISR 内部重新使能中断，允许 Step Port Reset 中断（Timer0）抢占式更精确地控制脉冲宽度。
  - 执行 Bresenham 算法：对 X/Y/Z 三轴分别累加 `counter`，当 `counter > step_event_count` 时输出一个脉冲并回减。
- **嵌入式意义**：这是你第一次见到"在中断里再次打开中断"的手法（嵌套中断）。原因是脉冲宽度由独立的 Timer0 中断控制，比主 Timer1 更精确。理解这种多层中断设计对嵌入式性能优化至关重要。

### 6.4 `planner_recalculate()`（planner.c）
- **作用**：反向遍历环形缓冲，从尾到头计算每个块的最大安全入口速度。
- **关键细节**：使用了 trapezoidal 速度剖面——先加速到名义速度、匀速、减速。入口速度受两个限制：(a) 本轴的加速度限制，(b) 前一模块和本模块的转向夹角限制（junction deviation）。
- **嵌入式意义**：前瞻算法是一个典型的"写在前、算在后"的延迟计算模式。新块入队时不立即计算，而是等块积累够才反向计算——这是流水线（pipeline）思维在嵌入式中的应用。

### 6.5 `gc_execute_line()`（gcode.c）
- **作用**：把一行 ASCII G 代码字符串解析成内部结构化的运动指令。
- **关键细节**：状态机解析，识别 G0~G3、G38.x、M3/M5、F/S/T/X/Y/Z 等命令字。59KB 的庞大函数（包含注释），处理所有 G 代码模态和非模态命令。
- **嵌入式意义**：解析标准 G 代码协议（RS274/NGC）在 AVR 上需要极高的空间效率——函数体接近 60KB 但整个 Flash 只有 32KB？实际上 gcode.c 在编译优化后会被剪裁，但不是所有 G 代码功能都编译进去。理解编译期宏剪裁对于嵌入式固件开发至关重要。

## 7. 软件工程师类比解释

### 7.1 步进电机 ISR ≈ 数据库 WAL 写线程
- **类比**：步进电机 ISR 每 33 微秒被唤醒一次，必须在这个时间段内完成脉冲输出，和数据库的 WAL（Write-Ahead Log）写线程类似——必须在固定时隙内完成写入，不能延迟。
- **局限**：数据库 WAL 可以用缓冲区推迟写入，或者用 fsync 批量提交；步进电机 ISR 的时间窗口是物理硬约束——电机转子实实在在地转动，延迟一步就是失步，没有回退机会。

### 7.2 前瞻运动规划 ≈ Web 服务中的反向代理缓冲
- **类比**：规划器把新的 `plan_block_t` 放入环形缓冲后，从尾到头反向计算最优速度——类似反向代理（如 Nginx）先把请求缓冲下来，再决定后端服务的负载分配。
- **局限**：Nginx 缓冲的是网络请求，延迟只影响用户体验；前瞻缓冲的是物理运动，如果超前计算错误（比如高估了加速度限制），CNC 机器的机械结构会承受超过设计限度的加速度冲击，可能导致丢步或机械损坏。

### 7.3 段缓冲 + 环形队列 ≈ 消息队列（Kafka/RabbitMQ）
- **类比**：串口 -> 行缓冲 -> 规划器块缓冲 -> 段缓冲 -> ISR。每一层都是一个生产-消费者环形队列，生产者（主循环）放数据进去，消费者（ISR）取数据出来。如果消费者追不上生产者，队列就满了，生产者必须阻塞等待——和消息队列背压完全一致。
- **局限**：消息队列的背压会在消费者和处理能力之间建立弹性积压，数据可以持续堆积；段缓冲只有 32 个条目，满了就得阻塞整个主循环——这是一个硬上限，而不是弹性积压。在嵌入式系统中缓冲大小不可扩展，必须精心设计流水线深度来匹配最坏情况时序。

### 7.4 AMASS 自适应平滑 ≈ CSS 中的媒体查询
- **类比**：AMASS 在步进频率低的时候增加 Bresenham 分辨率（通过位扩展），在高频率时退回基准分辨率——类似 CSS 媒体查询根据屏幕宽度自适应布局。
- **局限**：CSS 媒体查询只是视觉呈现的调整，而 AMASS 直接影响了步进电机的物理脉冲输出精度和 CPU 负载。AMASS 的级别选择受限于嵌入式 CPU 在 33 微秒窗口内能完成多少整数运算，是一个硬实时调度问题，不是布局美学问题。

### 7.5 实时命令截获 ≈ Unix 信号处理
- **类比**：GRBL 在串口 ISR 中截获特殊字符（`?` `~` `!`），设置 volatile 标志，主循环在安全点检查执行——和 Unix 信号（SIGINT/SIGTERM）的 "延迟执行" 模式一致：中断只设置标志，不执行实际逻辑。
- **局限**：Unix 信号可以在任何用户代码位置触发，存在竞态条件，限制了信号处理函数能做的事情。GRBL 的实时截获只在串口 ISR 完成，主循环只在明确定义的检查点（`protocol_execute_realtime()`）查看标志，比信号处理更可控、更安全，但实时性也略低——从字符到达 ISR 到主循环处理，最多会延迟一个串口行传输的时间。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：ISR（中断服务程序）
- **一句话**：CPU 响应硬件事件的特殊函数，执行完自动返回打断点。
- **类比**：JS 中的 Promise.then 回调——主程序继续执行，事件来了自动处理。
- **场景**：GRBL 的 Timer1 中断每 33 微秒输出一个步进脉冲。
- **坑**：ISR 里不能阻塞、不能调用 printf 等重入不安全函数，越短越好。

### 卡片 2：Bresenham 线段算法
- **一句话**：只用整数加减法就能在离散网格上画出直线步进脉冲序列的算法。
- **类比**：在像素屏幕上画一条直线——每次决定走 X 还是走 Y。
- **场景**：GRBL 的 ISR 中同时控制 X/Y/Z 三轴脉冲，确保运动轨迹是直线。
- **坑**：多轴场景下非主导轴容易出现步进不均匀的"抖动"，需要用 AMASS 或类似机制平滑。

### 卡片 3：环形缓冲（Ring Buffer）
- **一句话**：固定大小的循环数组，生产者和消费者通过头尾指针解耦。
- **类比**：Java 中的 ArrayBlockingQueue。
- **场景**：规划器块缓冲（18 个块）、段缓冲（32 个段）都是环形缓冲。
- **坑**：满时有两种策略——覆盖旧数据或阻塞生产者；GRBL 选择阻塞。必须精确计算深度以避免死锁。

### 卡片 4：前瞻规划（Look-ahead）
- **一句话**：提前查看未来若干运动指令，规划出平滑的加减速曲线以避免机械冲击。
- **类比**：Web 应用的 CDN 缓存——提前拿到后续内容做优化决策。
- **场景**：`planner_recalculate()` 在每次新块入队时从头到尾反向重算连接速度。
- **坑**：前瞻深度受限（18 个块），一旦缓冲区满，主循环阻塞，串口数据会积压。

### 卡片 5：脉冲宽度（Pulse Width / Step Pulse）
- **一句话**：步进电机驱动芯片要求 step 引脚保持高电平的微秒数。
- **类比**：HTTP keep-alive 的 timeout 参数——太短连接断，太长浪费资源。
- **场景**：`settings.pulse_microseconds` 默认 10 微秒，GRBL 用 Timer0 精确控制。
- **坑**：不同驱动芯片要求不同脉冲宽度（1~20 微秒），设置错误会导致电机不动或发热。

### 卡片 6：定时器输出比较匹配（CTC）模式
- **一句话**：定时器计数到达预设值时自动触发中断或翻转引脚。
- **类比**：setInterval(fn, N) —— 每 N 毫秒执行一次 fn。
- **场景**：Timer1 的 OCR1A 匹配触发 ISR，产生周期性步进脉冲基准频率。
- **坑**：定时器预分频器和 OCR 值需要精确配合；主频变化（如用内部 RC 振荡器而不是外部晶振）会导致步进频率偏差。

### 卡片 7：门型速度剖面（Trapezoidal Speed Profile）
- **一句话**：运动过程分成加速段、匀速段、减速段，形状像梯形。
- **类比**：HTTP/2 流量控制从慢启动到稳定到退出。
- **场景**：GRBL 规划器为每个运动块计算加速/匀速/减速三阶段的步进数。
- **坑**：短距离运动可能只有加速和减速（三角形剖面），没有匀速段。

### 卡片 8：归零/原点开关（Homing Limit Switch）
- **一句话**：机械位置回零操作——先快速向限位开关移动，触碰到后减速退出再慢速接近确定精确零点。
- **类比**：Git 仓库的初始 commit——建立参照基准。
- **场景**：开机后 $H 命令触发归零周期，GRBL 的 limits.c 控制整个过程。
- **坑**：归零方向、搜索速度、碰触后退出距离、第二慢速接近距离都需要调参。错误配置会导致撞坏限位开关或归零位置不准确。

### 卡片 9：宏定义编译剪裁
- **一句话**：通过 #define / #ifdef 在编译阶段选择启用或禁用功能。
- **类比**：Webpack tree shaking —— 去掉不用的代码。
- **场景**：config.h 中定义 `VARIABLE_SPINDLE`、`USE_LINE_NUMBERS`、`ADAPTIVE_MULTI_AXIS_STEP_SMOOTHING` 等，被 #ifdef 在多个 .c 文件中引用。
- **坑**：配置宏的排列组合太多，不同组合之间可能产生未测试的交互 bug。编译时看不到死代码，但 Flash 只有 32KB，需要精确计算每个宏使能的 Flash 增量。

### 卡片 10：volatile 关键字与原子访问
- **一句话**：告诉编译器这个变量可能在中断中被修改，不要做读优化缓存。
- **类比**：Java 中的 volatile 关键字——保证可见性。
- **场景**：`sys_rt_exec_state`、`sys.position[]` 同时被主循环和 ISR 读写，加上 volatile 防止编译器优化掉读取。
- **坑**：C 的 volatile 不保证原子性——一个 32 位变量在 8 位 AVR 上需要多个指令读取，可能被中断破坏。需要额外用 ATOMIC_BLOCK 或关闭中断访问。

## 10. 3 道深度追问题

### 问题 1（时序/并发/资源限制）
GRBL 的规划器块缓冲只有 18 个条目。如果一个复杂 G 代码文件包含大量极短的直线微段（例如圆弧插补产生的 0.1mm 微段），会发生什么？从流水线角度分析主循环为什么可能阻塞，以及阻塞时串口数据会怎样？

### 问题 2（软件工程视角转嵌入式视角）
在 Web 后端开发中，如果请求处理速度跟不上到达速度，典型的方案是水平扩展（加更多实例）或者异步队列（Kafka 积压）。GRBL 运行在单个 AVR 芯片上，既不能水平扩展，也不允许无线积压（段缓冲只有 32 个条目）。请从"嵌入式单机硬实时"的角度，评价为什么缓冲深度限制是一个有意设计，而不是一个缺陷。

### 问题 3（工程判断）
假设你要把 GRBL 移植到另一个 MCU（比如 STM32F103）上，除了重写 `cpu_map.h` 的引脚映射，你还必须修改哪些核心点？提示：考虑定时器架构差异、Flash/SRAM 容量差异、中断优先级数量差异、以及外设寄存器接口差异。

## 12. 工程点评

- **可维护性**：⭐⭐⭐⭐（4/5）。模块划分清晰——serial、gcode、planner、stepper、motion_control 各司其职。但单个 gcode.c 接近 60KB，包含数千行的状态机，维护难度大。config.h 把所有编译配置集中在一起的设计值得学习。

- **可移植性**：⭐⭐⭐（3/5）。硬编码的 AVR 寄存器操作严重。虽然通过 cpu_map.h 隔离了引脚映射，但中断号、定时器配置、GPIO 寄存器地址仍然与 AVR 绑定。移植到 ARM Cortex-M 需要重写 stepper.c 和 serial.c 的硬件相关部分。

- **可靠性**：⭐⭐⭐⭐⭐（5/5）。得益于裸机设计，没有调度器不确定性。ISR 最坏执行时间 25 微秒（<33 微秒周期），有明确的数学保证。实时命令截获通过中断实现零延迟响应。限位开关和急停都是最高优先级中断。

- **可测试性**：⭐⭐（2/5）。硬件耦合度高，难以做纯软件单元测试。虽然有 Simulator 项目，但非官方维护的。`#ifdef` 编译剪裁使得同一个代码库产生多个变体，测试覆盖率难以保证。

- **量产/运维角度**：⭐⭐⭐⭐（4/5）。GRBL 被大量商业 CN芯片 产品采用，参数通过 `$` 命令可调，不需要重新编译。EEPROM 保存设置，掉电不丢。但缺少在线固件升级（OTA）能力，固件更新需要通过 ISP 编程器或 Arduino 引导加载程序。

## 13. 明天可以延展的方向

1. **GRBL 的 Gnea 分支（gnea/grbl）**：目前更活跃的维护分支，支持了更多 CPU 和功能。可以对比 grbl/grbl 的差异，理解一个开源项目如何演进。
2. **GrblHAL**：一个将 GRBL 核心移植到硬件抽象层（HAL）的分支，让你能在不同 MCU（STM32、Teensy、Raspberry Pi Pico）上运行 GRBL。这是理解"嵌入式可移植性"的最佳下一站。
3. **基于位置环的闭环控制**：GRBL 是开环系统（只发脉冲，不读取编码器反馈）。下一步可以研究"步进电机 + 编码器 -> 闭环位置控制"的经典嵌入式控制工程话题。
