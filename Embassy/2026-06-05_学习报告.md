# 每日嵌入式学习：Embassy

## 1. GitHub 项目链接与一句话介绍

- 仓库链接：https://github.com/embassy-rs/embassy
- 一句话介绍：Embassy 是一个用 Rust async/await 构建嵌入式应用的框架，目标是在无操作系统、无堆分配、低功耗的微控制器上，更安全、更高效地组织并发任务、硬件驱动、网络、USB、时间和启动升级能力。

## 2. 为什么今天选它

Embassy 今天值得选，首先是因为它近期仍然活跃。通过仓库浅克隆看到最新 HEAD 为 `4d3cb9b`，提交时间是 2026-06-05 08:29:33 +0000，提交信息为合并 USB DFU manifest 修复相关 Pull Request。这说明它不是一个停留在概念阶段的实验仓库，而是在持续处理真实外设、协议和启动升级细节。

它的工程完整度也很高。仓库根目录下包含 `embassy-executor`、`embassy-time`、`embassy-stm32`、`embassy-nrf`、`embassy-rp`、`embassy-net`、`embassy-usb`、`embassy-boot`、`embassy-sync` 等多个 crate，还包含大量 examples 和 tests。对软件工程师来说，这不是只看一个 LED 闪烁 demo，而是能看到一个嵌入式框架如何把运行时、硬件抽象、定时器、通信协议、启动加载、芯片族适配和测试样例组合成工程体系。

它特别适合从纯软件转向嵌入式的人学习，因为 Embassy 把很多熟悉的软件工程概念带进了微控制器世界：异步任务像协程，HAL 像平台适配层，feature flag 像编译期配置，executor 像轻量调度器，网络和 USB 栈像基础设施组件。但它又不断提醒你，嵌入式不是“把后端搬到小机器上”：这里没有无限内存，没有垃圾回收，任务存储是静态分配，CPU 空闲时要睡眠，定时精度受硬件定时器影响，很多错误要在编译期和链接期暴露。

## 3. 项目解决的问题

真实产品场景可以这样理解：你要做一个电池供电的传感器、蓝牙设备、USB 外设、联网控制器或工业小网关。它需要同时做多件事，比如周期采样、响应 GPIO 中断、维护网络连接、处理 USB 请求、写 flash、低功耗休眠和唤醒。如果用传统裸机写法，很容易变成中断、状态机和全局变量交织在一起；如果直接上 RTOS，又要处理任务栈大小、上下文切换、锁和优先级配置。Embassy 要解决的就是这类“资源很小但并发很多”的产品工程问题。

和普通软件项目相比，它多了几类约束。硬件约束包括不同芯片的寄存器、外设版本、DMA 能力、低功耗模式和中断机制都不一样。资源约束包括 RAM 很小，不能随便分配堆内存，也不能给每个任务开一个大栈。时序约束包括定时器 tick、唤醒延迟、中断响应、低功耗唤醒和通信协议超时都可能影响功能正确性。可靠性约束包括设备可能长期无人值守运行，断电、升级失败、通信异常、时钟漂移和 flash 磨损都必须提前考虑。

## 4. 技术架构分析

从公开文件看，Embassy 的主线可以分成几层。第一层是异步执行器，也就是 `embassy-executor`。README 明确说明它面向嵌入式 async/await，特点是 no alloc、无需堆、任务静态分配、任务大小在编译期计算、链接期暴露 RAM 不足问题、无忙等轮询，空闲时通过中断或 WFE/SEV 让 CPU 睡眠，并且支持多个 executor 实例以表达不同优先级。

第二层是时间系统，也就是 `embassy-time`。它提供 `Instant`、`Duration`、`Timer`、`Ticker`、`with_timeout` 和 `with_deadline` 等抽象。对软件工程师来说，这像是把后端服务里的 sleep、timeout、deadline 变成嵌入式可用的 Future；但底层必须绑定到硬件 time driver，tick 频率、16 位定时器溢出、低功耗 RTC 精度都会影响实际行为。

第三层是 HAL，也就是硬件抽象层。以 `embassy-stm32` 为例，它试图覆盖所有 STM32 芯片家族，公开文件说明其做法是通过 feature flag 指定芯片，依靠 `stm32-metapac` 和 `stm32-data` 在编译期生成寄存器类型，再根据外设版本选择正确实现。`embassy-stm32/src/lib.rs` 里可以看到大量按条件编译打开的外设模块，例如 gpio、dma、rcc、timer、adc、can、eth、flash、i2c、spi、usart、usb、rng、rtc 等。这体现了嵌入式工程的典型特征：同一个“驱动 API”背后是大量芯片差异和编译期裁剪。

第四层是协议与产品组件，包括 `embassy-net`、`embassy-usb`、`embassy-boot`、`embassy-sync` 等。网络栈处理 Ethernet、IP、TCP、UDP、ICMP、DHCP 等；USB 相关 crate 覆盖设备、主机、DFU、日志等；boot 相关 crate 则进入固件升级和启动可靠性问题。资料不足的部分需要标注为基于公开目录和 README 推断：这些模块共同把 Embassy 从“异步运行时”扩展成“可做实际产品的嵌入式框架”。

典型运行流程可以概括为：应用使用 `#[embassy_executor::main]` 标记异步入口，初始化芯片配置和外设，拿到 `Spawner` 后启动任务；任务通过 `.await` 等待定时器、外设中断、网络事件或同步信号；executor 只轮询被唤醒的任务；没有任务可运行时，平台实现让 CPU 睡眠；硬件中断或 timer queue 再把任务唤醒。这个流程与普通 async 服务器相似，但它的调度、内存和唤醒都要与 MCU 硬件绑定。

## 5. 源码陪读：今天只读哪几个文件

- `README.md`：先读项目定位，重点理解 Rust、async、no runtime、no garbage collector、低功耗、HAL、网络、USB、boot 这些关键词如何组成 Embassy 的产品主线。
- `embassy-executor/README.md`：第二步读执行器设计，重点看 no alloc、静态任务、集成 timer queue、无 busy-loop polling、多 executor 优先级这些设计目标。
- `embassy-executor/src/raw/mod.rs`：第三步读 raw executor，重点看 `TaskHeader`、`TaskStorage`、任务状态流转、timer queue item 和 poll 逻辑。
- `embassy-executor/src/raw/run_queue.rs`：第四步读运行队列，重点看原子队列、批量 dequeue、公平性，以及 scheduler-priority / scheduler-deadline feature 下的排序逻辑。
- `embassy-time/src/timer.rs` 与 `examples/stm32wl55cm4-lp/src/bin/blinky.rs`：最后读时间抽象和一个低功耗 blink 示例，理解 `Timer::after_millis(...).await` 如何从应用层落到硬件时间和睡眠唤醒。

推荐阅读顺序是先 README，后 executor README，再 raw/mod.rs，再 run_queue.rs，最后把 timer.rs 和示例连起来看。这样不会一上来陷入 unsafe 和链表细节，而是先建立“为什么要这样设计”的上下文。

## 6. 核心模块/函数导读

`#[embassy_executor::main]` 是应用入口的关键配置点。它把一个 async main 连接到具体平台入口，例如示例中指定 executor 为 `embassy_stm32::executor::Executor`，entry 为 `cortex_m_rt::entry`。这相当于在嵌入式里把启动向量、运行时入口和异步主函数接起来，而不是像普通服务那样由操作系统启动进程。

`Spawner` 和 `SpawnToken` 是任务启动的关键。`spawner.rs` 里说明，调用一个被 `#[embassy_executor::task]` 标记的 async 函数不会立刻运行，而是返回 `SpawnToken`，必须交给 `Spawner::spawn()`。`SpawnError::Busy` 也很有嵌入式味道：默认一个任务一次只能有一个实例运行，要多个实例就需要设置 pool_size，代价是更多 RAM。

`TaskStorage` 是 Embassy 静态内存模型的核心。`raw/mod.rs` 里说明每个任务的 Future 存在 `TaskStorage` 中，而且任务存储必须具有 `'static` 生命周期。任务宏内部会为任务分配静态数组。软件工程师要特别注意，这里不是后台服务里的动态 goroutine 或协程池，而是把任务容量和内存占用提前变成编译期、链接期问题。

`RunQueue` 是调度路径的核心。`run_queue.rs` 显示它使用很简单的 lock-free 链表队列，将任务入队，dequeue 时一次性取出批次处理。注释中特别强调，即便某个任务马上唤醒自己，也不能阻止其他任务获得运行机会。启用 priority 或 deadline scheduler feature 时，它还会把待运行任务放进排序链表，按优先级或 deadline 取出。

`Timer` 和 `with_timeout` 是时序控制的核心。`timer.rs` 中 `Timer::after` 把当前 `Instant::now()` 加上 Duration，形成一个 Future；`with_timeout` 则把业务 Future 和 Timer 一起 poll，谁先完成就返回谁的结果。这里的设计和后端超时很像，但它依赖硬件 tick 和 timer queue，且 README 提醒 STM32 使用 1MHz 默认 tick 时可能让 16 位定时器在临界区延迟超过 32ms 后出问题，因此需要选择更合理的 tick，例如 32.768kHz。

## 7. 软件工程师类比解释

- 异步 executor 像后端里的事件循环。它等待事件、唤醒任务、轮询 Future。这个类比的局限是，后端事件循环背后通常有操作系统、线程和充足内存，而 Embassy 需要直接考虑中断、睡眠指令、静态任务内存和硬件计时。
- HAL 像平台适配层或驱动接口。应用通过稳定 API 操作 GPIO、USART、SPI、I2C 等外设。这个类比的局限是，硬件外设并不总能抽象成完全一致的接口，不同芯片族的寄存器、DMA、时钟树和低功耗限制会让 API 能力不同。
- feature flag 像后端构建配置或按环境开关功能。这个类比的局限是，嵌入式 feature flag 往往直接决定哪些寄存器、外设、芯片支持和内存布局进入最终固件，不只是运行时配置开关。
- Timer timeout 像服务调用超时。这个类比的局限是，嵌入式超时受硬件 timer 精度、tick 频率、中断延迟和低功耗唤醒影响，不能只看代码里的毫秒数。
- 静态任务池像容量固定的连接池。这个类比的局限是，后端连接池容量不足常常可以扩容机器或调配置，而 MCU 上 RAM 固定，容量设计错误可能直接链接失败或运行 Busy。

## 8. 今日 10 个嵌入式概念卡片

### 1. no_std

- 概念名：no_std
- 一句话解释：Rust 程序不依赖完整标准库，适合没有操作系统的 MCU。
- 软件工程类比：像只保留最小运行时的精简容器。
- 常见工程场景：裸机固件、bootloader、低资源传感器节点。
- 容易误解/坑点：no_std 不等于没有库，而是很多依赖 OS 的能力不能直接用。

### 2. no alloc

- 概念名：no alloc
- 一句话解释：不依赖堆分配，减少碎片和运行时内存失败。
- 软件工程类比：像禁止动态扩容的固定容量系统。
- 常见工程场景：长期运行、RAM 很小、可靠性要求高的设备。
- 容易误解/坑点：不用堆不代表不用内存，内存只是转移到静态区、栈或编译期计算。

### 3. 静态任务分配

- 概念名：静态任务分配
- 一句话解释：任务 Future 的存储在编译期或静态区确定。
- 软件工程类比：像预先声明好 worker 数量和每个 worker 的内存预算。
- 常见工程场景：Embassy 的 `#[task]` 和 pool_size。
- 容易误解/坑点：任务越多不是免费并发，会直接增加 RAM 占用。

### 4. 协作式调度

- 概念名：协作式调度
- 一句话解释：任务在 await 点让出执行权，由 executor 轮询被唤醒的任务。
- 软件工程类比：像 Node.js 或 async Rust 服务的事件循环。
- 常见工程场景：等待定时器、串口、网络包、GPIO 中断。
- 容易误解/坑点：如果任务长时间不 await，其他同级任务可能被饿住。

### 5. 中断唤醒

- 概念名：中断唤醒
- 一句话解释：硬件事件通过中断让睡眠中的 CPU 或等待中的任务继续运行。
- 软件工程类比：像 epoll 收到 socket 可读事件。
- 常见工程场景：串口收到数据、定时器到期、GPIO 边沿触发。
- 容易误解/坑点：中断里不能随意做重活，否则会影响时序和系统响应。

### 6. Timer queue

- 概念名：Timer queue
- 一句话解释：管理多个定时 Future，到期后唤醒对应任务。
- 软件工程类比：像服务端定时任务小根堆或延迟队列。
- 常见工程场景：周期采样、通信超时、低功耗唤醒。
- 容易误解/坑点：定时精度取决于硬件 timer 和 tick 配置，不只是 API 名字。

### 7. HAL

- 概念名：硬件抽象层 HAL
- 一句话解释：用安全、统一的接口包装寄存器和外设操作。
- 软件工程类比：像数据库驱动或云厂商 SDK。
- 常见工程场景：GPIO、SPI、I2C、USART、ADC、DMA 驱动。
- 容易误解/坑点：抽象层不能消除所有硬件差异，仍需查具体芯片文档。

### 8. DMA

- 概念名：DMA
- 一句话解释：让外设和内存之间搬数据，不总是占用 CPU。
- 软件工程类比：像旁路的数据搬运通道。
- 常见工程场景：高速串口、SPI 屏幕、ADC 采样、网络收发。
- 容易误解/坑点：DMA 也有对齐、缓存一致性、生命周期和并发访问问题。

### 9. 低功耗睡眠

- 概念名：低功耗睡眠
- 一句话解释：无任务可运行时让 MCU 进入低功耗状态，等待中断唤醒。
- 软件工程类比：像服务空闲时自动降频或暂停 worker。
- 常见工程场景：电池设备、间歇采样、远程传感器。
- 容易误解/坑点：睡眠会影响调试、时钟、外设可用性和唤醒延迟。

### 10. 编译期芯片选择

- 概念名：编译期芯片选择
- 一句话解释：通过 feature flag 指定目标芯片，生成或启用对应寄存器和外设代码。
- 软件工程类比：像为不同平台构建不同二进制。
- 常见工程场景：`embassy-stm32` 按 STM32 型号选择 HAL 实现。
- 容易误解/坑点：选错芯片 feature 不只是功能错误，可能导致寄存器、引脚和外设能力完全不匹配。

## 12. 工程点评

从可维护性看，Embassy 的模块拆分非常清晰，executor、time、HAL、network、USB、boot、sync 各自成 crate，便于学习和复用。它用 Rust 类型系统和宏把很多错误提前到编译期，这是可维护性的强项，但也提高了初学者理解 feature flag、生命周期、unsafe 边界和生成代码的门槛。

从可移植性看，它通过 HAL 和 chip-specific feature 支持多个芯片族，尤其 STM32 的设计体现了“外设版本复用加寄存器生成”的思路。局限是嵌入式可移植性永远不是完全免费，不同芯片的低功耗、DMA、外设细节和文档差异仍会穿透抽象层。

从可靠性看，no alloc、静态任务、链接期暴露 RAM 不足、无 busy-loop polling、低功耗唤醒和 timeout 机制都很适合长期运行设备。需要警惕的是 async 并不自动保证实时性，任务是否及时 await、优先级是否合理、临界区是否过长、timer tick 是否匹配硬件，仍然要靠工程判断。

从可测试性看，仓库包含大量 examples 和 tests，且 crate 边界清楚，有利于分别验证 executor、HAL 和协议模块。但嵌入式测试仍受硬件矩阵限制，单靠桌面测试无法覆盖所有芯片外设、低功耗模式和中断时序。

从量产和运维看，Embassy 的 `embassy-boot`、USB DFU、日志和网络相关模块给产品化留下了空间。真正进入量产时，还要继续关注固件升级回滚、掉电保护、日志可观测性、版本兼容、设备唯一身份、功耗曲线和现场故障恢复。

- 深入 `embassy-net`：学习无操作系统环境下的 TCP/IP 栈、DHCP、socket 抽象和异步网络并发。
- 深入一个具体 HAL，例如 `embassy-stm32` 的 GPIO、USART 或 DMA：从应用 API 一路追到寄存器配置和中断处理。
