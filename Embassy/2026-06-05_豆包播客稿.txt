今天我们学习的项目是 Embassy。它是一个用 Rust 异步能力来构建嵌入式应用的框架，目标是在没有完整操作系统、没有垃圾回收、甚至不依赖堆分配的微控制器上，让开发者更安全、更高效地组织并发任务、硬件驱动、网络、USB、时间管理和启动升级能力。如果你过去主要写后端、全栈或者基础设施软件，可以把 Embassy 先理解成一个面向微控制器的异步运行时加硬件平台框架；但马上要补一句，这个类比并不完整，因为嵌入式里的并发不是运行在资源充足的服务器上，而是运行在 RAM 很小、功耗敏感、时序敏感、还要直接面对中断和寄存器的设备上。

今天选择 Embassy，有一个很直接的原因：它不是一个停留在概念阶段的仓库。通过公开仓库的浅克隆可以看到，它在二零二六年六月五日仍然有新的合并提交，提交内容还涉及 USB DFU manifest 的修复。也就是说，它仍在处理真实工程里的外设、协议和固件升级细节。更重要的是，它的工程完整度很高。仓库里不仅有异步执行器，还有时间系统、STM32、Nordic、RP 系列等硬件抽象层，还有网络、USB、启动加载、同步工具、大量示例和测试。对一个正在从纯软件转向软硬一体的人来说，这样的项目比单独看一个 LED 闪烁程序更有价值，因为你能看到一个嵌入式框架怎样把运行时、硬件抽象、定时器、通信协议、启动升级、芯片族适配和测试样例组合成一个工程体系。

Embassy 要解决的问题，可以放到一个真实产品场景里理解。假设你要做一个电池供电的传感器、蓝牙设备、USB 外设、联网控制器或者工业小网关。它可能要周期性采样，要响应按键或者 GPIO 中断，要维护网络连接，要处理 USB 请求，要写 flash，还要在空闲时进入低功耗睡眠。用传统裸机写法，这些逻辑很容易变成中断、状态机和全局变量缠在一起。直接上实时操作系统，又要面对任务栈大小、上下文切换、锁、优先级和调度配置。Embassy 的核心思路，是把 Rust 的 async 和 await 带到这个场景里，让等待定时器、等待外设、等待网络事件这些动作都变成自然的异步任务，同时又尽量避免堆分配和运行时内存失败。

和普通软件项目相比，Embassy 所处的环境多了几类硬约束。第一类是硬件约束，不同芯片的寄存器、外设版本、DMA 能力、低功耗模式和中断机制都不一样。第二类是资源约束，微控制器的 RAM 很小，不能随便分配内存，也不能给每个任务开一个很大的独立栈。第三类是时序约束，定时器 tick、唤醒延迟、中断响应、低功耗唤醒、通信协议超时，都会影响功能是否正确。第四类是可靠性约束，设备可能长期无人值守运行，断电、升级失败、通信异常、时钟漂移和 flash 磨损，都不是上线之后再慢慢修的小问题，而是在设计阶段就要考虑。

从架构上看，Embassy 可以分成几层。第一层是异步执行器，也就是 embassy executor。它面向嵌入式 async 和 await，强调不需要堆分配，任务静态分配，任务大小在编译期计算。如果任务放不进 RAM，问题尽量在链接期暴露，而不是运行一段时间后突然崩掉。它还强调没有忙等轮询，CPU 在没有任务可运行时可以睡眠，再由中断或硬件事件唤醒。对后端工程师来说，这有点像事件循环；但在这里，事件循环背后不是操作系统帮你兜底，而是要和中断、睡眠指令、硬件定时器以及静态内存布局配合。

第二层是时间系统，也就是 embassy time。它提供 Instant、Duration、Timer、Ticker、with timeout 和 with deadline 这些抽象。你可以把 Timer after millis await 理解成嵌入式里的异步 sleep，把 with timeout 理解成给一个异步操作加超时。但这里的时间不是云服务器里的系统时钟那么简单，它要绑定到硬件 time driver。tick 频率、十六位定时器溢出、临界区延迟、低功耗 RTC 精度，都会影响最终表现。比如 STM32 相关说明里就提醒，默认一兆赫 tick 对某些十六位定时器可能太激进，如果临界区让中断延迟超过几十毫秒，就可能出问题，所以实际工程里要选择更合理的 tick，例如三十二点七六八千赫这种低功耗场景常见频率。

第三层是硬件抽象层，也就是 HAL。以 embassy stm32 为例，它试图覆盖大量 STM32 芯片家族。公开文件里说明，它的做法是通过编译期 feature flag 指定芯片，再依靠芯片数据和寄存器描述生成对应类型，最后由 HAL 根据外设版本选择正确实现。你可以把 HAL 类比成数据库驱动或者云厂商 SDK，应用通过相对稳定的接口操作 GPIO、串口、SPI、I2C、ADC、DMA 等外设。但这个类比的局限是，硬件外设并不总能被抽象得完全一致。不同芯片的时钟树、DMA 通道、引脚复用、低功耗限制和外设版本，都会穿透抽象层，变成你必须理解的工程事实。

第四层是协议与产品组件。Embassy 仓库里有网络栈、USB 相关模块、boot 启动加载相关模块和同步工具。网络栈覆盖以太网、IP、TCP、UDP、ICMP、DHCP 等能力；USB 模块覆盖设备、主机、DFU 和日志等方向；boot 模块则进入固件升级、启动可靠性和断电保护这些产品化问题。基于公开目录和说明可以推断，Embassy 的定位并不只是一个小型 async demo，而是希望成为能支撑真实产品开发的嵌入式框架。

一个典型运行流程可以这样想象。应用用 embassy executor main 标记异步入口，在入口里初始化芯片配置和外设，然后拿到 Spawner 启动任务。任务里通过 await 等待定时器、外设中断、网络事件或者同步信号。执行器只轮询被唤醒的任务；没有任务可运行时，平台实现让 CPU 睡眠；硬件中断或者 timer queue 再把任务唤醒。这个流程和普通 async 服务器确实相似，但嵌入式特有的地方在于，任务数量、任务内存、唤醒来源和睡眠策略都必须提前设计，不能假设资源无限。

今天的源码陪读建议从项目说明开始，不要一上来钻进 unsafe 细节。第一步读 README，抓住 Rust、async、无运行时、无垃圾回收、低功耗、HAL、网络、USB 和 boot 这些关键词。第二步读 embassy executor 的说明，重点看不依赖堆、静态任务、集成 timer queue、无忙等轮询、多 executor 优先级这些设计目标。第三步读 executor 的 raw 模块，关注 TaskHeader、TaskStorage、任务状态流转、timer queue item 和 poll 逻辑。第四步读 run queue，也就是运行队列，看看它怎样把被唤醒的任务入队，怎样批量取出任务，怎样保证一个任务即使频繁唤醒自己，也不能完全阻止其他任务运行。最后把 embassy time 的 timer 文件和一个 STM32 低功耗 blink 示例连起来看，理解 Timer after millis await 为什么不是普通阻塞 sleep，而是把执行权交还给执行器，让系统有机会睡眠和唤醒。

这里有几个核心点值得特别说明。第一个是应用入口。示例里可以看到，异步 main 会通过宏连接到具体平台的 executor 和芯片启动入口。这在后端里可能只是 main 函数启动服务，但在嵌入式里，它实际上连接了启动向量、运行时入口、芯片初始化和异步执行器。第二个是 Spawner 和 SpawnToken。调用一个被 task 宏标记的异步函数，并不代表任务立刻开始运行，它先返回一个代表待启动任务的 token，然后必须交给 Spawner 去启动。这个设计让任务启动成为一个显式动作，也让任务实例数量可以被控制。

第三个核心点是 TaskStorage。Embassy 的任务 Future 存在静态任务存储里，任务存储需要长期存在。宏内部会为任务准备静态存储，默认一个任务一次只有一个实例在运行。如果你想允许多个实例，就要设置 pool size，但代价是更多 RAM。如果容量不够，你会遇到 Busy 或者链接期内存问题。这和后端里请求来了就开一个协程的思维不同。在微控制器上，任务并发不是免费的，任务数量本身就是内存预算的一部分。

第四个核心点是运行队列。Embassy 的 run queue 使用很简单的链表队列。任务被唤醒后入队，执行器批量取出待运行任务，然后逐个 poll。注释里有一个很重要的公平性判断：即使某个任务马上唤醒自己，它也不能阻止当前批次里的其他任务获得运行机会。启用优先级或者 deadline 调度特性时，队列还可以按优先级或截止时间排序。这提醒我们，嵌入式里的调度策略不仅影响性能，还直接影响通信超时、控制响应和低功耗唤醒。

第五个核心点是 Timer 和 timeout。Timer after 会把当前时间加上一段 Duration，形成一个到期后完成的 Future。with timeout 则把业务 Future 和 Timer 一起轮询，业务先完成就返回成功，Timer 先完成就返回超时。后端工程师很熟悉这个模式，但嵌入式里的超时要更加谨慎。你写下五百毫秒，并不代表设备一定以理想精度在五百毫秒醒来。硬件 timer、tick 频率、临界区、中断优先级和低功耗模式都会影响这个数字。

用软件工程类比，Embassy 里的异步 executor 像事件循环，HAL 像平台适配层，feature flag 像构建配置，Timer timeout 像服务调用超时，静态任务池像容量固定的连接池。但每个类比都有局限。事件循环在服务器上通常有操作系统和内存兜底，在 MCU 上没有。平台适配层在软件里可以隐藏大部分差异，在硬件里却会被时钟、DMA、引脚和外设版本打破。构建配置在后端常常只是开关功能，在嵌入式里会决定寄存器、外设和最终固件内容。超时在服务端主要受系统调度和网络影响，在嵌入式里还受硬件 tick 和唤醒延迟影响。连接池在服务器上可以通过扩容缓解，在 MCU 上则直接受固定 RAM 限制。

今天可以记住十个概念。no std 表示程序不依赖完整标准库，适合没有操作系统的 MCU，但它不等于没有库，而是很多依赖操作系统的能力不能直接用。no alloc 表示不依赖堆分配，可以减少碎片和运行时内存失败，但内存只是转移到静态区、栈或者编译期计算，并不是消失。静态任务分配表示任务 Future 的存储提前确定，任务越多就越占 RAM。协作式调度表示任务在 await 点让出执行权，如果一个任务长时间不 await，同级任务就可能被饿住。中断唤醒表示硬件事件让睡眠中的 CPU 或等待中的任务继续运行，但中断里不能随便做重活。Timer queue 负责管理多个定时 Future，到期后唤醒对应任务，但精度取决于硬件和 tick。HAL 用安全接口包装寄存器和外设，但不能抹平所有芯片差异。DMA 让外设和内存之间搬数据，不总占用 CPU，但它会带来对齐、缓存一致性和生命周期问题。低功耗睡眠让设备空闲时省电，但会影响调试、时钟、外设可用性和唤醒延迟。编译期芯片选择通过 feature flag 决定目标芯片和外设代码，选错不只是配置错误，可能让寄存器、引脚和能力完全不匹配。

今天的动手任务不需要硬件。你可以用三十到六十分钟完成一次架构阅读和任务内存模型练习。先读项目说明和 executor 说明，摘出不依赖堆、静态任务、Timer、低功耗和多个 executor 五个关键词。再读 spawner 相关源码，画出 task function 到 SpawnToken，再到 Spawner spawn，再到 raw executor 的调用链。接着读 TaskHeader 和 TaskStorage，标出 state、run queue item、executor、poll function、timer queue item 和 future 分别负责什么。然后读一个 blink 示例，解释为什么 Timer after millis await 不是阻塞 sleep，而是给执行器和低功耗机制留下空间。最后假设你要增加一个每十秒采样一次传感器并通过串口输出的任务，写一段伪代码，说明需要一个 task、一个 Timer、一个外设句柄，并思考如何避免多个任务同时占用串口。

验收标准也很明确。你应该能用自己的话说明，Embassy 为什么不需要为每个任务配置独立任务栈；能解释 SpawnError Busy 和 pool size 的关系；能指出 async 任务如果长时间不 await 会带来什么调度问题；还能说清楚低功耗设备里 Timer 的精度为什么和硬件时钟有关。如果今天没有板子，也完全可以只做源码阅读和伪代码。今天的重点不是跑通某块开发板，而是建立工程模型。

最后留三个追问。第一个，如果一个 Embassy 任务在循环里持续做 CPU 计算，几百毫秒都没有 await，同一个 executor 上的其他任务会发生什么？如果它负责通信协议超时，会带来哪些时序风险？第二个，从后端视角看，你可能习惯请求来了就开一个 task；在 Embassy 里为什么不能无限 spawn？pool size、RAM、链接期检查和 Busy 分别对应什么工程判断？第三个，如果你要把同一套应用从 STM32 移到 RP2040 或 nRF52，你希望 HAL 帮你屏蔽什么？又有哪些部分必须重新评估，比如时钟树、DMA、中断、低功耗、flash 和引脚复用？

从工程角度点评，Embassy 的可维护性很强，模块拆分清楚，executor、time、HAL、network、USB、boot 和 sync 各自成体系。它用 Rust 类型系统、宏和编译期配置把很多错误提前暴露，这是优点，但也提高了初学者理解 feature flag、生命周期、unsafe 边界和生成代码的门槛。可移植性方面，它通过 HAL 和芯片 feature 支持多个芯片族，尤其 STM32 的做法体现了外设版本复用和寄存器生成的思路，但嵌入式移植永远不是免费午餐，低功耗、DMA、外设细节和文档差异仍然会穿透抽象。可靠性方面，不依赖堆、静态任务、链接期暴露内存不足、无忙等轮询和低功耗唤醒都很适合长期运行设备。需要警惕的是，async 并不自动等于实时，任务是否及时 await、优先级是否合理、临界区是否过长、timer tick 是否匹配硬件，仍然要靠工程判断。

今天的复盘问题是：你以前理解的并发，可能主要是在操作系统和云资源充足的环境里成立。当并发发生在一个只有几十 KB 到几百 KB RAM、需要睡眠省电、还要响应中断的 MCU 上时，你会如何重新定义任务数量、任务边界和超时策略？明天可以继续沿着三个方向延展：一是深入 boot，学习固件升级、双分区、回滚和断电保护；二是深入网络栈，学习无操作系统环境下的 TCP/IP、DHCP 和异步网络并发；三是选一个具体 HAL，比如 STM32 的 GPIO、串口或者 DMA，从应用接口一路追到寄存器配置和中断处理。
