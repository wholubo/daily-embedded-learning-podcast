# 每日嵌入式学习：MCUboot

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**: https://github.com/mcu-tools/mcuboot
- **一句话介绍**: MCUboot 是一个为 32 位微控制器设计的通用安全引导加载程序，让设备可以在运行时安全地升级固件，同时验证启动镜像的完整性和真实性，防止被篡改或被刷入恶意固件。

## 2. 为什么今天选它

MCUboot 是目前物联网设备中最广泛使用的开源安全引导加载程序之一，由 Linaro、JUUL Labs、Arm 和 Nordic Semiconductor 等公司持续维护。截至 2026 年 6 月，项目有 1927 个星标、935 个复刻，最新版本 v2.4.0 发布于 2026 年 4 月，上游仍在活跃开发，最近一次代码提交在 2026 年 6 月 12 日。

**适合软件工程师学习嵌入式的理由**：
- OTA（空中升级）是嵌入式产品量产后面临的最核心工程挑战之一，MCUboot 给出了工业级解决方案。
- 代码用纯 C 编写，清晰模块化，没有 RTOS 强依赖，适合理解底层嵌入式设计模式。
- 它抽象了 flash 分区、镜像验证、安全签名、故障注入防护等关键嵌入式概念。
- 软件工程师可以把自己熟悉的"版本管理/签名验证/回滚机制"知识，对照到嵌入式环境下的具体实现。

## 3. 项目解决的问题

### 真实场景

物联网设备部署到用户手中后，固件几乎一定会需要升级——修复漏洞、增加功能、更新配置。不像手机 App 那样有强大的操作系统管理更新，微控制器上通常没有完整文件系统，甚至连用户交互界面都没有。如果升级过程中掉电、网络中断、甚至被恶意攻击，设备就可能变砖。MCUboot 解决了这个"如何在极其受限的环境中安全地更换系统固件"的问题。

### 相比普通软件的额外约束

| 约束维度 | 普通软件（云服务） | 嵌入式（MCUboot 场景） |
|---|---|---|
| 存储介质 | 磁盘/SSD，GB 级随机访问 | Nor Flash，KB～MB 级，按扇区擦写 |
| 升级原子性 | 文件系统保证，可回滚 | 掉电可能卡在擦写中间，需要防断电设计 |
| 安全基础 | 操作系统提供进程隔离 | 无 MMU，攻击者可读整个 flash |
| 回滚机制 | 容器/包管理天然支持 | 需要双区（A/B）或多区布局 |
| 签名验证 | 操作系统或包管理器负责 | 必须在裸机启动时自己完成 |
| 升级触发 | UI 或后台静默 | 往往只通过串口/BLE/网络加载固件 |

## 4. 技术架构分析

### 整体架构

MCUboot 分为两层：

1. **bootutil 库** (`boot/bootutil/`)：核心引导逻辑，包含镜像验证、交换管理、加密解密、TLV 解析等。不依赖具体硬件平台，可做单元测试。
2. **boot 应用** (`boot/<port>/`)：各 RTOS/平台的适配层，处理 flash 驱动、启动跳转、串口恢复等平台相关操作。

### 镜像格式

MCUboot 定义了标准镜像格式：

```
[image_header (32字节)] + [镜像载荷] + [protected TLV区] + [TLV区]
```

- `image_header`：幻数、加载地址、头大小、镜像大小、标志位、版本号
- TLV 区：类型-长度-值序列，包含签名、SHA256 哈希、密钥哈希、安全计数器、依赖声明等
- 保护 TLV（`ih_protect_tlv_size`）：被哈希覆盖但不被加密保护的区域，用于测量启动（measured boot）

### 三种交换策略

MCUboot 支持三种固件交换方式，可在编译时选择：

1. **Swap-Using-Move**（推荐）：将主区内容逐步搬移到临时区，同时把新固件写回主区。抗断电，最多需要额外一个扇区。
2. **Swap-Using-Scratch**（传统）：用一个 scratch 扇区做缓冲区，逐步交换主区和备区。抗断电但擦写次数较大。
3. **Swap-Using-Offset**（新引入）：在备区尾部预留偏移，不需要 scratch 扇区。节省空间，适合存储极度受限的设备。

无论哪种策略，核心都是"先复制/交换一个扇区，记录状态到 status flash，继续下一个扇区"的事务性操作。如果掉电，下次启动时从 status 恢复，继续未完成的交换。

### 镜像验证流程

1. 读取镜像头，检查幻数（`IMAGE_MAGIC = 0x96f3b83d`）
2. 解密（如果开启了加密镜像）
3. 计算镜像的 SHA256/SHA384/SHA512 哈希
4. 从 TLV 区提取签名，用内置公钥验证哈希
5. 检查安全计数器（防回滚攻击）
6. 检查镜像依赖（多镜像系统）

### 故障注入防护

MCUboot 用 `fih_ret` 类型和 `FIH_DECLARE/FIH_ASSERT` 宏实现**故障注入硬化**（Fault Injection Hardening）。关键比较结果用 volatile 变量和双重检查，防止攻击者通过电磁/激光干扰跳过安全检查。

### Flash 分区布局

```
[Bootloader] [Primary Slot] [Secondary Slot] [Scratch Area] [Status Area]
```

- Primary Slot：当前运行的主固件
- Secondary Slot：OTA 下载的新固件
- Scratch Area：交换用的临时缓冲扇区
- Status Area：记录交换进度的 metadata

### 构建系统与工具

- 支持 CMake、Zephyr 构建系统、Mynewt 构建系统
- `imgtool`（Python 脚本）：签名固件镜像、加密密钥管理、生成 TLV 区
- CI 包含模拟器（`sim/`）测试，可在主机上运行完整引导流程测试

## 5. 源码陪读：今天只读哪几个文件

推荐按以下顺序阅读：

**文件 1**: `boot/bootutil/src/bootutil_loader.c`
- **阅读目标**: 理解 MCUboot 的启动入口和决策流程。函数 `boot_check_header_erased`、`boot_check_header_valid`、`boot_read_image_headers`、`boot_check_image` 构成了完整的镜像检查管线。这里的 `boot_go` 函数是整个引导过程的顶层编排器。

**文件 2**: `boot/bootutil/src/swap_move.c`
- **阅读目标**: 理解 MCUboot 推荐的 Swap-Using-Move 策略的具体实现。看 `find_last_idx`、`swap_read_status_bytes`、`boot_read_image_header`（交换中断恢复版）如何保证掉电安全。这是嵌入式"事务性操作在 flash 上的实现"的绝佳案例。

**文件 3**: `boot/bootutil/src/image_validate.c`
- **阅读目标**: 理解镜像验签的全流程。关注 `bootutil_img_hash` 如何计算哈希，以及验证函数如何组装哈希+签名+公钥。注意 `EXPECTED_SIG_TLV` 的编译时配置以及 `MCUBOOT_HW_KEY` 和 `MCUBOOT_BUILTIN_KEY` 两种密钥管理方式。

**文件 4**: `boot/bootutil/src/bootutil_misc.c`
- **阅读目标**: 理解 flash 区域管理、镜像头读取、状态读写等底层辅助函数。这些函数是上面的核心逻辑与底层 flash 驱动之间的桥梁。

**文件 5**: `scripts/imgtool.py`
- **阅读目标**: 理解固件签名的完整流程。这个 Python 工具负责：生成密钥对、签名固件、添加 TLV 区、加密镜像。看完它你就知道一个嵌入式镜像在 PC 端是如何被"打包"成设备可接受的格式的。

## 6. 核心模块/函数导读

### 1. `boot_go()` — `boot/bootutil/src/bootutil_loader.c`
- **角色**: MCUboot 的顶层调度函数，是整个引导程序的入口。它依次调用：读取镜像头 → 检查镜像有效性 → 决定是否交换 → 执行交换 → 跳转。
- **与工程的关系**: 这是"引导流程"的起点，任何系统初始化（时钟、外设、内存）完成之后会调用它。它调用 `boot_validated_swap_copy_go`（实际上执行交换的底层函数），最后用 `context->Bootstrap()` 或直接跳转到主镜像入口。

### 2. `swap_status_entries()` + `swap_read_status_bytes()` — `swap_move.c` / `swap_priv.h`
- **角色**: 管理交换过程的持久化状态。每次交换一个扇区后，更新 status flash 中的一个 entry。下次上电时读取这些 entries 恢复中断的操作。
- **与工程的关系**: 这是"掉电安全"的关键。每次擦写一个扇区后立即写入状态标记，形成一个可恢复的"事务日志"。如果掉电在擦写中间，重启后再读状态就知道哪些扇区已经交换过。

### 3. `bootutil_img_hash()` — `boot/bootutil/src/bootutil_img_hash.c`
- **角色**: 计算整个镜像的哈希值，输入范围包括 `image_header` + 镜像载荷 + protected TLV 区。
- **与工程的关系**: 这是安全验证的第一道防线。哈希值随后被签名算法覆盖，或者直接作为完整性摘要（无签名配置下）。注意这里的哈希计算排除了未保护的 TLV 区，因为签名和密钥 TLV 在验证前尚未被信任。

### 4. `IMAGE_TLV_*` 系列常量 — `boot/bootutil/include/bootutil/image.h`
- **角色**: 定义镜像尾部 TLV 区的类型码，包括签名类型（RSA2048、ECDSA、Ed25519）、加密密钥类型（AES-KW、ECIES）、安全计数器、依赖声明等。
- **与工程的关系**: TLV 格式是 MCUboot 镜像结构的核心设计。它灵活、可扩展——需要添加一个新属性（如硬件绑定、UUID 约束）只需定义新类型码。这种设计类似于 Protocol Buffers 的嵌入式版本，但更轻量。

### 5. `imgtool` CLI — `scripts/imgtool.py`
- **角色**: 主机端工具，对固件进行签名、加密、添加 TLV 元数据、管理密钥。
- **与工程的关系**: 这是整个 OTA 流程的"生产端"。固件在 CI/CD 流水线中先编译，再被 imgtool 签名，最后部署到 OTA 服务器。设备端只有 MCUboot 负责验证。理解 imgtool 的输入输出就能画出完整的 OTA 工程链路。

## 7. 软件工程师类比解释

### 类比 1：镜像格式（Image Header + TLV）→ HTTP 请求头 + body + 自定义头
镜像的 `image_header` 类似 HTTP 请求头（method、path、content-length），TLV 区类似自定义头（X-Signature、X-Digest）。MCUboot 先用 header 解析基本信息，再用 TLV 做安全验证。
- **局限**: HTTP 头是纯文本、可变长度、网络传输；镜像头是固定大小的二进制结构，存在 flash 上，由裸机代码解析，任何解析错误直接导致设备变砖。

### 类比 2：Swap 策略（Move/Offset/Scratch）→ Git 仓库的原子提交
Git 提交的"写对象 + 更新 ref"是原子操作，掉电后 ref 要么指向旧提交要么指向新提交。MCUboot 的 swap 就是把每个扇区的交换当作一个"提交"，status flash 就是"reflog"。
- **局限**: Git 提交在文件系统层面由内核保证原子性；MCUboot 在裸机 flash 层面要自己管理擦写顺序和状态记录，并且必须在无 OS、无文件系统、无 mmu 的环境下完成。

### 类比 3：Fault Injection Hardening（fih_ret）→ Web 应用的输入验证双重检查
Web 后端用双重检查（客户端验证 + 服务端验证）防止绕过。MCUboot 的 FIH 用 volatile 变量、双重检查和延迟随机化防止攻击者通过电磁脉冲跳过签名检查。
- **局限**: Web 双重检查防的是逻辑绕过；FIH 防的是物理层攻击——攻击者用激光照射芯片特定位置让某个 if 判断跳过。这超出绝大多数软件工程师的威胁模型。

### 类比 4：镜像签名验证 → Docker 镜像签名（Docker Content Trust）
Docker 用 Notary 签名镜像，拉取时验证摘要和签名。MCUboot 的验证流程几乎一样：计算镜像哈希 → 用公钥验证签名 → 检查版本防回滚。
- **局限**: Docker 只在拉取时验证一次，运行时由 OS 保护；MCUboot 在每次启动都要验证，而且验证环境（裸机 C 代码）比 Notary（Go + Linux）受限得多，可用内存通常只有 4～32 KB。

### 类比 5：Flash 分区布局 → Kubernetes Pod 中的 Init Container + Sidecar
Primary Slot = main container，Secondary Slot = 新版本待部署，Bootloader = Init Container
- **局限**: Kubernetes 可以创建、删除容器，可以依赖 etcd 做分布式状态管理；MCUboot 只有两个固定分区加一个 scratch 扇区，升级策略只有"替换主区"一种。

## 8. 今日 10 个嵌入式概念卡片

### 1. Secure Boot（安全引导）
- **一句话**: 从硬件复位到主固件运行的整个链路上，每一步都验证下一段的完整性和真实性。
- **软件工程类比**: SSH 登录时验证 host key 的指纹。
- **常见工程场景**: 物联网门锁、汽车 ECU、医疗设备固件启动。
- **容易误解的坑**: 安全引导不等于加密启动。安全引导只是验证签名，不一定加密镜像数据。

### 2. Flash Sector / Page（闪存扇区/页）
- **一句话**: Nor Flash 的最小擦除单位和最小写入单位，通常是 4KB（扇区）和 256B（页）。
- **软件工程类比**: 数据库的页结构——不能只更新一行，要读整个页再写回。
- **常见工程场景**: OTA 固件写入、参数存储。
- **容易误解的坑**: 写入前必须先擦除（把位从 0 变 1 只能通过擦除），而且擦除次数有限（通常 10 万次）。

### 3. Swap（交换升级）
- **一句话**: 将新固件和旧固件在 flash 上逐扇区对调，掉电可恢复。
- **软件工程类比**: kubectl rollout undo 的回滚机制。
- **常见工程场景**: 物联网设备 OTA 升级。
- **容易误解的坑**: 交换会反复擦写同一块 flash 区域，加速 flash 磨损。Swap-Move 策略对此做了优化。

### 4. TLV（Type-Length-Value）
- **一句话**: 一种紧凑的自描述数据格式，MCUboot 用它存放签名、哈希、密钥等尾部元数据。
- **软件工程类比**: Protocol Buffers 的字段编码（field_number + wire_type + value）。
- **常见工程场景**: 镜像签名、文件元数据、网络协议扩展。
- **容易误解的坑**: TLV 的 T 和 L 宽度是固定的（2字节 + 2字节），不是可变长编码，有大小限制。

### 5. Fault Injection（故障注入）
- **一句话**: 通过电磁脉冲、激光、电压毛刺等物理手段干扰芯片运行，试图让安全检查"刚好"跳过。
- **软件工程类比**: 对 Web 后端进行 DDoS 试图让 WAF 失效（但物理攻击的精度高得多）。
- **常见工程场景**: 支付终端、汽车电子、加密钱包的破解攻击。
- **容易误解的坑**: 这不是漏洞，而是物理攻击。软件加固（如双重比较、延迟随机化）只能提高攻击成本，不能完全消除。

### 6. Rollback Protection（回滚防护）
- **一句话**: 禁止设备从高版本固件退回到低版本，防止攻击者利用旧版本已公开的漏洞。
- **软件工程类比**: Kubernetes 强制最低 API 版本，不允许退化。
- **常见工程场景**: 已修复 CVE 的固件更新。
- **容易误解的坑**: 安全计数器（Security Counter）是一维单调递增值，不依赖版本号字符串。如果开发者在 debug 阶段频繁刷不同版本，可能烧掉安全计数器导致无法升级。

### 7. Measured Boot（测量启动）
- **一句话**: 启动时逐层记录各组件的哈希值，不阻止加载（与 Secure Boot 的区别），只供远程验证设备状态。
- **软件工程类比**: CI 构建中记录每个步骤的 checksum 并上传到公证服务器。
- **常见工程场景**: 远程证明（Remote Attestation）、可信计算基线检测。
- **容易误解的坑**: Measured Boot 不阻止恶意固件启动，它只记录。阻止需要 Secure Boot 或 TPM 配合。

### 8. Image Encryption（镜像加密）
- **一句话**: 固件在 flash 中以密文存储，防止通过物理读取 flash 提取固件知识产权。
- **软件工程类比**: 给 Docker 镜像做全加密，只有运行环境的密钥才能解密。
- **常见工程场景**: 防止竞品抄板、保护私有算法。
- **容易误解的坑**: 加密密钥必须在设备端存储或派生，攻击者仍可通过 JTAG/SWD 读取密钥。真正的安全需要硬件安全模块（HSM/SE）。

### 9. Flash Map（Flash 分区表）
- **一句话**: 一种抽象层，把物理 flash 地址映射为逻辑分区（bootloader、primary、secondary、scratch）。
- **软件工程类比**: fstab 或 Kubernetes PV/PVC 的映射关系。
- **常见工程场景**: 多分区设备、OTA 升级方案设计。
- **容易误解的坑**: 分区表通常是硬编码在链接脚本或设备树中的，动态调整极难。设计时必须预先留好冗余空间。

### 10. Boot Status（启动状态区域）
- **一句话**: Flash 中的一小块区域，记录当前交换操作的进度（写了哪个扇区、写到哪一步），掉电后恢复用。
- **软件工程类比**: 数据库的 WAL（Write-Ahead Log）或者 Redis 的 AOF 文件。
- **常见工程场景**: 掉电安全的 OTA 升级。
- **容易误解的坑**: Boot status 区域本身也可能在写的过程中掉电损坏，MCUboot 用冗余写入（每个 status 写两次到不同偏移）来缓解，但不能 100% 保证。

## 9. 30～60 分钟动手任务

### 目标
用 MCUboot 的模拟器（sim）在主机上完整运行一次安全启动流程：签名一个镜像 → 模拟升级 → 验证掉电恢复。

### 步骤

1. **克隆仓库**
   ```bash
   git clone https://github.com/mcu-tools/mcuboot.git
   cd mcuboot
   ```

2. **安装依赖**
   ```bash
   pip install --user cryptography cbor2
   ```

3. **编译模拟器**
   ```bash
   mkdir build && cd build
   cmake -DCMAKE_C_COMPILER=gcc -DMCUBOOT_TARGET=sim ..
   make -j$(nproc)
   ```

4. **生成并签名测试镜像**
   ```bash
   # 用 imgtool 生成一个测试固件并签名
   python3 ../scripts/imgtool.py sign \
     --key ../root-ec-p256.pem \
     --algo ecdsa-p256 \
     --version 1.0.0 \
     --hex-addr 0x10000 \
     --slot-size 0x40000 \
     --max-align 8 \
     --header-size 0x200 \
     <(echo "firmware image payload") \
     signed_image.bin
   ```

5. **运行模拟器测试**

## 10. 3 道深度追问题

### 问题 1（时序/并发/资源限制）
MCUboot 的 Swap-Using-Move 策略在处理交换时，会把主区的旧固件逐扇区搬移到临时区域。假设主区有 16 个扇区，你在交换第 7 个扇区的**中间**掉电了（即已经擦除了该扇区但还没来得及把主区数据写回去）。下次启动时 MCUboot 从 status 恢复——请描述恢复过程如何保证数据一致性，status 的哪个字节对应第 7 扇区的哪个状态。

### 问题 2（软件工程 → 嵌入式视角转换）
在 Web 后端开发中，你需要给用户文件上传功能添加"病毒扫描 + 格式验证"。如何在理念上把这个流程映射到 MCUboot 的"镜像验证 + 签名校验 + 防回滚"流程？注意这个映射必须考虑：嵌入式环境中没有操作系统、没有临时文件系统、也没有网络调用查杀毒。你的映射方案中哪些步骤可以"离线"做（在 PC 端打包时），哪些必须在设备上在线做？

### 问题 3（工程设计判断）
MCUboot 的 imgtool 用 Python 3 编写，依赖 `cryptography` 和 `cbor2` 库。假设你的嵌入式产品 CI/CD 流水线安装在纯内网环境中，不允许 pip 到外网拉取依赖——设计一套方案，让 CI 可以离线完成固件签名。你至少需要包含：如何离线分发密钥、如何在没有 pip 时管理 Python 依赖、如何与 MCUboot 的签名格式兼容。

## 11. 今日复盘问题

你是一个软件工程师，正在为你的第一个嵌入式产品设计 OTA 升级方案。你选择了 MCUboot，但在设计 flash 分区大小时，发现芯片 flash 总共只有 2MB，而你的固件已经有 800KB。按照 A/B 双区 + scratch 的设计，你必须为每个 slot 预留至少 800KB，scratch 区至少 4KB（一个扇区），再加上 bootloader 本身约 64KB——总需求已经接近 2MB。你的产品经理要求再加一个"安全日志分区"来记录 OTA 事件。**是向 PM 要求换更大 flash 的芯片（增加 BOM 成本），还是压缩固件大小（减少功能），还是改造分区方案（例如改用单区 + 压缩升级）？请从技术难度、维护成本和量产风险三个维度论证你的选择。**

## 12. 工程点评

**可维护性**: ★★★★☆
MCUboot 的代码注释完善、模块划分清晰、有模拟器可做自动化测试。每种 RTOS 的移植都有独立目录，新增平台只需实现 `flash_map_backend` 接口。`BOOT_LOG_MODULE_DECLARE` 提供了可按模块开关的日志系统。

**可移植性**: ★★★★★
这是 MCUboot 的最大优势。它用纯 C 编写，不依赖 RTOS，通过 `flash_map_backend` 抽象 flash 操作，已经在 Zephyr、Mynewt、NuttX、RIOT、Mbed、ESP-IDF、Cypress 上验证。主机模拟器进一步降低了移植验证成本。

**可靠性**: ★★★★☆
三种交换策略都抗断电，有完整的 status 恢复机制。Fault Injection Hardening 增加了物理攻击的难度。主要扣分点是 boot status 区域本身在掉电时可能损坏——虽然有冗余写入，但不能完全免疫。

**可测试性**: ★★★★★
模拟器可以在 PC 上完整模拟 flash、swap、签名验证、掉电恢复等全流程。单元测试覆盖 `bootutil` 库的所有核心路径。

**量产/运维**: ★★★★☆
`imgtool` 工具成熟，支持多种签名算法和加密方式。但缺少官方的 OTA 服务器端参考实现——MCUboot 只负责设备端的引导，服务器端的 OTA 协调、设备注册、密钥分发需要自己搭建（如用 Eclipse hawkBit 或 AWS IoT OTA）。
