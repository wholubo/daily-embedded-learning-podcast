# 每日嵌入式学习：ArduPilot

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/ArduPilot/ardupilot
- **一句话介绍**：ArduPilot 是目前最成熟的开源自动驾驶仪软件，能驱动从多旋翼、固定翼飞机到无人车、无人船和潜艇几乎任何类型的无人载具。

## 2. 为什么今天选它

ArduPilot 是嵌入式工程领域的"庞然大物"——它在一个统一的代码库内实现了六种载具类型（Copter、Plane、Rover、Sub、AntennaTracker、Blimp），拥有超过 15,000 颗 GitHub Star，131 个 AP_ 前缀的专业库，持续开发超过 15 年。代码全程用 C++ 编写，运行在从 STM32 到 Linux SBC 等多种硬件平台上。

**为什么适合从软件工程转向嵌入式的人**：
- 它是一个"把软件跑在真实物理世界"的极致案例——控制算法不是模拟输入，而是真实电调、舵机和传感器。
- 它展示了如何用纯 C++ 在资源受限（几百 KB RAM）的 MCU 上构建一个实时控制系统。
- 它的 HAL（硬件抽象层）是软件工程中"依赖反转/接口隔离"原则的教科书级实现。
- 它有完整的 CI/CD、自动测试、参数系统、日志系统，和一个软件工程师会感到熟悉的工程体系——但一切都在实时的、物理安全的约束下运行。

## 3. 项目解决的问题

### 真实产品场景

ArduPilot 解决的核心问题是：**如何让一个飞行器或地面载具，在无人操控或半自动操控下，安全、稳定地完成预定任务**。

具体场景包括：

- **农业无人机**（植保喷洒、测绘）
- **搜救无人机**（热成像搜索、物资投放）
- **水下勘测**（潜艇自主航线测绘）
- **无人船**（水文监测、环境采样）
- **固定翼 BVLOS 超视距飞行**（巡检、物流）

### 与普通软件项目相比多了什么

1. **实时性约束**：控制循环必须在严格时间窗口内完成（ArduCopter 主循环 400Hz，快速线程高达 1-2kHz）。延迟超过 1ms 就可能导致无人机失控。
2. **硬件多样性**：需要在不通用的 MCU 上运行，每种 MCU 有不同的外设、内存布局、中断优先级。AP_HAL 层就是为了解决这个问题存在。
3. **安全关键**：软件崩溃意味着物理坠毁。因此 ArduPilot 有看门狗定时器、故障安全模式、参数校验系统。
4. **传感器融合**：需要用卡尔曼滤波把 IMU、GPS、磁力计、气压计、光流等噪声各异的数据融合成一致的状态估计。
5. **资源极度受限**：典型的 Pixhawk 飞控只有 1MB Flash 和 512KB RAM，所有功能必须塞进这个空间。

## 4. 技术架构分析

### 总体架构（基于公开文件推断）

ArduPilot 的架构分为四层：

```
┌─────────────────────────────────────────────┐
│  车辆应用层 (ArduCopter / ArduPlane / ...)    │
│  ┌─────────────────────────────────────────┐ │
│  │  Mode 状态机 (Stabilize / Loiter / Auto) │ │
│  └─────────────────────────────────────────┘ │
├─────────────────────────────────────────────┤
│  核心库层 (AP_* Libraries, ~131个库)          │
│  AP_AHRS AP_NavEKF AP_InertialSensor        │
│  AP_Motors AC_AttitudeControl AP_Scheduler   │
│  AP_Param AP_Logger AP_GPS AP_Compass ...    │
├─────────────────────────────────────────────┤
│  硬件抽象层 (AP_HAL)                         │
│  AP_HAL_ChibiOS / AP_HAL_Linux / AP_HAL_SITL│
├─────────────────────────────────────────────┤
│  硬件层 (MCU + 传感器 + 执行器)                │
│  STM32F4/F7/H7 / ESP32 / Linux SBC          │
└─────────────────────────────────────────────┘
```

### 主要模块

1. **AP_HAL（硬件抽象层）**：定义了 UART、SPI、I2C、GPIO、PWM、RCOutput、Scheduler 等纯虚接口。各平台（ChibiOS、Linux、SITL）分别实现这些接口。这是 ArduPilot 跨平台移植能力的基石。

2. **AP_Scheduler（调度器）**：主循环的调度核心。ArduPilot 不使用传统 RTOS（如 FreeRTOS），而是用 400Hz 的大循环 + 一个可选的快速线程（1-2kHz）来处理高速控制。Scheduler 维护一张"任务表"，每个任务指定执行间隔（比如 EKF 每 2.5ms 执行一次，GPS 数据解析每 20ms 执行一次），大循环里顺序遍历任务表，根据时间戳决定哪些任务该跑了。这是一种"协作式调度"——所有任务在同一个线程里轮流执行，不存在任务优先级抢占。

3. **AP_NavEKF2 / AP_NavEKF3（扩展卡尔曼滤波器）**：这是 ArduPilot 中算法最复杂的模块。EKF 接收 IMU（加速度计+陀螺仪）、GPS、磁力计、气压计等多传感器数据，输出位置、速度、姿态的最优估计。EKF3 是较新的版本，在数值稳定性和调参便利性上更好。这部分代码在 `libraries/AP_NavEKF3/` 下。

4. **AC_AttitudeControl（姿态控制）**：实现从期望姿态（欧拉角/四元数）到电机推力的控制算法。核心是 PID 控制器 + 角度-角速率-加速度三级级联控制。对多旋翼和直升机分别有不同的实现。

5. **AP_Motors（电机混控）**：根据姿态控制器的输出（roll/pitch/yaw 力矩 + 总推力）计算出每个电机的 PWM 值。不同的机体构型（四轴、六轴、Y6、同轴、倾转旋翼等）对应不同的混控矩阵。

6. **AP_Param（参数系统）**：ArduPilot 通过"参数"来实现运行时配置。参数存储在 EEPROM 中，通过 MAVLink 协议的 PARAM_REQUEST_LIST 和 PARAM_SET 消息可以远程读取和修改。这套系统类似于"嵌入式版的 etcd"——把配置持久化到片内 Flash。

7. **AP_Logger（日志系统）**：以二进制格式把传感器数据、控制状态、GPS 位置等记录到 SD 卡或内部 Flash。是事后分析飞行问题和调试的核心工具。日志格式基于 MAVLink 的勘测日志标准（.bin/.log）。

### 车辆层（以 ArduCopter 为例）

ArduCopter 本身是一个有限状态机：

- **初始化**：`Copter::init_ardupilot()` 初始化所有子系统（传感器、GPS、电池、电调校准、参数加载）。
- **主循环**：`Copter::fast_loop()`（由 AP_Scheduler 调度）→ 读取传感器 → 运行 EKF → 运行当前飞行模式（如 Loiter、Auto）→ 执行姿态控制 → 输出电机 PWM。
- **飞行模式切换**：通过遥控器通道或 MAVLink 命令切换。每个模式是一个 Mode 派生类（ModeLoiter、ModeAuto 等），实现 `init()` 和 `run()`。

## 5. 源码陪读：今天只读哪几个文件

### 建议阅读顺序

1. **`libraries/AP_HAL/AP_HAL.h`** → 先看硬件抽象层的接口定义。理解 UARTDriver、Scheduler、GPIO、RCOutput 这些抽象类，这是理解 ArduPilot 跨平台能力的第一步。

2. **`ArduCopter/Copter.cpp`** → 看主入口 `fast_loop()` 和 `init_ardupilot()`。理解初始化流程的执行顺序——什么子系统先初始化、参数什么时候加载、电调什么时候校准。

3. **`ArduCopter/system.cpp`** → 看调度器的任务表定义（`AP_SCHEDULER_TASK` 宏）和 `allocate_motors()`。这里你能看到电机对象是如何根据 frame_class 参数动态创建的——这是嵌入式里策略模式的经典用法。

4. **`ArduCopter/mode_loiter.cpp`** → 看一个完整的飞行模式实现。`ModeLoiter::init()` 和 `ModeLoiter::run()` 构成了从输入处理到位置保持的完整控制链。特别关注 `precision_loiter_xy()` 方法——它展示了如何用视觉定位来做精确悬停。

5. **`libraries/AP_NavEKF3/AP_NavEKF3.cpp`** → 如果不怕算法，可以看看 EKF 的核心更新函数 `Update()》。不需要完全理解卡尔曼方程，只要能看懂输入是什么（IMU 测量值）、输出是什么（位置/速度/姿态）以及协方差矩阵在更新循环中的作用就够了。

## 6. 核心模块/函数导读

### 1. `Copter::fast_loop()` - ArduCopter 主循环

- **位置**：`ArduCopter/Copter.cpp`
- **角色**：所有周期性逻辑的起点。被 AP_Scheduler 以 400Hz 调用。每次调用会执行：读取新传感器数据 → 更新姿态/位置估计 → 运行当前飞行模式 → 更新电机输出。
- **嵌入式意义**：这是"实时控制循环"的典型代表。循环周期必须稳定在 2.5ms，超出意味着姿态估计滞后，直接导致飞行不稳定。

### 2. `ModeLoiter::run()` - 悬停模式运行函数

- **位置**：`ArduCopter/mode_loiter.cpp`
- **角色**：当飞控在 Loiter 模式时，每个控制周期调用此函数。它读取遥感输入、计算目标位置偏差、调用位置控制器和姿态控制器来保持悬停点。
- **嵌入式意义**：展示了传感器输入 → 控制计算 → 执行器输出的完整闭环。注意 `update_simple_mode()` 中的坐标系旋转——飞控一直在做"机体坐标系→世界坐标系"的转换。

### 3. `AP_Scheduler::run()` - 调度器主函数

- **位置**：`libraries/AP_Scheduler/AP_Scheduler.cpp`
- **角色**：遍历任务表，检查每个任务是否到执行时间了，如果是则执行。使用 `_micros()`（微秒级计时器）计算时间差。
- **嵌入式意义**：这是协作式调度的实现。注意它**没有操作系统**——所有任务在一个线程里顺序执行。好处是简单、可预测、没有锁竞争问题；坏处是一个任务阻塞就拖慢所有任务。这跟 RTOS 的抢占式调度形成鲜明对比。

### 4. `AP_NavEKF3::Update()` - EKF 状态更新

- **位置**：`libraries/AP_NavEKF3/AP_NavEKF3.cpp`
- **角色**：接收 IMU 测量值，预测状态（位置/速度/姿态），然后如果有 GPS/磁力计/气压计等外部观测，则用观测值修正预测。这本质上是一个"先猜后修正"的循环。
- **嵌入式意义**：EKF 是嵌入式传感器融合的标配技术。软件工程师可能更熟悉 "Sensor Fusion" 这个概念，但在嵌入式世界，你会直接和矩阵运算、协方差传播、数值稳定性打交道——资源受限下，double 都不能用，完全用 float。

### 5. `AP_Param::send_pending()` / `load()` - 参数系统

- **位置**：`libraries/AP_Param/AP_Param.cpp`
- **角色**：管理所有可通过 MAVLink 远程修改的配置参数。参数以键值对形式存储在 EEPROM（Flash 模拟的持久化存储）中。`load_object_from_eeprom()` 在启动时从 EEPROM 恢复所有参数。
- **嵌入式意义**：Flash 写入有寿命限制（~10万次擦写），所以参数系统需要做"磨损均衡"——不能每次修改都擦写同一块。这也是嵌入式开发中常见的"EEPROM 模拟"技术。

## 7. 软件工程师类比解释

### 类比 1：AP_HAL → 依赖注入容器 / ORM 接口层

AP_HAL 定义了 UART、SPI、I2C、GPIO 等纯虚类，不同硬件平台提供不同的实现。这就像后端工程里定义了一个 `Repository` 接口，然后有 `PostgresRepository`、`MySQLRepository`、`SQLiteRepository` 分别实现。

**类比的局限**：后端工程中的"实现切换"通常发生在启动时（依赖注入配置），而 AP_HAL 的实现选择发生在编译时——通过条件编译（`#ifdef HAL_CHIBIOS`）决定链接哪个平台的实现。这是嵌入式中"零开销抽象"的典型做法，不能在运行时动态替换。

### 类比 2：AP_Scheduler 协作式调度 → Node.js 的事件循环 / Python asyncio

协程/事件循环也是"单线程 + 周期性检查任务是否该执行"的模式。很像 ArduPilot 的调度器。

**类比的局限**：Node.js 的事件循环支持异步 I/O（文件/网络不会阻塞主线程），而 ArduPilot 的调度器中的任务执行的是纯 CPU 操作——没有异步 I/O 的概念。如果某个任务的 while 循环忘了退出，整个无人机就失控了。而且事件循环可以跳出（await），调度器不行。

### 类比 3：AP_Param 参数系统 → etcd / Redis 作为配置中心

ArduPilot 通过 MAVLink 远程读写参数，参数持久化到 EEPROM。这跟后端用配置中心（etcd/Consul）管理服务配置、应用启动时拉取的思路高度相似。

**类比的局限**：EEPROM（Flash）写入至少耗时 5-10ms，且第 10 万次写入后 Flash 单元可能损坏。后端配置中心没有这种"写入寿命"限制。另外 ArduPilot 的参数系统没有事务保障——如果写入过程中断电，参数可能部分写入、部分丢失（系统有校验和但不可原子回滚）。

### 类比 4：EKF 传感器融合 → 后端事件溯源的聚合重建

EKF 的"先预测（IMU 积分）→ 再修正（GPS/磁力计观测）"类似于事件溯源中的"先 apply 事件 → 再补偿修正"。

**类比的局限**：事件溯源可以重新播放全部历史恢复到任意时刻的状态。EKF 不行——卡尔曼增益矩阵的传播不是可逆的，你不能"撤销"一次 GPS 观测来恢复之前的姿态估计。而且 EKF 是实时系统，观测必须在预测后的极短时间内到达才算有效。

### 类比 5：飞行模式状态机 → 前端路由 + 中间件链

每个飞行模式（Stabilize / Loiter / Auto / RTL）类似于前端的一个路由页面，而 AttitudeController 像中间件——不管哪个页面，都要经过相同的姿态控制 Pipeline。

**类比的局限**：前端的路由切换是毫秒级且无安全后果的。ArduPilot 的模式切换必须严格检查安全条件——`arm()` 需要 GPS 锁定、电池电压正常、传感器校准完成、倾斜角度不超过阈值。模式切换失败意味着无人机不能起飞。另外模式间的切换有严格的状态转换约束（例如从 Land 直接切到 Auto 可能被拒绝）。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：PWM（脉冲宽度调制）
- **一句话**：通过改变方波高电平的宽度（占空比）来模拟输出不同的模拟电压值。
- **软件类比**：类似于用二分法发消息——01 表示最小功率，99 表示最大功率，50 表示一半。
- **常见场景**：电调调速（无人机电机）、舵机角度控制、LED 亮度调节。
- **坑点**：频率选错了电机尖叫不转；占空比范围与具体电调/舵机协议相关；PWM 信号需要定时器硬件支持，纯软件模拟精度不够。

### 卡片 2：IMU（惯性测量单元）
- **一句话**：包含 3 轴加速度计 + 3 轴陀螺仪（有时还有磁力计），用于测量自身的加速度和角速度。
- **软件类比**：相当于一个闭着眼睛的人体——不知道自己在哪里，但知道自己在朝哪个方向加速和旋转。
- **常见场景**：飞控的姿态估计、手机的屏幕旋转、运动追踪。
- **坑点**：加速度计测不到匀速运动，陀螺仪有温漂和零偏。IMU 数据必须经过滤波和融合才能用。

### 卡片 3：AHRS（航姿参考系统）
- **一句话**：把 IMU 数据+磁力计数据融合成稳定的航向、俯仰、横滚角度。
- **软件类比**：相当于前端框架的状态管理器（Redux/MobX）——多个"脏"输入经过 reducer 产生一致的全局状态。
- **常见场景**：无人机姿态显示、自动驾驶姿态反馈。
- **坑点**：AHRS 依赖磁力计做航向参考，但电机大电流产生的磁场会严重干扰磁力计读数（compassmot 问题）。

### 卡片 4：EKF（扩展卡尔曼滤波）
- **一句话**：一种递归算法，把多个有噪声的传感器数据融合成精确的状态估计。
- **软件类比**：A/B 测试中的贝叶斯更新——先验 belief（预测）加上新的实验数据（观测）得到后验 belief（修正）。
- **常见场景**：GPS+IMU 融合定位、视觉惯性里程计、电池 SOC 估算。
- **坑点**：EKF 假设噪声是高斯分布——实际上不是；调 EKF 的协方差矩阵是门玄学。

### 卡片 5：电调 / ESC（电子调速器）
- **一句话**：接收飞控的 PWM 信号，将其转换成驱动无刷电机旋转的三相交流电。
- **软件类比**：类似于 GpuPgrm——上层应用发"渲染这个三角形"的指令（PWM），ESC 负责把这个指令翻译成 GPU 硬件控制信号（三相逆变桥）。
- **常见场景**：任何无刷电机驱动（无人机、RC 车船、电动工具）。
- **坑点**：不同协议的 PWM 信号（PWM OneShot DShot）时序不同；电调油门校准必须在第一次使用前做；电调损坏往往因为过流而非过热。

### 卡片 6：Pixhawk（飞控硬件参考设计）
- **一句话**：ArduPilot/PX4 生态的标准飞控硬件平台，基于 STM32F4/F7/F7+H7 双芯片架构。
- **软件类比**：类似于 Kubernetes 的参考实现——不强制用，但大家默认兼容。
- **常见场景**：作为 ArduPilot 的主要部署目标硬件。
- **坑点**：Pixhawk 标准有几代了（FMUv2-v6），不兼容的硬件和外设接口变化很大。

### 卡片 7：MAVLink（微型无人机通信协议）
- **一句话**：无人机与地面站之间传输遥测、命令、参数的消息协议，一条消息几字节到几十字节。
- **软件类比**：类似于 protobuf/gRPC 的二进制 RPC——但是带宽极窄（915MHz 无线电，几十 kbps），所以每个位都要精打细算。
- **常见场景**：Mission Planner / QGroundControl 与飞控通信、数传模块中继。
- **坑点**：MAVLink 1 和 2 不兼容；消息签名和加密是可选的（很多人不启用）。

### 卡片 8：SITL（软件在环仿真）
- **一句话**：在 PC 上编译运行 ArduPilot，用模拟的传感器和物理模型代替真实硬件来调试飞控代码。
- **软件类比**：相当于在后端写单元测试/集成测试时 mock 数据库、mock 外部服务，只是 ArduPilot 还 mock 了地球重力场和空气动力学。
- **常见场景**：在不炸机的前提下测试飞行逻辑、调试新功能。
- **坑点**：仿真永远不等于真实——IMU 噪声模型、电调响应延迟、风扰是理想化的。

### 卡片 9：混控矩阵（Motor Mixer）
- **一句话**：一个数学矩阵，把 roll/pitch/yaw 力矩命令乘以一个混控矩阵，得到每个电机的推力输出。
- **软件类比**：类似后端负载均衡器把请求按权重分配到多台服务器上。
- **常见场景**：四轴飞行器的电机输出计算、矢量推进器的推力分配。
- **坑点**：某些构型（Y6、OctoQuad）的混控矩阵不是方阵；无效解意味着某些姿态无法实现；出厂前必须设置正确的 frame_class 参数。

### 卡片 10：Watchdog Timer（看门狗定时器）
- **一句话**：硬件定时器，如果 CPU 在一定时间内没有"喂狗"（复位定时器），它会自动重启整个系统。
- **软件类比**：类似于 Kubernetes 的 liveness probe / 健康检查——容器不响应就重启。
- **常见场景**：飞控的故障保护——如果代码死锁了，看门狗会在 1-2 秒内自动重启飞控。
- **坑点**：不要在中断服务函数里喂狗（中断返回后主循环可能还是卡住的）；太短的看门狗间隔容易误触发。

## 9. 30～60 分钟动手任务

### 目标：在 SITL 仿真中运行 ArduPilot Copter，并观察飞行模式切换

**不需要真实无人机，没有硬件也能做。**

### 步骤

1. **安装 SITL 环境**：
```bash
git clone https://github.com/ArduPilot/ardupilot
cd ardupilot
git submodule update --init --recursive
./Tools/environment_install/install-prereqs-ubuntu.sh -y
```
（如果当前环境没有图形界面，可以用 `--no-graphics` 参数）

2. **启动仿真**：
```bash
sim_vehicle.py -v ArduCopter --map --console
```
等待飞控初始化完成。你会看到 MAVProxy 命令行和地图窗口（如果无图形则只有控制台）。

3. **执行基本操作**：
   - 输入 `arm throttle` 解锁电机
   - 输入 `mode guided` 切换到 Guided 模式
   - 输入 `takeoff 10` 让无人机起飞到 10 米高度
   - 飞行中切换模式：`mode loiter`（悬停）→ `mode stabilize`（自稳）→ `mode rtl`（返航）

4. **观察日志**：
   - 在 MAVProxy 中执行 `module log` 开启日志记录
   - 飞行结束后，查看 `logs/` 目录下的 `.bin` 日志文件
   - 尝试用 `mission planner` 或 `mavlogdump.py` 解析日志

### 验收标准
- [ ] 飞控成功启动并显示 GPS 锁定（3D Fix）
- [ ] 成功解锁（arm）并起飞
- [ ] 至少切换了 3 个不同的飞行模式
- [ ] 无人机通过 RTL 模式成功返航并降落

如果你的环境无法安装 SITL，可以通过阅读 ArduCopter 的 `mode_loiter.cpp` 和 `mode_auto.cpp` 代码，画一张"飞行模式状态机转换图"作为替代。至少画出 5 个模式并标注触发转换的事件。

## 10. 3 道深度追问题

### 问题 1（时序/并发/资源限制）
ArduPilot 的主循环在 400Hz（2.5ms/周期）下运行。假如你需要在每 20ms 的传感器更新中新增一个"运行一个复杂的路径规划算法"，该算法在最坏情况下需要 8ms。你会怎么设计方案？（提示：考虑 AP_Scheduler 的协作式调度特性、快速线程、任务优先级和看门狗定时器的时间窗口。）

### 问题 2（软件工程 → 嵌入式视角转换）
你在后端微服务中习惯用 REST API + JSON 来做服务间通信。现在要设计 ArduPilot 的某个新外设（例如一个激光雷达）的驱动通信协议，数据速率只有 115200 波特（~11.5KB/s）。请描述你的设计决策会有哪些不同——至少从序列化格式、协议状态机、错误重试策略、缓冲区管理几个角度展开。另外，如果这个外设的数据必须在 10ms 内送达飞控主控制器（否则避障无效），你的设计会怎么变化？

### 问题 3（工程判断）
ArduPilot 的 EKF3 使用 float（32 位浮点）而不是 double（64 位浮点）做所有矩阵运算。在 Pixhawk（STM32F4，带 FPU）上，float 运算大约是 double 的 2-4 倍快，且占用更少的内存带宽。但 float 的精度在某些场景下不够（例如长时间 GPS 漂移后的小数位误差）。如果你要决定是否在 EKF3 的关键协方差更新步骤中局部使用 double：该从哪些维度评估这个代价？你会用什么具体测试来验证切换是安全的？

## 12. 工程点评

- **可维护性**：⭐⭐⭐⭐ 131 个 AP_ 库的组织结构清晰，命名一致。但整体代码量极大（~150 万行 C++），新手入门曲线陡峭。
- **可移植性**：⭐⭐⭐⭐⭐ AP_HAL 层堪称 C++ 硬件抽象实现的典范，从 STM32 到 ESP32 到 Linux SBC 都能跑。
- **可靠性**：⭐⭐⭐⭐ 有看门狗、故障安全模式、参数校验、超过 15 年的 bug 修复和实战验证。但社区维护的 ESP32 HAL 等部分不如主平台稳定。
- **可测试性**：⭐⭐⭐⭐ SITL 仿真 + 完整的 CI 自动化测试套件（Copter/Plane/Rover/Sub/ChibiOS 编译测试）。但硬件在环（HITL）测试的门槛高，需要真实飞控。
- **量产/运维**：⭐⭐⭐ 通过参数系统 + MAVLink 协议可以远程配置和日志下载。但固件更新流程对新手不友好（需要单独下载/编译 bootloader + 固件）。参数调优依赖经验（PID 调参等）。
