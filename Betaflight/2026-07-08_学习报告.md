# 每日嵌入式学习：Betaflight 飞控固件

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/betaflight/betaflight
- **一句话介绍**：Betaflight 是一套开源的飞行控制器固件，专为多旋翼无人机和固定翼飞行器设计，核心是实时姿态解算与 PID 控制回路，目标是让穿越机飞得更快、更稳、更精准。

## 2. 为什么今天选它

**活跃度**：11,189 Stars，3,880 Forks，最新推送 2026-07-07，有稳定的半年制发布周期（2025.12 已发布，2026.6 在 RC 阶段）。325 个 open issues，社区极活跃。

**工程完整度**：从传感器驱动（加速度计/陀螺仪/磁力计/气压计/光流）、遥控器协议（SBus/CRSF/Spektrum/FrSky/ELRS）、电机协议（DShot/Multishot/Oneshot）、PID 控制器、混控器、黑盒日志、OSD 到 USB MSC 存储，是一个完整的嵌入式固件产品。

**适合软件工程师的原因**：
- 没有使用 RTOS，而是实现了一个**自定义协作式调度器**——这对理解"实时系统在无 RTOS 下怎么跑"极其有教育意义。
- PID 控制理论在嵌入式中的落地——软件工程师熟悉"反馈回路"概念（如监控告警系统），但实时 PID 有采样率、积分饱和、微分噪声等硬件约束。
- 传感器融合（IMU）展示了如何用有限精度浮点做姿态解算，没有卡尔曼滤波器那样的数学强度，用互补滤波加 DCM 实现，更亲民。
- 模块化驱动层设计——抽象的 bus 层（SPI/I2C/QuadSPI）让添加新传感器时只需写芯片特定驱动，其余复用。

## 3. 项目解决的问题

Betaflight 解决的核心问题是：**把飞行员的手柄输入（遥控信号）转化为四个（或更多）电机的精确转速，让飞行器按指令保持或改变姿态。**

真实工程场景：
- 穿越机竞速（需要极低延迟、极高 PID 刷新率，典型 8KHz～32KHz 陀螺采样）
- 航拍无人机（需要平稳悬停，GPS 定高定点和 GPS Rescue）
- 固定翼飞行（需要混控器支持各种翼型）

相比普通软件系统，多了以下约束：

| 约束 | 说明 | 软件类比 |
|------|------|----------|
| **实时性** | 控制回路必须在采样周期内完成计算（如 125μs），超时就炸机 | 比任何 web API 响应时间要求都严格 |
| **资源受限** | STM32 F4/F7/H7，Flash 128KB～2MB，RAM 64KB～1MB，无 MMU | 相当于在 80 年代计算机上跑现代控制算法 |
| **时序确定性** | 调度器必须保证 gyro 任务准时启动，抖动超过 1μs 会影响控制质量 | 和数据库主从延迟完全不是一个量级 |
| **硬件耦合** | 传感器驱动直接操作 SPI 寄存器、DMA 通道、GPIO 中断、TIM 定时器 | 没有操作系统为你抽象硬件 |
| **安全关键** | 代码 bug 直接导致物理损坏（炸机），更新规格无热修复 | 和金融交易系统的可靠性要求同级，但测试手段更原始 |
| **功耗与散热** | H7 跑 400MHz 做浮点运算，散热靠空气流过 PCB | 后端服务有数据中心冷却，这里是"物理风冷" |

## 4. 技术架构分析

基于公开文件和源码推断，Betaflight 的架构如下：

### 整体架构分层

```
┌──────────────────────────────────────────────┐
│                  应用层                        │
│  PID 控制器 · 姿态估计 · 混控器 · 遥测 · OSD │
├──────────────────────────────────────────────┤
│                  功能层                        │
│  RX 协议解析 · 黑盒日志 · 电池管理 · GPS     │
├──────────────────────────────────────────────┤
│              任务调度层（自定义调度器）         │
│  Gyro 任务（最高优先） · PID 任务 · RX 任务    │
│  LED 任务 · 遥测任务 · OSD 任务 · Beeper 任务 │
├──────────────────────────────────────────────┤
│                 硬件抽象层                     │
│  Bus I2C/SPI · IO · Timer · DMA · ADC · PWM  │
├──────────────────────────────────────────────┤
│                MCU 外设/寄存器层               │
│  STM32 F4/G4/F7/H7 · RP2350                  │
└──────────────────────────────────────────────┘
```

### 核心模块

1. **调度器（scheduler）**：Betaflight 没有使用 FreeRTOS 或任何第三方 RTOS。它实现了一个基于优先级的协作式调度器，核心逻辑在 `src/main/scheduler/scheduler.c`。主循环 `run()` 就是一个 `while(true) { scheduler(); }`。每个任务有 `checkFunc`（判断是否该运行）和 `taskFunc`（实际执行），调度器计算任务是否到期、动态调整优先级。Gyro 任务拥有最高优先级，由定时器中断触发。

2. **陀螺仪驱动（gyro）**：`src/main/sensors/gyro.c` 是整个系统的"心跳"。陀螺采样通过 SPI 读取传感器寄存器（如 BMI270、ICM42688 等），经过滤波（低通 + 动态陷波）后，传递给 PID 控制器。Betaflight 支持 8KHz 和 32KHz 的陀螺采样率——32KHz 意味着每 31.25μs 就要完成一次 SPI 读取、滤波、姿态更新。

3. **IMU 姿态解算（imu）**：`src/main/flight/imu.c` 实现姿态估计。使用方向余弦矩阵（DCM）和互补滤波，融合陀螺仪的角速度积分和加速度计的倾角测量，得到横滚、俯仰、偏航角。不使用卡尔曼滤波（计算量太大），而用 PI 控制器修正漂移。

4. **PID 控制器（pid）**：`src/main/flight/pid.c` 是整个飞行控制的核心。实现标准 PID + 前馈 + 动态滤波。包含 Airmode（在空中也能保持 PID 生效）、Anti-Gravity（油门大时调低 I 增益防下沉）、动态 D 衰减（高频噪声抑制）、iTerm 退饱和（防止积分饱和）等特性。代码约 65K 字符，是项目中最复杂的模块。

5. **混控器（mixer）**：`src/main/flight/mixer.c` 把 PID 的输出（横滚/俯仰/偏航/油门四轴控制量）映射到实际电机的 PWM/DSHOT 信号。支持 Quad X、Quad +、Hex、Octo、三旋翼、固定翼等数十种混控模式。混控矩阵是一个简单的线性方程组。

6. **RC 输入与解析（rx）**：`src/main/rx/` 目录下实现了 SBus、CRSF（ExpressLRS 协议）、FrSky X 与 D 系列、Spektrum、SumD、IBUS、GHST 等几十种遥控协议。每个协议有自己的解析器，输出统一的结构化遥控通道数据。

7. **参数持久化（pg）**：`src/main/pg/` 实现参数组系统。类似 Linux 的 sysfs 但更轻量，每个参数组有默认值、ID 和序列化/反序列化函数。参数存储在 EEPROM 或 Flash 上，通过 Betaflight Configurator 配置。

8. **配置系统（config）**：`src/main/config/` 处理 EEPROM 的读写、参数验证、固件升级后的配置迁移。

### 数据流

```
遥控接收机 → RX 协议解析 → RC 平滑滤波
    → RC 指令处理（rates/exp/mix）→ 设定点生成
    → 陀螺仪采样（8K/32KHz）→ 滤波
    → PID 控制器（测量值 vs 设定点 → 输出值）
    → 混控器（4 轴控制量 → N 电机值）
    → DSHOT/PWM 驱动 → 电调 → 电机
```

## 5. 源码陪读：今天只读哪几个文件

以下按阅读顺序推荐：

### 1️⃣ `src/main/main.c`
**阅读目标**：理解整个固件的入口和启动流程。看 `main()` 函数如何初始化系统、调用 `init()`、然后进入 `run()` 主循环。这是理解"嵌入式固件从哪里开始跑"的最佳起点。

### 2️⃣ `src/main/scheduler/scheduler.c`
**阅读目标**：理解 Betaflight 的协作式调度器。重点关注 `scheduler()` 函数、任务优先级队列、`taskGuard`（任务保护时间）、以及 gyro 任务被"粘"在高优先级的原因。对比 FreeRTOS 的抢占式调度，理解为什么这里不需要抢占。

### 3️⃣ `src/main/sensors/gyro.c`
**阅读目标**：理解嵌入式传感器驱动怎么写。重点看 gyro 初始化流程（`gyroInit()`）、SPI 读取时序、滤波链（低通 → 动态陷波 → RPM 滤波）、溢出检测和 yaw spin recovery。看 FAST_DATA 和 FAST_CODE 这些性能优化宏。

### 4️⃣ `src/main/flight/pid.c`
**阅读目标**：理解嵌入式 PID 控制器的工程实现。重点看 `pidApply()` 或类似的核心计算函数，看代码如何处理积分限幅（`pidRuntime.iTermError`）、微分项前馈、动态滤波和 anti-gravity 逻辑。软件工程师会惊讶于嵌入式 PID 的代码比教科书复杂得多。

### 5️⃣ `src/main/flight/imu.c`
**阅读目标**：理解无卡尔曼的姿态解算怎么做。重点看 DCM（方向余弦矩阵）更新和互补滤波的实现。软件工程师可以把 DCM 类比为数据库里的"增量状态更新"，互补滤波类比为"两个数据源的加权平均"。

## 6. 核心模块/函数导读

### `int main(int argc, char *argv[])` — `src/main/main.c`
**角色**：固件入口。执行硬件初始化、参数加载、任务注册，然后进入 `run()` 无限循环。`init()` 函数（在 `fc/init.c`）完成几乎所有初始化工作：从时钟配置到外设驱动到传感器校准。软件工程师可以类比为 Spring Boot 的 `@SpringBootApplication` + 自动配置——但这里一切都是显式、线性的初始化序列，没有依赖注入。

### `void scheduler(void)` — `src/main/scheduler/scheduler.c`
**角色**：Betaflight 的"操作系统内核"。遍历所有注册的任务，用 `checkFunc` 判断是否到期，执行 `taskFunc`。调度器会跟踪每个任务的执行时间和周期，动态调整优先级。关键设计决策：**没有任务抢占**——所有任务主动让出 CPU。为什么？因为 gyro 任务必须无抖动执行，抢占会引入不可预测的延迟。

### `void FAST_CODE gyroUpdate(void)` — `src/main/sensors/gyro.c`
**角色**：每次调度时读取陀螺仪数据。实际上由定时器中断驱动（不是调度器轮询）。读取的原始数据经过一系列滤波器（DYN notch、RPM filter、低通），得到最终角速度。这个函数的执行时间直接决定了控制回路能达到的最大频率。8KHz = 125μs 要完成全部读取+滤波。

### `pidApply(void)` — `src/main/flight/pid.c`
**角色**：核心 PID 计算。接收当前的姿态角速度（来自 gyro）和期望的角速度（来自遥控器设定点），输出三个轴的控制量。实际代码中包含大量工程优化：Airmode（非零油门时保持 PID 活跃）、iTerm 退饱和抗锁定、动态 D 衰减减少噪声放大、前馈加速响应。

### `void mixTable(void)` — `src/main/flight/mixer.c`
**角色**：将 PID 计算出的横滚/俯仰/偏航/油门四通道值混合成每个电机的输出值。混控矩阵是一个简单的 4×N 线性变换。支持 motor_stop 功能、反重力补偿、crash flip 模式下的混控切换。软件工程师可以把它类比为一个"数据管道转换器"——输入 4 个值，输出 N 个 PWM 占空比。

### `void pgResetFn_xxx(const pgConfig_t *)` — `src/main/pg/` 各文件
**角色**：参数组（Parameter Group）系统的默认值初始化。每个参数组（PID 参数、串口配置、传感器启用）都有一个 reset 函数，在配置丢失或固件升级时恢复默认值。系统通过参数 ID 做序列化和版本迁移。这是嵌入式世界里少见的"结构化参数持久化"实现。

## 7. 软件工程师类比解释

### 类比 1：调度器 vs 事件循环/协程调度

**类比**：Betaflight 的自定义调度器类似于 Node.js 的事件循环或 Python 的 asyncio 协程调度——都是单线程协作式并发。任务通过 `checkFunc` + `taskFunc` 的机制，类似 `epoll` + 回调。Gyro 任务是"微任务"，必须在下一次事件触发前完成。

**局限性**：Node.js 的事件循环可以通过 `worker_threads` 或 `child_process` 做 CPU 密集任务，而嵌入式单核 CPU 没有这样的选项。而且嵌入式调度器的一个任务超时会让**所有**任务延迟——后端服务超时只是 502，无人机超时就是炸机。

### 类比 2：PID 控制器 vs 监控告警系统中的线性反馈

**类比**：PID 的三个分量可以映射到后端监控系统：P（比例）= 当前错误直接告警，I（积分）= 累积错误（比如半小时内错误率持续升高），D（微分）= 错误变化率（突然飙升）。你日常调 Prometheus 告警规则就是在调"PID 参数"。

**局限性**：后端告警阈值可以慢慢调，调完了重启就行。嵌入式 PID 参数需要在实际飞行的**几千分之一秒内**做出响应。P 太大导致「振荡」，在软件里只是告警抖动，在无人机里就是"炸机"。而且嵌入式 PID 还要处理传感器噪声、采样延迟、执行器饱和等物理问题，比告警监控复杂得多。

### 类比 3：混控器 vs API 网关的请求转换

**类比**：混控器把 4 通道控制量转换为 N 电机输出，类似于 API 网关把 HTTP 请求转换为后端微服务能理解的格式。你收到一个 `/checkout` 请求，网关把它拆成 `order.create()` + `payment.charge()` + `inventory.reduce()`——对应着 PID 的横滚/俯仰/偏航输出被`分发`到 4 个电机。

**局限性**：API 网关的转换是纯软件的，HTTP 请求延迟 50ms 正常。混控器必须在几微秒内完成，而且涉及物理约束（电机只能正转、不能瞬间改变转速、DSHOT 协议有固定位时间等）。

### 类比 4：参数组（pg）系统 vs 数据库 Schema 迁移

**类比**：pg 系统的参数组 ID + 序列化 + 版本迁移，类似于数据库的 migration 机制。固件升级后，新的 pg 代码可以识别旧版本的参数结构，在加载时做字段迁移。`pgResetFn` 就像给数据库表设 DEFAULT 值。

**局限性**：数据库迁移可以回滚，有成熟的 ORM 工具；pg 系统在嵌入式世界是手动实现的，且错误可能导致飞控无法解锁。而且嵌入式的 EEPROM 有写入寿命限制（通常 10 万次），不能频繁写入。

### 类比 5：DSHOT 协议 vs WebSocket

**类比**：DSHOT 是一种数字电机控制协议，通过单线传输 16 位数据包（包含油门值和遥测请求），类似于 WebSocket 发送的二进制帧。两者都有帧结构、校验和、握手机制。

**局限性**：DSHOT 工作在有刷/无刷电调的硬件定时器层面，位时间精确到纳秒级，信号直接用 GPIO 和 DMA 产生——没有 TCP 重传、没有缓冲区、没有 OS 介入。而且 DSHOT 还必须兼容不同电调厂商的实现差异，类似 Web 浏览器兼容性，但更底层。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：PID 控制器
- **一句话**：比例-积分-微分控制器，根据当前误差、历史累计误差和误差变化率调整输出。
- **软件类比**：Prometheus 告警阈值 + 累积告警计数 + 突增检测组合。
- **常见场景**：无人机姿态稳定、电机速度控制、温度控制系统。
- **坑**：积分饱和会导致系统大幅超调；微分会放大传感器噪声；三个参数相互耦合，不能独立调优。

### 卡片 2：DCM（方向余弦矩阵）
- **一句话**：姿态旋转的数学表示，用 3×3 矩阵描述机体坐标系到世界坐标系的旋转。
- **软件类比**：多态路由表中将"请求路径"映射到"后端服务"——只是这个映射在三维空间。
- **常见场景**：IMU 中融合陀螺仪和加速度计数据做姿态估计。
- **坑**：DCM 需要正交化校正（Gram-Schmidt 过程），否则矩阵漂移会导致估计发散；计算量比四元数大，但在没有浮点加速的 MCU 上更直观。

### 卡片 3：互补滤波
- **一句话**：融合高频可信的陀螺仪数据和低频可信的加速度计数据，各取其优。
- **软件类比**：两个不同精度的数据源加权合并——高性能低延迟的源用高通，高精度低频率的源用低通。
- **常见场景**：姿态估计、传感器融合。
- **坑**：截止频率的选择很关键，太低则陀螺仪漂移累积，太高则加速度计噪声混入；没有自适应能力。

### 卡片 4：DSHOT 协议
- **一句话**：一种数字电机控制协议，用 PWM 脉宽编码 0 到 2047 的油门值，支持双向双向遥测和 RPM 滤波。
- **软件类比**：自定义的二进制线协议，类似 Protobuf over TCP，但位时序由硬件定时器保证。
- **常见场景**：穿越机电调控制，替代传统的 PWM/Oneshot/Multishot。
- **坑**：不同 DSHOT 速率（150/300/600）的位时间不同，要求 CPU 定时器精度和 DMA 配置正确；长线缆信号容易受干扰。

### 卡片 5：Airmode
- **一句话**：在零油门（电机怠速）时仍然保持 PID 控制器活跃的飞控特性。
- **软件类比**：开发环境的特性开关——关闭时会优化某些行为但牺牲控制精度。
- **常见场景**：穿越机在做翻横滚动作时，需要瞬间降到零油门但保持姿态控制。
- **坑**：Airmode 开启时飞机在地面也可能因为 PID 输出而通电启动电机，容易割伤手脚。

### 卡片 6：动态陷波滤波器
- **一句话**：自动跟踪并滤除特定频率的噪声，比如电机振动产生的频率成分。
- **软件类比**：自动扩展的网络防火墙规则——发现攻击频率后自动添加过滤规则。
- **常见场景**：陀螺仪信号遇到桨叶振动（通常是电机转速的倍数频率）时滤除。
- **坑**：如果滤波器的跟踪速度跟不上转速变化，反而会在振动频率附近引入相位延迟，降低控制质量。

### 卡片 7：黑盒日志（Blackbox）
- **一句话**：飞控非易失存储上的飞行数据记录器，记录陀螺仪、PID 输出、RC 信号等到 Flash/SD 卡。
- **软件类比**：后端应用的 APM 日志 + tracing 数据，可以离线分析性能瓶颈。
- **常见场景**：调试 PID 参数、"炸机"后分析原因、对比不同飞控算法效果。
- **坑**：日志速率很高（32KHz 采样就是每秒 32,000 条记录），写入带宽和存储空间严重受限；日志解析需要专用工具（blackbox_decode）。

### 卡片 8：I-Term 退饱和
- **一句话**：当执行器达到极限（如电机满油门），停止积分项累加，防止积分饱和。
- **软件类比**：微服务的熔断器——下游挂了就别再发请求积累队列，否则恢复后大量请求涌入瞬间压垮。
- **常见场景**：PID 控制器的所有应用场景。
- **坑**：退饱和实现太激进会导致稳态误差；实现太保守会导致大幅超调。

### 卡片 9：RP2350 多核（USE_MULTICORE）
- **一句话**：Betaflight 开始支持 RP2350 的双核架构，可以在第二个核心上运行初始化序列或部分计算。
- **软件类比**：从单核升级到多线程——但这里的"多线程"是裸金属级别的显式核心分配。
- **常见场景**：新飞控硬件用 RP2350 替代 STM32，提升算力。
- **坑**：多核共享外设时的一致性（cache coherence）需要软件手动管理；中断路由到哪个核需要仔细设计。

### 卡片 10：SPI 时序约束
- **一句话**：SPI（串行外设接口）通信的时序由 MCU 时钟分频决定，传感器有最大速率限制。
- **软件类比**：API 接口的频率限制（rate limit），但这里限制是以 MHz 计的时钟频率，且需要看懂数据手册的时序图。
- **常见场景**：陀螺仪/加速度计/Flash 存储通过 SPI 连接 MCU。
- **坑**：SPI 时钟频率设置过高会导致 CRC 错误或数据错位；SPI 走线过长会出现信号反射，需要调整驱动能力和匹配电阻。软件工程师往往只关心"通信成功"，但嵌入式工程师还要关心"信号完整性"。

## 9. 30～60 分钟动手任务

### 无硬件方案：阅读并理解 Betaflight 的调度器

**目标**：理解一个无 RTOS 的嵌入式系统是如何实现实时调度的。

**步骤**：
1. 打开 `src/main/scheduler/scheduler.c`，找到 `scheduler()` 函数。
2. 找到任务结构体定义 `cfTask_t`，理解每个字段的含义（`checkFunc`、`taskFunc`、`desiredPeriodUs`、`staticPriority`、`taskExecutionTimeUs`）。
3. 找到 `taskGyro` 的定义（在 `fc/tasks.c` 中），看它的 `desiredPeriodUs` 和 `staticPriority`。
4. 理解 `getTaskDeltaTime()` 和 `getGyroDeltaTime()` 在调度器中的作用。
5. 查看 `rescheduleTask()` 和 `setTaskEnabled()` 如何动态调整任务调度。
6. 阅读 `src/main/main.c` 的 `run()` 函数，理解主循环结构。

**验收标准**：
- 能画出调度器的流程图。
- 能解释为什么 gyro 任务不需要 `checkFunc`。
- 能回答"如果 PID 任务耗时超过 gyro 采样周期会怎样"。
- 能对比 Betaflight 调度器和 FreeRTOS 的抢占式调度差异。

### 有硬件方案（如果有飞控板）

**目标**：编译并烧录 Betaflight，通过 CLI 查看参数结构。

1. 安装 Docker，用 devcontainer 编译 `make TARGET=SPEEDYBEEF405WING`。
2. 烧录到飞控板，连接 Betaflight Configurator。
3. 在 CLI 中输入 `dump` 查看完整配置。
4. 修改 PID 参数，对比默认值和你的修改。

## 10. 3 道深度追问题

### 问题 1（时序/并发/资源限制）
Betaflight 支持 32KHz 的陀螺采样率，也就是每 31.25μs 就要完成一次 SPI 读取、滤波和 PID 计算。假设你升级到 64KHz 采样，发现 MCU 算力不够。在不能用更高频 MCU 的前提下，你会从哪些方面做优化？请至少从以下角度考虑：SPI 时钟速率、DMA 配置、滤波器阶数裁剪、浮点转定点运算、任务优先级重排。

### 问题 2（软件视角 → 嵌入式视角）
你是一个后端工程师，习惯了"加机器就解决性能问题"。现在你面对的问题："无人机在 8KHz 控制回路下，陀螺仪噪声导致飞控抖动。" 请把这个嵌入式问题**翻译**成你熟悉的软件领域类比，然后解释为什么软件工程常规方案（重试、缓冲、异步、水平扩展）在这里不适用。最后给出嵌入式领域的工程方案。

### 问题 3（工程判断）
Betaflight 的调度器是协作式的，没有任务抢占。这是设计决策而非工程疏忽。请推断设计团队做此决策的**三个**核心理由，以及它带来的**两个**严重缺点。如果你要在 MCU 上设计一个新的飞控调度器，你会保留协作式还是换抢占式？为什么？（提示：考虑中断延迟、上下文切换开销、共享数据保护）

## 12. 工程点评

### 可维护性
模块化做得非常好：驱动层（`drivers/`）和功能层（`flight/` `rx/` `telemetry/` `io/`）严格分离。新增传感器只需在 `drivers/` 下加一个文件和一个头文件，上层代码基本不用改。参数系统（`pg/`）的设计也很优雅——每个参数组有独立的 reset 和序列化函数。

扣分点：`pid.c` 65K 字符体量偏大，部分函数内部嵌套条件编译宏过多，阅读困难。

### 可移植性
通过 `src/main/target/` 中目标板配置文件实现硬件抽象。不同 MCU 系列的差异用 `platform.h` 和条件编译隔离。支持 STM32 F4/G4/F7/H7 和 RP2350。Bus 层（SPI/I2C/QuadSPI）的抽象非常好——驱动代码不关心是哪个 MCU 的 SPI 外设。

### 可靠性
- 有 watchdog 定时器防止死锁。
- 陀螺仪溢出检测和恢复机制。
- 空难/失效安全（failsafe）——遥控信号丢失后自动着陆。
- 参数校验和 CRC 保护。
- 飞行状态机严格检查：arm/disarm 有前置条件。

### 可测试性
缺乏自动化单元测试是主要短板。虽然有 `src/test/` 目录，但覆盖率不高。嵌入式固件测试一直是行业难题——SIMULATOR_MULTITHREAD 模式可以在 PC 上模拟运行，但传感器信号需要模拟输入。

### 量产/运维
- 通过 Betaflight Configurator（PWA 应用）配置，用户体验好。
- OTA 更新通过 USB 或 DFU。
- 不同飞控板有标准化的目标配置文件，厂商可以提交 PR 添加新硬件。
- 没有云端管理（这是社区项目的正常限制）。

## 13. 明天可以延展的方向

1. **深入 IMU 算法**：对比 Betaflight 的 DCM+互补滤波和 ArduPilot 的 EKF（扩展卡尔曼滤波）在姿态估计上的不同取舍。为什么穿越机用简单的互补滤波，自动驾驶用 EKF？可以从计算开销、精度、收敛速度角度分析。

2. **学习电机控制协议**：研究 DShot 协议的物理层实现，理解 Bitbang DShot 和非 Bitbang DShot 的区别。然后扩展到 FOC（磁场定向控制）和 SimpleFOC 项目，理解"控制电机"和"控制飞控"的不同抽象层级。

3. **研究调度器设计空间**：对比三种调度方案的实际效果——Betaflight 的协作式调度、FreeRTOS 的抢占式调度、以及混合式（中断驱动的硬实时任务 + 协作式轮询的软实时任务）。哪种方案更适合什么样的嵌入式系统？
