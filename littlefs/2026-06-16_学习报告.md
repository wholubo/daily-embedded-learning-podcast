# 每日嵌入式学习：littlefs

## 1. GitHub 项目链接与一句话介绍
- 仓库链接：https://github.com/littlefs-project/littlefs
- **一句话介绍**：littlefs 是 ARM 为微控制器设计的一个轻量、掉电安全、自带动态磨损均衡的文件系统，专为 SPI NOR Flash 等嵌入式存储介质而生。

## 2. 为什么今天选它
- **活跃度**：项目创建于 2017 年，至今持续维护，最新提交为 2026 年 3 月。GitHub 上超过 6700 个 Star、990 个 Fork，社区活跃。
- **工程完整度**：包含完整 C 源码（lfs.h 约 800 行、lfs.c 约 6500 行）、配套工具（FUSE 挂载、Python 镜像工具、JS 模拟器）、详尽的设计文档（DESIGN.md）和磁盘规范文档（SPEC.md），还有完善的测试套件和 PC 模拟块设备。
- **学习价值**：对于一个从后端转嵌入式的工程师来说，文件系统是一个天然的熟悉领域——你每天都在用 ext4、NTFS、tmpfs，但它们在嵌入式场景下完全不可用。littlefs 用不到一万行 C 代码实现了掉电安全、磨损均衡、无递归、RAM 严格有界，是理解"嵌入式约束如何推导出独特数据结构和算法"的绝佳例子。
- **避重说明**：历史列表中已有 SPIFFS（在第 7 天讲 Trice 时提及）、FatFs（在 Zephyr 报告中谈及）、TinyUSB（独立项目），littlefs 是不同的文件系统实现，采用不同设计思路，不在历史列表中。

## 3. 项目解决的问题
littlefs 解决的核心问题是：在一个只有 32KB RAM、512KB ROM、若干 MB Flash 的 32 位微控制器上，如何安全可靠地存储文件？

和普通软件项目相比，它多了几个嵌入式特有约束：

- **掉电安全**：嵌入式设备可能在任何时刻断电——没有关机流程、没有 fclose、没有 fsync 保证。文件系统必须在任意写入操作中途掉电后，仍然能恢复到最近一个已知完好状态。
- **Flash 物理特性**：Flash 写入前必须先擦除（将位设为 1），编程（将位设为 0）可以逐字节进行，但擦除必须整块操作且耗时较长。一个块大约可擦除 1 万到 10 万次，反复写同一块会提前报废。
- **RAM 严格有界**：不能使用任何随文件系统增长而增长的数据结构，不能有递归遍历。比如，一个目录下有 1000 个文件，遍历时不能在堆上分配 1000 个条目的链表。
- **ROM 限制**：代码体积必须尽量小，很多 MCU 的 Flash 连 firmware 本身都要精打细算，不可能塞进完整 ext4 驱动。

## 4. 技术架构分析

### 整体架构

littlefs 采用"两层蛋糕"架构：

**底层**：元数据对（Metadata Pairs，简称 mdir）。这是一对块（两个物理擦除块），以环形日志（log）形式存储目录条目、文件属性、目录结构等元数据。每次写操作以追加方式写入日志，当日志快满时进行压缩（compaction），把有效条目合并到一个新块。掉电时，总是有一个完整副本可用。

**上层**：文件数据通过 COW（Copy-on-Write，写时复制）结构存储。littlefs 使用一种巧妙的 CTZ（Count Trailing Zeros，数末尾零）跳表结构来索引文件数据块。这不是传统 B-tree 或 linked list，而是一种源于 CTZ 算法推导的、占用常量内存的链式索引——每个块存储指向前面块的指针，指针数等于该块地址二进制末尾零的个数。

**链接层**：一个公共的块分配器（block allocator），负责分配和回收块。它使用 lookahead buffer（向前看缓冲区）来高效跟踪空闲块，并且通过限制每个块的擦除次数（block_cycles 参数）来实现动态磨损均衡——擦除次数超标的块会被优先冷落，让磨损均匀分布到整个存储介质。

### 运行流程

1. `lfs_format()`：格式化空设备，写超级块（superblock）。
2. `lfs_mount()`：挂载文件系统，从超级块读取配置，定位根目录元数据对。
3. 文件操作：以 POSIX 风格 API 暴露：
   - `lfs_file_open()`：在元数据中查找文件条目，如果文件内容被内联（inline）则直接从元数据中读取，否则通过 CTZ 结构读取块链。
   - `lfs_file_write()`：写入时触发 COW——新数据写到新分配的块，元数据对中的指针在新写入提交（close/sync）时才原子切换。
   - `lfs_file_close()` / `lfs_file_sync()`：提交写操作到存储，确保掉电后能恢复。
4. 目录操作：`lfs_mkdir()`、`lfs_dir_open()`、`lfs_dir_read()` 等操作目录元数据对。

### 配置系统

通过 `struct lfs_config` 暴露所有可调参数：read/prog/erase/sync 函数指针、块大小、块数量、缓存大小、磨损均衡周期、内联文件上限、只读模式等。用户可在编译时通过宏（`LFS_READONLY`、`LFS_NAME_MAX`、`LFS_NO_MALLOC` 等）裁剪功能，减少 ROM 占用。

### 通信/存储

文件系统本身不直接操作硬件——所有底层存储操作通过用户提供的回调（read、prog、erase、sync）对外部块设备发出调用。这个解耦让 littlefs 可以跑在 SPI Flash、SD 卡、NOR Flash、NAND Flash（通过 FTL 层）、甚至内存模拟盘上。

## 6. 核心模块/函数导读

- **`lfs_mount()` 与 `lfs_format()`**：一个文件系统的始和终。`lfs_format()` 初始化超级块和根元数据对；`lfs_mount()` 从存储中恢复状态。理解它们的执行顺序等于理解 littlefs 的启动生命周期。

- **`lfs_dir_compact()`**：元数据对压缩函数，是 littlefs 最核心的内部操作。当日志快满时（元数据对中有效条目占满块），触发压缩——遍历日志中的条目，丢弃失效条目，有效条目重新写入一个新块。掉电时如果压缩中断，旧块仍然完整可用。这个函数的正确性直接决定整个文件系统的掉电安全性。

- **`lfs_alloc()` 与 `lfs_alloc_lookahead()`**：块分配器。使用位图式的 lookahead buffer 跟踪空闲块，实现 O(1) 复杂度的块分配。通过 `block_cycles` 参数控制每个块的最大擦除次数，超限后该块不再用于分配元数据对中的新日志块，从而实现磨损均衡。

- **`lfs_ctz_find()` 与 `lfs_ctz_append()`**：CTZ 跳表的查找和追加操作。CTZ 结构是一种轻量索引：第 i 个数据块存储的向前指针数等于 count_trailing_zeros(i)。查找时只需要从尾部开始回溯，而不是像链表那样从头到尾遍历。这是 littlefs 在 RAM 有限情况下对文件随机访问的优雅解决方案。

- **`lfs_file_write()` 与 `lfs_file_sync()`**：文件写入链路。`lfs_file_write()` 先把数据写到内存缓存；`lfs_file_sync()` 触发 COW——分配新块，写入数据，更新 CTZ 结构，最后原子更新元数据对中的文件条目。掉电在任何一步中断都不会破坏已有数据。

## 7. 软件工程师类比解释

- **元数据对（Metadata Pair）类似 Write-Ahead Log（WAL）**：数据库的 WAL 在写入实际数据页之前把操作记录到日志，掉电后重放日志恢复。Littlefs 的元数据对也是先追加日志，压缩时原子切换。**局限**：数据库 WAL 通常可以无限增长（靠 checkpoint 回收），而 littlefs 的元数据对只有一对块，满了就必须压缩，设计约束完全不同。

- **CTZ 跳表类似不可变链表 + 稀疏索引**：如果后端工程师用 immutable 数据结构管理日志文件，每次追加产生新版本，那么 CTZ 结构类似——写时复制产生新版本，旧版本仍在。**局限**：CTZ 的索引结构不是树或哈希表，而是根据地址本身推导指针位置，省掉了存储索引的额外空间。

- **block_cycles（磨损均衡周期）类似 Kubernetes Pod 的 maxRestartCount**：集群调度器避免把 Pod 频繁调度到同一节点，类似 littlefs 避免反复分配同一 Flash 块。**局限**：K8s 调度是全局最优问题，littlefs 只做局部动态均衡，不能保证严格均匀磨损。

- **lfs_config 回调函数类似依赖注入**：read/prog/erase/sync 四个函数指针让 littlefs 不依赖任何具体硬件，使用者只需注入块设备驱动。**局限**：依赖注入在服务端是为了测试和扩展，在嵌入式这里是为了解耦硬件差异和让文件系统可在 PC 上模拟测试。

- **Lookahead buffer 类似 Redis 的 bitset 用作布隆过滤器**：它用位图追踪哪些块已被分配。类似 Redis 中用 BITFIELD 判断 key 是否存在。**局限**：lookahead buffer 只跟踪最近搜索过的区域，不是全量位图，所以分配器需要定期重新扫描空闲区域。

## 8. 今日 10 个嵌入式概念卡片

**卡片 1：掉电安全（Power-Loss Resilience）**
- 一句话：文件系统在任意时刻掉电后仍可恢复到一致状态。
- 类比：数据库的 ACID 中的 Durability 在嵌入式中的投射——但嵌入式通常没有 WAL 那样的磁盘空间来维护大日志。
- 常见场景：物联网设备电池耗尽、用户直接拔电源、OTA 升级中断。
- 坑点：认为 fsync 能解决所有问题——很多廉价 Flash 在硬件层面有写缓存，sync 时掉电仍然会丢数据。

**卡片 2：磨损均衡（Wear Leveling）**
- 一句话：让所有 Flash 块的擦除次数尽量接近，避免部分块提前报废。
- 类比：SSD 内部的 FTL 做同样的事，但 littlefs 是在软件层面实现的纯动态均衡。
- 常见场景：频繁写入日志文件（如计数器、温度记录）时，如果不做均衡，同一块会很快耗尽寿命。
- 坑点：动态均衡不能保证绝对均匀，极端写入模式下某些块仍可能先坏；静态均衡（搬走冷数据）才是更彻底但更复杂的方案。

**卡片 3：COW（写时复制，Copy-on-Write）**
- 一句话：写入时不在原地修改，而是写到新位置，然后原子更新指针。
- 类比：Git 的每次 commit 不修改已有对象，而是创建新对象并更新引用。
- 常见场景：文件系统、OTA 升级的 A/B 分区方案、ZFS/Btrfs。
- 坑点：COW 会产生碎片和放大写放大效应，在 Flash 上尤其需要平衡空间效率和写入寿命。

**卡片 4：元数据对（Metadata Pair）**
- 一句话：用两个物理块以环形日志方式存储文件系统元数据，任何一个块损坏都可从另一个恢复。
- 类比：两个数据库副本做冗余——但这里只有两块，且用奇偶校验而不是主从复制。
- 常见场景：目录结构、文件属性、文件名等必须原子更新的小数据。
- 坑点：元数据对大小受块大小限制，单个目录存储大量文件（如数万个小文件）会导致频繁压缩，影响性能。

**卡片 5：CTZ 跳表（CTZ Skip List）**
- 一句话：一种根据地址二进制末尾零个数决定向前指针数的链式索引，实现 O(log n) 的随机访问和 O(1) 的追加。
- 类比：类似于 skiplist 但节点里的指针数由节点序号本身决定，而非随机——更可预测，且省掉随机数的存储。
- 常见场景：文件数据块的索引。
- 坑点：写放大——追加一个块需要更新前面所有块的回溯指针。

**卡片 6：原地更新 vs 追加写入**
- 一句话：传统文件系统修改文件内容时覆盖原来的扇区（原地更新），嵌入式文件系统往往把新数据追加写到新位置（追加写入）。
- 类比：service 更新配置时不是修改原文件，而是写一个带版本号的新文件再切换软链接。
- 常见场景：频繁更新的小文件如配置、状态计数器。
- 坑点：追加写入会产生垃圾数据（stale data），需要定期垃圾回收（compaction）。

**卡片 7：内联文件（Inline Files）**
- 一句话：极小的文件（如几十字节的配置文件）不单独分配数据块，而是直接存在元数据对中。
- 类比：零宽字符串优化（SSO）——C++ 中小字符串直接存在栈上而不是堆上。
- 常见场景：WiFi 配置、设备 ID、小的计数器文件。
- 坑点：文件大小超过 `inline_max` 后不能再内联，但如果以后又缩小，也不会自动回到内联状态。

**卡片 8：Lookahead Buffer（向前看缓冲区）**
- 一句话：一个位图缓冲区，用于高效跟踪空闲块分配状态，大小固定不随存储增长。
- 类比：位图索引——用 1 字节跟踪 8 个块，500KB 的 Flash 可用小于 1KB RAM 完成分配追踪。
- 常见场景：块分配器的空闲空间管理。
- 坑点：位图只覆盖最近扫描过的区域，块较多时分配器需要周期性重新扫描，产生不确定的延迟。

**卡片 9：Superblock（超级块）**
- 一句话：文件系统最顶层的元数据块，存储版本、块大小、块数量、文件名长度上限等全局参数。
- 类比：数据库的 catalog 表——定义了库的 schema 和配置参数。
- 常见场景：挂载时首先读取，确认文件系统格式兼容。
- 坑点：超级块损坏则整个文件系统不可挂载。littlefs 的超级块存在元数据对的一个特殊条目中，受掉电保护。

**卡片 10：编译时配置宏（Compile-Time Configuration）**
- 一句话：通过 `-DLFS_READONLY`、`-DLFS_NO_MALLOC`、`-DLFS_NAME_MAX=64` 等宏在编译阶段裁剪 unused code 和 RAM 占用。
- 类比：Webpack 的 tree-shaking 或 Go 的 build tags——编译时移除不需要的功能。
- 常见场景：ROM 只有 128KB 的 MCU 上，去掉写功能可以省 35% 代码空间。
- 坑点：配置宏太多会增加测试矩阵复杂度——不同组合可能出现不同的隐藏 bug。

   ```bash
   pip install littlefs-python
   python3 -c "

   lfs = littlefs.LFS()
   lfs.mount()
   lfs.mkdir('/data')
   with lfs.open('/data/config.txt', 'w') as f:

   ```

## 10. 3 道深度追问题

1. **时序与并发问题**：littlefs 的 CTZ 跳表在追加一个新数据块时，需要更新前面块的回溯指针。如果在这个过程中掉电，未完全更新的指针链会导致文件数据部分不可读。littlefs 用 COW 机制解决这个问题，但 COW 本身会产生写放大。对于频繁写入小文件的场景（例如每秒写一次温度日志），你会如何权衡内联文件、metadata compaction 频率和块擦除寿命之间的关系？

2. **软件视角转嵌入式视角**：在后端系统中，如果一个文件系统返回 EIO 错误，通常做法是重试或报错。但在嵌入式场景中，Flash 块出现坏块（bad block）时，littlefs 可以通过 `LFS_ERR_CORRUPT` 从 prog/erase 回调中识别，并把这个块标记为不再使用。请分析这种"运行时坏块管理"和"运行时重试"在工程可靠性哲学上的根本区别——为什么嵌入式倾向于接受局部失效并继续运行，而服务端倾向于快速失败？

3. **资源限制问题**：littlefs 的 lookahead buffer 大小固定（例如 1KB 可以追踪 8192 个块的分配），但有一个隐含限制：当文件系统非常大（例如 512MB 的 Flash）时，单次 lookahead 扫描只能覆盖一小部分块，分配器可能需要多次扫描才能找到空闲块。这会导致写入延迟的 jitter。如果你在设计一个需要低延迟实时写入的系统（如音频录制），你会如何修改分配策略？

## 11. 今日复盘问题

**问题**：你在后端开发中可能习惯了"存储"就是"装在服务器里的 SSD/NVMe，容量几百 GB 起步，文件系统是 ext4/XFS，IOPS 够用"。littlefs 的整个设计都在和"只有几百 KB 的 RAM、几 MB 的 Flash、十万次擦除寿命"做斗争。那么，你觉得在"资源极其有限"这个限制条件之下，littlefs 的哪些设计选择（COW、元数据对、CTZ 跳表、内联文件）是你在后端系统设计中从没考虑过的？反过来，后端系统如果能借鉴 littlefs 的 RAM 有界理念，是否能设计出更可预测延迟的存储系统？

## 12. 工程点评

- **可维护性**：代码质量很高，lfs.h 中每个结构体、枚举、函数都有详细注释，超过三分之一行是注释。DESIGN.md 是嵌入式领域不可多得的设计文献。变量命名清晰一致。唯一不足是 lfs.c 中有些函数较长（如 compact），需要一定耐心阅读。

- **可移植性**：通过 `lfs_util.h` 的宏和回调函数，可以轻松移植到任意平台。已经有 Rust、JavaScript、Python、Nim、OCaml 的绑定或重实现。Mbed OS 将其作为官方文件系统。

- **可靠性**：掉电安全设计经过严格数学推理和测试验证。测试套件用全排列参数运行，模拟各种掉电时刻和坏块场景。SPEC.md 精确定义了每一位的磁盘布局，可用于独立实现和一致性校验。

- **可测试性**：PC 上可运行的模拟块设备（`lfs_emubd`）使得几乎所有测试不需要硬件。测试用例以 TOML 格式声明，参数覆盖全面。

- **量产/运维**：littlefs 已被大量产品使用（ESP32 的 SPIFFS 替代方案、Mbed OS 的默认文件系统、各种 IoT 设备）。没有内置文件系统检查工具（fsck），但有 littlefs-python 和 littlefs-fuse 可用于 PC 上检查镜像。

## 13. 明天可以延展的方向

1. **对比学习 SPIFFS**：SPIFFS 是另一个嵌入式文件系统，采用纯日志结构设计（log-structured），在与 littlefs 同样的约束下做了不同的取舍。对比两者的设计文档，理解"日志结构 vs 元数据对 + COW"的工程权衡。

2. **研究 littlefs 的坏块管理和磨损均衡实现细节**：深入看 `lfs_alloc()` 中的 block_cycles 如何影响分配决策，以及 bad block 检测的闭环流程。

3. **在自己的 MCU 上部署 littlefs**：如果手头有 STM32 或 ESP32 开发板，尝试移植 littlefs 到一块 SPI Flash 芯片上，并用逻辑分析仪观察 read/prog/erase 操作的时序。
