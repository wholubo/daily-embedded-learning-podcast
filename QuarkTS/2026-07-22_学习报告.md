# 每日嵌入式学习：QuarkTS

## 1. GitHub 项目链接与一句话介绍

**仓库地址**：https://github.com/kmilo17pet/QuarkTS

**一句话介绍**：QuarkTS 是一个面向资源受限微控制器的开源协作式（非抢占）操作系统，它填补了裸机编程和完整 RTOS 之间的空白，让开发者在不引入抢占式调度复杂性的前提下，获得事件驱动多任务、状态机、协程、软件定时器和任务间通信等现代嵌入式编程能力。

## 2. 为什么今天选它

**活跃度与成熟度**：截至 2026 年 7 月，QuarkTS 已发布到 v7.3.4，拥有 347 颗星，持续 7 年以上维护。CI/CD 流水线覆盖了静态分析、编译测试、文档生成，说明工程成熟度很高。

**工程完整度**：项目通过 MISRA C:2012 和 CERT 编码标准合规检查，Codacy、CodeFactor、DeepSource 三套静态分析全绿，代码质量经得起推敲。这对软件工程师是很大的信任信号。

**软件工程师学习嵌入式的最佳切入点**：大多数从后端/全栈转嵌入式的开发者，第一个困惑就是"我到底需不需要 RTOS？裸机循环有什么不好？"QuarkTS 恰好回答了这个问题——它用**协作式调度**（非抢占）展示了"比裸机多一点"的中间道路。你不需要理解抢占、临界区嵌套、优先级反转这些 RTOS 特有的复杂概念，就能写出事件驱动多任务程序。同时它的源码极精简（核心内核不足 2000 行），适合作为第一个深入研读的嵌入式操作系统源码。

## 3. 项目解决的问题

### 真实场景

在低成本 MCU（如 Cortex-M0、PIC、AVR，只有 4～32KB RAM）上，跑一个完整 RTOS 往往太奢侈——FreeRTOS 内核本身就占 4～12KB ROM，还要为每个任务分配独立栈空间。但裸机超级循环又难以管理超过 3～5 个任务，特别是事件驱动场景。QuarkTS 解决的就是这个两难局面。

典型应用场景包括：
- 汽车控制（门窗、灯光控制）——不需要抢占，但需要可靠的定时任务
- 工业监控（传感器数据采集、简单控制逻辑）——事件驱动自然
- IoT 终端节点——低功耗、低资源，同时需要有状态管理能力

### 额外约束

与普通软件项目相比，QuarkTS 所面对的问题域多了以下约束：

- **ROM/RAM 极度有限**：配置项中 `Q_DEFAULT_HEAP_SIZE` 默认只有 512 字节。这意味着你不能随心所欲地分配内存，所有数据结构和任务控制块都必须精心设计尺寸。
- **无 MMU/MPU**：没有虚拟内存，没有内存保护，一个野指针就能摧毁整个系统。所以 MISRA-C 合规是生存需求，不是锦上添花。
- **单核无抢占**：CPU 一次只能做一件事。协作式调度要求任务"自愿让出 CPU"，一个写得不合适的任务可能阻塞整个系统。
- **时序确定性**：嵌入式系统往往有实时性要求——传感器每 10ms 必须被读取一次。QuarkTS 的定时精度直接依赖 MCU 的硬件定时器。

## 4. 技术架构分析

### 整体模块架构

QuarkTS 采用模块化分层设计，核心层和扩展层通过统一的配置头文件 `qconfig.h` 裁剪。

**内核层（必须）**：
- `qkernel.c` — 内核控制块（KCB）、调度主循环、优先级队列管理
- `qtasks.c` — 任务生命周期管理（创建/删除/挂起/恢复）
- `qclock.c` — 时间基准和滴答提供者
- `qlists.c` — 内核内部双向链表
- `qcritical.c` — 临界区抽象（中断屏蔽）
- `qtypes.h` — 内核数据类型定义

**扩展层（可选，通过配置宏启用）**：
- `qfsm.c` — 层次化有限状态机（支持嵌套、历史模式、发布订阅）
- `qcoroutine.c` — 协程框架（带信号量同步）
- `qqueues.c` — 任务间消息队列
- `qstimers.c` — 软件定时器
- `qatcli.c` — AT 命令式命令行接口
- `qmemmang.c` — 静态内存管理器
- `qbsbuffers.c` — 字节缓冲区
- `qedgecheck.c` — 数字输入边沿检测
- `qioutils.c` — IO 工具（去抖、状态读取）
- `qresponse.c` — 响应处理器
- `qtrace.c` — 调试跟踪

### 调度模型：合作式 + 优先级轮转

调度算法是 QuarkTS 最核心的设计：

1. **运行至完成（RTC）模型**：每个任务从入口开始执行，直到主动返回（return）才交出 CPU。其间不会被其他任务打断。
2. **优先级轮转链式队列**：内核维护 `Q_PRIORITY_LEVELS`（默认 3）个就绪链表 + 1 个等待链表 + 1 个挂起链表。调度时从最高优先级链表取第一个节点，执行完后放回链表尾部——同优先级任务之间是轮转调度。
3. **事件队列**：`Q_PRIO_QUEUE_SIZE`（默认 10）的 FIFO 优先级队列，存放异步事件。内核会从中取出最高优先级的待处理事件。

### 运行流程

```
系统上电
  → qOS_Setup() 初始化 KCB、链表、时钟
  → 用户调用 qOS_Task_Add() 添加任务
  → qOS_Run() 进入调度主循环：
      循环：
        1. qOS_CheckForReadyTasks() — 遍历等待链表，将超时/事件触发的任务移回就绪链表
        2. 从就绪链表中按优先级选择下一个任务
        3. 调用任务回调函数，传入事件结构体
        4. 任务执行结束返回后，若配置了通知扩散，处理之
        5. 调用 idleCallback 做后台工作
        6. 调用 coreIdleFcn 让 CPU 进入休眠（可选）
```

### 硬件抽象

QuarkTS 不依赖特定硬件，通过 `qport.c` 和 `qport.h`（需用户实现）完成移植：
- `qGetTickFcn_t` — 提供系统滴答的获取函数
- 临界区通过 `qCritical_Enter()`/`qCritical_Exit()` 宏实现（通常对应关/开中断）
- 定时器基准由 `qClock_SetTimeBase()` 配置

### 配置系统

所有功能通过 `qconfig.h` 中的宏开关启用/禁用。例如 `Q_FSM` 控制是否编译状态机模块，`Q_QUEUES` 控制消息队列，`Q_ATCLI` 控制 AT 命令行。这种设计使最终 ROM 占用可精确裁剪到只含所需功能。

## 6. 核心模块/函数导读

### 6.1 `qOS_Setup()` — 内核初始化入口

**角色**：这是系统运行的起点。完成 KCB 清空、链表初始化、时钟提供者注册。如果开启了规范时间模式（`Q_SETUP_TIME_CANONICAL`），则假设滴答周期为 1ms。

**与启动流程的关系**：在 main() 中第一个调用。返回后系统滴答尚未开始，需要用户调用 `qOS_Run()` 才能启动调度。这种"两步启动"设计让用户有机会在调度启动前完成外设初始化和任务注册。

**关键设计决策**：所有链表在初始化时都是空链表，任务通过 `qOS_Task_Add()` 动态加入。内核不预分配任何任务槽位——这是与 FreeRTOS 等静态配置 RTOS 的重要区别。

### 6.2 `qOS_Run()` — 调度主循环

**角色**：永不返回的主循环。这是整个操作系统的引擎。

**实现要点**：
```
while(1) {
    if (qCoreFlags.NOTIFICATION_SPREAD) 处理通知扩散
    qOS_CheckForReadyTasks()    // 检查哪些等待任务就绪了
    qOS_DispatchTasks()         // 从就绪链表选任务执行
    if (idleCallback) 执行空闲回调
    if (cpuIdle) 进入低功耗
}
```

**与 RTOS 对比**：这里没有 PendSV 异常、没有上下文保存/恢复、没有栈切换——因为协作式调度不需要。每个任务运行在自己的"当前栈"上，函数返回即交还 CPU。

### 6.3 `qOS_Task_Add()` — 任务注册

**角色**：将用户定义的任务函数注册到内核调度表中。每个任务需要指定优先级、周期/触发模式、参数指针和事件标志需求。

**任务类型**：
- `qPeriodic`：定时循环执行（负值表示"相对时间"，正值表示"绝对值"）
- `qTask_Event`：由事件驱动（通知、队列、事件标志）
- `qTask_OneShot`：只执行一次

**与软件工程类比**：类似于在 Linux 上注册一个 cron 作业，但不同的是每个任务有自己的"事件槽"，可以接收异步信号。

### 6.4 `qStateMachine_Dispatch()` — 层次状态机调度

**角色**：在 FSM 模式下，每个状态机实例监听输入信号，按转换表查找匹配的源状态-信号对，执行转换动作。核心算法是标准的 HSM 转换协议：
1. 从当前状态向祖先链查找转换表
2. 找到匹配的转换后，从当前状态逐级 Exit 到 LCA
3. 执行转换动作
4. 从 LCA 逐级 Entry 到目标状态

**为什么重要**：大多数嵌入式系统天然是状态驱动的（传感器初始化→待机→测量→上报→休眠）。QuarkTS 的 FSM 模块把状态机直接集成到调度器里，每个"信号"可以是一个任务向另一个任务发的通知，实现了"事件驱动多任务 + 状态机"的组合。

### 6.5 `qCR_ExternControl()` — 协程外部控制

**角色**：允许外部代码控制一个协程的执行位置。可以重启（回到函数开头）、暂停（记住当前位置）、恢复（从上次暂停处继续）、跳转到指定位置。

**与软件工程类比**：协程的 `instr` 字段就是一个"程序计数器"，保存了上一个 `qCR_DELAY`、`qCR_WAIT_UNTIL` 或 `qCR_SEM_WAIT` 语句的下一条指令索引。这跟 Python 的 generator 中 `yield` 保存栈帧异曲同工，只是 QuarkTS 的做法更原始但 ROM 开销极低。

## 7. 软件工程师类比解释

### 7.1 协作式调度 ≈ Node.js 事件循环

在 Node.js 中，一个长时间运行的同步操作（比如 while 循环）会阻塞整个事件循环，其他回调处理不了。QuarkTS 的协作式调度完全一样——如果一个任务在函数里死循环，其他任务永远得不到 CPU。**这个类比的局限**：Node.js 的事件循环本质上是单线程异步 I/O 模型，I/O 操作靠底层 C++ 线程池完成；QuarkTS 是纯同步协作，没有任何后台线程。

### 7.2 优先级轮转链表 ≈ Kubernetes 调度队列

KCM 把不同优先级的 Pod 放入不同队列，高优先级队列的 Pod 先被调度。QuarkTS 的优先级就绪链表也一样，但更简单——Kubernetes 有几十种调度策略和抢占机制；QuarkTS 只有一条规则：执行完最高优先级链表的第一个任务，然后放到链表尾部。**这个类比的局限**：Kubernetes 是分布式调度器，节点独立；QuarkTS 是单核单线程，不存在分布式一致性问题。

### 7.3 层次状态机 ≈ React useReducer + 有限状态

React 的 `useReducer` 让组件状态变更变得可预测——给定当前状态和 action，返回新状态。HSM 的核心思想一样，只是多了嵌套和继承。**这个类比的局限**：React reducer 是纯函数，不涉及硬件寄存器状态；嵌入式 HSM 的 "entry/exit action" 可能操作 GPIO、启动 ADC、配置 DMA，是有副作用的。

### 7.4 静态内存管理器 ≈ 预分配连接池

后端应用通常用连接池避免每次请求都新建数据库连接。QuarkTS 的 `qmemmang` 做同样的事：在编译时或初始化时分配固定大小的内存池，运行时只做"分块分配"和"释放回池"，没有碎片问题。**这个类比的局限**：数据库连接池通常能动态扩容；QuarkTS 的内存池大小是编译期确定的，超出就分配失败。

### 7.5 临界区 ≈ 数据库事务隔离级别

数据库中的 `SERIALIZABLE` 隔离级别保证事务在完全隔离的环境中执行。QuarkTS 的临界区（`qCritical_Enter/qCritical_Exit`）也类似——进入后禁用中断，保证当前代码不会被中断服务程序干扰。**这个类比的局限**：数据库事务隔离通过 MVCC 或锁实现，可以并发执行；临界区是全局关中断，任何中断都会丢失，必须极短（微秒级）。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：RTC（Run To Completion）
- **一句话**：任务从开始执行到主动返回，中间不会被切换。
- **软件工程类比**：就像 Express.js 中的同步路由处理函数——必须调用 `res.send()` 才算完成。
- **常见工程场景**：协作式调度、事件驱动处理、简单控制回路。
- **误区**：RTC 不意味着"不能有 while 循环"——只是循环必须包含主动让出 CPU 的条件。

### 卡片 2：协作式调度（Cooperative Scheduling）
- **一句话**：任务自己决定何时交还 CPU，而不是被定时器中断强制切换。
- **软件工程类比**：协程的 `await` 点。
- **常见工程场景**：资源极受限的 MCU、安全和可靠性优先（避免抢占引入的数据竞争）。
- **误区**：协作式调度不能用在"有硬实时要求"的场景——其实如果每个任务执行时间确定且足够短，反而比抢占式调度更可预测。

### 卡片 3：TCB / KCB（任务/内核控制块）
- **一句话**：每个任务/内核有一个结构体存储其所有状态（优先级、事件、生命周期等）。
- **软件工程类比**：Linux 的 `task_struct`，或 Go 的 `g` 结构体。
- **常见工程场景**：任何 OS 的心脏数据结构。
- **误区**：TCB 越大功能越强——嵌入式系统中 TCB 必须精打细算，QuarkTS 的 `qTask_t` 才几十字节。

### 卡片 4：HSM（层次状态机）
- **一句话**：状态可以嵌套，子状态继承父状态的行为。
- **软件工程类比**：CSS 类继承——子元素继承父元素的样式，除非显式覆盖。
- **常见工程场景**：通信协议解析、UI 导航、设备初始化序列。
- **误区**：HSM 一定比平面状态机好——对于少于 10 个状态的场景，平面状态机更清晰。

### 卡片 5：临界区（Critical Section）
- **一句话**：不能被中断打断的代码段，通常通过关中断实现。
- **软件工程类比**：`sychronized` 块或 `spin_lock`。
- **常见工程场景**：操作共享变量、修改任务状态、读写外设寄存器。
- **误区**：临界区越长越好——实际上临界区应该在微秒内完成，否则会丢失实时事件。

### 卡片 6：软件定时器（Software Timer）
- **一句话**：由操作系统管理的一段时间后触发的回调机制，不需要硬件定时器资源。
- **软件工程类比**：JavaScript 的 `setTimeout()`。
- **常见工程场景**：周期性数据采集、看门狗刷新、LED 闪烁。
- **误区**：软件定时器精度和硬件定时器一样——实际上软件定时器精度受系统滴答周期限制（通常 1ms～10ms）。

### 卡片 7：任务通知（Task Notification）
- **一句话**：一个任务可以直接向另一个任务发送"信号"，后者可以在下次调度时处理。
- **软件工程类比**：Channels in Go——`ch <- data` → `<-ch`。
- **常见工程场景**：中断处理函数通知主循环数据就绪、传感器事件通知控制任务。
- **误区**：通知是带数据的——QuarkTS 支持"简单通知"（只发信号）和"排队通知"（带数据），容量受优先级队列大小限制。

### 卡片 8：优先级队列（Priority Queue）
- **一句话**：按优先级排列的异步事件缓冲，调度器每次从中取最高优先级的事件。
- **软件工程类比**：Redis 的 sorted set，按 score（优先级）排序。
- **常见工程场景**：中断服务程序在短时间内触发多个事件，系统按优先级依次处理。
- **误区**：优先级越高执行越快——在协作式调度中，"优先级"只决定下一个轮到谁，不改变当前任务的速度。

### 卡片 9：MISRA-C
- **一句话**：汽车工业制定的 C 语言安全编码标准，禁止使用容易出错的语法特性。
- **软件工程类比**：ESLint 的严格模式（`strict`）加上 Airbnb 风格指南。
- **常见工程场景**：汽车、医疗、航空航天等对代码安全性有强制性要求的行业。
- **误区**：MISRA-C = 慢代码——良好设计的 MISRA-C 代码性能与非 MISRA 代码无异，它只是限制了指针运算、递归、动态内存分配等危险操作。

### 卡片 10：硬件抽象层移植（Porting）
- **一句话**：让 OS 适配不同 MCU 所需的最小代码集（通常只有 2 个文件：`qport.c` 和 `qport.h`）。
- **软件工程类比**：Docker——一次封装，到处运行（但封装层越薄越好）。
- **常见工程场景**：把 RTOS/协作式内核从一个芯片移植到另一个芯片。
- **误区**：移植只是换几个宏定义——实际上每种 MCU 的中断系统、定时器、休眠模式可能差异巨大，需要仔细处理。

## 9. 30～60 分钟动手任务

### 任务：在模拟环境中运行 QuarkTS 并跟踪调度流程

**目标**：通过 QEMU 模拟 STM32 或使用 Linux 本地测试环境，实际运行 QuarkTS 的示例代码，观察协作式调度如何按优先级和时间片分配 CPU。

**步骤**：

1. **获取代码**：
```bash
git clone https://github.com/kmilo17pet/QuarkTS.git
cd QuarkTS
git submodule init
git submodule update
```

2. **阅读示例**：打开 `src/x86_test_do_not_use_it/x86_test.c`——这是专为 x86/Linux 设计的测试程序，可以在没有硬件的情况下直接运行。

3. **在 Linux 上编译运行**：
```bash
make -f Makefile TARGET=x86
./build/quarkts_x86_test
```

4. **观察输出**：注意各任务的打印顺序、时间戳、任务间的调度时序。

5. **修改配置**：在 `qconfig.h` 中将 `Q_PRIORITY_LEVELS` 从 3 改为 1，重新编译运行，观察调度行为如何退化为简单的轮转。

6. **添加自己的任务**：
```c
void myTask(qEvent_t e) {
    static int counter = 0;
    qClock_Delay(1000);  // 每 1 秒执行一次
    printf("My task: %d\n", counter++);
}

int main() {
    qOS_Setup(qClock_Tick, 1000, NULL, NULL);
    qOS_Task_Add(myTask, 0, Q_PERIODIC, NULL);
    qOS_Run();
    return 0;
}
```

**验收标准**：
- 能看到不同优先级的任务按预期顺序执行
- 修改优先级数量后，调度行为发生变化
- 你自己的任务每 1 秒打印一次计数

**无硬件替代**：上述步骤完全在 Linux 上完成，不需要任何 MCU 开发板。

## 10. 3 道深度追问题

### 问题 1（时序/并发）
假设你的 QuarkTS 系统中有三个任务：
- 任务 A：高优先级，每 5ms 执行一次，耗时 2ms
- 任务 B：中优先级，每 10ms 执行一次，耗时 1ms
- 任务 C：低优先级，每 20ms 执行一次，耗时 15ms

任务 C 一旦开始执行会持续 15ms，这期间即使任务 A 的 5ms 周期到了，也只能等待任务 C 返回。请问在这种协作式调度下，任务 A 的最大响应延迟是多少？如果要求任务 A 的延迟不超过 3ms，你会如何修改这个系统的设计？

### 问题 2（软件工程 → 嵌入式转换）
你是一名后端工程师，习惯了用 Python 的 asyncio 写异步代码。现在你要把一段后端服务（多个协程并发处理 HTTP 请求 + 定时任务清理缓存）移植到 QuarkTS 上运行。请列出至少 4 个从 asyncio 迁移到 QuarkTS 时必须注意的差异点，并说明为什么这些差异是嵌入式系统的根本约束。

### 问题 3（工程判断）
QuarkTS 的 `qfsm.c` 实现了发布-订阅信号机制（Publish-Subscribe）。它允许一个 FSM 实例向多个"订阅者"发送信号。但该机制受 `Q_FSM_PS_SIGNALS_MAX` 和 `Q_FSM_PS_SUB_PER_SIGNAL_MAX` 限制。如果让你设计一个工业生产线控制系统，有 20 个不同的信号类型和 50 个状态机实例，你会选择扩展这两个宏的数值，还是换用另一种通信架构？请说明你的判断依据。

## 12. 工程点评

### 可维护性 ⭐⭐⭐⭐⭐
QuarkTS 的代码风格统一、注释详尽、遵循 Doxygen 文档规范。MISRA-C 和 CERT 合规保证了代码的一致性和可读性。唯一的缺点是子模块结构（内核在独立分支）增加了初次上手的认知负担。

### 可移植性 ⭐⭐⭐⭐
移植层设计极简（只需 `qport.c/h`），配置宏的模块化裁剪使适配新 MCU 的工作量很小。但官方未提供常见 MCU 的现成移植示例，用户需要从零写 `qport.c`，这是减分项。

### 可靠性 ⭐⭐⭐⭐⭐
三重静态分析（Codacy、CodeFactor、DeepSource）+ MISRA-C 合规 + CERT 合规 + 持续集成的单元测试。对于一个 4000+ 行 C 代码的项目来说，这个质量保障体系是罕见的。

### 可测试性 ⭐⭐⭐
项目包含 x86 测试代码，但没有独立的单元测试框架（unity 目录存在但未集成到 CI）。对于安全关键系统，建议用户补充自己的测试套件。

### 量产/运维 ⭐⭐⭐
AT CLI 模块提供了运行时命令行交互能力，适合产测和现场调试。但缺少运行时错误统计、栈使用分析等高级诊断功能。适合资源极度受限的产品；对于资源稍丰富的系统，可以考虑 QuarkTS + 串口日志的轻量化运维方案。

## 13. 明天可以延展的方向

1. **抢占式 vs 协作式深度对比**：在 QuarkTS 的基础上加入一个抢占式 RTOS（如 FreeRTOS 或 RT-Thread）的源码陪读，对比两者的调度器实现、上下文切换开销和设计哲学。特别关注"什么场景必须用抢占"和"什么场景协作式反而更好"。

2. **HSM 在工程中的应用实践**：选择一个实际嵌入式协议（如 MQTT 客户端状态机、CAN 协议）用 QuarkTS 的 HSM 模块实现，然后对比使用平面状态机或第三方状态机库的实现差异。

3. **QuarkTS 的实时性分析**：用逻辑分析仪或示波器测量 QuarkTS 协作式调度在不同任务负载下的最大响应延迟，与 FreeRTOS 在同样硬件上的延迟做对比，验证"协作式调度延迟更可预测"这一理论。
