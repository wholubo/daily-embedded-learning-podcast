今天的嵌入式学习，我们来看一个很适合做系统化阅读的开源项目，名字叫 ZSWatch。它不是一个只会闪灯、显示一行字或者读取单个传感器的小 demo，而是一块从硬件到固件都开放出来的 Zephyr 智能手表。项目的核心是一颗 Nordic nRF5340 蓝牙芯片，旁边配着圆形触摸屏、惯性传感器、气压计、磁力计、外部闪存、光照传感器、实时时钟、麦克风和电源管理芯片。换句话说，它把一个可穿戴设备里经常遇到的问题，几乎都放到了同一个工程里。

今天选择 ZSWatch，有一个很重要的原因：它和之前讲过的 LVGL、Trice、Meshtastic Firmware 都不是同一个项目，也不是这些项目的改名版本或者分支。它的学习难度也比较合适。你最好已经会一点 C 语言，知道实时操作系统大概是什么，接触过单片机、蓝牙、传感器或者显示屏，但不需要具备特别重的行业背景。它真正有价值的地方，是让我们看到一个嵌入式产品怎样从板级配置、驱动、应用框架、手机通信、图形界面和电源管理这些环节一起工作。

从公开信息看，ZSWatch 是一个近期仍然很活跃的项目。GitHub 上的仓库不是 fork，主语言是 C，项目描述明确写着它是一个基于 Zephyr 的开源智能手表，包含硬件和固件。本次检索时，它大约有三千二百多个 star，三百多个 fork，开放问题接近三十个。仓库的话题标签包含 Zephyr、nRF5340、Nordic、蓝牙低功耗、显示、传感器和智能手表。最近一次推送发生在二零二六年五月二十三日，最近提交还在处理非常具体的工程问题，比如提高显示供电电压来修复最高亮度下的闪烁，以及更新上滑返回手势的文档。这类提交说明项目不是只停留在概念展示，而是在继续处理真实硬件和用户体验细节。

ZSWatch 想解决的问题，可以概括成一句话：怎样把一个现代智能手表做成可学习、可复现、可扩展的开源嵌入式系统。很多学习项目只覆盖一个点，比如点亮屏幕、读取加速度计，或者跑一个蓝牙外设服务。可是一个真正能戴在手腕上的设备，需要同时面对硬件选型、板级支持包、传感器采集、图形界面、手机通知、电池充电、外部存储、固件升级、调试接口、测试流程、外壳结构和用户交互。ZSWatch 的好处，就是把这些内容放在一套相对完整的 Zephyr 工程里，让我们能看到模块之间怎样配合。

先看硬件。项目使用的 nRF5340 是一颗双核蓝牙系统级芯片，可以简单理解为一个核心更偏应用处理，另一个核心更偏无线通信。README 里提到它有一百二十八兆赫兹双核、五百一十二 KB 内存、一 MB 片内闪存，并用三十兆赫兹的 SPI 去驱动显示屏。屏幕是二百四十乘二百四十的圆形 IPS 触摸屏。传感器方面，BMI270 惯性传感器负责手势导航、抬腕唤醒和计步；BMP581 气压计可以提供环境和高度相关数据；LIS2MDLTR 磁力计可以用在指南针功能里；六十四 MB 外部闪存可以放资源、文件或更大的固件内容；光照传感器可以做自动亮度；RTC 用来保持时间和闹钟；I2S 麦克风可以支持录音；nPM1300 电源管理芯片负责电源和系统管理。

这些器件放在一起以后，难点就不只是把某个传感器读出来，而是怎样让它们以可维护的方式进入系统。比如自动亮度听起来只是调屏幕亮一点或暗一点，背后却涉及光照传感器的采样周期、屏幕背光或供电策略、用户界面的亮度设置，以及电池续航。再比如通知弹窗看似只是 UI，实际上牵涉蓝牙低功耗连接、手机协议、消息队列、界面抢占和低功耗唤醒。ZSWatch 很适合训练这种系统视角。

软件架构上，ZSWatch 基于 Zephyr 实时操作系统，也采用了 Nordic nRF Connect SDK 常见的工程组织方式。项目里能看到 west 配置、工程配置文件、Kconfig、设备树覆盖文件、系统构建配置、静态分区配置，以及针对不同硬件版本和开发板的板级配置。也就是说，它不是把所有硬件差异都硬编码到 C 文件里，而是通过 Zephyr 的配置体系，把“这块板子有什么外设、启用什么功能、怎么分区、怎么构建”这些信息拆开表达。

仓库根目录下，README 是项目总览，app 是固件主工程，hardware 和相关硬件资料承载板级设计，production test 表示项目考虑过生产测试，power debug 说明它关注功耗调试，resources 保存图片和资源，website 则保存项目网站和文档。真正读固件时，最重要的是进入 app 目录。这里的 CMake、Kconfig、工程配置、设备树、sysbuild 和 west 文件，构成了 Zephyr 项目的骨架。

再往下看 app 里的 src 目录，main.c 是应用启动入口。applications 目录放手表上的具体应用，比如设置、音乐控制、指南针和其他功能；ble 负责蓝牙通信；sensors 和 sensor fusion 处理传感器采集和融合；ui 负责界面；managers 体现应用或系统资源管理；filesystem 和扩展驱动与外部存储相关；fuel gauge 和 basic battery 关心电池状态；events 用来做系统内部事件传递。还有时钟、闹钟、设置、命令行、看门狗和崩溃转储等文件，这些都很像真实产品里必须存在的工程化能力。

如果你想动手学习，可以按一个循序渐进的路线来。第一步，不要急着全仓库搜索，而是先读项目总览、入门文档和工具链文档，弄清楚它支持哪些硬件，为什么推荐 WatchDK，以及构建、烧录的大致流程。第二步，进入 app 目录，把 west、工程配置、Kconfig、boards 目录和 sysbuild 配置串起来，理解一个 Zephyr 固件是怎样由板级配置、功能开关、设备树和 CMake 拼出来的。第三步，从 main.c 开始追应用启动链，再去看 applications、ui、events 和 managers，搞清楚手表应用怎样注册、怎样切换、怎样显示页面和图标。第四步，选择一个硬件功能做精读，比如 BMI270 的计步和手势、BMP581 的气压数据、光照传感器的自动亮度，或者蓝牙通知。不要贪多，一次只把一条链路追完整。第五步，如果手里没有硬件，可以先尝试模拟构建或阅读脚本；如果有 WatchDK，就可以按文档编译、烧录、看日志，然后做一个小修改，比如改一个表盘文字、调整一个采样间隔，或者观察亮度策略。

这个项目里有十个概念特别值得今天记住。第一个是 Zephyr RTOS。它是一个面向嵌入式设备的实时操作系统，提供线程、驱动模型、设备树、配置系统、网络和蓝牙能力。在 ZSWatch 里，它是整个固件的底座。工程师需要懂它，因为复杂 MCU 产品越来越少只靠一个大循环解决问题。

第二个是 nRF5340。它是 Nordic 的双核蓝牙芯片，一个核心偏应用，一个核心偏无线网络。ZSWatch 用它做主控和蓝牙连接。理解这类芯片，有助于理解无线协议栈、应用逻辑和低功耗策略之间的关系。

第三个是 BLE，也就是蓝牙低功耗。手表要连接手机、接收通知、控制音乐或和 companion app 通信，都离不开它。可穿戴和物联网设备经常要在低功耗下传输少量但频繁的数据，所以 BLE 是非常核心的能力。

第四个是设备树覆盖文件。设备树用来描述硬件连接，覆盖文件用来针对某块板子或某种配置修改默认描述。ZSWatch 的 boards 目录里有很多这类文件。学会它，才能理解 Zephyr 下驱动和硬件之间是怎样绑定的。

第五个是 Kconfig。它是一套配置开关系统，用来决定启用哪些功能、驱动、日志和系统能力。嵌入式固件的体积、功耗和功能边界，很多时候都和这些配置直接相关。

第六个是 LVGL。它是轻量级图形库，常用于小屏幕嵌入式界面。ZSWatch 的表盘、应用界面和弹窗都需要图形框架。这里学习的重点不是重新讲 LVGL，而是看它如何和 Zephyr 的事件、显示驱动、应用管理结合起来。

第七个是 IMU，也就是惯性测量单元。ZSWatch 使用 BMI270 做手势、抬腕唤醒和计步。很多可穿戴、机器人和运动设备都需要从运动数据里判断用户动作，所以 IMU 是很常见的工程模块。

第八个是 PMIC，也就是电源管理芯片。ZSWatch 使用 nPM1300 来处理电源和系统管理。电池设备能不能稳定工作、续航能不能做好，很大程度取决于电源设计和软件策略。

第九个是外部闪存和 XIP。外部闪存可以放资源、文件或者更大的固件内容，XIP 则表示从外部存储中直接执行代码。带图形界面的设备很容易超过片内闪存容量，所以这部分能力很实用。

第十个是看门狗和崩溃转储。看门狗能在系统卡死时复位，崩溃转储能保存出错现场。ZSWatch 源码里能看到相关模块。真实产品不只是要能跑，还要在异常发生后能够恢复、定位和改进。

从工程角度评价，ZSWatch 的优点是结构完整、硬件资料开放、文档入口清晰、近期维护活跃，并且把开发板、最终手表、扩展硬件和手机应用生态都考虑进去了。它比普通裸机 demo 更接近产品工程，又比大型商业平台更容易从源码入手。它的挑战也很明显：配置体系比较多，牵涉 Zephyr、nRF Connect SDK、图形界面、蓝牙、传感器和电源管理，刚开始读可能会有点乱。另外它采用 GPL 三点零许可证，如果未来想借鉴到商业项目，需要认真评估许可义务。总体来说，ZSWatch 非常适合作为今天的嵌入式系统综合阅读材料，尤其适合已经写过一些 MCU 驱动、但想理解完整产品固件的人。
