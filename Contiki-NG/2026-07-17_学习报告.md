# 每日嵌入式学习：Contiki-NG

## 1. GitHub 项目链接与一句话介绍

- 仓库：https://github.com/contiki-ng/contiki-ng
- 一句话：Contiki-NG 是一个面向下一代物联网设备的开源跨平台操作系统，专注于可靠的低功耗通信和标准 IoT 协议（IPv6/6LoWPAN、6TiSCH、RPL、CoAP），让资源极度受限的 MCU 也能接入互联网。

## 2. 为什么今天选它

**活跃度**：截至 2026 年 7 月，项目有 1509 颗星、768 个 fork，三天前还有推送，CI 持续运行，社区活跃。从原来的 Contiki OS 彻底重写而来，代码质量高、架构清晰。

**工程完整度**：完整覆盖了 MCU 驱动层、RTOS 内核（protothread 事件驱动）、IPv6 网络栈、MAC 层（CSMA/TSCH/6TiSCH）、路由协议（RPL）、应用层（CoAP/MQTT/LwM2M）、存储系统、构建系统和模拟器（Cooja）支持。

**学习价值**：
- 对软件工程师而言，Contiki-NG 是理解"嵌入式网络栈怎么在 32KB RAM 的芯片上跑 IPv6"的绝佳教材。
- 它的 protothread 机制是嵌入式特有的"无栈线程"模式，和 RTOS 的线程模型完全不同，能帮软件工程师打开新思维。
- 代码用纯 C 写，清晰可读，没有复杂的模板或宏滥用。

**不重复确认**：历史项目列表中没有 Contiki-NG、Contiki、protothread 等关键词。

## 3. 项目解决的问题

**真实场景**：一个温度传感器节点，用纽扣电池供电，需要每隔几分钟上报数据到云端。它只有 16KB RAM、256KB Flash、没有 MMU、没有文件系统。Contiki-NG 让这样的设备能运行完整的 IPv6 协议栈，通过 6LoWPAN 压缩包头，通过 RPL 自动组网，通过 CoAP 和云端通信——整个过程功耗低到一颗电池撑两年。

**与普通软件项目的区别**：

- **内存约束**：整个 IPv6 栈加应用可能只需要几 KB RAM。在后端写 HTTP 服务时 1GB 内存都不够用，但在 Contiki-NG 上一个数据包缓冲区只有 128 字节。
- **时序敏感**：TSCH 时间同步 MAC 需要微秒级精度，数据发送必须在精确的时间槽内完成，错过窗口就只能等下一个周期。
- **无保护机制**：没有 MMU、没有用户态/内核态隔离。一个空指针就能让整个系统挂掉。
- **能量预算**：软件设计的核心指标是功耗，而不是吞吐量。CPU 大部分时间都在睡眠，只有在需要收发数据时才醒来。
- **网络不稳定**：无线链路随时可能丢失，路由拓扑自动变化，必须考虑丢包重传和路由修复。

## 4. 技术架构分析

Contiki-NG 的整体架构分为以下几层：

### 内核层 (os/sys/)
事件驱动的 protothread 内核。核心文件包括：
- `process.c` / `process.h`：进程管理和事件分发。维护一个全局事件队列，每个进程是一个 protothread。
- `pt.h`：protothread 实现——基于局部连续（local continuation）的轻量级无栈线程。
- `etimer.c` / `etimer.h`：事件定时器，定时到期后给指定进程发事件。
- `ctimer.c` / `ctimer.h`：回调定时器，到期直接调用回调函数（不经过事件队列）。
- `rtimer.c` / `rtimer.h`：实时定时器，用于硬实时场景（如 TSCH 时间槽）。
- `autostart.c`：系统启动时自动拉起注册的进程列表。
- `log.h`：按模块按级别的日志系统。
- `clock.c` / `clock.h`：时钟抽象，提供 `clock_time()` 和 `CLOCK_SECOND`。

### 网络栈 (os/net/)
这是 Contiki-NG 最核心的部分：
- `ipv6/`：完整的 IPv6 + 6LoWPAN 实现。包括邻居发现（NDP）、重复地址检测（DAD）、ICMPv6。
- `mac/`：MAC 层，支持 CSMA（载波侦听多路访问）、TSCH（时间同步信道跳频）、6TiSCH（IETF 标准化确定性网络）。
- `routing/`：路由协议，主要是 RPL（IPv6 低功耗有损网络路由协议），有 `rpl-lite`（轻量版）。
- `app-layer/`：应用层协议，主要是 CoAP（受约束应用协议）。
- `nullnet/`：最简单的 raw 数据包收发接口。
- `security/`：链路层安全（IEEE 802.15.4 安全帧）。

### 硬件抽象层 (arch/)
- `cpu/`：支持 ARM Cortex-M 系列（CC2538、CC26xx、nRF52840、Gecko、SimpleLink）、MSP430 等。
- `platform/`：各种开发板平台支持，包括 Cooja 模拟器（可以在 PC 上模拟运行）。
- `dev/`：通用设备驱动（radio、sensor、storage、ethernet）。

### 服务层 (os/services/)
包括：shell（命令行调试）、RPL Border Router（将 6LoWPAN 网络桥接到普通 IP 网络）、orchestra（TSCH 调度）、lwm2m（设备管理）、deployment（固件部署/OTA）、nat64（IPv6 到 IPv4 转换）。

### 存储层 (os/storage/)
- `cfs/`：Coffee File System，为 NOR Flash 设计的文件系统。
- `antelope/`：嵌入式数据库管理系统（DBMS）。

### 构建系统
基于 GNU Make，使用 `Makefile.include` 统一管理，自动识别目标平台、跨平台编译。支持 `make TARGET=xxx` 一次编译所有模块。

### 配置系统
通过 `contiki-conf.h` 和 `project-conf.h` 以宏定义方式配置。所有功能开关、缓冲区大小、定时器精度等都在编译期决定，避免运行时动态分配。

## 5. 源码陪读：今天只读哪几个文件

### 推荐阅读顺序和文件

**文件1：`os/sys/process.h` 和 `os/sys/process.c`**
- 阅读目标：理解 Contiki-NG 的事件驱动内核。看进程如何定义、如何启动、事件如何投递和分发、轮询机制如何与中断协作。
- 这是整个系统的"心脏"，所有业务逻辑最终都以进程形式跑在这上面。

**文件2：`os/sys/pt.h` 和 `os/sys/lc.h`**
- 阅读目标：理解 protothread（原型线程）的实现原理。只有几十行宏定义，却实现了完整的阻塞语义。注意 `LC_SET` 和 `LC_RESUME` 这对局部连续是如何用 C 语言 `switch` 语句模拟出来的。
- 这是 Contiki-NG 最独特的设计，也是理解"无栈并发"的关键。

**文件3：`os/sys/etimer.c` 和 `os/sys/etimer.h`**
- 阅读目标：理解"事件定时器"的工作流。看 `etimer_process` 本身是一个进程，维护一个定时器链表，每次 `process_run()` 循环中检查哪个定时器到期，然后给目标进程投递 `PROCESS_EVENT_TIMER` 事件。
- 这是一个内核服务以进程形式存在的范例。

**文件4：`os/net/ipv6/uip-ds6.h` 和 `os/net/routing/rpl-lite/rpl.c`**
- 阅读目标：理解嵌入式 IPv6 网络栈的数据结构设计。`uip-ds6.h` 中定义了邻居缓存、默认路由器列表、前缀列表等表结构。RPL 路由协议在低功耗网络中的实现值得关注，特别是 DODAG 构建和路由度量。

**文件5：`examples/hello-world/hello-world.c`**
- 阅读目标：完整的 protothread 应用示例。看 `PROCESS_THREAD` 宏如何定义一个进程入口、`PROCESS_BEGIN` 到 `PROCESS_END` 之间的代码如何实现阻塞等待、`etimer_set` 如何触发定时事件。
- 这是最简单的切入点，读完就能跑起来。

## 6. 核心模块/函数导读

### 6.1 `process_run()` —— 系统主循环
在 `main()` 中无限调用。它做两件事：先遍历所有请求轮询（poll）的进程，执行它们的轮询处理；然后从事件队列取出一个事件，调用目标进程的 `PROCESS_THREAD` 入口。返回值是队列中剩余事件数，主调方能据此决定是否让 CPU 进入睡眠。

工程角色：相当于 RTOS 中的调度循环。因为没有抢占式调度，所以"调度"就是顺序处理事件队列。这也意味着某个进程处理事件不能阻塞太久，否则整个系统的实时性都会受影响。

### 6.2 `process_post()` / `process_post_synch()` —— 事件投递
异步地投递一个事件到进程。事件队列是用固定大小环形缓冲区实现的，满了会返回 `PROCESS_ERR_FULL`。同步版本 `process_post_synch()` 不会经过队列，直接在当前上下文中调用目标进程的事件处理函数——这实际上是一个同步函数调用，但有完整的进程上下文切换。

工程角色：进程间通信的唯一机制。因为 Contiki-NG 的进程协作式运行，事件投递实际上就是调度手段。

### 6.3 `PROCESS_THREAD()` 宏与 `PT_BEGIN()` / `PT_END()`
每个进程必须用 `PROCESS_THREAD(name, ev, data)` 定义入口函数。内部用 `PT_BEGIN()` 到 `PT_END()` 包裹，中间可插入 `PT_WAIT_UNTIL(condition)` 等阻塞原语。编译器展开后其实是一个巨大的 `switch` 语句，每个阻塞点是一个 `case` 标号。这就是 protothread 的神奇之处——没有栈切换，没有线程上下文保存，仅靠局部连续实现函数级别的挂起和恢复。

工程角色：把事件驱动的状态机代码写成顺序逻辑，大大降低编程复杂度。但代价是局部变量不能跨越阻塞点（因为栈不保存）。

### 6.4 `etimer_set()` —— 设置事件定时器
创建一个定时器，指定到期时间（以 `CLOCK_SECOND` 为单位的 ticks）。当定时器到期时，事件定时器内核进程向设置定时器的进程投递 `PROCESS_EVENT_TIMER` 事件，进程在 `PROCESS_THREAD` 中通过条件 `ev == PROCESS_EVENT_TIMER` 捕获。

工程角色：时间是嵌入式系统中最重要的资源之一。`etimer` 把"定时"概念抽象成了"事件"，让应用代码不需要自己管理 ticks 计数。注意它有 `etimer_expired()` 查询接口、`etimer_restart()` 重启接口。

### 6.5 `autostart_start()` —— 自动启动进程
遍历 `autostart_processes` 数组，依次调用 `process_start()`。`autostart_processes` 是一个 `const` 指针数组，由 `AUTOSTART_PROCESSES()` 定义，放在 `process_list` 链表的末尾。

工程角色：系统启动流程的入口。所有"开机就要跑"的进程都在这里声明。对嵌入式系统而言，启动顺序是有讲究的——驱动进程需要先于网络进程启动，而网络进程需要先于应用进程启动。

## 7. 软件工程师类比解释

### 类比1：Protothread（原型线程） vs Python 协程（async/await）
- **类比**：Python 中 `async def` 定义的协程在 `await` 处挂起、恢复时继续执行。Contiki-NG 的 protothread 也是类似：一个函数在 `PT_WAIT_UNTIL(condition)` 处挂起，当条件满足时从同一位置恢复执行。两者都是协作式多任务，都没有真正的并行。
- **这个类比的局限**：Python 协程有自己的栈和堆分配的状态对象（`__await__`），而 protothread 根本没有栈——它只保存一个程序计数器（用 `switch` case 标号模拟）。Python 协程可以在内层函数中 `await`，protothread 的阻塞只能发生在 `PROCESS_THREAD` 函数的直接体内，不能跨函数调用。

### 类比2：Contiki-NG 事件队列 vs Node.js 事件循环
- **类比**：Node.js 中 `process.nextTick()` 或 `setTimeout()` 把回调放到事件队列里，事件循环按顺序取出执行。Contiki-NG 的 `process_post()` 也是把事件放进一个全局队列，`process_run()` 逐一取出分发给对应进程。两者都是单线程事件驱动模型。
- **这个类比的局限**：Node.js 的事件循环支持微任务和宏任务的多级队列，有优先级区分。Contiki-NG 的事件队列是一个简单的环形缓冲，所有事件优先级相同。另外 Node.js 的异步 I/O 背后是真正的线程池（libuv），而 Contiki-NG 的 I/O 是主动轮询或中断驱动的。

### 类比3：RPL 路由协议 vs Kubernetes 服务发现和负载均衡
- **类比**：RPL 让低功耗无线网络中的节点自动形成树状拓扑（DODAG），根节点通常是边界路由器，叶子节点是传感器。当一个节点加入网络，它自动找到父节点并建立路由。这类似于 Kubernetes 中 Pod 注册到 Service，自动获得 DNS 名称和负载均衡路由。
- **这个类比的局限**：K8s 的路由建立在稳定的数据中心网络上，链路可靠、带宽充足。RPL 必须应对无线链路的时断时续、节点移动、电池耗尽等不稳定因素，它需要不断修复破损链路、重新计算路由度量，同时控制协议开销以避免耗尽网络带宽。

### 类比4：Coffee File System（CFS） vs 关系型数据库中的 B-Tree
- **类比**：CFS 是为 NOR Flash 设计的日志结构文件系统，写入时追加新数据，标记旧数据为"脏"，垃圾回收时统一擦除。这类似于数据库的 WAL（预写日志）和 MVCC（多版本并发控制）——写入先记日志，读的时候能看到历史快照。
- **这个类比的局限**：CFS 的"日志"是直接写在 Flash 上的，它必须处理 Flash 的物理特性：需要先擦除再写入、擦除次数有限、按页写入按块擦除。数据库通常运行在有 FTL（闪存转换层）的 SSD 上，FTL 替你处理了这些物理约束。

### 类比5：6LoWPAN 包头压缩 vs HTTP/2 头部压缩（HPACK）
- **类比**：6LoWPAN 把 IPv6 的 40 字节包头压缩到只有几个字节，因为知道在同一 6LoWPAN 网络中很多字段（如链路本地前缀）是共享的。这类似于 HPACK 利用静态表+动态表+Huffman 编码将 HTTP 头部从几百字节压缩到几十字节。
- **这个类比的局限**：HTTP/2 的 HPACK 假设连接两端有协商状态，可以复用上下文。6LoWPAN 是无连接的，每个数据包都是独立的无线传输，压缩/解压在每次发送/接收时独立完成。而且 IEEE 802.15.4 的最大帧只有 127 字节，IPv6 最小 MTU 是 1280 字节——所以 6LoWPAN 不只是压缩，还要做分片和重组。

## 8. 今日 10 个嵌入式概念卡片

### 卡片1：Protothread（原型线程）
- **一句话**：一种无栈的轻量级线程，每个线程只保存一个程序计数器（几字节 RAM），通过 C 语言的 switch 技巧实现挂起和恢复。
- **软件工程类比**：类似于 Go 的 goroutine，但 goroutine 有动态栈（可增长到几 MB），protothread 完全没有栈。
- **常见工程场景**：事件驱动的协议栈（如 6LoWPAN 分片重组）、设备驱动中的状态机（如 I2C 读写序列）。
- **容易误解/坑点**：局部变量不能在 protothread 的阻塞点之间保持有效，必须用 `static` 或结构体成员。很多新手在这里栽跟头。

### 卡片2：无栈线程（Stackless Thread）
- **一句话**：线程不维护自己的调用栈，所有线程共享同一个 C 调用栈，挂起时只保存恢复点（program counter）。
- **软件工程类比**：类似于 Python generator（yield 语句），挂起时只保存执行位置。
- **常见工程场景**：内存极度受限的 MCU（< 4KB RAM）、几千个并发连接的 IoT 场景。
- **容易误解/坑点**：不能在阻塞点内调用普通函数，因为调用的函数有自己的栈帧，挂起后栈帧会被覆盖。

### 卡片3：6LoWPAN（IPv6 over Low-Power Wireless Personal Area Networks）
- **一句话**：让 IPv6 数据包能在 IEEE 802.15.4 这样的低功耗无线网络上传输的适配层协议，负责包头压缩、分片和重组。
- **软件工程类比**：TCP 分片 + HTTP 压缩的组合体。
- **常见工程场景**：智能家居（Zigbee 的 IP 替代方案）、工业无线传感器网络、城市照明控制。
- **容易误解/坑点**：6LoWPAN 不是一种无线技术，而是一个适配层；它可以在任何最大帧小于 IPv6 MTU 的链路上工作，不止限于 802.15.4。

### 卡片4：CoAP（Constrained Application Protocol）
- **一句话**：专为受限设备设计的 HTTP 替代品，使用 UDP 传输、二进制编码、支持观察者模式和资源发现。
- **软件工程类比**：把 RESTful API 搬到嵌入式设备上。GET 就像 HTTP GET，但 CoAP 的消息头只有 4 字节而不是几百字节。
- **常见工程场景**：传感器数据读取（GET /temperature）、设备控制（PUT /actuator）、固件 OTA 通知。
- **容易误解/坑点**：CoAP 使用 UDP 和重传机制实现可靠传输，不是 TCP。不要期望像 HTTP 那样有稳定长连接。

### 卡片5：TSCH（Time-Synchronized Channel Hopping）
- **一句话**：在多个频率信道上按时隙跳频的 MAC 协议，提供高可靠性和可预测延迟，是工业物联网 6TiSCH 标准的基础。
- **软件工程类比**：类似于分配了固定时隙的 TDMA（时分多址），再加上了 CDMA 的频率分集。
- **常见工程场景**：工业过程自动化、石油天然气管道监测，要求数据包到达率 > 99.999%。
- **容易误解/坑点**：TSCH 需要全网时间同步（通常由边界路由器发送信标实现），同步精度要求微秒级，软件工程师容易低估硬件定时器和中断延迟的影响。

### 卡片6：RPL（IPv6 Routing Protocol for Low-Power and Lossy Networks）
- **一句话**：让低功耗、高丢包率网络中的节点自动形成树状路由拓扑（DODAG）的距离矢量路由协议。
- **软件工程类比**：类似 BGP（边界网关协议）的链路状态和距离矢量混合体，但专门为链路时断时续的网络做了优化。
- **常见工程场景**：多个传感器节点通过多跳路由汇聚数据到网关。
- **容易误解/坑点**：RPL 的路由度量不是简单的跳数，还包括 ETX（预期传输次数）、延迟、能量等。只考虑跳数可能会选出一条丢包率极高的"捷径"。

### 卡片7：事件驱动内核（Event-driven Kernel）
- **一句话**：操作系统内核不主动调度线程，而是等待事件（定时器、数据到达、外部中断）发生后，按顺序处理事件队列中的事件。
- **软件工程类比**：浏览器的 JavaScript 事件循环，所有用户点击和网络回调都在同一个线程中顺序执行。
- **常见工程场景**：传感器网络节点、低功耗蓝牙设备——大部分时间在睡眠，事件来了才唤醒处理。
- **容易误解/坑点**：事件处理不能阻塞。如果你在事件处理里做 `while(1)` 或者 `delay(1000)`，整个系统就卡死了。这和 Linux 的多线程思维完全不同。

### 卡片8：局部连续（Local Continuation）
- **一句话**：用 C 语言的 `switch` 和函数内 `static` 变量实现的执行状态保存机制，是 protothread 的底层基础。
- **软件工程类比**：C 语言版的 setjmp/longjmp，但不需要跳转缓冲区。
- **常见工程场景**：contiki-ng 的 protothread 就是用局部连续实现的——实际上就是 `static int lc; switch(lc) { case 0: ... }` 模式。
- **容易误解/坑点**：由于依赖 C 语言 switch，阻塞点不能出现在内层函数中，只能出现在宏展开的同一函数内。也不能使用嵌套 switch（会冲突）。

### 卡片9：Coffee File System（CFS）
- **一句话**：为串行 NOR Flash 设计的日志结构文件系统，按页写入（256 字节）按扇区擦除（64KB），支持垃圾回收和磨损均衡。
- **软件工程类比**：类似于 LevelDB 的 LSM-Tree（日志结构合并树）——追加写入，后台合并压缩。
- **常见工程场景**：保存传感器配置参数、记录日志、缓存固件 OTA 包。
- **容易误解/坑点**：CFS 不能像 Linux ext4 那样"覆盖写入"一个文件。更新一个字节也要先标记旧页无效，再写入新页到空闲位置。而且 Flash 有擦除寿命（通常 10 万次）。

### 卡片10：Cooja 网络模拟器
- **一句话**：Contiki-NG 自带的 Java 编写的网络模拟器，可以在 PC 上模拟几十到几百个传感器节点的运行，逐步调试每个节点的代码。
- **软件工程类比**：类似于 mininet（网络拓扑模拟器）或 Docker Compose（多容器编排模拟），但 Cooja 可以在指令级模拟 MCU 执行。
- **常见工程场景**：在烧录到真实硬件之前，先在 Cooja 中验证网络协议的正确性和性能。
- **容易误解/坑点**：模拟器中的时间不是真实时间——一个模拟"小时"可能在 PC 上运行 30 秒。不要拿模拟器中的时序指标直接估算真实性能。

## 10. 3 道深度追问题

**问题1（时序/并发/资源限制）**：
假设你有一个 Contiki-NG 传感器节点，需要同时处理三个周期性任务：每 100ms 读取温度传感器、每 500ms 上报数据到服务器、每 1s 检查电池电压做低电量预警。三个任务都用 `etimer_set` 驱动。问：如果 `process_run()` 在处理温度读取时花费了 60ms（因为传感器 I2C 通信等待），这会影响到 500ms 上报任务的定时精度吗？如果会，如何在不增加硬件定时器中断个数的前提下解决？

**问题2（软件工程视角转嵌入式视角）**：
你在后端写过微服务，每个服务是一个独立进程，进程间通过 gRPC 通信，负载高了可以横向扩容。现在把同样的"微服务架构"思维移植到 Contiki-NG 上：每个功能模块（读传感器、网络上报、电池管理）拆成独立的 `PROCESS`，它们之间通过 `process_post` 传递事件通信。问：这种设计在嵌入式系统中的主要缺陷是什么？什么时候拆得太细反而比单进程更差？

**问题3（工程判断）**：
你的团队需要为一个产品选择 IoT 通信方案：方案 A 用 Contiki-NG + 6LoWPAN/RPL/CoAP（完整 IP 栈），方案 B 用简单的非 IP 私有协议（如基于 802.15.4 的原始帧直接通信）。方案 A 更标准、互操作性更好；方案 B 更简单、代码更少、RAM 占用更低、调试更容易。对于只有 8KB RAM、电池寿命要求 3 年的温湿度传感器节点，你会选哪个方案？给出具体理由，并说明在什么条件下你才会改变选择。

## 11. 今日复盘问题

当你阅读 `os/sys/process.c` 中 `process_run()` 的实现时，注意到它每次只处理一个事件然后就返回。这种"一次一个事件"的设计和 RTOS 中的"调度器跑完所有就绪线程"有什么本质区别？如果把这个循环改成一次处理完事件队列中的所有事件再返回，对系统的响应延迟和功耗会有什么影响？

## 12. 工程点评

### 可维护性 ★★★★☆
Contiki-NG 的代码风格统一，注释质量高（特别是 protothread 和网络栈的 Doxygen 注释）。模块划分清晰：`os/sys/` 是内核、`os/net/` 是网络、`arch/` 是硬件。不同 CPU 的驱动隔离得好。但部分网络协议代码（如经典的 uIP）历史包袱较重，宏嵌套深度大，新手上手有一定难度。

### 可移植性 ★★★★☆
通过 `arch/cpu/` 和 `arch/platform/` 两层抽象，移植到新 MCU 只需要实现 `clock`、`rtimer`、`radio` 等几个底层接口。构建系统通过 `TARGET` 变量自动切换。不足的是文档中对移植流程的说明不够详细，新手需要参照已有平台逆向推断。

### 可靠性 ★★★★☆
TSCH/6TiSCH 的确定性调度提供了工业级可靠性保证。事件队列用静态分配的环形缓冲区，不会出现动态内存泄漏。protothread 的协作风避免了竞态条件（没有抢占式调度就没有真正的竞争）。但是，没有内存保护（MMU）且单线程运行，一个野指针就能让整个系统崩溃。

### 可测试性 ★★★★★
这是 Contiki-NG 的突出亮点。Cooja 模拟器允许指令级调试和网络级模拟，每个节点都可以设置断点。还有 `tests/` 目录下有系统的自动化测试（编译测试、模拟运行测试、IPv6 一致性测试）。对于嵌入式项目来说，这种测试覆盖度相当优秀。

### 量产/运维角度 ★★★☆☆
Contiki-NG 有一定量产的工业部署案例（特别是在智慧城市、工业物联网领域），但与 FreeRTOS 或 Zephyr 相比，商业支持和生态规模明显较小。OTA 固件更新支持存在但不够成熟。日志系统功能完善，但远程调试和诊断能力不如商业 RTOS。

## 13. 明天可以延展的方向

1. **对比学习 Zephyr vs Contiki-NG**：Zephyr 使用标准的 RTOS 线程（有栈），Contiki-NG 使用 protothread（无栈）。对比两者的调度模型、内存开销、编程模型差异，能帮你深刻理解"什么时候该用有栈线程、什么时候无栈就够了"。

2. **深入 TSCH 实现**：本周选的 Contiki-NG 提到了 TSCH，但未深入时间槽调度算法的细节（如 Orchestra 的自治调度）。可以从 `os/services/orchestra/` 开始读，了解 TSCH 调度如何在无中心节点的情况下由各节点自协调完成。

3. **6LoWPAN 边界路由器搭建**：用一块 CC2538 或 nRF52840 开发板搭建一个真实的 RPL Border Router，把 6LoWPAN 网络接入家中 WiFi，然后从浏览器中用 CoAP 读取传感器数据。这是"嵌入式物联网全栈"的典型动手项目。
