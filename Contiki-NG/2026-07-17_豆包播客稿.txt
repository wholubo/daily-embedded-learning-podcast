今天我们要探索的不是一个普通的开源项目，而是一个真正意义上让微控制器联网的操作系统，叫做 Contiki-NG。它的全称是 Contiki Next Generation，也就是下一代 Contiki。这是专门为下一代物联网设备设计的操作系统，它解决的核心问题特别有意思：怎么让一个只有十几 KB 内存、电池还得撑好几年的微型芯片，跑起完整的互联网协议栈，还能跟云服务器通信。

Contiki-NG 的 GitHub 仓库目前有一千五百多颗星，将近八百个分支，而且非常活跃，就在几天前还有代码推送。它原本是从经典的 Contiki OS 分叉出来的，但经过了彻底的现代化重写，代码质量非常高。它的主要设计思路是围绕可靠的低功耗通信和标准物联网协议展开的，包括 IPv6 协议在低功耗网络上的压缩版本六楼波兰，时间同步信道跳频技术 TSCH，低功耗路由协议 RPL，以及受限制应用协议 CoAP。

咱们先从 Contiki-NG 的背景说起。想象一个场景：一个温度传感器节点，用纽扣电池供电，每隔几分钟要把数据上报到云端。它只有十六 KB 的 RAM，两百五十六 KB 的 Flash，没有内存管理单元，也没有文件系统。Contiki-NG 就是在这个极端的资源约束下，让设备能够运行完整的 IPv6 协议栈，通过六楼波兰压缩 IP 包头，通过自治组网形成网络拓扑，通过 CoAP 协议和云端通信，整个过程功耗低到一颗电池能用两年。

说到这里，你们可能会问，这不就跟咱们之前聊过的 Zephyr 或者 FreeRTOS 差不多吗？其实不一样。Contiki-NG 最大的特色是它的两个核心设计：一个是事件驱动内核，另一个是叫 Protothread 的并发模型。大多数传统 RTOS 的线程每个都有自己的独立栈空间，一个线程几百字节甚至几 KB 都不稀奇。但 Protothread 一个线程只占用两到十二个字节的状态存储，基本上就只保存了一个程序计数器。它是怎么做到的呢？它巧妙利用了 C 语言的 switch 语句，每个阻塞点对应一个 case 标号，函数被再次调用时从那个标号继续执行。这背后的原理叫做局部连续，实际上就是一个 static 变量加一个 switch 分支。

这个机制对软件工程师来说应该不陌生，它跟 Python 的 async 协程有点像。协程在 await 那里挂起，恢复后从挂起点接着跑。Protothread 的本质也是类似，但区别在于协程有自己的栈空间，而 Protothread 完全没有栈。这就带来了一个重要的约束：局部变量不能在阻塞点之间保持有效。如果你在 PT_WAIT_UNTIL 之前定义了一个局部变量，在它之后再去读，那个值可能已经不存在了。所以开发者必须把需要跨阻塞点的变量声明为 static，或者放到进程结构体成员里面。这个坑几乎每个初学 Contiki-NG 的人都会踩一次。

好，现在我们来看看 Contiki-NG 的架构是怎么组织的。整个系统最核心的是内核层，目录在 os 下面的 sys 目录。这里包含了进程管理、事件队列、定时器和 Protothread 的底层实现。其中 process.c 是系统的中央调度器，它维护一个全局的事件队列，所有进程间的通信都通过事件投递来完成。process_run 这个函数是整个系统的引擎，每次被调用时，它会先处理所有被标记为轮询的进程，然后从事件队列取出一个事件，分发给对应的进程。这个架构跟 Node.js 的事件循环非常相似，每次只处理一个事件，然后返回给调用者。

再往上是网络层，这是 Contiki-NG 最精华的部分，在 os 下面的 net 目录。这里面包含了完整的 IPv6 协议栈实现，而且还不是普通的 IPv6，是经过六楼波兰压缩的版本。普通 IPv6 包头有四十个字节，而 IEEE 八百零二点十五点四标准的最大帧只有一百二十七个字节，所以必须压缩。六楼波兰把共享的链路层信息从包头中去掉，压缩到只有几个字节。同时它还负责分片和重组，因为 IPv6 的最小 MTU 是一千二百八十个字节，而无线帧只有一百二十七个字节，需要把一个大 IP 包切成多个小片发送。

网络层还有 MAC 子层，支持两种模式。一种是传统的 CSMA，就是发送前先侦听信道，空着就发，忙就等。另一种是 TSCH，时间同步信道跳频。TSCH 把时间切成固定的时隙，全网节点时间同步，每个时隙在特定的频率信道上收发数据，而且每过一个时隙就换一个频率。这样做的效果是，不仅避免了节点间的碰撞，还能有效抵抗无线干扰和多径衰落。这是工业级确定性网络 6TiSCH 的底层技术。

再往上就是路由层，主要是 RPL 协议。RPL 让节点自动形成树状的网络拓扑，叫做 DODAG。根节点通常是边界路由器，叶子节点是传感器节点。当一个新节点加入网络时，它会监听周围邻居的信标消息，然后选择最优的父节点加入拓扑。这个最优不是简单地看跳数，而是看 ETX，也就是预期传输次数。如果一个链路质量差、经常丢包，那它的 ETX 就高，节点就会回避这个链路选别的父节点。

最上层是应用层协议 CoAP。你可以把它理解成 HTTP 的嵌入式版本。CoAP 使用 UDP 传输，消息头只有四个字节，支持 GET、PUT、POST、DELETE 这些 RESTful 操作。它还有一个特别有用的特性叫观察者模式，客户端可以订阅一个资源，当资源变化时服务器主动推送通知，不需要客户端轮询。这对电池供电的设备来说太重要了，因为每次轮询都要唤醒无线模块，而唤醒一次可能比发十次数据还费电。

存储层有 Coffee 文件系统，这是为串行 NOR Flash 特别设计的日志结构文件系统。它跟普通文件系统最大的区别是，Flash 不能覆盖写入，必须先擦除再写入，而且擦除是按块进行的，一个块可能六十四 KB。所以 Coffee 采用追加写入的方式，更新的数据写到一个新位置，旧位置标记为脏，后台的垃圾回收进程统一回收脏块。这跟 LevelDB 的 LSM 树原理很像。

硬件抽象层在 arch 目录下，支持 ARM Cortex-M 全系列芯片，包括 CC2538、CC26xx、nRF52840，以及 MSP430 等。每个 CPU 目录下有完整的启动代码、链接脚本和外设驱动。平台目录则对应具体的开发板。特别值得一提的是 Cooja 模拟器，它用 Java 编写，可以在电脑上模拟几十到几百个传感器节点，每个节点都能在指令级单步调试。这意味着你可以在没有硬件的情况下完成绝大部分网络协议的开发和测试。

今天推荐先读五个源码文件。第一个是 os 下面的 sys 目录下的 process.h 和 process.c，这是整个系统的引擎，看进程如何定义、启动，事件如何投递和分发。第二个是 pt.h 和 lc.h，Protothread 和局部连续的实现，只有几十行宏定义，却实现了完整的阻塞语义，非常精巧。第三个是 etimer.c，事件定时器的工作流，它本身也是一个进程，维护一个定时器链表，每次 process_run 检查哪些定时器到期，给对应进程发事件。第四个是 net 下面的 ipv6 目录下的 uip-ds6.h，理解嵌入式 IPv6 的邻居缓存、路由器列表等数据结构设计。第五个是 examples 下的 hello-world，一个完整的应用示例，读完了就能跑起来。

好，接下来咱们做几个嵌入式概念的类比，方便从软件工程转型的同学理解。第一个类比是把 Protothread 比作 Python 的 async await 协程，两者都是协作式多任务，都在阻塞点挂起恢复。但这个类比有一个重要的局限：Python 协程有自己独立的运行时栈，可以在内层函数中嵌套 await，而 Protothread 根本没有栈，阻塞只能发生在 PROCESS_THREAD 函数的直接体内，不能跨函数调用。第二个类比是把事件驱动内核比作 Node.js 的事件循环，process_post 就像 process.nextTick，process_run 就像事件循环的 tick。但 Node.js 的 I/O 背后有 libuv 线程池，而 Contiki-NG 是纯单线程，I/O 全靠中断和轮询。第三个类比是把 RPL 路由协议比作 Kubernetes 的服务发现，新加入的传感器节点自动找到父节点，就像 Pod 注册到 Service。但 K8s 运行在稳定可靠的数据中心网络上，RPL 必须应对无线链路时断时续、节点移动甚至电池耗尽这些极端不稳定因素。第四个类比是把 Coffee 文件系统比作数据库的 WAL 写入，都是追加写入。但 Coffee 必须直接面对 Flash 的物理擦写约束，而数据库通常跑在带了 FTL 闪存转换层的 SSD 上，FTL 帮它处理了这些底层细节。第五个类比是把六楼波兰的包头压缩比作 HTTP 二的 HPACK 头部压缩，两者都是利用上下文消除冗余信息。但六楼波兰是无连接的，每个数据包独立压缩解压，而 HPACK 依赖两端协商的状态。

好，接下来是今天的十个嵌入式概念卡片。第一个概念是 Protothread 原型线程，一种无栈的轻量级线程，每个线程只保存一个程序计数器，通过 C 语言的 switch 技巧实现挂起恢复。它的坑是局部变量不能跨越阻塞点。第二个概念是无栈线程，所有线程共享同一个 C 栈，挂起时只保存恢复点，类似 Python 的生成器。但它不能在阻塞点里调用普通函数，因为被调函数的栈帧会被覆盖。第三个概念是六楼波兰，不是一个无线技术，而是 IPv6 在低功耗网络上的适配层，负责包头压缩和分片重组。第四个概念是 CoAP 协议，HTTP 的嵌入式替代品，用 UDP 传输，消息头只有四字节，支持观察者模式。它跟 HTTP 最大的不同是，CoAP 靠应用层重传保证可靠性，而不是 TCP 的滑动窗口。第五个概念是 TSCH，时间同步信道跳频，提供工业级可靠性。全网时间同步精度要求微秒级，这对习惯了毫秒级精度的软件工程师来说是个全新的挑战。第六个概念是 RPL 协议，距离矢量路由协议，用 DODAG 树状拓扑组织网络。它的路由度量是 ETX 预期传输次数，不是跳数。第七个概念是事件驱动内核，不主动调度线程，而是等事件来了顺序处理。事件处理绝对不能阻塞，否则系统就卡死了。第八个概念是局部连续，Protothread 的底层实现，本质就是 static 变量加 switch case。第九个概念是 Coffee 文件系统，为 NOR Flash 设计的日志结构文件系统，不能覆盖写入，需要垃圾回收。第十个概念是 Cooja 模拟器，Java 写的网络模拟器，可以指令级模拟 MCU。但模拟时间不是真实时间，模拟一小时可能在电脑上只跑三十秒，不要拿模拟数据直接估算真实性能。

接下来有一个动手任务，三十分钟可以完成。在 Linux 环境里克隆 Contiki-NG 仓库，进入 examples 下的 hello-world 目录，用 make TARGET=native 编译原生版本，直接运行看看效果。然后打开 hello-world.c，找到 PROCESS_THREAD 函数，理解 etimer 如何控制周期性打印，然后修改成每秒钟打印一次模拟传感器读数。重新编译运行验证。验收标准是编译无错误，程序每秒输出一行读数。最后试着解释为什么修改后的代码里如果用了计数变量，必须声明为 static。如果你不想装环境，也可以纯阅读完成：第一，读懂 process.c 中 process_run 的执行路径。第二，画一个事件从 process_post 投递到 PROCESS_THREAD 入口被调用的流程图。第三，解释 PT_WAIT_UNTIL 为什么最终展开为一个 switch case。

最后是三道深度问题。第一道关于时序和并发。假设你有一个传感器节点需要同时处理三个周期任务：每百毫秒读取温度传感器、每五百毫秒上报数据到服务器、每一秒检查电池电压做低电量预警。三个任务都用 etimer 驱动。如果 process_run 在处理温度读取时因为 I2C 通信等待花了六十毫秒，这会影响到五百毫秒上报任务的定时精度吗？如果会，怎么在不增加硬件定时器中断个数的前提下解决？这个问题需要你理解 Contiki-NG 的事件驱动本质和 etimer 的到期检查时机。第二道关于架构思维迁移。你在后端写过微服务，每个服务是独立进程，通过 gRPC 通信，负载高了可以横向扩容。现在把同样思维移植到 Contiki-NG 上，每个功能模块拆成独立 PROCESS，之间用 process_post 传递事件。这种设计的缺陷是什么？什么时候拆太细反而比单进程更差？第三道关于工程判断。你的团队需要为一个产品选 IoT 通信方案。方案 A 用 Contiki-NG 完整 IP 栈，方案 B 用简单私有非 IP 协议。对于只有八 KB RAM、电池寿命要求三年的温湿度传感器节点，你选哪个？什么条件下你会改变选择？

今天的复盘问题来自 process.c 的 process_run 实现：注意到它每次只处理一个事件就返回。这种一次一个事件的设计和 RTOS 调度器跑完所有就绪线程有什么本质区别？如果改成一次处理完队列里所有事件再返回，对系统的响应延迟和功耗会有什么影响？这个问题没有标准答案，但能帮你深入理解事件驱动内核和抢占式内核在设计哲学上的根本差异。尝试从 CPU 唤醒和睡眠的角度思考：每次 process_run 返回零时，系统就知道没有待处理事件了，可以让 CPU 进入深度睡眠直到下一个中断唤醒。如果改成一次处理完所有事件，CPU 的活跃时间会更长但上下文切换次数会变少。你觉得哪种方式更适合电池供电的物联网设备呢？
