# 每日嵌入式学习：CherryUSB

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**：https://github.com/cherry-embedded/CherryUSB
- **一句话介绍**：CherryUSB 是一个轻量、高性能、可移植的嵌入式 USB 主机与设备协议栈，帮助 MCU 开发者以极小的代码量实现 USB 通信功能（如串口、U盘、键盘、音视频等）。

## 2. 为什么今天选它

CherryUSB 是近年来嵌入式 USB 协议栈中的一匹黑马：

- **活跃度高**：1921 stars，416 forks，2026年6月18日仍有提交，社区活跃。
- **工程完整性好**：同时支持 Device（从机）和 Host（主机）模式，覆盖 CDC、HID、MSC、UVC、UAC、RNDIS、DFU、MTP 等十余种标准 Class，还支持复合设备、多级 HUB、多总线。
- **学习价值极高**：代码以"树状化编程"为设计哲学，逻辑清晰、分层明确，没有复杂 C 语法，非常适合有软件工程背景的人阅读。USB 协议本身也是一个极好的"分层协议"设计案例。
- **软件工程师友好**：API 设计理念是"像使用 UART DMA 一样使用 USB"，收发接口无长度限制、零拷贝、支持 DMA，抽象层次恰到好处。
- **与 TinyUSB 形成对比**：之前讲过 TinyUSB，而 CherryUSB 是另一个设计风格迥异的 USB 栈，二者对比学习能加深对 USB 协议栈设计的理解。

## 3. 项目解决的问题

USB 是嵌入式产品中最普遍的接口之一，但直接操作 USB 硬件 IP（如 DWC2、EHCI、XHCI、MUSB 等）非常复杂：

**真实场景**：一个物联网网关需要通过 USB 连接 4G 模块（CDC ACM 虚拟串口），同时自身作为 USB 从机给 PC 提供虚拟串口和 U 盘功能。如果没有现成协议栈，开发者需要自行处理：
- USB 枚举和描述符解析
- 端点（Endpoint）管理与调度
- 控制传输、批量传输、中断传输、同步传输四种传输类型
- 复合设备接口协调
- 不同 USB IP 的寄存器差异

**嵌入式特有约束**：
- **资源极度有限**：usbd_core.c 仅约 4KB FLASH，一个 CDC ACM 类驱动仅约 400 字节
- **无堆分配**：核心数据结构在非缓存 RAM 段静态分配，零动态内存使用
- **中断上下文**：分包逻辑在中断中执行，不能在中断中做阻塞操作
- **缓存一致性**：启用 D-Cache 时需处理对齐和缓存刷新
- **实时性要求**：USB 是时间敏感协议，枚举超时、帧起始（SOF）等都有严格时序

## 4. 技术架构分析

基于公开文件推断，CherryUSB 采用分层树状架构：

### 4.1 整体架构

```
┌──────────────────────────────────────────────┐
│                 Application                   │
├──────────────────────────────────────────────┤
│        Class Drivers (HID/MSC/CDC/...)       │
├──────────────────────────────────────────────┤
│       USB Core (usbd_core / usbh_core)       │
├──────────────────────────────────────────────┤
│      Porting Layer (DCD/HCD Adaptation)      │
├──────────────────────────────────────────────┤
│            USB IP Registers (DWC2/EHCI/...)  │
└──────────────────────────────────────────────┘
```

### 4.2 主要模块

1. **core 层**：`usbd_core.c` 和 `usbh_core.c` 分别实现设备栈和主机栈。处理标准设备请求（如 GetDescriptor、SetConfiguration）、枚举流程、端点管理和事件分发。

2. **class 层**：按 USB Class 划分，每个 Class 独立为一个目录。device/ 和 host/ 下各有一套类驱动，如 `usbd_cdc_acm.c`、`usbh_hid.c`。Class 驱动通过注册回调函数挂接到 Core 层。

3. **common 层**：USB 协议常量定义、标准描述符结构体、日志宏、对齐宏、链表实现。相当于"基础设施"。

4. **port 层**（Porting）：分为 DCD（Device Controller Driver）和 HCD（Host Controller Driver）。各 USB IP 在此实现底层寄存器操作。CherryUSB 当前支持 DWC2、EHCI、OHCI、XHCI、MUSB、CDNS2、CDNS3、RP2040 等十余种 IP。

5. **osal 层**（OS Abstraction Layer）：封装了 RTOS 相关的信号量、互斥锁、消息队列、线程创建等接口，可适配 FreeRTOS、RT-Thread、ThreadX、Zephyr 等。

6. **config 系统**：`cherryusb_config_template.h` 是一个集中配置头文件，通过宏开关控制功能裁剪（如是否启用 MSC 线程、是否开启测试模式等）。

### 4.3 关键设计点

- **内存零拷贝**：数据收发直接在用户缓冲区操作，不经过中间缓冲
- **注册驱动表**：Host 端用链表维护 `usbh_class_info`，枚举完成后自动匹配 Class 驱动
- **总线模型**：支持多总线（CONFIG_USBDEV_MAX_BUS / CONFIG_USBHOST_MAX_BUS），单芯片可处理多个 USB 控制器
- **复合设备**：Device 端通过 `usbd_add_interface` 将多个 Class 接口注册到链表，实现复合 USB 设备

## 6. 核心模块/函数导读

### 6.1 `usbd_setup_packet_handler`（usbd_core.c）
**角色**：USB 设备栈的中枢调度器。每当主机发来 Setup 包（控制传输的第一阶段），此函数被调用。它解析 bmRequestType 判断是标准请求、Class 请求还是厂商请求，然后分派给对应处理函数。
**工程关联**：控制传输是 USB 中最复杂的传输类型。此函数体现了"请求→解析→分发→响应"的经典服务器设计模式。

### 6.2 `usbh_enumerate`（usbh_core.c）
**角色**：主机枚举流水线。对新插入的 USB 设备，依次执行：复位→获取设备描述符→分配地址→设置地址→获取完整配置描述符→解析接口/端点→装载 Class 驱动。
**工程关联**：这是 USB 主机栈最核心的流程。类比后端服务中的"设备发现→握手→注册→加载插件"流程。

### 6.3 `usbd_ep_open` / `usbd_ep_close`（通过 dcd 接口）
**角色**：端点物理配置。每个 USB 端点对应硬件中的一个 FIFO/缓冲区，usbd_ep_open 将描述符中的端点属性（方向、类型、最大包长）写入 USB IP 寄存器。
**工程关联**：这是"软件配置→硬件生效"的典型接口。在软件中，端点像是"端口"；在硬件上，它对应一组寄存器和 DMA 通道。

### 6.4 `usbh_find_class_driver`（usbh_core.c）
**角色**：Class 驱动匹配器。枚举解析出设备描述符后，遍历 `usbh_class_info_table` 表，按 Class/SubClass/Protocol/VID/PID 匹配对应驱动。
**工程关联**：类似后端中的 SPI（Service Provider Interface）或路由器中的路由表匹配。驱动表是静态注册的，体现了嵌入式中的"编译时确定性"理念。

### 6.5 `usbd_add_interface`（usbd_core.h）
**角色**：复合设备接口注册。将 Class 驱动实例挂到 `g_usbd_core[busid].intf[]` 数组，支持最多 16 个接口。
**工程关联**：这是复合设备的关键。在软件工程中，类似"插件管理器注册插件"。嵌入式限制：数组大小编译时固定，不能动态增长。

## 7. 软件工程师类比解释

### 7.1 USB 端点（Endpoint）⇔ TCP 端口
USB 端点就像 TCP 端口——每个端点有唯一编号和通信方向（IN/OUT），特定类型的端点用于特定类型的数据传输。
**局限**：TCP 端口是网络层概念，端口号由操作系统动态绑定；USB 端点在设备描述符中静态声明，数量极少（通常最多 16 个），由 USB 控制器硬件直接管理。

### 7.2 USB 枚举（Enumeration）⇔ DHCP + DNS
设备插入后，主机分配地址、读取描述符、装载驱动——就像新设备接入网络后 DHCP 分配 IP、DNS 解析服务。
**局限**：DHCP/DNS 是网络层协商，可重试、可超时；USB 枚举是同步、严格时序的，必须在几十毫秒内完成，否则主机认为设备异常。

### 7.3 描述符（Descriptor）⇔ API Schema / Swagger
USB 描述符（设备描述符、配置描述符、接口描述符、端点描述符等）就像 REST API 的 OpenAPI 规范——用结构化数据声明设备的能力和接口。
**局限**：描述符是硬件设备向主机声明自身能力的只读数据结构，存储在设备 FLASH 中，编译时决定；API Schema 是运行时可动态生成或修改的。

### 7.4 DCD/HCD Porting 层 ⇔ 数据库驱动层
DCD（设备控制器驱动）/ HCD（主机控制器驱动）的抽象接口类似 JDBC/ODBC——上层代码通过统一接口操作不同的 USB IP 硬件。
**局限**：JDBC 驱动层性能损耗通常可接受（毫秒级），而 USB Porting 层的零拷贝、内联函数、寄存器直接操作都是为了纳秒级的性能优化。

### 7.5 OSAL（OS 抽象层）⇔ Docker 容器接口
OSAL 封装了 RTOS 的系统调用（信号量、互斥锁、消息队列），让 CherryUSB 可以无缝运行在 FreeRTOS、RT-Thread、Zephyr 等不同 RTOS 上。
**局限**：Docker 容器的抽象层通过系统调用转发实现，开销相对较大；OSAL 是编译时绑定的宏和内联函数，零运行时开销。

## 8. 今日 10 个嵌入式概念卡片

### 8.1 USB 描述符（Descriptor）
**一句话**：描述 USB 设备能力和配置的数据结构树，从设备描述符到端点描述符共五层嵌套。
**类比**：就像 HTTP 协议的 Content-Type + Accept 头，声明自己能做什么。
**场景**：每次主机请求设备信息时，设备返回描述符数据。
**坑点**：描述符必须按严格顺序返回，长度字段要精确，否则主机拒绝识别设备。

### 8.2 端点（Endpoint）
**一句话**：USB 通信的最小逻辑单元，每个端点有方向（IN/OUT）、类型（控制/批量/中断/同步）和最大包长。
**类比**：就像 Unix 管道，但分方向且类型多样。
**场景**：鼠标用中断端点，U盘用批量端点，摄像头用同步端点。
**坑点**：端点 0 是默认控制端点，所有设备必须支持；其他端点在配置描述符中声明。

### 8.3 控制传输（Control Transfer）
**一句话**：USB 的控制通道，分为 Setup→Data（可选）→Status 三个阶段，用于设备枚举和命令下发。
**类比**：就像 HTTP POST 请求（Setup 是请求行，Data 是请求体，Status 是响应码）。
**场景**：主机通过控制传输设置设备地址、获取描述符、配置设备。
**坑点**：控制传输的最大数据长度受端点 0 最大包长限制（通常 8/16/32/64 字节），大数据需分多次传输。

### 8.4 DMA（直接内存访问）
**一句话**：硬件直接在内存和外设间搬运数据，不经过 CPU，USB 的 DMA 还要处理分包。
**类比**：就像 Node.js 的异步 I/O——数据传输不需要主线程参与。
**场景**：U盘读写时，DMA 从内存到 USB FIFO 传输数据块。
**坑点**：DMA 要求缓冲区地址对齐（通常 32 字节），且需处理缓存一致性问题。

### 8.5 缓存一致性（Cache Coherence）
**一句话**：CPU 缓存和 RAM 中的数据不一致时造成的问题，DMA 传输时特别突出。
**类比**：就像分布式系统中的缓存与数据库不一致问题——你读到的数据是过期的。
**场景**：DMA 向 RAM 写入数据时，CPU 可能从缓存读到旧值。
**坑点**：CherryUSB 用 `USB_NOCACHE_RAM_SECTION` 将关键数据结构放到非缓存 RAM 区来规避这一问题。

### 8.6 ISR（中断服务例程）
**一句话**：当 USB 硬件完成一次数据传输或需要处理事件时触发的中断处理函数。
**类比**：就像 Webhook/回调函数——事件发生时有系统自动调用你的处理代码。
**场景**：USB 端点收到数据时触发中断，ISR 中把数据转发到应用层消息队列。
**坑点**：ISR 中不能做阻塞操作（如等待信号量、分配内存、打印日志），应尽量短小，复杂逻辑推到线程中处理。

### 8.7 复合设备（Composite Device）
**一句话**：一个 USB 设备同时提供多个功能，比如虚拟串口 + U盘 + 音频。
**类比**：就像一台服务器同时运行 HTTP + SSH + FTP 服务。
**场景**：开发调试工具同时提供 CDC ACM（日志输出）和 MSC（U盘升级固件）。
**坑点**：复合设备需在一个配置描述符中声明多个接口（Interface），每个接口需要独立的驱动，主机端也要能同时加载多个驱动。

### 8.8 USB 帧/微帧（Frame / Microframe）
**一句话**：USB 的时间同步单位，全速/低速每 1ms 一个帧，高速每 125μs 一个微帧。
**类比**：就像实时音频系统中的音频缓冲区——所有传输按固定时间片调度。
**场景**：同步传输（如摄像头/麦克风）按帧预留带宽。
**坑点**：等时传输没有重传机制，丢包不重发，适用于容忍少量丢包的实时数据。

### 8.9 Kconfig / CMake 配置裁剪
**一句话**：通过 Kconfig 或 CMake 选项开关特性、配置缓冲区大小和堆栈大小，静态编译时的功能定制。
**类比**：就像 Dockerfile 中的构建参数——一次配置，编译时决定最终镜像大小。
**场景**：不需要 Host 功能则关掉 CONFIG_USBHOST 节省几 KB FLASH。
**坑点**：配置项互相依赖（如打开 MSC 需打开 Bulk 传输支持），需要理解依赖关系树。

### 8.10 零拷贝（Zero Copy）
**一句话**：数据直接从外设到用户缓冲区，或从用户缓冲区到外设，不经过中间缓冲拷贝。
**类比**：就像 Linux 的 sendfile() 系统调用——文件到 socket 数据不经过用户空间。
**场景**：CherryUSB 的 usbd_ep_start_write 直接使用用户传进来的缓冲区指针。
**坑点**：零拷贝要求缓冲区必须是常驻内存的（不能是栈临时变量），且对齐和持久化由使用者保证。

## 12. 工程点评

**可维护性**（8/10）：代码风格统一，命名规范，注释较充分。回调式架构让模块间耦合度低。但部分函数长度偏长（如 usbd_setup_packet_handler），可进一步拆分。

**可移植性**（9/10）：OSAL 抽象层 + DCD/HCD 标准接口，移植到新 RTOS 或 USB IP 只需实现约 10 个函数。已支持主流 RTOS 和十余种 USB IP。

**可靠性**（7/10）：有 CI 构建测试，但缺乏单元测试和硬件在环测试描述。依赖静态配置减少运行时错误，但运行时错误处理（如描述符格式错误）主要靠 assert。

**可测试性**（6/10）：代码紧耦合硬件寄存器，纯软件模拟测试较难。但各个 Class 驱动的行为可以通过 USB 协议分析仪验证。

**量产/运维**（7/10）：资源占用极小，适合成本敏感的 MCU。支持 DFU（固件升级）和 MSC（U盘升级场景）。但缺少 OTA 升级适配。

## 13. 明天可以延展的方向

1. **深入 USB 协议标准**：阅读 USB 2.0 规范中关于控制传输和描述符的章节，结合 CherryUSB 源码理解规范到代码的映射关系。
2. **对比学习 TinyUSB 与 CherryUSB**：两个开源 USB 栈的设计哲学差异，包括调度策略、内存管理方式、Class 驱动加载机制，能极大加深对 USB 协议栈设计的理解。
3. **研究 CherryUSB Host 栈的多级 HUB 支持**：USB HUB 级联（最多 7 级）的实现是 USB 协议中最复杂的部分之一，包括事务拆分（Split Transaction）、轮询调度等。
