# 每日嵌入式学习：QMK Firmware

## 1. GitHub 项目链接与一句话介绍

- **仓库链接**: https://github.com/qmk/qmk_firmware
- **一句话介绍**: QMK（Quantum Mechanical Keyboard）是目前全球最大的开源键盘固件项目，让你可以用纯软件的方式完全自定义机械键盘上每一个按键的行为，从简单改键到复杂的宏、多层布局、RGB灯光和音频输出。

## 2. 为什么今天选它

QMK Firmware 在 GitHub 上有超过 20,500 颗星，是嵌入式领域最活跃的开源项目之一。它的代码库极其成熟，经历了十多年的迭代和社区打磨，工程完整度极高。对于有软件工程背景、正在转向嵌入式开发的工程师来说，QMK 是一份教科书级别的教材：

- **它把嵌入式底层复杂性封装得很优雅**：你不用一开始就面对晦涩的数据手册，就能看到矩阵扫描、USB HID 协议、事件驱动架构这些核心概念在真实产品中的工程实现。
- **用户态与内核态的分离感**：QMK 的 "quantum" 层就像后端的中间件，tmk_core 像操作系统内核，keyboard 和 keymap 像业务代码——这种分层对软件工程师极其自然。
- **丰富的文档体系**：官方文档站点 docs.qmk.fm 有完整的架构说明、代码导读、API 参考和最佳实践。

## 3. 项目解决的问题

QMK 解决的核心问题是：**让键盘不再是固定的输入设备，而是一个可编程的交互终端**。

在真实产品/工程场景中，机械键盘用户在以下场景需要 QMK：
- 程序员想把 Caps Lock 改成 Ctrl，把 Esc 放在波浪键位置
- 游戏玩家需要一键连招、快速切换层（比如 MO（1）按住变第二层）
- 极客想要用键盘发送快捷键序列、启动应用、甚至当宏键盘用
- 分体式键盘需要左右手通过串口或 I2C 通信同步状态

和普通软件项目相比，QMK 需要应对的嵌入式特有约束包括：
- **资源极度受限**：AVR 单片机只有 2KB RAM 和 32KB Flash，代码必须精打细算到每个字节
- **实时性要求**：矩阵扫描每秒钟要跑几十次，按键延迟不能超过几毫秒，否则用户会感知到卡顿
- **无操作系统**：没有标准库、没有堆管理、没有线程调度——一切靠裸机主循环
- **USB 协议栈**：必须精确遵循 USB HID 规范，与操作系统内核的输入子系统握手
- **多个硬件平台**：同时支持 AVR 和 ARM 两种架构，需要在编译时做平台抽象

## 4. 技术架构分析

QMK 的架构可以清晰地分为四层，从底部到顶部：

### 第一层：tmk_core（底层核心）
继承自 TMK 项目，是 QMK 的前身。这一层负责硬件接口抽象、USB 协议栈（LUFA/ChibiOS/V-USB 三种实现）、矩阵扫描基础逻辑和按键动作处理。
- `tmk_core/protocol/`：USB HID 协议的实现，支持 LUFA（AVR）、ChibiOS（ARM）和 V-USB（软件 USB）
- `tmk_core/common/`：通用的工具函数和宏，如定时器、延时、按键码映射

### 第二层：quantum（量子层）
QMK 在 tmk_core 之上添加的增强层，所有"魔法"功能都在这里。这一层扮演中间件的角色：它接收 tmk_core 传来的原始按键事件，经过一系列可插拔的处理器链处理，最终输出处理后的按键动作。
- `quantum/main.c`：程序的入口，初始化硬件后进入 `while(true)` 主循环
- `quantum/keyboard.c`：键盘核心逻辑——调用矩阵扫描、检测状态变化、分发给按键处理链
- `quantum/action*.c`：动作处理系统，实现按键的下压、释放、保持、连击等语义
- `quantum/process_keycode/`：按键码处理器目录，每个功能是一个独立模块，包括 Tap Dance、组合键、Unicode、MIDI、RGB 灯光、Leader 键、Space Cadet 等 30 多个可插拔模块
- `quantum/quantum.c`：量子层的核心入口，定义了 Process Record 链的编排顺序

### 第三层：keyboard（键盘定义层）
每个键盘型号一个目录，定义硬件矩阵布局、引脚分配、背光配置、驱动等。这一层相当于"板级支持包"。
- `keyboards/<vendor>/<model>/`：包含 `config.h`（编译时配置）、`info.json`（键盘元信息）、`<model>.h`（LAYOUT 宏定义矩阵到物理按键的映射）、`<model>.c`（自定义初始化代码）

### 第四层：keymap（用户自定义层）
用户配置层，这是 QMK 的灵魂。用户在这里定义层布局、快捷键、宏和各种功能绑定。这一层完全分离了硬件细节。

### 主循环工作流程
```
电源上电 → main() 初始化硬件 → while(true):
  1. keyboard_task() 调用 matrix_scan()
  2. matrix_scan() 扫描按键矩阵，返回当前状态位图
  3. 与上一次状态比较，检测按下的键和释放的键
  4. 对每个状态变化的按键调用 action_exec()
  5. action_exec() 经过 pre_process_record → process_record → post_process_record 链
  6. process_record 链中按顺序调用各处理器模块
  7. 最终按键码通过 USB HID 报告发送给主机
```

注意：QMK 不使用 RTOS，完全靠裸机超级循环完成所有任务。矩阵扫描频率通常在 10ms 到 1ms 级别，取决于键盘矩阵大小和 MCU 速度。

## 5. 源码陪读：今天只读哪几个文件

以下推荐按顺序阅读，从入口到核心，逐步深入：

### 文件 1：`quantum/main.c`
**阅读目标**：理解嵌入式 C 程序的完整启动流程。你会看到从 main() 函数开始的硬件初始化调用序列，然后进入永不退出的主循环。这是一个极其简洁的嵌入式入口范例，不到 60 行有效代码。

### 文件 2：`quantum/keyboard.c`
**阅读目标**：理解键盘固件的核心循环——keyboard_task() 函数。关注它如何协调矩阵扫描、状态变化检测、LED 状态管理和鼠标处理。尤其注意它把"按键事件"和"按键状态"区分开的工程思路。

### 文件 3：`quantum/quantum.c`
**阅读目标**：理解 Process Record 链——这是 QMK 的事件分发机制核心。看它是如何把原始按键动作依次传递给 Tap Dance、组合键、Unicode、RGB 灯效等 30 多个处理器模块的。每个处理器返回 false 可以终止后续处理，类似于 Web 后端的中间件洋葱模型。

### 文件 4：`quantum/action_tapping.c`
**阅读目标**：这是 QMK 最精妙也最容易出 bug 的部分——Tap Dance 和连击检测。看 QMK 如何处理"短按"和"长按"的歧义：你按下一个键，是不确定是点按还是按住？QMK 用 TAPPING_TERM 这个超时窗口来解决这个经典的嵌入式时序歧义问题。这是软件工程师理解"嵌入式实时"思维的绝佳入口。

### 文件 5：`quantum/matrix.c`
**阅读目标**：理解矩阵扫描的硬件原理。看 QMK 如何通过逐行拉高/拉低 GPIO 电平、逐列读取状态来检测按键。留意去抖（debounce）算法的实现——机械按键在物理接触瞬间会有几十毫秒的抖动，如何稳定地检测到"真正按下"是嵌入式系统中最经典的问题之一。

## 6. 核心模块/函数导读

### 1. `keyboard_task()`（quantum/keyboard.c）
**工程角色**：整个固件的心脏。每次主循环迭代都会调用它。它的责任包括：
- 调用 `matrix_scan()` 获取当前按键矩阵状态
- 与上一次状态做 XOR 异或比较，生成"变化事件"列表
- 对每个变化事件调用 `action_exec()`
- 最后处理 LED、鼠标等辅助功能
**与启动流程的关系**：在 main() 初始化硬件后首次被调用，之后就无限循环。

### 2. `process_record()` 链（quantum/action.c + quantum/quantum.c）
**工程角色**：事件处理管道，类似于 Express.js 的中间件链或 Kafka 的消息处理器链。每个按键事件会流经：
1. `pre_process_record_quantum()` → 预处理钩子
2. `process_record_quantum()` → 核心处理，这里遍历所有 `process_*()` 模块
3. `post_process_record_quantum()` → 后处理钩子
**与协议和硬件的关联**：这是 USB HID 协议与用户意图之间的桥梁——原始矩阵坐标在这里被翻译成 USB 扫描码，再被拼装成 HID 报告发送给操作系统。

### 3. `matrix_scan()`（quantum/matrix.c + tmk_core 中的具体实现）
**工程角色**：硬件抽象的关键接口。QMK 提供了默认的通用实现，但每个键盘也可以覆盖。这个函数的任务是：
- 逐行驱动 GPIO 输出（行扫描）
- 逐列读取 GPIO 输入（列检测）
- 返回一个矩阵状的位图，1 表示按键按下
**与持久化和构建系统的关系**：矩阵的物理定义在键盘的 `info.json` 或 `config.h` 中配置，构建时通过 Makefile 选择对应的矩阵扫描实现。

### 4. `action_tapping.c` 中的 `process_tapping()` 
**工程角色**：解决嵌入式系统中最棘手的"时序歧义"问题。当用户按下并释放一个键时，固件不知道这是一个"短按"还是"长按"的开始。QMK 引入 TAPPING_TERM（默认 200ms）超时机制：如果在超时时间内释放，认为是一次点按；如果超过超时时间没有释放，认为进入了"按住"模式。
**与 RTOS 的关系**：虽然 QMK 没有 RTOS，但这个机制展示了嵌入式系统中典型的时间敏感型决策——是实时系统思维的优秀教学案例。

### 5. `LAYOUT()` 宏（键盘目录中的 .h 文件）
**工程角色**：编译时的布局映射。它在物理键盘矩阵和逻辑按键布局之间建立映射关系。例如一个 4x4 的矩阵可能定义了 16 个按键位置，但物理键盘只有 12 个按键（大空格占用了多个矩阵位置），LAYOUT 宏就把矩阵中不存在的物理位置映射为 KC_NO（空键）。
**与构建系统的关联**：LAYOUT 宏的定义存储在键盘头文件中，构建时被 keymap.c 引用，编译优化器会将其展开为数组字面量并存入 Flash。

## 7. 软件工程师类比解释

### 概念 1：矩阵扫描 ↔ Web 后端的心跳轮询
键盘固件以固定频率遍历按键矩阵的行列，读取每个交叉点的通断状态。这类似于后端服务每隔几秒轮询数据库检查新任务。区别在于：轮询是网络 I/O 操作，而矩阵扫描是直接操作 GPIO 寄存器，延迟从毫秒级降到微秒级。**局限**：矩阵扫描必须与去抖算法配合，否则物理抖动会被当成"有效数据"，而数据库轮询不会有这种物理噪声问题。

### 概念 2：LAYOUT 宏 ↔ TypeScript 的类型映射（mapping type）
当你在 keymap.c 中写 LAYOUT(KC_A, KC_B, ...)，编译器在编译期就把这些逻辑位置映射到矩阵的物理行列上。这就像 TypeScript 中定义一个映射类型，把枚举值映射到另一个类型上。**局限**：这是编译期展开的宏，不能热替换或动态绑定；如果键盘矩阵变了，必须重新编译固件并烧录。

### 概念 3：Process Record 链 ↔ Express.js 的中间件链
QMK 的按键事件从 pre_process → process（经过 30+ 处理器）→ post_process，每个处理器可以拦截事件（return false）或继续传递。和 Express 的 app.use(middleware) 几乎一模一样。**局限**：Express 中间件可以异步（await/next），QMK 的处理链是同步阻塞的——一次按键处理的耗时直接影响下一次矩阵扫描的间隔。

### 概念 4：PROGMEM 和分页 Flash ↔ 前端代码分割（code splitting）
QMK 使用 PROGMEM 宏将常量数据放在 Flash 而不是 RAM 中。在 AVR 平台上，RAM 只有 2KB，而 Flash 有 32KB。这就像前端应用用代码分割把不常用的页面放在 CDN 上而不是主 bundle 中。**局限**：读取 PROGMEM 数据需要调用 pgm_read_word() 这类特殊函数，不能像 RAM 变量一样直接指针引用，编译器不会帮你自动处理这个差异。

### 概念 5：Layer 叠加 ↔ Git 分支合并
QMK 的多层键盘布局就像 Git 的分支系统。基础层是 master 分支，功能层是 feature 分支。按 MO(1) 进入层 1 就像 git checkout feature-branch，松开就回到 master。层有优先级（高数字层优先），类似 Git 的 merge 冲突解决策略。**局限**：Git 合并可以三路合并、内容合并，QMK 的层叠加只是简单的"高优先级层有值就覆盖"，不支持条件合并或冲突解决。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：GPIO（通用输入输出引脚）
- **一句话解释**：MCU 上可以配置为输入或输出的通用数字引脚。
- **软件工程类比**：类似于你操作一个文件的 O_RDONLY 或 O_WRONLY 模式，只不过操作的是物理电平。
- **常见工程场景**：读取按键状态、控制 LED 亮灭、驱动蜂鸣器。
- **容易误解/坑点**：输入模式下引脚是高阻态，不能直接驱动负载；输出模式下驱动的电流有限，通常只能到几毫安，大负载需要外部晶体管或 MOSFET。

### 卡片 2：矩阵扫描（Matrix Scanning）
- **一句话解释**：用较少的 GPIO 检测大量按键的技术——N 行 + M 列可以检测 N×M 个按键。
- **软件工程类比**：类似用 MapReduce 的思路，分两轮轮询来减少数据采集的复杂度。
- **常见工程场景**：键盘、薄膜面板、触控面板。
- **容易误解/坑点**：必须处理好"鬼键"和"串扰"问题——当三个按键同时按下形成一个矩形时，第四个角可能会被误检测为按下。

### 卡片 3：去抖（Debounce）
- **一句话解释**：机械按键在按下和释放瞬间会多次弹跳，去抖电路或算法通过延迟采样来过滤掉弹跳产生的虚假信号。
- **软件工程类比**：类似于前端输入框的 debounce 防抖——用户输入时等 300ms 不输入再触发搜索。
- **常见工程场景**：所有机械开关输入、编码器、按钮。
- **容易误解/坑点**：去抖延迟不能太长（否则按键感觉迟钝），也不能太短（滤不掉抖动）。AVR 上的去抖通常 5~20ms。

### 卡片 4：USB HID（人机交互设备协议）
- **一句话解释**：USB 标准中专门定义的一类设备协议，键盘、鼠标、游戏手柄都走这个协议。
- **软件工程类比**：类似于 HTTP REST API 的规范——定义了数据格式、端点（endpoint）和报告（report）的结构。
- **常见工程场景**：键盘向主机发送按键扫描码、鼠标发送移动向量。
- **容易误解/坑点**：键盘发送的是扫描码而不是字符；操作系统根据键盘布局表把扫描码映射成字符。所以键盘固件里改的是"按 A 键发送扫描码 0x04"，而不是发送字母 'a'。

### 卡片 5：PROGMEM
- **一句话解释**：AVR 架构上的一个属性宏，指示编译器将数据存储在 Flash（程序存储器）而不是 RAM 中。
- **软件工程类比**：相当于用 SQLite 的磁盘存储来缓存数据，把热数据放在内存、冷数据在磁盘。
- **常见工程场景**：存放键盘布局表、字体点阵、音频样本等大量只读数据。
- **容易误解/坑点**：PROGMEM 数据不能用普通指针读取，必须用 pgm_read_word()/pgm_read_byte() 族函数。ARM 架构不需要 PROGMEM（Flash 和 RAM 统一编址），但 QMK 用宏把它透明化了。

### 卡片 6：HID 报告描述符（Report Descriptor）
- **一句话解释**：告诉操作系统"我的键盘有几个按键、多少 LED、用什么格式发数据"的元信息。
- **软件工程类比**：类似于 OpenAPI/Swagger 规范文档——告诉客户端接口的数据结构是什么。
- **常见工程场景**：每个 USB 键盘都需要向主机发送报告描述符，声明它是键盘、报告 ID、每个字节的含义。
- **容易误解/坑点**：报告描述符写错了会导致操作系统不识别设备或部分按键无效；调试 HID 报告问题需要 USB 分析仪或专门软件。

### 卡片 7：EEPROM（电可擦除可编程只读存储器）
- **一句话解释**：MCU 上的一种非易失性存储器，断电不丢数据，可以按字节擦写但写入次数有限（通常 10 万次）。
- **软件工程类比**：类似于浏览器的 localStorage——容量小（通常几百字节到几 KB）、持久化存储、但写入有寿命限制。
- **常见工程场景**：保存键盘的当前层状态、RGB 颜色配置、按键映射持久化。
- **容易误解/坑点**：EEPROM 写入时间较长（几毫秒），不能在中断里写；频繁写同一个地址会磨损。做 OTA 升级时千万别频繁写 EEPROM。

### 卡片 8：Bootloader（引导加载程序）
- **一句话解释**：MCU 上电后最先执行的固件，负责检查是否需要更新固件，不需要就跳转到主程序。
- **软件工程类比**：类似于 Docker 的初始化进程或 Linux 的 GRUB——决定加载哪个"操作系统"（主程序）。
- **常见工程场景**：QMK 刷写固件时按复位键再按键盘进入 DFU 模式，就是由 bootloader 控制的。
- **容易误解/坑点**：Bootloader 在写 Flash 时不能跑应用代码（Flash 是共享的）；刷写过程中断电会导致变砖，所以有些 bootloader 有双备份。

### 卡片 9：主循环（Super Loop / Bare-metal Loop）
- **一句话解释**：没有操作系统的情况下，嵌入式固件在 main() 的 while(1) 里不停顺序执行的编程模型。
- **软件工程类比**：类似于 Node.js 的事件循环——不断处理事件回调，代码不能阻塞。区别在于 Node 有异步 I/O 和回调队列，裸机循环是纯同步。
- **常见工程场景**：QMK、Arduino 默认的 loop()、很多简单的 IoT 设备。
- **容易误解/坑点**：主循环中任何一个函数的执行时间加在一起不能超过你最紧急任务的周期——如果矩阵扫描周期是 10ms，键盘处理链的总耗时必须远小于 10ms，否则会丢帧。

### 卡片 10：TAPPING_TERM
- **一句话解释**：区分"点按"和"长按"的超时阈值（QMK 默认 200ms）。
- **软件工程类比**：类似于双击 vs 单击的 time threshold——操作系统的双击阈值通常 500ms，少于它是双击，多是单击。
- **常见工程场景**：按住 Shift 键是修饰键（长按），按一下 Caps Lock 是切换键（短按）。在 QMK 里同一个键可以同时做两种事。
- **容易误解/坑点**：不同用户的打字速度不同，TAPPING_TERM 需要根据打字习惯调整。设置太短会导致长按容易失败，太长会导致短按有滞后的感觉。

## 9. 30～60 分钟动手任务

### 主题：搭建 QMK 开发环境并定制一个最小键位

**目标**：在本地搭建 QMK 固件编译环境，修改一个已有键盘的 keymap，编译并验证。

**步骤**：

1. **安装 QMK 工具链**：
   ```bash
   python3 -m pip install qmk
   qmk setup
   ```
   这个命令会自动 clone qmk_firmware 仓库并安装工具链。

2. **创建你的第一个 keymap**（即使没有物理键盘也能编译验证）：
   ```bash
   # 以 planck 键盘为例创建你自己的 keymap
   qmk new-keymap -kb planck/ez
   # 输入你的 GitHub 用户名作为 keymap 名称
   ```

3. **修改 keymap.c**：打开 `keyboards/planck/ez/keymaps/<你的用户名>/keymap.c`，把默认布局改成：
   - 将左下角 Ctrl 改成 Shift
   - 将 Tab 键改成 "按下时切换到功能层（MO(1)），单按发送 Tab"

4. **编译**：
   ```bash
   qmk compile -kb planck/ez -km <你的用户名>
   ```

5. **阅读编译输出**：观察 FIRMWARE SIZE 信息——了解编译后的固件占了多少 Flash、用掉了多少 RAM。

**验收标准**：
- 编译成功无错误无警告
- 你能在编译输出中看到固件大小，并理解 Flash 使用率
- 你能描述你的 keymap 中 2 个改动分别对应哪个宏/函数调用

**无硬件替代方案**：
- 使用 QMK Configurator（https://config.qmk.fm）在线编辑 keymap 并下载 hex/elf 文件
- 用 `qmk json2c` 把 JSON 配置转成 C 代码并理解其中结构
- 阅读 QMK 文档中的 Keymap 章节，理解每个宏的含义

## 10. 3 道深度追问题

### 问题 1（时序/并发）：
在 QMK 的主循环中，matrix_scan() 返回当前矩阵状态。但如果你按住一个键不松手，下一次扫描又会返回同样的"按下"状态。QMK 如何避免在每次主循环迭代时都对同一个"一直按着"的键重复触发按键事件？请从代码层面描述两种可能的实现方案并比较优劣。

### 问题 2（软件工程 → 嵌入式视角转换）：
假设你在写一个高并发的后端 API，每秒处理几万个请求。你的应用有一个中间件链，每个中间件做一些验证和转换。现在你把同样的架构思路迁移到 QMK 的 process_record 链上（30 多个处理器顺序执行）。作为嵌入式工程师，你发现老手们都把宏展开成直接函数调用链，而不是用动态注册的表驱动方式。为什么？从 ROM/RAM 占用、执行时间、可预测性三个角度分析。

### 问题 3（资源限制与权衡）：
QMK 的 Layer 系统支持最多 32 层。层状态的存储方式是 `layer_state`（一个 32 位整数的位图），每一层是一个 `MATRIX_ROWS × MATRIX_COLS` 的 16 位 keycode 数组。假设一个 6x17 的键盘每层有 102 个键，每个 keycode 占 2 字节，全部 32 层如果全部放在 RAM 里需要约 6.5KB（远超 AVR 的 2KB RAM）。QMK 把层数据存储在 Flash（PROGMEM）中。请计算：如果改成动态加载（按需从 Flash 读取当前层的 keycode），预期的性能开销是多少？你会怎么设计这个缓存策略？

## 11. 今日复盘问题

**QMK 的设计中，quantum 层作为 tmk_core 之上的一层"中间件"，增加了很多有趣功能但也增加了代码体积和执行路径。如果你是 QMK 的维护者，面对一个只有 32KB Flash 和 2KB RAM 的 AVR 键盘，你如何决定哪些处理器应该默认启用、哪些应该留给用户可选？你靠什么指标做这个决策——ROM 大小？RAM 使用？还是执行时间预算？这个问题没有标准答案，但你可以从编译器的 -Os 优化、用户调研数据、以及"最常用 20% 功能覆盖 80% 用户"的帕累托原则来思考。**

## 12. 工程点评

- **可维护性**：⭐⭐⭐⭐⭐ QMK 的代码注释非常丰富，文档完善，模块划分极其清晰。每个 process_* 文件都是独立的功能模块，新增一个功能只需要添加一个文件和一两个钩子调用。社区贡献流程成熟，有代码审查、自动构建和测试。
- **可移植性**：⭐⭐⭐⭐ 支持 AVR 和 ARM 两大架构，通过抽象层隔离了硬件差异。但 AVR 和 ARM 之间的性能差异巨大，有些高级功能（如 OLED 驱动、音频输出）在 AVR 上跑不动。构建系统通过 rules.mk 的 enable/disable 机制处理这个差异。
- **可靠性**：⭐⭐⭐⭐ 核心逻辑经过十几年迭代，数以万计的用户每天都在使用。USB 协议栈经过多种操作系统的验证。唯一风险点在于：用户自定义的 process_record_user() 可以做任何事情，不当的实现可能导致键盘卡死。
- **可测试性**：⭐⭐⭐ 项目有自动化测试框架（tests/ 目录），使用 QEMU 模拟器模拟 ARM 芯片进行功能测试。但由于涉及 USB 硬件和物理按键输入，端到端测试覆盖率有限。大部分测试集中在逻辑层（action 处理、键码映射），硬件相关的测试依赖社区手工验证。
- **量产/运维角度**：⭐⭐⭐⭐ ⭐ QMK 的刷写流程非常成熟——通过 USB 进入 DFU/bootloader 模式，然后用 qmk flash 或 dfu-programmer 烧录。支持在键盘上动态配置（VIA 协议）、热插拔键位映射，不需要重新编译。对于量产场景，QMK 有厂家配置机制（rules.mk 的默认设置），但需要键盘厂商做一定适配。

## 13. 明天可以延展的方向

1. **研究 QMK 的分体键盘通信协议**：QMK 的 `split_common` 模块实现了左右手键盘之间的串口/I2C 通信，包括按键同步、LED 同步和握手协议。这本质是一个简单的"分布式系统"在单片机上的实现，值得深入阅读代码。

2. **深入 HID 协议 && QMK 的 USB 报告模式**：你可以研究 QMK 的 NKRO（N-Key Rollover）实现——为什么键盘要支持同时按下任意数量的键？这涉及 USB HID 的 Boot Protocol 和 Report Protocol 的区别，以及与操作系统的兼容性问题。

3. **从键盘固件到"智能输入设备"**：QMK 的 VIA 运行时配置（量子/ 目录中的 via.c）展示了一个嵌入式设备如何通过 USB 与 PC 端的配置工具通信并实时改变行为。这个双向通信架构可以类比到任何需要"现场配置"的物联网设备上——是理解嵌入式设备的运维接口的绝佳案例。
