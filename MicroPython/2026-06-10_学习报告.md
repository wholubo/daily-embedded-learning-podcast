# 每日嵌入式学习：MicroPython

## 1. GitHub 项目链接与一句话介绍

- 仓库链接：https://github.com/micropython/micropython
- 一句话介绍：MicroPython 是面向微控制器、嵌入式系统和其他受限平台的 Python 3 实现，让开发者能用接近 Python 的方式控制 GPIO、串口、网络、文件系统和板级外设。

## 2. 为什么今天选它

MicroPython 今天值得学习有三个原因。第一，它仍然非常活跃。GitHub API 显示仓库最近更新时间为 2026-06-10，最新提交包括 `esp32: Rename the sdkconfig.spiram_xxx config snippets.`、`esp32: Enable PSRAM support on ESP32_GENERIC_C5.` 等，说明项目持续跟进具体芯片、板级配置和内存能力。第二，它工程完整度高：仓库不是单一脚本解释器，而是包含虚拟机、垃圾回收、交叉编译器、多个芯片端口、外设模块、测试、文档、构建系统和示例的完整嵌入式运行时。第三，它对软件工程师转嵌入式特别友好：你熟悉的语言运行时、模块系统、异常、调度、文件系统，在这里都要被压缩进几十 KB 到几百 KB RAM 的设备里，这能很好地暴露嵌入式世界的资源约束、启动流程、硬件抽象和可移植性问题。

它没有出现在历史去重列表中，也不是列表里已讲项目的同一仓库、fork 或明显同源变体。

## 3. 项目解决的问题

真实产品场景里，很多传感器节点、教学开发板、小型自动化控制器、原型验证设备和边缘网关，并不一定需要完整 Linux，也不一定适合用纯 C 从零写所有业务逻辑。MicroPython 解决的是“在微控制器上快速、安全、可交互地开发逻辑”的问题。它把 Python 解释器、内存管理、模块系统和板级外设访问封装到固件里，开发者烧录固件后可以通过 REPL 或脚本快速控制硬件。

和普通软件项目相比，它多了几类硬约束。硬件上，不同 MCU 的启动代码、时钟、Flash 布局、串口、USB、Wi-Fi、蓝牙和中断模型都不同。资源上，堆内存、栈空间、Flash 空间都非常有限，动态对象和字符串常量不能像服务器上一样随意膨胀。时序上，外设操作、软定时器、中断回调和解释器调度可能互相影响，不能只考虑吞吐量，还要考虑响应延迟。可靠性上，设备可能长时间无人值守运行，断电、掉线、Flash 磨损和异常恢复都是真实问题。

## 4. 技术架构分析

基于公开仓库文件分析，MicroPython 可以按五层理解。

第一层是核心语言运行时，主要在 `py/` 目录。这里包含解析、编译、字节码执行、对象模型、运行时、异常、调度器、垃圾回收和配置头文件。对软件工程师来说，这相当于把一个小型语言虚拟机做成可裁剪库，但它的目标不是服务器进程，而是 MCU 固件。

第二层是可移植扩展模块，主要在 `extmod/` 和 `shared/`。例如网络、TLS、虚拟文件系统、压缩、正则、机器相关接口、软定时器和标准输入输出辅助逻辑。它们尽量写成跨端口复用的代码，但会通过端口层接入具体硬件能力。

第三层是端口层，主要在 `ports/`。`ports/stm32/`、`ports/rp2/`、`ports/esp32/`、`ports/unix/` 等目录把同一个 MicroPython 核心适配到不同平台。端口层负责启动、板级初始化、内存区域声明、外设驱动绑定、文件系统挂载、REPL 输入输出、构建选项和板卡配置。

第四层是构建与交叉编译。`mpy-cross/` 负责把 Python 源码预编译成 MicroPython 字节码文件，减少目标设备上的解析成本。Makefile、CMake、板级配置和端口配置一起决定最终固件包含哪些模块、占用多少 Flash、面向哪块板子。

第五层是测试、示例和文档。`tests/` 用来验证语言行为和模块行为，`examples/` 给出使用方式，`docs/` 解释端口、库和开发流程。嵌入式项目里测试难点在于同一套逻辑可能要覆盖 Unix 模拟端口和真实硬件端口，MicroPython 通过 Unix port 提供了一个低成本入口。

典型运行流程可以概括为：芯片复位后进入端口启动代码，初始化时钟、内存、底层外设和堆；然后初始化 MicroPython 运行时，挂载文件系统，启动交互式 REPL 或执行板上脚本；脚本调用 `machine`、`network`、`uos` 等模块时，再通过端口层转到具体驱动和硬件抽象。

## 5. 源码陪读：今天只读哪几个文件

建议阅读顺序如下。

1. `README.md`：先确认项目定位、支持特性、端口范围和构建入口，建立全局地图。
2. `py/mpconfig.h`：理解 MicroPython 如何用大量编译期开关裁剪功能，这是嵌入式可配置性的核心。
3. `py/gc.c`：阅读垃圾回收实现，重点看堆初始化、块管理、标记清扫和为什么嵌入式动态内存必须谨慎。
4. `py/scheduler.c`：理解异步调度、挂起任务和中断上下文之间的边界，重点关注“不能在中断里做太重的事”的工程含义。
5. `ports/stm32/main.c` 或 `ports/rp2/main.c`：任选一个端口看启动流程，关注板级初始化、文件系统、REPL 和运行时初始化如何串起来。

如果时间只有 30 分钟，先读 `README.md`、`py/mpconfig.h` 和一个端口的 `main.c`；如果有 60 分钟，再加上 `py/gc.c` 和 `py/scheduler.c`。

## 6. 核心模块/函数导读

`py/mpconfig.h` 是功能裁剪中枢。它定义大量默认配置，例如是否启用某类对象、模块、优化、错误信息和运行时能力。端口可以覆盖这些默认值。它和构建系统、Flash 占用、RAM 占用直接相关。

`py/gc.c` 是 MicroPython 的堆内存生命线。普通 Python 开发者通常不关心对象到底在哪块内存里，但 MCU 上堆很小，碎片、临时对象和长生命周期对象都会影响稳定性。读它时要把垃圾回收看成“嵌入式内存调度器”，不是纯语言内部细节。

`py/scheduler.c` 管理待调度回调。它与中断、软定时器、异步执行和主解释器循环有关。嵌入式代码里常见模式是中断只记录事件或投递任务，真正复杂逻辑放到安全上下文执行，这个文件正好体现这种边界。

`shared/runtime/pyexec.c` 负责执行脚本和 REPL 相关逻辑。它把用户输入、脚本文件、异常输出和运行时状态串起来，是理解“固件烧进去以后用户代码怎么跑起来”的好入口。

`ports/stm32/main.c` 或 `ports/rp2/main.c` 是端口主流程。它体现了嵌入式固件不像服务器程序那样从 `main` 直接进入业务循环，而是要先完成硬件、内存、文件系统、USB/串口和运行时初始化。

## 7. 软件工程师类比解释

第一个类比：MicroPython 的 `ports/` 像后端项目里的适配器层或驱动插件层。核心运行时尽量保持统一，端口负责接入不同芯片和板卡。这个类比的局限是什么：后端适配器通常只面对 API 差异，而嵌入式端口还要面对寄存器、启动地址、中断、内存映射和电气外设差异。

第二个类比：`mpconfig.h` 像大型 SaaS 的 feature flags。通过开关决定启用哪些能力。这个类比的局限是什么：服务器上的 feature flag 多半是运行时策略，MicroPython 的很多配置是编译期决定，一旦烧录到设备里就不能随便改。

第三个类比：垃圾回收像服务端的内存管理和后台清理任务。它回收不再使用的对象，避免内存耗尽。这个类比的局限是什么：服务器内存大且有操作系统兜底，MCU 内存小，GC 停顿、碎片和栈扫描边界都会直接影响业务时序。

第四个类比：REPL 像生产环境里的远程控制台。它让你现场查看状态、执行命令、调试逻辑。这个类比的局限是什么：嵌入式 REPL 可能通过串口或 USB 暴露，受带宽、安全、启动模式和现场连接条件限制，不能等同于云端 SSH。

第五个类比：Unix port 像本地模拟环境。它让很多语言层测试不依赖开发板。这个类比的局限是什么：模拟环境无法完全覆盖真实中断、外设时序、电源抖动、Flash 擦写和无线网络行为。

## 8. 今日 10 个嵌入式概念卡片

### 1. Port（端口）
- 一句话解释：把同一套核心代码适配到某个芯片、操作系统或板卡的实现层。
- 软件工程类比：数据库驱动或云厂商适配器。
- 常见工程场景：同一固件框架支持 STM32、RP2040、ESP32、Unix。
- 容易误解/坑点：端口不是简单换编译参数，往往涉及启动、链接脚本、中断和外设。

### 2. HAL（硬件抽象层）
- 一句话解释：把寄存器和外设差异包装成相对统一的接口。
- 软件工程类比：平台 SDK 或抽象接口。
- 常见工程场景：统一访问 GPIO、UART、SPI、I2C。
- 容易误解/坑点：抽象会泄漏，不同芯片能力和时序差异不能完全抹平。

### 3. REPL
- 一句话解释：读取、执行、打印循环，用来交互式运行代码。
- 软件工程类比：Python shell 或运维控制台。
- 常见工程场景：板卡调试、现场验证传感器、快速执行脚本。
- 容易误解/坑点：REPL 方便但可能带来安全和资源占用问题。

### 4. GC（垃圾回收）
- 一句话解释：自动找出不再使用的对象并回收内存。
- 软件工程类比：托管语言运行时的内存管理。
- 常见工程场景：脚本创建临时对象、字符串、列表、字典。
- 容易误解/坑点：自动回收不等于无限内存，GC 停顿和碎片仍需关注。

### 5. Heap（堆）
- 一句话解释：运行时动态分配对象使用的内存区域。
- 软件工程类比：进程堆内存。
- 常见工程场景：创建 Python 对象、缓冲区、网络包。
- 容易误解/坑点：MCU 堆通常很小，临时对象太多会造成不可预测问题。

### 6. Flash
- 一句话解释：非易失存储，用来放固件、文件系统和持久化数据。
- 软件工程类比：服务器磁盘的一小块只读或低频写区域。
- 常见工程场景：保存脚本、配置、证书、日志。
- 容易误解/坑点：Flash 有擦写粒度和寿命限制，不适合高频小写入。

### 7. Soft Timer（软定时器）
- 一句话解释：由软件调度维护的定时任务机制。
- 软件工程类比：事件循环里的定时任务。
- 常见工程场景：周期采样、超时检测、延迟执行。
- 容易误解/坑点：软定时器精度受主循环、中断和系统负载影响。

### 8. Interrupt（中断）
- 一句话解释：硬件事件打断当前执行流，进入预设处理逻辑。
- 软件工程类比：高优先级事件回调。
- 常见工程场景：按键、串口接收、定时器、DMA 完成。
- 容易误解/坑点：中断里不能随意分配内存、打印日志或执行复杂逻辑。

### 9. VFS（虚拟文件系统）
- 一句话解释：用统一接口访问不同底层文件系统或存储介质。
- 软件工程类比：操作系统文件系统抽象。
- 常见工程场景：在 Flash、SD 卡或 ROM 文件系统上运行脚本。
- 容易误解/坑点：嵌入式文件系统受掉电一致性、磨损和容量限制影响。

### 10. Cross Compiler（交叉编译器）
- 一句话解释：在开发机上生成目标设备可执行内容的工具。
- 软件工程类比：CI 里为不同平台打包产物。
- 常见工程场景：用 `mpy-cross` 生成 `.mpy` 字节码，或为 ARM/RISC-V 构建固件。
- 容易误解/坑点：能编译通过不代表能在目标板上稳定运行，还要验证内存、时序和外设。

## 9. 30～60 分钟动手任务

目标：不用真实硬件，先通过 Unix port 理解 MicroPython 的构建、运行和 REPL，再阅读一个真实 MCU port 的启动流程。

步骤：
1. 克隆仓库后阅读 `README.md` 中的构建说明。
2. 在支持的开发环境中尝试构建 Unix port：进入 `ports/unix/`，按项目文档安装依赖并执行构建命令。
3. 启动生成的 Unix 版 MicroPython，进入 REPL，执行简单的对象创建、异常处理和模块导入。
4. 打开 `ports/rp2/main.c` 或 `ports/stm32/main.c`，把启动流程画成五步：硬件初始化、堆初始化、运行时初始化、文件系统/输入输出准备、执行用户代码或 REPL。
5. 打开 `py/mpconfig.h`，找三个你认为会影响固件体积的配置项，记录它们可能带来的功能取舍。

验收标准：
- 能说明 Unix port 与 MCU port 的差异。
- 能解释为什么 MicroPython 需要 `mpy-cross`。
- 能指出至少一个“服务器 Python 习惯在 MCU 上不安全”的例子，例如大量动态创建对象、频繁写 Flash、在中断回调里做复杂工作。

无硬件替代方案：只构建 Unix port，并把 MCU port 的 `main.c` 当作源码阅读材料；如果本地依赖不足，则只完成文件阅读和流程图。

## 12. 工程点评

可维护性方面，MicroPython 的核心、端口、扩展模块和测试分层清晰，适合长期演进；但配置开关很多，新人容易迷失，需要从一个端口切入。可移植性方面，`ports/` 结构是亮点，它展示了同一运行时跨芯片复用的工程方法；但移植成本并不低，硬件差异会渗透到启动、内存、外设和构建。可靠性方面，项目提供 GC、异常、文件系统和 REPL，提升开发效率，但动态语言运行在 MCU 上必须谨慎控制内存、时序和持久化写入。可测试性方面，Unix port 给了软件工程师很好的入口，可以先在宿主机测试语言层逻辑，再迁移到硬件；但真实外设和电源问题仍需要板级验证。量产与运维方面，MicroPython 适合快速迭代、教学、可配置设备和中低复杂度产品，但如果业务强实时、资源极紧或安全面极敏感，需要严格裁剪模块、限制 REPL、设计升级和回滚策略。
