# 每日嵌入式学习：nanopb（嵌入式 Protocol Buffers）

## 1. GitHub 项目链接与一句话介绍

- 仓库地址：https://github.com/nanopb/nanopb
- 一句话：nanopb 是一个专为微控制器和嵌入式系统设计的 Protocol Buffers 编解码库，让资源受限设备也能用上 Google Protobuf 的高效序列化格式，同时将代码体积压缩到最小。

## 2. 为什么今天选它

nanopb 体现了"在极端资源约束下重新设计成熟协议"的嵌入式工程思维，对于有软件工程背景的人来说是理解嵌入式序列化范式的绝佳案例。

- **活跃度**：GitHub 上 5434 颗星、1350+ Forks。2026 年 6 月仍有活跃提交。Petteri Aimonen 作为核心维护者持续参与十余年，社区发布稳定，最新版本 0.9.x。议题和 PR 响应及时。
- **工程完整度**：完整的 protobuf 编解码生成器（作为 protoc 插件）、C 语言运行时库、多种内存分配策略（静态/动态/回调）、支持 proto2 和 proto3、支持 oneof、map、packed fields、自定义选项等几乎所有 protobuf 特性。
- **学习价值**：对软件工程师来说，可能在后端用过 protobuf/gRPC，但从未思考过"在只有 2KB RAM 的 MCU 上怎么用 protobuf"。nanopb 展示了：如何用 C 预处理器宏生成类型安全的编解码函数、如何用回调流式处理无法一次性放入内存的大数据、如何用静态分配避免堆碎片。这些是嵌入式序列化的核心思维。
- **Why Now**：此前讲过了 JSON（Mongoose 中的 JSON 处理）、MQTT 协议、lwIP 网络栈。但序列化格式是物联网应用层协议的关键一环——设备端采集传感器数据，必须用高效的二进制格式上传。nanopb 比 JSON 体积小 5-10 倍、解析速度快一个数量级，是真正的嵌入式首选。

## 3. 项目解决的问题

**真实场景**：你有一个基于 STM32G030（主频 48MHz、RAM 8KB、Flash 32KB）的温湿度传感器节点，需要将采集的温度、湿度、电池电压、信号强度等数据通过 NB-IoT 模组发送到云端。云端的后端服务用标准的 protobuf 解析。如果直接在 MCU 上用完整的 protobuf C++ 库，代码体积轻松超过 100KB Flash、需要数 KB 的运行时堆。这个 MCU 根本跑不动。你需要一个解析器只生成"刚好够用"的编解码代码——只针对你定义的那两三个消息类型，而不是一个通用的 protobuf 解析器。

nanopb 解决的核心问题是：**如何让 protobuf 序列化格式在 KB 级 MCU 上可用**。

它面临的嵌入式特有约束：

- **存储**：生成的编解码函数只包含你实际使用的 .proto 消息类型的代码。如果 proto 文件中只有两个 message，生成的 C 代码只有几百行，不包含任何"运行时加载 schema"的开销。这与 protobuf-c 或 C++ protobuf 的"运行时反射"形成鲜明对比。
- **内存**：支持三种分配模式——静态分配（编译时固定最大消息大小，分配在栈或全局变量上，零堆开销）、动态分配（使用 malloc，适合消息大小不确定的场景）、回调模式（通过回调流式处理字段，几乎不占用额外 RAM）。
- **可靠性**：编解码函数对输入做严格的边界检查，确保不会因为损坏的输入数据导致缓冲区溢出。这在嵌入式设备接受来自不可靠无线网络的二进制数据时至关重要。
- **可移植性**：纯 C89/C99 代码，无平台依赖，支持从 8 位 AVR 到 32 位 ARM Cortex-M 到 x86 Linux 的任何平台。

**与普通软件项目的差异**：后端工程师在微服务之间使用 protobuf，依赖的是 Google 官方的 C++/Java/Go 运行时库，几百 MB 内存随意分配、由操作系统负责内存保护、出错时有完整的异常处理机制。而在嵌入式场景中，一个 protobuf 解析错误就可能导致看门狗复位或内存越界破坏关键数据。nanopb 没有异常、没有虚函数、没有 RTTI、没有动态内存管理（可选情况下），一切错误通过返回值报错。

## 4. 技术架构分析

**基于公开文件和源码推断。**

### 整体架构

nanopb 由两个主要部分组成：**代码生成器（Python）** 和 **运行时库（C）**。

```
.proto 文件（定义消息格式）
    ↓
nanopb 生成器（protoc 插件，Python）
    ↓
.pb.h + .pb.c（生成的 C 头文件和源文件）
    ↓
    应用程序代码（调用生成的编解码函数）
    ↓
nanopb 运行时库（pb_encode / pb_decode）
    ↓
    原始字节流（通过 UART/I2C/SPI/无线模块发送）
```

### 目录结构

| 目录/文件 | 用途 |
|-----------|------|
| `pb.h` / `pb.c` | 核心运行时库：编码器、解码器、流接口、字节序处理 |
| `pb_common.h` / `pb_common.c` | 公共类型和辅助函数（field 迭代、扩展字段处理） |
| `pb_encode.h` / `pb_encode.c` | 编码器的具体实现 |
| `pb_decode.h` / `pb_decode.c` | 解码器的具体实现 |
| `generator/` | Python 代码生成器（包含 nanopb_generator.py、protoc 插件入口） |
| `generator/proto/` | nanopb 的 .proto 文件（描述生成选项） |
| `tests/` | 测试套件（包含数十个测试用例） |
| `examples/` | 示例代码（simple、network_server、uart 等） |
| `docs/` | 文档（概念、迁移指南、API 参考） |

### 关键设计模式

1. **生成即定制**：nanopb 不像 protobuf-c 那样提供"通用"的编解码器。它的生成器针对每个 message 类型生成专门的 C 结构体和编解码函数。例如 proto 中定义了一个 `SensorReading` 消息，生成器就会生成 `SensorReading` 结构体、`pb_encode_SensorReading()` 和 `pb_decode_SensorReading()` 函数。这些函数是直接操作结构体字段的 C 代码，而不是通过反射表查找字段。这就是它们能编译到很小体积的原因。

2. **pb_ostream_t / pb_istream_t 流抽象**：nanopb 定义了统一的流接口。编码器写入 `pb_ostream_t`，解码器从 `pb_istream_t` 读取。流接口包含一个函数指针 `callback`，底层可以是内存缓冲区、UART 发送函数、文件系统写入函数、无线电模块发送函数等。用户实现自己的回调，编解码器不需要知道底层是什么。这类似于 Go 的 `io.Writer` / `io.Reader` 接口。

3. **静态与动态分配可选**：在 .proto 文件中，可以用 `nanopb` 选项为每个字段指定分配策略。对于固定长度的字符串/字节数组，可以编译时设定最大长度（`max_size` 选项）。对于 repeated 字段，可以指定最大元素个数（`max_count`）。对于真正可变大小的数据，可以启用 `callback` 模式，让用户提供一个回调函数，字段数据到达时逐段调用回调，避免一次性分配整个缓冲区。

4. **pb_field_t 描述表**：每个生成的 message 类型对应一个 `pb_field_t` 数组，描述了每个字段的编号、类型、偏移量、大小、标签等信息。编解码函数遍历这个数组来序列化或反序列化。这个描述表在编译时生成，存放在 Flash（ROM）中，不占用 RAM。这与"运行时构建反射信息"的库（如 protobuf-c）有本质区别。

### 编码/解码流程

```
编码流程：
1. 应用程序填充结构体字段（如 sensor.temperature = 25.4f）
2. 创建 pb_ostream_t 输出流（指向内存缓冲区或自定义输出）
3. 调用 pb_encode(&stream, SensorReading_fields, &sensor)
4. pb_encode 遍历 SensorReading_fields 描述表
5. 对每个字段，根据类型调用对应的编码函数（如 pb_encode_varint、pb_encode_fixed32、pb_encode_submessage）
6. 编码结果写入输出流
7. 返回 true/false 表示成功/失败

解码流程：
1. 创建 pb_istream_t 输入流（指向收到的缓冲区或自定义输入）
2. 调用 pb_decode(&stream, SensorReading_fields, &sensor)
3. pb_decode 从输入流读取字节，解析 varint/tag
4. 根据 tag 查找字段在描述表中的位置
5. 调用对应的解码函数，将值写入结构体对应字段
6. 对未知字段，默认跳过（或通过回调捕获）
7. 返回 true/false 表示成功/失败
```

### 字节序处理

nanopb 使用小端字节序作为内部表示。在固定长度字段（fixed32、fixed64、float、double）上，如果目标平台的字节序不同，nanopb 会在编码/解码时自动转换。对于 varint 类型，字节序不影响表示。

## 6. 核心模块/函数导读

### 1. `pb_encode()`（`pb_encode.c`）
- **角色**：nanopb 的顶层编码入口。接收一个输出流、一个 field 描述表指针、一个消息结构体指针，将结构体序列化为 protobuf 二进制格式。
- **关键点**：该函数遍历 `pb_field_t` 数组，对每个字段调用对应的编码函数。如果字段是 submessage 类型，递归调用自身。对 oneof 字段，根据 `which_xxx` 标志选择编码哪个成员。对 packed repeated 字段，先计算总长度再编码。
- **关系**：依赖 `pb_encode_varint`（编码变长整数）、`pb_encode_tag`（编码 field tag 和 wire type）、`pb_encode_submessage`（递归编码子消息）。这些底层函数共同构成编码流水线。

### 2. `pb_decode()`（`pb_decode.c`）
- **角色**：nanopb 的顶层解码入口。从输入流读取 protobuf 二进制数据，根据 field 描述表解析到对应的结构体字段中。
- **关键点**：解码器是一个基于 TLV（Type-Length-Value）的状态机。先读 tag（varint），从 tag 中提取 field_number 和 wire_type。用 field_number 在 field 描述表中查找对应的字段信息。根据 wire_type 和字段类型决定如何解析 value。对未知字段（描述表中找不到的 field_number），默认跳过。
- **关系**：依赖 `pb_decode_varint`（解码变长整数）、`pb_decode_tag`（解析 tag）、`pb_decode_fixed32` 等。最关键的子函数是 `pb_decode_fieldtype()`，它根据 wire_type 分发到不同的解析逻辑。

### 3. `pb_ostream_from_buffer()` / `pb_istream_from_buffer()`（`pb.h` 或 `pb.c`）
- **角色**：创建基于内存缓冲区的输入/输出流。这是最常用的流接口。
- **关键点**：`pb_ostream_from_buffer(buf, size)` 创建一个写向固定缓冲区的流。如果写入的数据超过缓冲区大小，`bytes_written` 仍然增加但数据被截断，`errno` 被置为 `ENOMEM`。调用者必须在编码后检查流状态。`pb_istream_from_buffer(buf, size)` 创建一个从固定缓冲区读取的流，读取超过大小后返回 false。
- **关系**：流抽象是 nanopb 的可扩展性基础。除了内存缓冲区，用户可以创建基于 UART、文件系统、加密模块的自定义流，只需设置 `callback` 函数指针。

### 4. `pb_field_t` 结构体与 field 描述表（`pb.h` 生成的 .c 文件）
- **角色**：每个 message 类型对应一个 `pb_field_t` 数组（例如 `SensorReading_fields`），描述了该消息所有字段的元数据。
- **关键点**：每个 `pb_field_t` 包含字段编号（tag）、数据类型（`PB_LTYPE_*`）、数据大小（`sizeof`）、在结构体中的偏移量（`offsetof`）、数组元素个数、标签标志（required/optional/repeated/oneof）等。这个数组在编译时由生成器创建，存放在 Flash 中。
- **关系**：编码器和解码器都不需要知道具体的 message 类型，它们通过 field 描述表来操作任意消息。这类似于 C++ 的 vtable 概念，但在编译时静态生成。

### 5. `pb_callback_t` 与回调模式（`pb.h`）
- **角色**：当字段数据量太大无法预分配缓冲区时，使用回调模式逐段处理。对编码，回调函数在需要数据时被调用；对解码，回调函数在每次接收到一块数据时被调用。
- **关键点**：`pb_callback_t` 结构体包含 `funcs.encode` 和 `funcs.decode` 两个函数指针，以及一个 `arg` 指针（用户上下文）。例如，一个包含 10 万条日志记录的 repeated 字段，在回调模式下解码时，每条记录到达时调用一次回调，用户可以决定是写入 SD 卡、通过网络发送还是过滤掉，而不需要一次性分配容纳所有记录的内存。
- **关系**：回调模式是纳米级设备使用 protobuf 的关键能力。不使用回调意味着你必须分配足以容纳最大消息的内存——在 RAM 只有 4KB 的设备上，这可能是不可接受的。

## 7. 软件工程师类比解释

### 1. pb_field_t 描述表 → Java 的反射（Reflection）API
Java 反射允许你在运行时查看类的字段和方法信息。nanopb 的 `pb_field_t` 数组提供了类似的功能——它描述了结构体的每个字段的偏移量、类型、标签——但它是**在编译时静态生成**的，不存在运行时性能开销。
- **类比的局限**：Java 反射是运行时能力，你可以用任意类的名字来查询其成员。nanopb 的 field 描述表是编译时就固定的，无法在运行时动态加载一个新的 message 类型。要支持一个新消息类型，必须重新运行 protoc 生成代码、重新编译固件。

### 2. pb_ostream_t / pb_istream_t → Go 的 io.Writer / io.Reader 接口
Go 的 `io.Writer` 定义了一个 `Write(p []byte) (n int, err error)` 方法，不同的实现（文件、网络连接、缓冲区）满足这个接口。nanopb 的流抽象完全一样——`pb_ostream_t` 有一个 `callback` 字段，用户提供自己的写入函数，编码器通过这个回调输出数据，完全不知道底层是 UART、SPI Flash 还是内存缓冲区。
- **类比的局限**：Go 的接口是动态分发的（运行时通过 itable 查方法表）。nanopb 的流回调是函数指针调用，虽然也是运行时间接调用，但没有类型安全检查——如果你传入了错误的回调函数指针，编译器不会报错。此外，Go 的 Writer 支持 `Write` 方法的标准错误返回，nanopb 只通过 `errno` 和 bool 返回值传递错误信息。

### 3. 静态内存分配（max_size / max_count）→ TypeScript 的静态类型检查
在 TypeScript 中你定义一个 `age: number` 变量，编译器保证你不会把一个字符串赋给 age。nanopb 的静态内存分配类似——你在 .proto 中声明 `string name = 1 [(nanopb).max_size = 40]`，生成器就分配一个 `char name[40]` 的固定数组，编译器保证 name 字段不会超过 40 字节。任何超过 39 字符的字符串在编码时会被截断，由运行时库报告错误。
- **类比的局限**：TypeScript 的类型检查对所有使用代码都生效。nanopb 的 max_size 只对生成的结构体有效——如果你手动通过 memcpy 覆盖结构体字段，绕过生成器的保护，没有编译器能帮你检查。在嵌入式开发中，经常有人为了省事直接操作内存，然后引入 bug。

### 4. pb_encode 的递归 submessage 编码 → HTML DOM 的递归渲染
浏览器渲染一个 HTML 元素时，如果该元素包含子元素，浏览器递归地渲染每个子元素。nanopb 对 submessage 字段的处理完全一样——当编码器遇到 `PB_LTYPE_SUBMESSAGE` 类型的字段时，它递归调用 `pb_encode()` 来处理子消息，子消息再包含子消息则继续递归。
- **类比的局限**：HTML DOM 的深度可以很大（几十层嵌套），浏览器有栈空间处理。MCU 的栈通常只有 1-4KB，如果 proto 定义中有深度嵌套的消息（比如 A 包含 B，B 包含 C，C 包含 D……），递归编解码很容易栈溢出。nanopb 没有做尾递归优化或递归深度限制检查——这是开发者在定义消息结构时必须意识到的约束。

### 5. generator/nanopb_generator.py → Webpack/Vite 的代码打包
Webpack 分析你的 import 语句，只打包你用到的模块，移除 dead code（tree-shaking）。nanopb 的生成器做类似的事情——它只针对你 .proto 文件中定义的 message 类型生成代码，不包含任何"通用的 protobuf 解析器"。生成的 .c 文件只包含这几种消息的编解码逻辑，没有运行时加载 .proto 文件的能力。
- **类比的局限**：Webpack 的 tree-shaking 可以移除未使用的导出，但运行时仍加载同一个 bundle。nanopb 的"编译时定制"更加彻底——根本不存在"通用解析器"代码，每个消息的解析代码是内联的、特化的。代价是每次修改 .proto 都要重新运行生成器，而不能像后端那样运行时加载新的 .proto 文件。

## 8. 今日 10 个嵌入式概念卡片

### 卡片 1：TLV（Type-Length-Value）编码
- **一句话**：protobuf（和 nanopb）使用的二进制数据格式——每个字段编码为"字段类型和编号"（Type）+ "数据长度"（Length，仅变长类型）+ "数据值"（Value）。
- **软件类比**：HTTP 的 Content-Type 头 + Content-Length 头 + Body。Type 告诉你这是什么字段，Length 告诉你有多长，Value 是实际数据。
- **常见场景**：MCU 将传感器数据序列化为二进制字节流，通过无线模块发送到网关。网关解析 TLV 格式提取各个字段。
- **易错点**：protobuf 的 tag 包含 field_number 和 wire_type，wire_type（0=varint、1=64bit、2=length-delimited、5=32bit）决定了数据如何解析。如果 sender 和 receiver 对 wire_type 的理解不一致（比如 proto 版本不匹配），解码器会解析出错误数据，但 nanopb 只检查 tag 中的 field_number 是否匹配，不会验证 wire_type 与字段类型是否一致——这是 protobuf forward/backward compatibility 的设计，但也意味着不会帮你发现这种错误。

### 卡片 2：varint（变长整数编码）
- **一句话**：用 1-10 个字节表示整数的编码方式，数值越小占用的字节越少。每个字节的最高位（MSB）表示是否还有后续字节，低 7 位是数据。
- **软件类比**：UTF-8 编码——ASCII 字符用 1 个字节，中文用 3 个字节。varint 也是类似的"小数据用少字节，大数据用多字节"的设计。
- **常见场景**：protobuf 中所有 int32、int64、uint32、uint64、enum、bool 类型都编码为 varint。传感器值如温度（int32，通常 ±100 以内）编码为 1-2 个字节。
- **易错点**：负的 int32 会编码为 10 个字节（因为 protobuf 使用 signed varint 编码 `sint32`，而 `int32` 对负值使用完整的 64 位表示）。如果你传输的数据可能为负（如温度 -5°C），应该用 `sint32` 而不是 `int32`，否则每个负值都会浪费 9 个额外字节。

### 卡片 3：编译时配置（.proto 中的 nanopb 选项）
- **一句话**：在 .proto 文件中通过 `[(nanopb).max_size = ...]`、`[(nanopb).max_count = ...]`、`[(nanopb).type = PB_LTYPE_FIXED_LENGTH_BYTES]` 等选项来指定每个字段的内存分配策略，这些选项在编译时被生成器读取并影响生成的 C 代码。
- **软件类比**：Java 或 C# 中的属性/注解（Annotation）。你在字段上加一个注解，编译器插件读取注解生成不同的代码。
- **常见场景**：`string name = 1 [(nanopb).max_size = 40]` 将字符串限制为 40 字节固定数组；`repeated float samples = 2 [(nanopb).max_count = 100]` 限制 repeated 字段最多 100 个元素，使用静态数组而非动态数组。
- **易错点**：max_size 设定过小会导致生产环境中截断重要数据；设定过大会浪费 RAM。嵌入式开发中，你必须对实际数据大小有准确的预估——这在软件开发中很少需要操心的事情，在嵌入式开发中直接影响 BOM 成本。

### 卡片 4：流抽象（pb_ostream_t / pb_istream_t）
- **一句话**：nanopb 的 I/O 接口抽象层，定义了一个包含函数指针回调的结构体，编码器/解码器通过回调与底层传输介质通信，不需要关心具体是内存、UART、SPI Flash 还是网络 socket。
- **软件类比**：UNIX 的"一切皆文件"哲学——open/read/write/close 统一接口，底层可以是普通文件、管道、socket、设备。
- **常见场景**：将传感器数据编码后直接写入 UART 发送（使用自定义 `uart_write` 回调），而不需要先在内存缓冲区内，再拷贝到 UART 发送寄存器。
- **易错点**：流回调函数中不能做长时间阻塞操作（如等待 Flash 写入完成），否则会阻塞整个编码/解码流程。如果回调返回的错误码不够精确（nanopb 只区分"成功"和"失败"），上层无法区分是"缓冲区满了"还是"硬件错误"。

### 卡片 5：堆 vs 静态内存分配
- **一句话**：嵌入式开发中两种主要的内存分配方式——堆分配（malloc/free）灵活但有碎片和不确定性；静态分配（全局/栈数组）可预测但不够灵活。nanopb 同时支持两种方式。
- **软件类比**：JavaScript 的自动 GC（堆分配）vs Rust 的栈优先+显式生命周期管理。前者方便但不可预测，后者可控但需要更多设计。
- **常见场景**：产品原型阶段用堆分配减少开发时间；量产固件用静态分配确保确定性和零碎片风险。nanopb 允许在 .proto 层面对每个字段独立选择分配方式。
- **易错点**：堆分配在长时间运行的嵌入式设备中会产生碎片。"跑了三个月后 malloc 失败"是嵌入式领域的经典问题。静态分配虽然安全，但如果在编译时设定的最大消息大小不够，编码会静默截断数据——解码端根本不知道数据被截断了。

### 卡片 6：protobuf 的 forward/backward compatibility
- **一句话**：protobuf 格式设计保证了新老版本消息的兼容性——发送方可以添加新字段，老版本接收方自动忽略未知字段；发送方可以删除字段（保留 tag 号不重用），老版本接收方读取到该字段时该字段保持默认值。
- **软件类比**：REST API 的版本管理。你可以在 API 响应中添加新字段，老客户端忽略它们；你可以废弃旧字段但仍然返回它们以保证兼容。protobuf 用 tag 号实现类似的能力，但不需要 API 版本号。
- **常见场景**：OTA 固件升级后，传感器上报消息新增了一个字段。云端的新服务能读取新字段，旧设备仍然使用旧格式也能被新服务正确解析。不需要协调升级所有设备。
- **易错点**：forward compatibility 依赖接收方正确初始化结构体。如果你在解码前没有用 `memset` 或生成的 `xxx_init` 函数初始化结构体，未赋值的字段包含栈上的随机值，nanopb 不会重置这些字段，解码后你会得到 garbage 值。

### 卡片 7：代码生成器（protoc 插件模式）
- **一句话**：nanopb 的生成器作为 protoc 的插件运行，读取 .proto 文件后生成对应的 C 头文件和源文件。不同 .proto 文件生成独立的 .h/.c 对，不产生额外的运行时 schema 文件。
- **软件类比**：GraphQL 的代码生成工具（如 graphql-codegen）从 schema 生成 TypeScript 类型和 hooks。都是"从定义文件生成类型安全代码"的模式。
- **常见场景**：项目中有 `sensor.proto` 和 `config.proto` 两个文件，分别生成 `sensor.pb.h/.c` 和 `config.pb.h/.c`。应用程序只 `#include "sensor.pb.h"` 来使用传感器数据结构。
- **易错点**：生成器生成的代码依赖 nanopb 运行时库的版本。如果库版本升级（如 0.4.x → 0.5.x），必须重新运行生成器更新生成代码，否则可能出现结构体偏移量不匹配导致的静默内存损坏。在 CI/CD 中应该确保生成器版本与运行时库版本一致。

### 卡片 8：回调模式处理大数据
- **一句话**：当消息中有大数据字段（如日志列表、大块文件数据）时，使用回调模式避免一次性分配大缓冲区——每处理一个元素就调用一次用户回调，数据随到随处理。
- **软件类比**：Node.js 中处理大文件时的流式 API（`fs.createReadStream().on('data', ...)`）。不是等整个文件读完再处理，而是每读到一个 chunk 就触发回调处理。
- **常见场景**：传感器节点要上传 10000 个历史采样点。如果用静态 repeated 字段，需要 `max_count = 10000`，每个采样点 4 字节 float，仅这个字段就需要 40KB RAM——很多 MCU 根本没有这么多内存。用回调模式，每次解码一个采样点就调用回调发送出去，内存占用只有几十字节。
- **易错点**：回调函数的执行时间不能太长。如果在 UART 发送回调中等待发送完成（轮询 TX 寄存器），会阻塞后续字段的解码。应该使用 DMA 或中断驱动的发送，让回调快速返回。

### 卡片 9：字段描述表（pb_field_t）与 Flash 存储
- **一句话**：每个消息类型编译时生成的 `pb_field_t` 数组，描述了该消息所有字段的元数据，存放在 Flash（ROM）中而非 RAM 中。编码/解码时通过遍历这个数组来操作消息字段。
- **软件类比**：C++ 的 vtable 表——每个类有一个虚函数表，在编译时生成，存储在只读内存中。pb_field_t 数组类似于 vtable，但它描述的是数据字段而非虚函数。
- **常见场景**：你定义了一个包含 20 个字段的消息，生成器生成一个包含 20 个 `pb_field_t` 条目的数组（外加一个结束标记），总计约 20 × 12 = 240 字节，全部存放在 Flash 中，不占用宝贵的 RAM。
- **易错点**：字段描述表的顺序必须与结构体中字段的定义顺序一致。如果你手动修改了生成的结构体（不推荐），但没有重新生成 field 描述表，编码/解码会读取错误偏移量的数据，导致数据损坏。永远不要手动修改生成的文件。

### 卡片 10：errno 与 bool 返回值错误处理
- **一句话**：nanopb 不使用异常或 setjmp/longjmp 来处理错误，而是通过函数的 bool 返回值（true=成功/false=失败）和全局/流对象中的 `errno` 变量来传递错误信息。调用者必须在每次编解码后检查返回值。
- **软件类比**：Go 语言的 `(result, error)` 返回值模式——不是抛出异常，而是显式返回错误。但它比 Go 更原始，没有 error 类型，只有一个整数 errno。
- **常见场景**：编码时缓冲区满了（`pb_encode` 返回 false，流中的 `errno` 为 `ENOMEM`）；解码时遇到损坏的数据（`pb_decode` 返回 false，`errno` 为 `EBADMSG`）。
- **易错点**：在嵌入式开发中，最常见的 bug 就是"忘记检查返回值"。一个不检查 `pb_encode()` 返回值的函数，在缓冲区溢出时仍然继续执行后面的代码，发送的是截断的、损坏的数据包。建议的做法是使用 `assert()` 或条件编译的 CHECK 宏，在开发阶段捕获所有未检查的错误。

## 9. 30～60 分钟动手任务

**任务：在本地编写并运行一个 nanopb 的传感器数据编解码程序**

### 目标
不依赖任何硬件，在 Linux 上通过 CMake 编译 nanopb 运行时库，编写一个 .proto 文件定义传感器消息，生成 C 代码，编写一个测试程序完成编码和解码的往返验证（round-trip）。

### 步骤

1. **准备 nanopb 源码**（已存在于 `/tmp/nanopb_repo` 或手动下载）：
```bash
git clone --depth=1 https://github.com/nanopb/nanopb.git /tmp/nanopb_lab
cd /tmp/nanopb_lab
```
注意：如果 git clone 有网络问题，可以直接从 GitHub 页面下载 ZIP 包或使用已有的缓存。

2. **编写 .proto 文件**：
创建 `/tmp/nanopb_lab/sensor.proto`：
```protobuf
syntax = "proto3";
import "nanopb.proto";

message SensorData {
    uint32 device_id = 1;
    sint32 temperature = 2 [(nanopb).max_size = 1];  // -128~127，1字节
    uint32 humidity = 3 [(nanopb).max_value = 100];   // 0~100
    float battery_voltage = 4;
    string location = 5 [(nanopb).max_size = 32];
    repeated float fft_bins = 6 [(nanopb).max_count = 8];  // 最多8个FFT bin
}
```

3. **生成 C 代码**：
```bash
cd /tmp/nanopb_lab
# nanopb 生成器需要 protoc 和 Python 依赖
python3 generator/nanopb_generator.py sensor.proto
# 生成 sensor.pb.h 和 sensor.pb.c
```

4. **编写测试程序**：
创建 `/tmp/nanopb_lab/main.c`：
```c
#include <stdio.h>
#include <string.h>
#include "pb_encode.h"
#include "pb_decode.h"
#include "sensor.pb.h"

int main() {
    uint8_t buffer[256];
    
    // === 编码 ===
    SensorData msg = SensorData_init_zero;
    msg.device_id = 42;
    msg.temperature = -5;
    msg.humidity = 68;
    msg.battery_voltage = 3.35f;
    strcpy(msg.location, "living_room");
    msg.fft_bins_count = 4;
    msg.fft_bins[0] = 0.1f;
    msg.fft_bins[1] = 0.5f;
    msg.fft_bins[2] = 2.3f;
    msg.fft_bins[3] = 0.8f;
    
    pb_ostream_t stream = pb_ostream_from_buffer(buffer, sizeof(buffer));
    if (!pb_encode(&stream, SensorData_fields, &msg)) {
        printf("Encode failed: %s\n", PB_GET_ERROR(&stream));
        return 1;
    }
    printf("Encoded %zu bytes\n", stream.bytes_written);
    
    // === 解码 ===
    SensorData decoded = SensorData_init_zero;
    pb_istream_t istream = pb_istream_from_buffer(buffer, stream.bytes_written);
    if (!pb_decode(&istream, SensorData_fields, &decoded)) {
        printf("Decode failed: %s\n", PB_GET_ERROR(&istream));
        return 1;
    }
    
    // === 验证 ===
    printf("device_id: %u\n", decoded.device_id);
    printf("temperature: %d\n", decoded.temperature);
    printf("humidity: %u\n", decoded.humidity);
    printf("battery: %.2f\n", decoded.battery_voltage);
    printf("location: %s\n", decoded.location);
    printf("fft bins: ");
    for (int i = 0; i < decoded.fft_bins_count; i++)
        printf("%.2f ", decoded.fft_bins[i]);
    printf("\n");
    
    // 验证一致性
    if (decoded.device_id != 42 || decoded.temperature != -5) {
        printf("FAIL: values don't match!\n");
        return 1;
    }
    printf("PASS: round-trip verification successful!\n");
    return 0;
}
```

5. **编译并运行**：
```bash
cd /tmp/nanopb_lab
gcc -I. -Igenerator/proto main.c sensor.pb.c pb_encode.c pb_decode.c pb_common.c -o sensor_test
./sensor_test
```

### 验收标准
- 能编译通过，运行输出编码字节数（通常在 20-40 字节之间）。
- 解码后的所有字段值与编码前一致。
- 验证 `temperature` 字段为 -5（验证 `sint32` 正确处理负值）。
- 尝试将 `humidity` 设为 101（超过 max_value），观察编码器是否拒绝（如果没有拒绝，说明 max_value 选项需要在 .proto 中手动检查——这是一个学习点）。

### 无硬件替代方案
如果编译环境有问题，可以直接阅读生成的 `sensor.pb.h` 和 `sensor.pb.c` 文件，观察：
- 结构体如何排列字段（注意字节对齐）。
- `SensorData_fields` 数组的结构。
- `pb_encode_SensorData()` 和 `pb_decode_SensorData()` 的函数签名。

也可以直接阅读 nanopb 源码中的 `tests/` 目录下的测试用例，那里有大量现成的编解码测试代码。

## 10. 3 道深度追问题

**问题 1**：你在一个产品中使用了 nanopb 编码传感器数据，通过 NB-IoT 模块发送到云端。量产 10000 台设备后，发现约 0.1% 的设备在某些温度条件下上报的数据中，温度字段有时会丢失（温度值为 0 而不是实际值）。你已经确认云端的解码代码和固件中的编码代码使用完全相同的 .proto 文件。你会从哪几个维度排查这个问题？注意，温度可能是 0°C（合法的真实值），你如何区分"真 0°C"和"字段丢失"？

**问题 2**（资源限制）：你的 MCU 有 16KB RAM。消息结构体中有一个 `repeated float spectrum[1024]` 字段（4096 字节），加上其他字段和一些协议开销，结构体总大小约 5.5KB。用静态分配可以工作，但每次编解码都要在栈上分配 5.5KB 的结构体——你的栈总共只有 2KB。你有哪些方案来解决这个栈溢出问题？请从"静态全局变量""堆分配""回调模式""拆分消息"四个方案中分析各自的利弊。

**问题 3**（时序/并发）：你的设备在一个 RTOS 上有三个任务：传感器采集任务（每 100ms 采集一次）、数据处理任务（将数据编码为 protobuf）、网络发送任务（通过无线模块发送编码后的数据）。nanopb 的运行时库是纯函数式的（没有全局状态），但编码/解码过程中会修改调用者传入的结构体。在这个三任务架构中，需要注意哪些并发安全问题？如果使用静态分配的全局 pb_ostream_t 会有什么风险？

## 12. 工程点评

**可维护性**：9/10。nanopb 的代码风格清晰，函数命名一致（`pb_encode_xxx` / `pb_decode_xxx`），错误处理通过返回值 + errno 模式一目了然。生成的 .pb.h 文件有完善的注释，不需要看文档就能理解如何使用。唯一不足是 Python 代码生成器代码较长（2000+ 行），修改生成逻辑需要深入理解 protobuf 的 IR 结构。

**可移植性**：10/10。纯 C89/C99 代码，无任何平台依赖。支持从 8 位 AVR（Arduino Uno）到 32 位 ARM Cortex-M 到 64 位 x86 Linux 的所有平台。编译时只需要一个 C 编译器，不需要 CMake 或其他构建系统（虽然官方也提供 CMake 支持）。这是 nanopb 最大的优势之一。

**可靠性**：9/10。编解码器对所有输入做严格的边界检查，缓冲区溢出防护是核心设计。IAR、Keil、GCC 等主流嵌入式编译器下长期量产验证。唯一的潜在问题是：如果用户没有正确初始化结构体（忘记调用 `xxx_init_zero`），解码器不会清空未赋值的字段，这可能导致未定义行为。但这是使用问题而非库本身的问题。

**可测试性**：8/10。nanopb 在 GitHub Actions 上有 CI 测试，包含数十个测试用例覆盖各种边界情况和错误路径。测试目录中有 fuzz 测试。在同一平台的编译和运行都比较顺畅。不足是测试覆盖的 MCU 平台数量有限（主要在 x86 上测试），在一些小众编译器上可能存在未发现的问题。

**量产/运维**：8/10。nanopb 本身不提供运行时遥测或日志功能，但它的设计使得在生产环境中排查问题相对容易——编码结果是可预测的二进制数据，可以使用标准的 protobuf 工具（`protoc --decode_raw`）在云端解析验证。问题排查的难点通常不在 nanopb 本身，而在通信链路（无线模块、丢包、重传）上。

## 13. 明天可以延展的方向

1. **对比学习：nanopb vs cJSON vs tinyCBOR vs mpack**：从编码效率（二进制大小）、解析速度、内存占用、代码体积、易用性四个维度实测比较这四个嵌入式序列化库。写一个基准测试，在同样的 MCU 上序列化同样的数据（100 个字段的传感器数据），对比 Flash 占用、RAM 占用、编码时间和解码时间。

2. **protobuf 在 OTA 升级中的应用**：研究如何使用 protobuf（通过 nanopb）来设计结构化的 OTA 升级协议——包括固件版本协商、分块传输、CRC 校验、回滚机制。这与简单的"下载整个 bin 文件并写入 Flash"的方式相比，在可靠性和兼容性方面有什么优势？

3. **nanopb 在 Zephyr RTOS 中的集成**：Zephyr 内置了对 nanopb 的支持（`CONFIG_NANOPB`）。研究如何在 Zephyr 项目中启用 nanopb，以及 Zephyr 的构建系统如何自动处理 .proto 文件到 C 代码的生成。这展示了"现代嵌入式 RTOS + 现代序列化"的最佳实践。
