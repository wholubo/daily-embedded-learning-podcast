# 每日嵌入式学习：QP/C 实时事件框架

## 1. GitHub 项目链接与一句话介绍

- 仓库：https://github.com/QuantumLeaps/qpc
- QP/C 是一个为微控制器（尤其是 ARM Cortex-M）量身定制的轻量级实时事件框架，它实现了异步事件驱动的"活动对象/角色"（Active Object / Actor）模型，并结合了层次状态机（Hierarchical State Machine），让嵌入式开发者能用结构化的状态机来编写并发、可靠的固件。

## 2. 为什么今天选它

QP/C 是一个非常特别的项目。大多数嵌入式框架走的是"RTOS + 裸机"路线，而 QP/C 从根上走的是**事件驱动 + 状态机 + 角色模型**路线。它有以下几个值得学习的理由：

- **活跃且成熟**：从 2005 年维护至今，目前已到 v8.1.4，有 1325+ GitHub Stars，持续更新和发布。
- **工程完整度极高**：有完整的文档（在线 HTML 手册、应用笔记、视频教程）、安全认证套件（SafeQP/C 用于工业/医疗/航空安全关键系统）、QS 软件跟踪调试子系统、QM 建模工具。
- **适合软件工程师转型嵌入式**：活动对象模型与后端系统里的 Actor 框架（如 Akka、Erlang）理念相通，但实现却必须考虑裸机 MCU 的资源限制——没有堆、没有标准库、没有操作系统。这种"概念熟悉但实现受限"的对比，是理解嵌入式约束的绝佳切入点。
- **可直接学习 C 语言在嵌入式中的高级用法**：QP/C 大量使用 C11 标准特性（位域、内联函数、原子操作、指定初始化器）来实现面向对象风格（继承、多态、虚函数表），几乎是 C 语言嵌入式 OOP 的最佳实践教材。

## 3. 项目解决的问题

在传统嵌入式开发中，一个典型 MCU 项目往往用大循环（super-loop）加中断来处理所有任务。随着系统变复杂，这种方法导致的问题包括：

- 全局变量满天飞，状态难以追踪。
- 中断和主循环之间共享数据的竞态问题无法系统性地解决。
- 业务逻辑和硬件驱动耦合在一起，难以测试和移植。
- 缺乏统一的并发模型，多任务协同依赖开发者"人工调度"。

QP/C 用三个核心设计解决了这些问题：

1. **活动对象（Active Object）作为并发单元**：每个活动对象运行在自己的事件队列中，通过消息传递（事件发布/订阅）进行通信，避免了共享内存的锁问题。这在概念上跟微服务架构中的 Actor 模式完全一样。

2. **层次状态机（HSM）作为行为建模工具**：每个活动对象的行为由一张 UML 状态图定义，支持状态嵌套、历史状态、进入/退出动作。状态机让状态转换显式化，相比传统 switch-case 状态机更易维护、扩展和安全。

3. **零中断延迟的内核设计**：QP/C 的 QK 抢占式内核（及 QV 协作式内核）专门为事件驱动设计，不提供传统的线程/信号量机制，而是把 CPU 的控制权交给事件调度器。它在临界区内的操作极其精简（仅几条指令），使得中断延迟接近零——这对实时嵌入式系统至关重要。

相比传统软件项目，QP/C 面临的硬件和时序约束非常突出：必须工作在无 MMU 的 Cortex-M 上（不能动态内存映射），使用固定大小的内存池（event pool）而非通用堆分配器，所有操作必须在微秒级完成的 ISR 环境中执行，而且需要为一个 64KB Flash / 8KB RAM 级别的芯片提供完整的事件驱动框架。

## 4. 技术架构分析

QP/C 框架由多个组件层构成，以下分析基于公开源码和文档：

### 4.1 整体架构

```
+-------------------------------------------------------------+
|  Application Active Objects (用户编写的角色)                   |
|  ┌────────────┐  ┌────────────┐  ┌────────────┐            |
|  │  Philo[0]  │  │  Philo[1]  │  │  Table AO  │  ...       |
|  └──────┬─────┘  └──────┬─────┘  └──────┬─────┘            |
+---------|---------------|---------------|--------------------+
          |  事件发布      |  事件发布      |  事件订阅
          v               v               v
+-------------------------------------------------------------+
|  QF (QP Framework) - 核心框架层                              |
|  ├─ QActive: 活动对象基类、注册表、优先级位图调度              |
|  ├─ QEvt: 事件结构（含位域 sig、poolNum、refCtr）             |
|  ├─ QTimeEvt: 时间事件（定时器），挂接在时间滴答链表          |
|  ├─ QEQueue: 事件队列（FIFO）                                |
|  ├─ QMPool: 内存池（固定大小块分配器）                        |
|  ├─ 发布-订阅机制：按信号类型分发事件到订阅者                  |
|  └─ QPSet: 优先级位图集（快速查找最高优先级就绪角色）          |
+-------------------------------------------------------------+
|  QEP (QP Event Processor) - 层次状态机引擎                    |
|  ├─ QHsm/QMsm: 层次状态机执行引擎                            |
|  ├─ QStateHandler: 状态处理函数指针                           |
|  └─ QMTranActTable: 状态转换动作表（用于 QM 建模工具生成代码） |
+-------------------------------------------------------------+
|  QK / QV - 实时内核                                          |
|  ├─ QV: 协作式内核（非抢占，事件队列空时 idle）               |
|  ├─ QK: 抢占式内核（支持基于优先级的主动对象抢占）             |
|  └─ QXK: 扩展内核（支持传统阻塞式线程，商业版"QP/C Extras"） |
+-------------------------------------------------------------+
|  QS (Q-SPY) - 软件跟踪子系统                                 |
|  └─ 运行时事件日志、时序记录、错误检测                         |
+-------------------------------------------------------------+
|  平台移植层 (Ports)                                          |
|  ├─ ARM Cortex-M0/M3/M4/M7/M33 等                            |
|  ├─ TI MSP430, Renesas RL78, RX                             |
|  ├─ POSIX (用于桌面测试)                                      |
|  └─ Zephyr RTOS 集成                                         |
+-------------------------------------------------------------+
```

### 4.2 关键设计点

**事件模型**: QEvt 仅 8 字节，用位域紧凑编码信号、池编号和引用计数。动态事件从固定大小的内存池（event pool）分配，而不是 malloc。事件有引用计数，支持多个活动对象共享同一事件实例（零拷贝发布）。

**调度器**: QP/C 不使用传统 RTOS 时间片轮转。它维护一个 QPSet（优先级位图），记录哪个优先级有活动对象就绪。调度时用内置 QF_LOG2 快速找到最高优先级。在 QK 抢占模式下，更高优先级的活动对象发布事件时可以立即抢占当前执行的较低优先级对象。

**状态机引擎**: QHsm 是手工编码的状态机基类，QAsm 引入虚函数表（vptr）实现多态。状态处理函数通过返回 Q_RET_SUPER / Q_RET_HANDLED / Q_RET_TRAN 等编码值来驱动引擎执行状态层次遍历。QMsm 配合 QM 建模工具，使用预计算的转换动作表（QMTranActTable）实现更高执行效率。

### 4.3 配置系统

所有配置通过编译时宏完成（`qf_port.h`）：
- `QF_MAX_ACTIVE`：最多活动对象数（默认 32，最大 64）
- `QF_MAX_TICK_RATE`：支持的时间滴答率数（默认 1，最大 15）
- `QF_MAX_EPOOL`：事件内存池数（默认 3，最大 15）
- `QF_TIMEEVT_CTR_SIZE`：定时器计数器大小（1/2/4 字节）

这种编译时配置避免了运行时动态分配，完全符合 MISRA 和嵌入式安全编码规范。

## 5. 源码陪读：今天只读哪几个文件

以下按推荐阅读顺序排列：

### 文件 1: `include/qp.h`
**阅读目标**：理解整个框架的数据结构体系。先读这个头文件（约 790 行），了解 QEvt、QActive、QTimeEvt、QPSet、QAsm 等所有核心类型的定义和 API。这是整个框架的"地图"。

### 文件 2: `src/qf/qf_qact.c`
**阅读目标**：理解活动对象的生命周期。这个文件实现了 `QActive_ctor`、`QActive_register_`、`QActive_unregister_`，以及 QPSet（优先级位图）的全部操作。重点看 `QActive_register_` 中注册前后的资源检查和 `QActive_ctor` 中 vtable 的设置方式——这是一个纯 C 实现继承和多态的经典手法。

### 文件 3: `src/qf/qf_dyn.c`
**阅读目标**：理解嵌入式事件内存管理。这个文件实现了事件池初始化 `QF_poolInit`、事件分配 `QF_newX_`、回收 `QF_gc` 和引用计数管理。重点看它是如何用固定大小的内存池替代通用堆分配的，以及引用计数如何实现零拷贝事件发布。

### 文件 4: `src/qf/qf_time.c`
**阅读目标**：理解嵌入式定时器设计。这个文件实现了 `QTimeEvt` 的时间滴答驱动。重点看时间事件链表的管理、`QTimeEvt_tick_` 如何处理滴答中断、以及超时事件的到期分发机制。

### 文件 5: `src/qf/qep_hsm.c`
**阅读目标**：理解层次状态机执行引擎。这个文件是 QP/C 最核心的代码之一，实现层次状态机的 init 和 dispatch。重点看状态处理函数如何通过返回值驱动引擎进行状态层次遍历（super/substate 关系）、进入动作、退出动作的执行顺序。

## 6. 核心模块/函数导读

### 6.1 `QActive_ctor` 和 `QActive_start`（src/qf/qf_qact.c）

`QActive_ctor` 初始化活动对象的状态机和虚函数表。`QActive_start` 启动活动对象：分配优先级、注册到全局注册表、绑定事件队列和栈空间、设置优先级阈值（preemption-threshold），最后通过 `QACTIVE_START` 宏启动底层线程。这两个函数是每个嵌入式 QP 应用的入口点，理解它们是上手 QP/C 的第一步。

### 6.2 `QPSet_findMax`（src/qf/qf_qact.c）

这个函数是实现 QK 抢占式调度的核心——用位运算在 O(1) 时间内找到当前最高优先级的就绪活动对象。它利用 `QF_LOG2`（内置查找表 + 二分移位）在纯 C 中实现了一个高效的优先级搜索算法。对于从软件工程转来的开发者，这相当于 Redis 的 `IDLETIME` 排序或操作系统调度器的 runqueue，但实现代价只有几十条指令。

### 6.3 `QF_newX_` 和 `QF_gc`（src/qf/qf_dyn.c）

`QF_newX_` 从事件内存池分配新事件。它按事件大小遍历预初始化的池，找到能容纳的最小池，然后调用 `QF_EPOOL_GET_` 宏进行实际分配。`QF_gc` 是垃圾回收函数，通过引用计数决定是否将事件归还到池。这两者一起实现了零 malloc、零碎片的事件生命周期管理——这是嵌入式内存管理区别于服务器后端的核心差异。

### 6.4 `QTimeEvt_tick_`（src/qf/qf_time.c）

驱动 QP/C 时间子系统的心脏。通常从 SysTick 中断中调用。它遍历该 tick rate 上的所有活动定时器，递减计数器，到期后将超时事件投递到对应的活动对象事件队列中。注意：它在临界区中操作链表，这在中断上下文中执行，必须极端谨慎。对于后端工程师，这相当于 "epoll 超时管理 + 定时任务调度" 在 MCU 上的实现——但操作不能阻塞、不能分配内存、必须在几微秒内完成。

### 6.5 `QHsm_dispatch_`（src/qf/qep_hsm.c）

层次状态机的核心引擎。当事件到达时，这个函数会从当前状态开始向上遍历状态层次，直到找到能处理该事件的状态。返回时执行所有经过状态的退出动作、目标状态的进入动作和执行转换动作。它通过状态处理函数的返回值（Q_RET_SUPER/TRAN/HANDLED 等）来驱动状态层次遍历——不需要额外的配置表就可以实现完整的状态嵌套行为。

## 7. 软件工程师类比解释

### 7.1 活动对象（Active Object）→ 后端微服务

每个活动对象就像一个微服务实例：有自己的消息队列（对应微服务的请求队列）、有独立的行为逻辑（对应业务处理）、通过事件发布-订阅通信（对应消息队列/事件总线）。

**局限**：活动对象运行在单核 MCU 上，没有真正的并行执行，只有基于优先级的抢占式调度；微服务的独立性靠进程隔离保障，活动对象共享同一个地址空间。

### 7.2 层次状态机（HSM）→ UML 状态图 / BPMN 工作流

HSM 类似于后端系统中的工作流引擎（如 Temporal、Camunda）。状态嵌套等价于子流程（sub-process），进入/退出动作相当于流程节点的 onEntry/onExit 回调。

**局限**：HSM 在 C 代码中的实现是通过函数返回值和虚函数表完成的，没有工作流引擎那样的持久化和补偿机制。HSM 中的状态是纯内存的，系统掉电即丢失。

### 7.3 事件内存池（QMPool）→ 对象池模式

QP/C 的事件从固定大小的内存池分配，就像后端系统中的连接池或对象池——预分配、固定大小、无碎片。引用计数实现零拷贝跨活动对象传递，类似于共享指针（shared_ptr）的简单版本。

**局限**：内存池的大小在编译时确定，无法像服务器端 JVM 堆那样动态增长或收缩。池碎片不是内存碎片问题，而是"大事件撑爆小池"的分配失败问题。

### 7.4 QPSet 优先级位图 → Linux 内核 O(1) 调度器 / Redis 跳表

QPSet 用位图来标记哪些优先级有就绪活动对象，`QPSet_findMax` 用二分查找+查找表在 O(1) 时间内找到最高优先级。类似于 Linux 的 `bitmap_find_next_zero_area` 或 Redis 有序集合的跳表头指针。

**局限**：QPSet 只支持最多 64 个静态优先级（编译时配置），而 Linux 调度器支持动态优先级调整和 CFS 完全公平调度。QP/C 的调度策略是纯静态优先级抢占——简单、可预测、但不够灵活。

### 7.5 发布-订阅事件机制 → Kafka / Redis Pub-Sub

QP/C 内置信号级别的发布-订阅：活动对象通过 `QActive_subscribe(sig)` 注册对某类事件的兴趣，调用 `QActive_publish_` 后事件被分发到所有订阅者的事件队列。

**局限**：订阅关系在初始化时静态配置，不支持运行时动态订阅/取消。没有消息持久化、没有分区机制、没有重试/死信队列。一个订阅者处理慢会影响发布者吗？QF 的事件发布到队列是非阻塞的，但如果队列满了，发布者可以选择等待或丢弃。

## 8. 今日 10 个嵌入式概念卡片

### 卡 1: 临界区（Critical Section）
- **一句话**：执行时不允许被中断打断的代码区域，通常通过关中断实现。
- **软件类比**：数据库事务中的锁——在临界区内执行的操作要么全部生效，要么全部不生效。
- **工程场景**：QP/C 中修改活动对象注册表 `QActive_registry_[]` 时必须进入临界区；中断服务例程和主循环之间共享任何数据都要临界区保护。
- **坑点**：临界区应极短（GPOS 中几毫秒，嵌入式中应为十几条指令），否则会显著增加中断延迟，导致实时响应失败。

### 卡 2: 事件队列（Event Queue）
- **一句话**：一个 FIFO 缓冲区，存放等待活动对象处理的事件。
- **软件类比**：消息队列（如 RabbitMQ 的队列），活动对象是消费者。
- **工程场景**：QP/C 中每个活动对象有一个属于自己的 `QEQueue`，`QActive_post_` 将事件压入该队列，`QActive_get_` 取出。
- **坑点**：事件队列长度必须在编译时确定，运行时不能动态扩展。队列满时 `post` 可以配置为"等待"或"丢弃"，没有死信队列。

### 卡 3: 内存池（Memory Pool）
- **一句话**：预分配固定大小内存块的分配器，无碎片、确定时间。
- **软件类比**：数据库连接池——连接的数量和大小在启动时确定。
- **工程场景**：QP/C 用 `QF_poolInit` 初始化多个大小递增的内存池，`QF_newX_` 选择合适池分配事件。
- **坑点**：必须保证最大事件大小不超过最大池块大小，否则会断言失败。池不能动态增长，设计时必须估算好峰值用量。

### 卡 4: 抢占阈值（Preemption-Threshold）
- **一句话**：每个活动对象设置一个"最大可被抢占的优先级上限"，控制嵌套抢占深度。
- **软件类比**：在微服务中设置请求超时和熔断阈值，防止级联故障。
- **工程场景**：QP/C 中 `QActive.pthre` 字段，防止高优先级活动对象长时间霸占 CPU 或导致堆栈溢出。
- **坑点**：阈值设置过高意味着更少抢占，实时性下降；设置过低意味着更深嵌套，堆栈需求增加。需要在实时性和堆栈之间做权衡。

### 卡 5: 层次状态机（Hierarchical State Machine）
- **一句话**：状态可以嵌套，子状态继承父状态的行为，支持"从任何子状态处理父事件"。
- **软件类比**：CSS 选择器的继承——子元素继承父元素的样式，但可以覆盖。
- **工程场景**：设备电源管理——"正常运行"状态下的"省电子状态"、"通信子状态"；休眠/唤醒事件在父状态统一处理。
- **坑点**：状态层次过深会导致状态机难以理解和调试。QP/C 中状态处理函数的返回值驱动遍历，返回值错误可能导致状态机跑飞。

### 卡 6: 软件跟踪（Software Trace / QS）
- **一句话**：在运行时把关键事件（状态切换、事件发布、ISR 进入/退出）记录到缓冲区，在线分析。
- **软件类比**：分布式追踪（Jaeger/Zipkin）中的 span 日志，但 QS 是本地缓冲区，不依赖网络。
- **工程场景**：调试难以单步跟踪的时间敏感场景——如中断上下文中的事件发布顺序、活动对象调度的时序。
- **坑点**：QS 输出通过宏（`#ifdef Q_SPY`）选择编译，生产固件不包含跟踪代码。跟踪数据格式需要通过 QSPY 主机工具解析，自己写调试脚本需要理解 QS 协议。

### 卡 7: 可剥夺调度（Preemptive Scheduling）
- **一句话**：高优先级任务可以打断正在运行的低优先级任务，抢占 CPU。
- **软件类比**：操作系统的中断处理——硬件中断的优先级高于任何用户态进程。
- **工程场景**：QP/C 的 QK 内核支持活动对象间抢占。一个高优先级 AO 发布事件后，会立即尝试抢占正在执行的低优先级 AO。
- **坑点**：抢占导致的活动对象栈深度不可预测（嵌套深度 = 最多同时就绪的活动对象数）。需要用抢占阈值限制嵌套深度，否则堆栈需求会爆炸。

### 卡 8: 时间事件（Time Event / QTimeEvt）
- **一句话**：嵌入式中的"定时器"——基于系统滴答（tick）计数的一次性或周期性事件。
- **软件类比**：JavaScript 的 setTimeout/setInterval，或 Linux 的 timer_list。
- **工程场景**：设备定期上报传感器数据、看门狗刷新、LED 闪烁、通信超时重试。
- **坑点**：时间事件在临界区中管理链表，中断上下文中不能执行阻塞操作。如果滴答周期是 1ms，精度就是 1ms；不能用浮点或大数运算来计算超时。

### 卡 9: 位域（Bit-Field）与紧凑数据设计
- **一句话**：C 语言允许将一个字节/字中的不同位域分配给不同变量，节省 RAM。
- **软件类比**：Protocol Buffers 的变长编码或后端的列式存储，但位域是硬件的思考方式。
- **工程场景**：QP/C 的 QEvt 结构在一个 32 位字中同时编码了 sig（16 位）、poolNum（8 位）、refCtr（8 位）。
- **坑点**：位域的存储顺序（endianness）由编译器决定，不同架构间不可移植。位域的赋值是非原子的，多线程（中断）环境下需要保护。

### 卡 10: 端口的可移植性层（Port Layer）
- **一句话**：框架与具体 MCU/编译器之间的适配层，通常包含 `qp_port.h`、启动代码、中断向量配置。
- **软件类比**：后端框架的"适配器模式"或"驱动"（如 JDBC 驱动、ORM 方言适配）。
- **工程场景**：QP/C 的 `ports/` 目录包含了从 ARM Cortex-M 到 Zephyr RTOS 的各种移植。添加新 MCU 支持时只需要实现几个关键的宏和函数（如临界区、栈初始化、线程启动）。
- **坑点**：移植层的正确性直接影响框架的实时性。临界区的 `QF_CRIT_ENTRY/EXIT` 实现必须优先考虑中断嵌套场景。armv7m 上关中断只需一条 `cpsid i`，但需要处理 `BASEPRI` 寄存器支持可屏蔽中断优先级。

## 12. 工程点评

**可维护性**：★★★★☆
QP/C 的整体代码质量非常高。统一的编码风格、完整的 Doxygen 注释、严格的前置条件检查（Q_REQUIRE/Q_ASSERT）、代码模块化清晰。每个 .c 文件不超过 500 行。但是层次状态机的学习和维护曲线比传统裸机代码陡峭——团队成员都需要理解 Actor + HSM 范式。

**可移植性**：★★★★★
移植层（ports）被严格隔离，添加一个新 MCU 只需要实现约 10 个宏和 3 个函数。框架本身不依赖任何硬件特性（除了关中断指令）。POSIX 移植更是让桌面端开发和测试成为可能。

**可靠性**：★★★★★
QP/C 是极少数在开源领域就配备了安全认证套件的嵌入式框架。SafeQP/C 通过了 IEC 61508 / ISO 26262 / DO-178C 等级别的认证。框架内部使用了"契约式设计"（Design by Contract），每个函数都有前置条件、后置条件断言。引用计数和内存池的防御性检查非常彻底。

**可测试性**：★★★★☆
活动对象的独立性使得单元测试相对容易——你可以向一个活动对象注入事件，观察它的状态转移和输出事件。QP/C 的 QUTest 测试框架特别为此设计，支持 100% MC/DC 覆盖率。但必须指出：在真实硬件上的时序测试仍然困难，毕竟活动对象之间的抢占行为取决于优先级配置和中断时序。

**量产/运维**：★★★☆☆
框架本身很成熟，有过大量量产案例。但运维方面缺乏 OTA、远程日志、性能监控等开箱即用的功能（这些需要用户自己搭建）。QS 软件跟踪可以用于现场的诊断数据采集，但需要配套的 QSPY 主机工具链做分析。
